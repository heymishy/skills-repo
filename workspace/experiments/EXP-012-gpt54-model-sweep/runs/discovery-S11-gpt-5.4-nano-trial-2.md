Below is a structured design/discovery-style write-up for the consent management API, based on your requirements, highlighting key decisions, entities, API shape, consent lifecycle, accreditation enforcement, enriched insights scoping, and operational controls.

---

## 1) Goals & non-goals

### Goals
- Give customers granular control over which **data types** a consent grants.
- Allow **accredited third parties** to access only the data they’re permitted to.
- Enforce **time-limited consents (up to 12 months)** with **instant revocation**.
- Support **fine-grained enforcement per API call**, including **blocking de-accredited parties**.
- Include **enriched insights** (generated by internal analytics; not raw core banking data) in launch scope.

### Non-goals (until expanded)
- No customer onboarding flows inside this API (assuming member portal handles UI and consent capture).
- No account-holder identity verification mechanisms inside this API beyond validating a consent token / customer linkage (depending on your existing auth model).

---

## 2) Actors
- **Customer**: grants/revokes consent via member portal.
- **Third Party (TPP)**: accredited party calling the API.
- **Accreditation Authority**: source of truth for accreditation status.
- **Consent Service / API**: records consents, enforces limits, validates accreditation per request, and gates access.

---

## 3) Data types & sharing rules

Define a canonical set of “scopes” (data types) that can be independently consented:

1. **TransactionHistory**
   - Up to **12 months**.
2. **AccountBalancesAndCredit**
   - Current balances + available credit.
3. **CreditCardSummary**
   - Statement balance, payment due date, utilisation.
4. **EnrichedInsights** *(launch-day available; analytics-derived)*
   - Spending category breakdowns
   - Estimated monthly income
   - 90-day projected balance

### Rule: No over-sharing
For each consent, the API must:
- Only return data types explicitly granted.
- Only return the time window (for transactions) that the consent covers (<= 12 months).
- Ensure enriched insights are treated as a separate consentable scope (even if derived from other internal models).

---

## 4) Consent model (granular, time-limited, revocable)

### Core concepts
- **Consent**: customer authorization for a specific TPP to access specific scopes.
- **Consent Grant**: the selected scopes + validity.
- **Revocation**: customer can revoke at any time; revocation must stop future access and trigger deletion.

### Recommended consent status lifecycle
- `PENDING` (optional, if you need intermediate state)
- `ACTIVE`
- `REVOKED`
- `EXPIRED`
- (Optional) `DELETION_PENDING` for asynchronous deletion workflows

### Time limits
- Maximum consent duration: **12 months**
- Revocation: effective **immediately** for new reads.

### Deletion requirement on revocation
- “On revocation, the third party must delete all data received under that consent.”
This is a contractual/data-governance obligation enforcement area. Technically, you should:
- Stop reads immediately after revocation.
- Require a **deletion attestation** callback/message from TPP (recommended).
- Store a **deletion deadline** and track acknowledgement (optional but helpful for audits).

---

## 5) Accreditation enforcement (per API call)

### Accreditation requirement
- TPP must be accredited before receiving any data.
- Accreditation must be validated on **each API call**, not only at onboarding.
- If accreditation is lost, block new access immediately.

### Implementation approach
- Maintain (or query) an accreditation status endpoint/cache:
  - Use either:
    - synchronous call to authority (slower, strongest correctness), or
    - signed accreditation status snapshots with short TTL (faster, requires careful security/TTL).
- Add a step in every request:
  1. Identify `TPP_ID` from auth credentials.
  2. Check accreditation status (including required permissions, if your accreditation is scope-based).
  3. If not accredited → return `403 Forbidden`.

### Suggested response codes
- `401 Unauthorized` (bad/missing auth)
- `403 Forbidden` (not accredited OR consent missing OR scope not granted)
- `404 Not Found` (optional: hide existence of consent IDs to reduce leakage)
- `409 Conflict` (if client attempts operations inconsistent with consent state)
- `429 Too Many Requests` (rate limiting)
- `410 Gone` (expired/revoked consent for clarity)

## 6) API surface (discovery-level proposal)

You’ll typically split responsibilities into:
1. **Consent lifecycle endpoints** (create/activate/revoke) — might be driven by member portal.
2. **Consent query endpoints** (optional for TPP)
3. **Data endpoints** gated by consent and accreditation

### A) Consent endpoints (primarily portal-driven)
**POST /consents**
- Creates a consent record from portal input (customer + TPP + selected scopes + duration).
- Returns `consentId`, `status`, `expiresAt`.

**POST /consents/{consentId}/revoke**
- Marks consent revoked.
- Returns revocation timestamp.

**POST /consents/{consentId}/deletion-confirmation** *(recommended)*
- Lets TPP attest that data has been deleted post-revocation/expiry.
- You can require this as part of compliance tracking.

*(If your portal already exists, these endpoints may be called internally by portal rather than by TPPs.)*

### B) Data endpoints (TPP-driven)
All data calls should include:
- `Authorization` (TPP credentials)
- `Consent-Id` header OR consent token in request body
- The requested scopes should map 1:1 with the consent scopes.

Example:

**GET /data/transaction-history**
- Query params: `consentId`, maybe `from`, `to` (but must remain within <= 12 months and within consent window)
- Server verifies:
  - Consent is ACTIVE
  - Consent includes `TransactionHistory`
  - Requested date range is within allowed window
  - TPP is accredited currently

**GET /data/account-balances-and-credit**
- Requires consent scope `AccountBalancesAndCredit`

**GET /data/credit-card-summary**
- Requires consent scope `CreditCardSummary`

**GET /data/enriched-insights**
- Requires consent scope `EnrichedInsights`

### C) Consent introspection endpoint (optional)
**GET /consents/{consentId}**
- Might be restricted (only portal or the TPP that owns the consent).
- Should not leak more than necessary (e.g., show which scopes are permitted, expiration time).

## 7) Enriched insights handling (launch-day scope)

Because EnrichedInsights are analytics-derived:
- Treat them as a distinct consent scope (`EnrichedInsights`).
- Do not derive them unless consent includes this scope.
- Consider versioning for analytics logic:
  - e.g., `enrichedInsightsModelVersion` returned in metadata so TPPs can understand consistency changes over time.
- Ensure you can reproduce/justify why a value was generated (auditability):
  - store the model version and generation timestamp in the response metadata.

## 8) Security & privacy controls (minimum checklist)

- **Strong authentication for TPPs** (mTLS and/or signed tokens; use your enterprise standard).
- **Per-call authorization**:
  - Verify accreditation status
  - Verify consent existence and ACTIVE state
  - Verify scope match
  - Verify requested parameters are within consent limits
- **Least privilege at response level**
  - Only return requested data types.
- **Audit logs**
  - Log: `TPP_ID`, `consentId`, data type(s), timestamp, accreditation result, customer id (hashed/pseudonymised), outcome.
- **Data minimisation**
  - Avoid returning raw identifiers not needed for the third party use case.
- **Revocation enforcement**
  - All reads check `consent.status == ACTIVE`.
- **Deletion compliance tracking**
  - Maintain deletion confirmation evidence from TPPs (and/or future automated compliance mechanisms).

## 9) Operational considerations

### Cohort launch: 10 partners in 6 months
Plan for:
- A partner onboarding process (not covered in discovery, but required).
- Contractual requirements:
  - encryption, logging, data retention limits, deletion attestation timing.
- Testing:
  - include enriched insights verification with deterministic fixtures where possible.

### Rate limiting & performance
- Enriched insights may be heavier than raw balances:
  - Consider caching (but only within privacy constraints).
- Support pagination for transaction history (even if max window is 12 months).

## 10) Open questions to resolve in discovery

1. **Consent representation**
   - Will the portal issue a **consent token** directly to the TPP, or do you require TPP to call a consent introspection step?
2. **Consent ID exposure**
   - Should TPPs receive a stable `consentId` or short-lived token?
3. **Revocation timing**
   - “Immediately” — do you require hard stop (synchronous check only) or also propagate revocation events to cache layers?
4. **Accreditation enforcement mechanism**
   - Will accreditation be queried live from the authority per request, or via signed cache with TTL?
5. **Deletion attestation enforcement**
   - Is deletion confirmation mandatory for compliance acceptance, and will you block future access if not confirmed?
6. **Analytics reproducibility**
   - Do you need to guarantee that the same response is repeatable within a given consent period, or is “best-effort current projection” acceptable?
