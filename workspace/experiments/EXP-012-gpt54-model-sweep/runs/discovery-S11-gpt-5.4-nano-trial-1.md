## 1) Purpose & scope (what we’re building)
**Consent management API** that lets customers grant/revoke time-limited, granular consent to **accredited third parties** to access:

1) **Transaction history** (max 12 months)  
2) **Current account balances & available credit**  
3) **Credit card summaries** (statement balance, payment due date, utilisation)  
4) **Enriched insights** (enterprise analytics output only)  
   - spending category breakdown  
   - estimated monthly income  
   - 90-day projected balance  

**Granularity**
- Consent is per **data type** and per **accredited third party**.

**Control**
- Consent: explicit, time-limited (up to 12 months)
- Revocation: at any time; third party must delete data received under that consent.

**Accreditation enforcement**
- Third parties must be accredited by a designated accreditation authority.
- Accreditation is validated **on every API call**, not only onboarding.
- If accreditation lapses, **new access is blocked immediately**.

**Privacy**
- Only shared when customer consent exists for that data type + third party.
- No over-sharing beyond consent.

---

## 2) Actors & responsibilities
### Customer
- Grants explicit consent via member portal.
- Chooses:
  - which data types
  - which third party
  - consent duration (≤ 12 months)
- Can revoke consent at any time.

### Third party (TP)
- Accredited TP calls the API on behalf of a customer.
- Must implement deletion on revocation (contractual + operational expectation).

### Member portal
- UI/workflow to create and revoke consent.
- Emits/updates consent state consumed by the API.

### Consent management service (this API)
- Stores consent grants, scopes access by data type.
- Enforces:
  - authorization (customer + TP)
  - data-type entitlement
  - time validity
  - revocation status
  - accreditation validity per request

### Accreditation authority / accreditation system
- Provides accreditation status for TP.
- Used for on-call validation.

### Data access / enrichment engines
- Provide the requested data types.
- Enriched insights are derived from internal analytics engine outputs.

---

## 3) Core user journeys
### A. Grant consent
1. Customer selects TP + data types + duration (≤12 months) in portal.
2. System creates a consent grant with:
   - consent ID
   - TP identifier
   - customer identifier
   - allowed data types
   - validity window
   - status = ACTIVE
3. TP can then call API to fetch only allowed data types for that customer during validity.

### B. Access data (per call)
For each API call:
1. Validate TP accreditation status (fresh check).
2. Validate consent exists:
   - belongs to that TP and customer
   - status ACTIVE (not revoked)
   - not expired
3. Validate the API endpoint/data type requested is within consent scope.
4. Return data only within permitted scope.

### C. Revoke consent
1. Customer revokes consent in portal (consent status → REVOKED, effective time recorded).
2. TP is blocked from further access immediately (subject to “how immediate” you define operationally).
3. TP must delete previously received data under that consent.

## 4) Functional requirements (what the API must do)

### 4.1 Consent lifecycle
- **Create consent grant** (triggered by portal)  
- **Revoke consent** (triggered by portal)
- Maintain consent states:
  - ACTIVE, REVOKED, EXPIRED, CANCELLED/FAILED (optional)

### 4.2 Consent granularity & enforcement
- Each consent grant includes:
  - allowed data types (subset selection)
  - allowed TP(s) (likely per consent record; described as “which third party”)
  - validity window
- Every data retrieval call must enforce:
  - consent exists for (customer, TP)
  - requested data type ∈ consent’s allowed set

### 4.3 Accreditation validation on every request
- For each API call:
  - verify TP accreditation is current and authorized for your programme
  - if not accredited: return **403/blocked** without serving data
- Need defined behaviour and caching TTL (see Open Questions).

### 4.4 Data type mapping to endpoints
You’ll need a clear mapping from “data type” to request/response model, e.g.:
- `/accounts/{id}/transactions?range=...`
- `/accounts/{id}/balances`
- `/cards/{id}/summary`
- `/insights/spending-categories`
- `/insights/income-estimate`
- `/insights/balance-projection?days=90` (or fixed)

### 4.5 Enriched insights tier included at launch
- Enriched insights must be deliverable at launch.
- Access control is the same:
  - enriched insight consent must exist for each customer+TP.
- Need SLAs / refresh policy (since they’re generated by analytics engine).

### 4.6 Audit & compliance
- Store:
  - who/what created consent (portal actor)
  - TP accesses (request logs)
  - consent revocation events
- Provide audit trails for privacy compliance.

## 5) Non-functional requirements
- **Security**
  - Strong authentication between TP and API (e.g., mTLS/OAuth2, depending on your standard)
  - Authorization checks for consent + accreditation
- **Availability**
  - Must tolerate accreditation lookups; define fallback behaviour.
- **Latency**
  - Per-call accreditation and consent checks must not cause unacceptable delays.
- **Scalability**
  - Initial 10 partners; but design for growth.
- **Data minimisation**
  - Ensure endpoint responses cannot return non-consented fields.
- **Deletion support**
  - Although TP must delete, you may also need internal tracking to support “revoked-at” semantics.

## 8) Enforcement logic per call (high-level)
For any TP data request:

1. Authenticate TP (and application/client identity).
2. Resolve `customer_id` and identify the appropriate consent (either via:
   - explicit `consent_id` supplied by TP, or
   - implicit mapping using customer session / token)
3. **Accreditation check (fresh):**
   - If accreditation invalid/expired/lost: block.
4. **Consent check:**
   - consent status ACTIVE
   - current time ∈ [start_time, end_time]
   - consent’s `allowed_data_types` includes the requested type(s)
5. Return only permitted fields.

## 9) Open questions (need answers to proceed)
1) **Consent reference model**
- Will third parties present a `consent_id` in requests, or is consent implicitly bound to an access token?
- Can a single consent cover multiple accounts/cards?

2) **“Within 12 months” definition**
- Is “up to 12 months” relative to now, or relative to consent start?
- Are exact date ranges supported (e.g., `from`/`to`)?

3) **Revocation timing**
- “Immediate” blocking: do you require strict real-time (<seconds), or is “near-real-time” acceptable (minutes)?
- On revocation, do you invalidate tokens, or rely purely on consent status checks?

4) **Accreditation lookup performance**
- What is the accreditation authority API latency and rate limits?
- Is it acceptable to cache accreditation status for a short TTL (e.g., 5–60 seconds) while still meeting “immediately”? If not, latency risk increases.

5) **Enriched insights behaviour**
- Refresh cadence: generated per request, scheduled, or cached?
- Do insights correspond exactly to the same 12-month window of transactions, or different data windows?
- Versioning: if analytics logic changes, do you expose version metadata?

6) **Consent deletion assurance**
- You’ve stated TP must delete on revocation. Do you require:
  - deletion attestation endpoint?
  - evidence/audit logs from TP?

7) **Field-level restrictions**
- For credit card summaries: are there any fields that might be considered “more sensitive” requiring additional sub-consent?
  - (e.g., due date vs utilisation vs statement balance)

## 10) Risks & mitigations
1) **Over-sharing via endpoint design**
- Mitigation: make data-type permissions explicit in code and ensure response schemas are generated from entitlement.

2) **Accreditation check causing outages/latency**
- Mitigation: use robust timeout/retry; define failure policy (e.g., “fail closed” → block if authority unreachable).

3) **Revocation race conditions**
- Mitigation: revocation status is the source of truth; data access always checks consent status.

4) **Enriched insights correctness & explainability**
- Mitigation: define calculation methodology, ensure consistent outputs, and document “estimated” nature.

5) **Deletion compliance**
- Mitigation: contractual controls + optional attestation + audit.

## 11) Launch plan implications (10 partners in 6 months)
To meet “enriched insights from day one,” prioritize:
- Consent CRUD + enforcement engine first
- Accreditation integration + “fail closed” policy
- Endpoint scaffolding with stubbed analytics outputs initially, then wire to internal engine
- Strict integration testing for:
  - consent not granted → 403
  - wrong data type → 403/400
  - consent expired → 403
  - revoked → 403
  - accreditation lost mid-cycle → blocked immediately
