## 4. SKILL.md changes shipped today

### `.github/skills/discovery/SKILL.md` — commit `27de6eb`

**Problem solved:** EXP-003 Config C runs showed regulated constraints (PCI DSS) could be mentioned in the input narrative but not elevated to named Constraints section entries. This was a discovery skill structural gap — the model treated unconfirmed constraints as prose rather than surfacing them explicitly.

**Three changes applied:**

**Change 1 — Constraint labelling rule (D5/D7):** New writing rule before Sections 6 and 7. Any constraint that cannot be confirmed from the input must be written as an explicit `[ASSUMPTION]` line: `[ASSUMPTION] [constraint statement] — unconfirmed, requires /clarify before scope is locked.` Confirmed constraints (from `product/constraints.md`, reference materials, or explicit operator statement) do not need the tag. This prevents regulatory constraints from being buried in prose.

**Change 2 — Observability minimum (D6):** New writing rule for Section 7 (Directional success indicators). Each success indicator must name a baseline value (or `[UNKNOWN BASELINE]`), a target value, and how it will be measured. Vague directional phrases ("improved adoption", "reduced friction") are prohibited.

**Change 3 — /clarify decision gate (Section 8a):** After Sections 6, 7, and 8, count every `[ASSUMPTION]` line. If two or more exist, end the artefact with a mandatory `/clarify recommendation` block listing each unresolved assumption by name. This surfaces scope lock risk before `/benefit-metric` runs with unresolved constraints.

### `.github/skills/definition/SKILL.md` — commit `4dae4e3`

**Problem solved:** EXP-003 Config C run 2 (F6/F7 finding): when /definition chose a vertical-slice decomposition, regulated constraints (PCI DSS C2) were assigned to preparation stories only. Implementation stories that actually triggered the regulated gate had no constraint AC. The self-check propagation table falsely declared "all constraints propagated" because C2 appeared somewhere in the feature — not necessarily in every triggering story.

**Change applied — Step 4a (Regulated constraint propagation check):**

New mandatory step inserted after Step 4 (story writing). Runs only when the discovery Constraints section contains at least one regulated constraint (external compliance framework, legal obligation, or third-party process gate). Three sub-steps:

- **Step 4a.1:** List each regulated constraint from discovery.
- **Step 4a.2:** For each constraint, enumerate triggering stories (stories whose implementation scope falls within the regulated gate — not documentation-only or vendor engagement stories). Present trigger assignment table to operator for confirmation. Record in `decisions.md`.
- **Step 4a.3:** For each (constraint, triggering story) pair, verify the constraint appears in the story's Architecture Constraints field. If missing, flag with `⚠️ REGULATED CONSTRAINT GAP` and apply fix immediately — do not defer to /review or /definition-of-ready.

**Fix validation:** A dedicated fix-validation run confirmed C2 definition chain score increased from 0.35 to 1.00 after Step 4a application.

## 7. EXP-003 summary

**Source:** `workspace/experiments/EXP-003-pipeline-eval/manifest.md`

**Metric:** Constraint Propagation Fidelity (CPF) — what fraction of constraints in the discovery artefact survive to the DoR contract and test plan NFR sections. Regulated threshold: ≥ 0.80. General threshold: ≥ 0.60. Below 0.80 on regulated = FAIL regardless of general score.

**Corpus:** S1 — financial services payment DR story with 5 constraints: C1 (RTO/RPO), C2 (PCI DSS QSA), C3 (AML/CFT 5-year retention), C4 (single DC baseline), C5 (hidden AML replication gap).

**Config results:**

| Config | Models | General CPF | Regulated CPF | Verdict |
|--------|--------|-------------|---------------|---------|
| A — Uniform Sonnet | All sonnet-4-6 | 1.00 | 1.00 | PASS |
| B — Opus front-loaded | Opus disc+def; Sonnet review+tp+dor | 1.00 | 1.00 | PASS (depth-of-extraction: Opus surfaces additional non-canonical constraints) |
| C — Cost-optimised | Run 1: Sonnet uniform (switch not executed); Run 2: Sonnet disc+def, Haiku review+tp+dor | Run 1: 0.60; Run 2: 0.68 chain | Run 1: 0.33 at source; Run 2: 0.675 chain | FAIL (both runs) |
| D — GPT-4o + Haiku | GPT-4o disc+def+review+tp; Haiku dor | Pending | Pending | Requires EXP-002a H5 confirmed |

**Findings:**

- **F1** — C2 (PCI DSS) dropped at discovery/definition in Config C runs. Regulatory constraint appeared in the narrative but was not elevated to the Constraints section.
- **F2** — Pipeline recovery pattern: /review and /definition-of-ready recovered C2 post-definition, but a coding agent dispatched post-definition (before review) would operate without any PCI DSS constraint. Recovery is not the same as at-source fidelity.
- **F3** — C4 (single DC) consistently stayed as background context rather than a formal Constraints entry across multiple runs.
- **F4** — Config C run 1: model switch not executed (Haiku not selected in model picker). All stages ran Sonnet. CPF gap is Sonnet variability, not Haiku failure.
- **F5** — CPF non-determinism: Config A and C both used Sonnet but got 1.00 vs 0.33 regulated-at-source. Sonnet CPF is not deterministically 1.00.
- **F6** — C2 drop is a slicing strategy effect: vertical-slice (Config C run 2) assigns C2 to preparation stories only, not to implementation stories. Risk-first slicing (Config A) creates a dedicated compliance epic forcing C2 into all dependent stories.
- **F7** — Self-check false positive: definition produced a propagation table claiming "ALL FIVE CONSTRAINTS PROPAGATED" — but the table validated feature-level presence, not per-story presence. Stories S1.2 and S2.2 had no C2 mention despite triggering the PCI DSS gate.

**Fix shipped:** Step 4a (per-story regulated constraint propagation check) added to /definition SKILL.md. Fix validation: C2 definition score 0.35 → 1.00. See commit `4dae4e3`.

## 12. Critical learnings (from `workspace/learnings.md`)

**Context pressure — effective session window is 55–60%, not 75%:** File-read-heavy phases (definition, review, trace) fill the Tool Results context bucket faster than the Messages bucket. By 55–60% context indicator, the Tool Results bucket may already be near threshold. Every session in Phase 1 ended at approximately 50–60% of intended scope. Invoke `/checkpoint` at 55% for all file-read-heavy phases.

**State.json write must be atomic full-file replacement:** Any write mechanism that appends, partially overwrites, or uses a non-truncating replace will produce concatenated JSON that fails `check-workspace-state.js` with "Unexpected non-whitespace character after JSON." Always write the complete JSON object; write to a temp file first if the tool cannot guarantee atomic overwrite. Do NOT use PowerShell here-strings for JSON writes — they produce UTF-8 with BOM, which breaks JSON.parse. Use `node -e` with `fs.writeFileSync(..., {encoding: 'utf8'})`.

**Parallel outer loop is correct model:** Review + test-plan + DoR phases can run concurrently across multiple chat windows for independent stories. Phase 1 cleared all 8 stories through three phases in 5 calendar hours using 4 parallel windows. Sequential assumption in the pipeline is incorrect for these phases.

**CPF is not deterministic at 1.00 for Sonnet:** Same model (Sonnet), same skill, same type of input produced CPF = 1.00 in Config A and CPF = 0.33 (regulated at source) in Config C. The variability is real and cannot be assumed away.

**Slicing strategy determines downstream CPF, not just model capability:** Vertical-slice decomposition assigns constraints to preparation stories; risk-first assigns them to a compliance epic that forces propagation into all dependent stories. This is a SKILL.md structural problem (F6), not a model problem. Step 4a fixes it with a per-story trigger assignment check.

**State update must precede completeStage():** For any handler that writes an artefact to disk and passes content to the next stage, write to disk first, then read back and use the disk content for handoff. Disk is canonical. (Source: ougl decisions.md, ADR-023.)

**Conflict marker scan before git add:** After any conflict resolution — merge, rebase, cherry-pick — run `Select-String -Pattern '<<<<<<|======|>>>>>>' <file>` before staging. The `>>>>>>> <sha>` tail line from cherry-pick resolutions is especially easy to miss and produces a SyntaxError that kills the entire CI test suite. (Source: wsm.3, learnings.md D40.)
