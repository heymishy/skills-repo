## 1. What this repo is

This is `heymishy/skills-repo`, the dogfood repository for an open-framework, governed software delivery pipeline. The pipeline encodes delivery standards, quality gates, and discipline-specific practices as versioned SKILL.md instruction files that AI agents execute. ### Step 4a finding (the fix that enabled Config C run 3 to pass)

**Root cause of Config C run 2 failure (F6/F7):** When /definition chose vertical-slice decomposition, regulated constraints (PCI DSS C2) were assigned to preparation stories only. Implementation stories that actually triggered the regulated gate (S1.2, S2.2) had no C2 Architecture Constraint. The definition self-check propagation table falsely declared "all constraints propagated" because C2 appeared somewhere in the feature — not necessarily in every triggering story (feature-scoped validation, not per-story validation).

**Fix shipped:** Step 4a added to `/definition` SKILL.md (commit `4dae4e3`, session 2026-05-14). Three sub-steps:
1. List all regulated constraints from discovery.
2. Enumerate triggering stories (stories whose implementation scope falls within the regulated gate) — present trigger assignment table to operator for confirmation; record in decisions.md.
3. For each (constraint, triggering story) pair, verify the constraint appears in that story's Architecture Constraints field. If missing, flag ⚠️ REGULATED CONSTRAINT GAP and apply fix immediately.

**Config A run 2 validation:** Step 4a fired during Config A run 2 (Sonnet, post-fix). One gap-fill: S2.1 C2 — failover automation framed as operational rather than CDE-scoped; Step 4a caught it. Regulated CPF = 1.00 across all 5 constraints. Config A run 2 vs Config C run 3: identical regulated CPF = 1.00.

|--------|-------------------------------|-------------|
| Config A — Uniform Sonnet | ~$1.50 | baseline |

**Conclusion:** Config C (Haiku downstream, Sonnet at /discovery only, Step 4a mandatory) is the confirmed cost-optimal routing policy for regulated-constraint stories.
