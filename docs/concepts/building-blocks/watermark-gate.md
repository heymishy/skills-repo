# Watermark Gate

## What it is

The watermark gate is a quality floor check that runs as part of the assurance gate on every PR. It verifies two things: (1) the eval suite pass rate meets the minimum floor threshold (70%), and (2) the full eval score does not regress below the best recorded score for this skill/surface combination. If either check fails, the watermark gate blocks the PR.

The watermark gate protects the platform from quality regression. Platform improvement proposals (SKILL.md diffs generated by the improvement agent) must pass the watermark gate before they can be applied — ensuring that a proposed improvement does not inadvertently regress the scenarios the eval suite guards.

## Why it exists

A platform that improves itself without regression protection will drift. Each proposed SKILL.md change fixes one failure pattern; without a floor check, it might reintroduce a previous one. The watermark gate is the structural protection against this: it records the best achieved eval score for each skill/surface combination and prevents any subsequent change from scoring below that watermark.

The floor pass rate (70%) ensures that even on a first run (before a historical watermark has been established), a minimum quality bar must be met. A proposal that scores below 70% on the eval suite is not ready for application regardless of what it fixes.

## How it works

The watermark gate reads `workspace/suite.json` (the eval suite) and runs all scenarios. It then:

1. Calculates the pass rate (passing scenarios / total scenarios)
2. If pass rate < 70% (the `FLOOR_PASS_RATE` constant): gate fails
3. Reads the watermark file for the current skill/surface combination
4. If the current score < the watermark (best recorded score): gate fails
5. If both checks pass: records the current score as the new watermark if it is higher

The watermark is stored per skill/surface combination, so a regression in one area does not block progress in another.

## What you do with it

The watermark gate runs automatically — you do not trigger it. If it fails:

- If the pass rate is below floor: investigate which scenarios are failing and why. A proposed change that reduces the pass rate below 70% needs to be revised.
- If the score is below watermark: investigate which scenarios regressed relative to the previously recorded best. The regression must be resolved before the proposal can proceed.

When reviewing improvement agent proposals, note the watermark gate result as part of your review. A proposal that passes the watermark gate has demonstrated it does not regress existing guarded scenarios; this is a necessary but not sufficient condition for acceptance.
