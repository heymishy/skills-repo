## Test Data Strategy

**Source:** Mocked — the seed script's DB connection is injected per D37 (this story explicitly calls out injectable adapter usage), so all tests run against an in-memory mock adapter, not a real Neon connection.
**PCI/sensitivity in scope:** Yes — the entire purpose of this story is guaranteeing zero real PII in generated data. Constraints below.
**Availability:** Available now — the mock adapter is written as part of the test file itself, no external dependency.
**Owner:** Self-contained

### Data requirements per AC

| AC | Data needed | Source | Sensitive fields | Notes |
|----|-------------|--------|-------------------|-------|
| AC1 | A mock adapter recording every insert call (table, tenant_id, row data) | In-memory mock, built in test file | None (synthetic) | Mock exposes an `inserts` array the test asserts against |
| AC2 | Same mock adapter, run twice in sequence, with unique-constraint simulation (mock throws on duplicate primary key) | In-memory mock | None | Mock's insert function checks a `Set` of already-seen keys and throws a constraint-violation-shaped error on duplicate, mimicking real Postgres behaviour |
| AC3 | The literal seed data values generated by the script (tenant names, emails, etc.) | Script's own output, inspected directly | None — this IS the check: asserting there are no real names/emails | Regex scan of generated string values for realistic-looking PII patterns vs the required synthetic pattern (e.g. `@example-staging.test`) |
| AC4 | The script's exported entrypoint function/CLI invocation | Script module itself | None | Confirms a single callable entrypoint exists; the "automatic" part is cross-referenced to S2.5 |

### PCI / sensitivity constraints

- Every seeded value must be clearly synthetic — this test plan enforces that via AC3's tests, not by trusting the implementation's intent
- No test in this plan ever reads from or writes to a real database — the injected adapter (D37) is mocked throughout

### Gaps

None.

## Unit Tests

### T1 — Seed produces at least 2 distinct synthetic tenants

- **Verifies:** AC1
- **Precondition:** Mock adapter is fresh (no prior inserts recorded)
- **Action:** Call the seed script's exported `seed(adapter)` function with the mock adapter injected
- **Expected result:** The mock adapter's recorded inserts contain at least 2 distinct `tenant_id` values (e.g. `tenant-demo-1`, `tenant-demo-2`)
- **Edge case:** No
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T2 — Each synthetic tenant has representative rows in every tenant-scoped table

- **Verifies:** AC1
- **Precondition:** Same as T1
- **Action:** After calling `seed(adapter)`, group the mock adapter's recorded inserts by `tenant_id`
- **Expected result:** For each of the ≥2 tenants, at least one row exists in `products`, `credits`, and `user_roles` (the tenant-scoped tables named in the story's AC1)
- **Edge case:** No
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T3 — Seeded rows carry the same `tenant_id` scoping shape as real data (ADR-025)

- **Verifies:** AC1 (architecture constraint — ADR-025 tenant-scoping model)
- **Precondition:** Same as T1
- **Action:** Inspect each recorded insert's row shape
- **Expected result:** Every row inserted into a tenant-scoped table includes a non-null `tenant_id` column matching one of the seeded synthetic tenant identifiers — no row is inserted without tenant scoping
- **Edge case:** No
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T4 — Running seed a second time produces an identical row count (idempotent)

- **Verifies:** AC2
- **Precondition:** Mock adapter has already been seeded once (T1 has run against it)
- **Action:** Call `seed(adapter)` a second time against the same mock adapter state
- **Expected result:** The total row count recorded by the mock adapter after the second run equals the row count after the first run — no new rows were added
- **Edge case:** Yes — this is the core idempotency contract
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T5 — Running seed a second time throws no unique-constraint error

- **Verifies:** AC2
- **Precondition:** Same as T4. The mock adapter is configured to throw a Postgres-shaped unique-violation error (`error.code === '23505'`) if an insert is attempted with a primary key already recorded.
- **Action:** Call `seed(adapter)` a second time
- **Expected result:** No error is thrown/rejected — the seed function's own idempotency logic (e.g. `INSERT ... ON CONFLICT DO NOTHING`, or a pre-check before inserting) avoids ever attempting a duplicate insert against the mock's constraint simulation
- **Edge case:** Yes
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T6 — Every seeded email matches the synthetic domain pattern

- **Verifies:** AC3
- **Precondition:** Seed has run once (T1)
- **Action:** Scan every string value in the mock adapter's recorded inserts for anything matching an email-shaped pattern (`\S+@\S+\.\S+`)
- **Expected result:** Every matched email ends in a clearly synthetic domain (e.g. `@example-staging.test` or `@example.com`) — none match a real-looking corporate/personal domain
- **Edge case:** No
- **Fails before implementation:** Yes — `seed()` does not exist yet

---

### T7 — No seeded value matches a known-real-customer-data pattern

- **Verifies:** AC3
- **Precondition:** Same as T6
- **Action:** Scan every string value in the mock adapter's recorded inserts against a denylist of patterns that would indicate real data leakage: any of this repo's actual product/tenant names as configured in `product/` docs, any string resembling a real GitHub login format already seen in this repo's own test fixtures (e.g. `testadmin`, real usernames from other tests), or any value containing the literal substring `real` inverted (i.e. absence of realistic non-synthetic identifiers)
- **Expected result:** Zero matches — every identifier is clearly and deliberately synthetic (`tenant-demo-N`, `engineer@example-staging.test`, etc.)
- **Edge case:** Yes — this is a denylist-style test, acknowledged as best-effort rather than exhaustive
- **Fails before implementation:** Yes — `seed()` does not exist yet

## Integration Tests

### IT1 — Seed script exposes a single CLI-invokable entrypoint

- **Verifies:** AC4 (partial — the invokability, not the automatic triggering)
- **Components involved:** The seed script module and its CLI wrapper (e.g. `node scripts/seed-staging.js`)
- **Precondition:** The script file exists at its expected path
- **Action:** Spawn the script as a subprocess (`child_process.spawnSync('node', [scriptPath], { env: { ...process.env, SEED_ADAPTER_MOCK: '1' } })` or equivalent test-mode flag that swaps in the mock adapter instead of a real `DATABASE_URL` connection)
- **Expected result:** The process exits with code `0` and its stdout includes a summary line reporting tenant/row counts (see NFR — Audit, below)
- **Fails before implementation:** Yes — the script does not exist yet
- **Note:** This test proves the script *can* be invoked as a single command a deploy step could call. It does not prove a real deploy pipeline actually calls it — that is S2.5's scope and this story's AC4 manual gap.

## NFR Tests

### NFR1 — Performance (completes within 30 seconds)

- **NFR addressed:** Performance — seed script completes in under 30 seconds as part of the post-deploy step
- **Measurement method:** Record `Date.now()` before and after calling `seed(mockAdapter)` (mocked — real DB round-trip time against Neon is not part of this measurement, see note)
- **Pass threshold:** `elapsed < 30000` milliseconds against the mocked adapter
- **Tool:** Node.js `Date.now()`
- **Note:** This measures the script's own logic time (data generation + insert-call construction), not real network/DB round-trip time against a live Neon branch. The 30-second budget against a real database is a lighter-weight version of the AC3/AC4-style External-dependency gap seen in S2.2 — flagged in Test Gaps below as a partial gap, not a full one, since the mocked measurement is still a meaningful lower-bound signal.

- **Tool:** Node.js regex scan

- **NFR addressed:** Seed script run is logged (success/failure, row counts) as part of the deploy pipeline's own logging
