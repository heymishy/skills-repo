**2026-07-10 | SCOPE | inner loop (bri-s3.2)**
**Decision:** bri-s3.2's implementation grew beyond "write one Playwright spec" to include several small, targeted production-code fixes, because the real signup → onboarding → first-feature journey could not be driven end-to-end without them. Recorded here for review visibility — these are not buried in code comments alone.

**What was added, beyond the DoR contract's estimated touch points (`tests/e2e/bri-s3.2-signup-onboarding-journey.spec.js` new; `routes/journey.js` and `src/enforcement/gate-map.js` read-only):**

1. **`GET /api/journey/:id` was missing `ownerId`** from its JSON response, despite this story's own ADR-024 architecture constraint naming it as a required field (`turns, stages, completedStages, stage, ownerId, activeSkill`). One-line fix in `routes/journey.js`'s `handleGetJourneyState`.
2. **S3.1's mock LLM gateway was built but never actually wired into the live chat-turn call site.** `htmlSubmitTurn` (`routes/skills.js`) called `_skillTurnExecutor` without the `meta.stage`/`meta.scenarioName` argument the gateway hook requires, and `mock-llm-gateway.js`'s own D37 adapter (`wireDefaultMockGatewayClient()`) was never invoked from `server.js` at all — so `isMockGatewayEnabled()` being true was not sufficient; every chat turn either called a real provider or threw "Adapter not wired: mockGatewayClient". Fixed both: wired `meta` through in `htmlSubmitTurn`, added a new `session.mockScenarioName` field (set by `journey.js`, default `'success'`), and wired `wireDefaultMockGatewayClient()` in `server.js`'s test-mode block.
3. **A real feature-level journey traverses 7 stages** (`discovery, benefit-metric, design, definition, review, test-plan, definition-of-ready` — `journey-store.js`'s `STAGE_SEQUENCE`), not the 5 named in this story's own AC2 text. `design` and `review` had no mock-gateway fixtures at all (bri-s3.1 deliberately scoped its fixture matrix to 5 outer-loop stages + branch-setup/branch-complete). Added `tests/e2e/fixtures/llm-gateway/design.success.json` and `review.success.json`, mirroring bri-s3.1's existing fixture schema — an additive extension of that fixture set, not a new mechanism.
4. **`POST /products/new` and `POST /products/confirm` (psh-s3) and `POST /api/journey/:id/stories` (ougl.6) had no request-body parsing** in the raw HTTP server — `req.body` is `undefined` at runtime for all three, so each would throw or silently 400 in real use, independent of this story. Added a small `_readBody` helper in `routes/products.js` (mirrors the existing per-file pattern already used by `routes/auth-email.js` and `routes/journey.js`'s own `_readFormBody`) and reused `journey.js`'s existing `_readFormBody` inside `handlePostStories`. Both short-circuit when `req.body` is already set, so existing Express-mock-style unit tests are unaffected (confirmed: `check-psh-s3-product-creation.js`, `check-psh-s4-navigation.js`, `check-ougl6-perstory-stage-routing.js` all still pass, including two pre-existing unrelated failures in the last file that were verified present on a clean `origin/master` checkout too).
5. **Several D37 adapters (bcrypt, the real non-streaming `skillTurnExecutor`, `generateProductDraft`) are only wired when `WIRE_SKILL_ADAPTERS=true`**, an env flag referenced in a `server.js` comment but never actually set by any Playwright config in this repo. Since bcrypt has no external dependency, the real `skillTurnExecutor` is safe because of point 2 above (it defers to the mock gateway rather than a real provider whenever `meta.stage` is set and the gateway is enabled), and `generateProductDraft` already no-ops safely with zero network calls when `ANTHROPIC_API_KEY` is unset, all three were wired unconditionally in the existing `NODE_ENV=test` block in `server.js` — additive only, gated to test mode, zero effect on any other spec or on production.
6. **No live Postgres is available in CI**, but AC1/AC2 require a real signup and a real product to exist. Added `src/web-ui/adapters/fake-test-db.js`, a narrow in-memory stand-in for exactly the `users`/`products` query shapes those two flows issue (not a general SQL engine — deliberately does not attempt to fake `journeys`, `credits`, `standards`, or any other table). Wired only when `NODE_ENV=test` and `DATABASE_URL` is unset; the real `DATABASE_URL`-backed path is untouched.
7. **The `/welcome` plan-selection gate has no test-mode bypass** and requires a real Stripe Checkout round-trip to clear — out of scope for this journey-testing story (owned by lab-s3.2's billing story) and unsafe to exercise for real in CI. Added `POST /test/complete-onboarding` (`NODE_ENV=test` only), mirroring the existing `/test/session` test-infrastructure pattern already in `server.js`.
8. **AC5's "zero real LLM calls" needed a way to observe the server's real HTTP calls from the Playwright process.** Added a small `https.request` wrapper in the test-mode block, counting only calls to `api.anthropic.com`/`githubcopilot.com`, exposed via `GET /test/real-llm-call-count` (`NODE_ENV=test` only).

**Alternatives considered:** Leave the spec unimplementable and log AC1–AC5 as blocked pending a separate infra story. Rejected — every gap above was small, narrowly scoped, test-mode-gated (points 5–8) or a one-line/one-file fix required by an existing named constraint (points 1, 3) or an existing broken call site with no working caller anywhere in the app (points 2, 4) — none of them is a new production feature; all are completions of half-wired D37 adapters or missing plumbing the existing code already assumed would exist.
