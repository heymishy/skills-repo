# Decisions: admin-credits-topup rate-limit bypass gap

## FINDING — admin-credits-topup.js never sends the existing staging rate-limit bypass header (2026-07-23)

**Context:** After `catc-s1` (PR #568) and `bssm-s1` (PR #572) both merged and were independently confirmed passing standalone against real `wuce-staging`, a full-suite real CI run (PR #572, run 30041728074) still showed a3's AC3 and a4's combined AC2&AC3 test skipping. Re-running the same tests standalone against the identical live environment moments later passed immediately — ruling out a real regression and pointing at run-shape-dependent contention.

**Root cause:** `auth-email.js`'s `_checkRateLimit()` (from `serlb-s1`) already implements a narrow, triple-gated bypass for the real 10-attempts/5-minute per-IP limiter (env secret + `x-e2e-rate-limit-bypass` header + `e2e-test-`-prefixed email). `staging-auth.js`'s `signUpEmail()` correctly sends this header. `admin-credits-topup.js`'s `_adminLogin()`/`_adminSignupOnce()` — called by a3, a4, and b1 to top up the same fixed `e2e-test-admin@example.test` identity's credits before every real-turn test — never send it, even though the identity's email already satisfies the third gate. When a1/a2/a3/a4 (and separately b1) all run in the same CI job, the admin identity's un-bypassed login attempts can accumulate past the real rate-limit threshold within the job's runtime window, causing the credits top-up to fail and the dependent test to skip — non-deterministically, depending on exactly how many spec files share the job and how quickly they run.

**Decision:** Fix by wiring the SAME existing bypass header into `admin-credits-topup.js`'s two HTTP calls — do not touch `auth-email.js`'s rate-limiter logic itself (already correct and appropriately narrow). See `stories/eatrl-s1.md` and `dor/eatrl-s1-dor.md` for the full fix and DoR sign-off.

**Verification performed:** Direct code inspection of `_checkRateLimit()`'s exact three gates (`auth-email.js` lines 55-140) and `admin-credits-topup.js`'s `_adminLogin()`/`_adminSignupOnce()` (confirmed no header sent in either). Reproducing evidence: real CI run 30041728074's compact reporter dot-pattern (`···°°°··°°·`) showing a3 AC3 (position 9) and a4 AC2&AC3 (position 10) skipped, contrasted against an immediate standalone pass on the same live environment. No fix applied yet in this pass — root cause confirmed and DoR signed off; implementation deferred to a dispatched coding agent per this repo's short-track pipeline.

## IMPLEMENTED — fix wired, real CI results (2026-07-24)

**What was done:** Exported `RATE_LIMIT_BYPASS_HEADER` and `STUB_SECRET` from `tests/e2e/fixtures/staging-auth.js`'s `module.exports` (additive; no existing caller affected), then imported and used them in `tests/e2e/fixtures/admin-credits-topup.js`'s `_adminLogin()` and `_adminSignupOnce()`, mirroring `signUpEmail()`'s existing pattern exactly. `auth-email.js` untouched. Branch `fix-forward-e2e-admin-topup-rate-limit-bypass`, draft PR #573.

**Local verification (AC1, AC3):** Code inspection confirms both functions now set `headers[RATE_LIMIT_BYPASS_HEADER] = STUB_SECRET` when `hasStubSecret()` is true (AC1). `npm test` locally: 36 failing files, all a subset of the 68-entry `tests/known-baseline-failures.json`, zero new regressions (AC3).

**Local repro of AC2 not possible:** `E2E_STAGING_AUTH_STUB_SECRET` is a GitHub Actions repository secret (confirmed present via `gh secret list`, name only — value inaccessible locally) and was not present in the local dev shell. Per this test plan's own documented contingency ("if unavailable locally, rely on the real CI job's own run for AC2/AC4 verification instead of a local repro"), AC2 and AC4 were verified via the real `scenario-a-staging-e2e`/`scenario-b-staging-e2e` CI jobs on PR #573 instead of a local repro.

**Real CI results (PR #573, run 30044703495):**
- **`scenario-b-staging-e2e`** (b1's spec, single file, `--workers=1`): **4 passed, 0 skipped, 0 failed.** All of b1's AC1-AC3 plus NFR-Security passed — previously these were expected to skip pending a separate credits-upsert sub-blocker (per `admin-credits-topup.js`'s own header comment), which appears to have been resolved by `catc-s1` (PR #568, merged since). No rate-limit-caused skip observed.
- **`scenario-a-staging-e2e`** (a1+a2+a3+a4, `--workers=1`, 11 tests total): dot-pattern `···°°°···F·` → **7 passed, 3 skipped, 1 failed.** The 3 skips are a2's own, unrelated, intentional manual-only skips (per this job's own name: "A2 manual-only"), not rate-limit related. **a3's AC3 — the specific test that skipped in PR #572's run — now passes cleanly.** a4's combined AC2&AC3 test **failed** (not skipped) with a real assertion mismatch: `expect(resumedThreadText).toContain(distinctiveDetail)` — the resumed session's chat log did not contain the distinctive pre-close message, showing stale/duplicated canvas-mapping content instead of the actual restored turn history. Critically, this is a genuine, substantive test *failure* after the credits top-up and login/signup succeeded (no `toppedUp: false` reason, no HTTP 429) — i.e. the rate-limit bypass fix worked exactly as intended (no rate-limit-caused skip on either a3 or a4), and this specific failure is a **separate, pre-existing, out-of-scope defect in a4's SSE-mid-stream session-resume turn-history restoration**, unrelated to credits, rate-limiting, or this story's scope. Recommend a dedicated follow-up bug-fix story for a4's session-resume turn-history defect.

**Unrelated CI-only unit-test finding:** the "Lint, typecheck, test, build" job's full-suite `npm test` run (Ubuntu CI runner) reported one failing file not in the checked-in baseline: `tests/check-p4-enf-second-line.js` (reproduced identically on a manual re-run of the same job — twice in a row). This file: (a) is untouched by this PR's diff (only `tests/e2e/fixtures/staging-auth.js` and `tests/e2e/fixtures/admin-credits-topup.js` were changed), (b) has no history since April 2026 and no dependency on either changed file, and (c) passes cleanly standalone (22 passed, 0 failed) when run outside the full aggregate suite. This matches the exact resource-contention flakiness pattern already documented in `tests/known-baseline-failures.json`'s own notes (the `check-md-3-adr.js`/`check-p3.5-validate-trace.js` pairing) — a full-suite-only, CI-runner-specific flake, not a regression introduced by this change. Recommend adding it to the baseline in a future baseline-refresh story rather than blocking this PR on it.

**Conclusion:** The rate-limit bypass fix is confirmed working as designed — no rate-limit-caused skip observed on a3, a4, or b1 across two real CI jobs. Two separate, out-of-scope findings were surfaced during verification (a4's session-resume turn-history defect; the `check-p4-enf-second-line.js` full-suite flake) and are recorded here for follow-up, per this story's own explicit "Out of Scope" boundary — not fixed as part of this story.
