# Story: Implement surface adapter model foundations — Path B git-native reference

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-3-distribution-surface-model.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the `execute(surface, context) → result` surface adapter interface defined with a git-native reference implementation and Path B `context.yml` surface type resolution,
So that surface-aware stories have a concrete, implemented contract to scope their ACs against — and Path A (EA registry) can be added in Phase 2 without redesigning the interface.

## Benefit Linkage

**Metric moved:** MM1 — Solo operator outer loop
**How:** Without a defined surface adapter contract and Path B resolution, story ACs for git-native implementation stories cannot reference a canonical surface type — they must either be surface-agnostic (losing precision) or hardcode assumptions (creating future debt). This story enables correctly scoped ACs in downstream stories, which reduces blocking lookups during the outer loop.

## Architecture Constraints

- Path A (EA registry surface type resolution) must not be foreclosed by Phase 1 design decisions — the interface must be extensible to accommodate the registry lookup in Phase 2 without a breaking change to the `execute` interface contract
- The git-native adapter is the reference implementation and must be a working implementation, not a stub — the surface adapter contract is validated by running it, not just defining it
- `context.yml` Path B declaration format must be agreed with and consistent with the distribution mechanism (P1.1) — the two stories share the `context.yml` format as a boundary; if worked in parallel, schema agreement must happen before either story's AC3 can be verified
- Credentials are structural — the git-native adapter must not read or write credentials directly; authentication is handled via the secrets store reference in `context.yml` (product constraint #12)

## Dependencies

- **Upstream:** `p1.1-distribution-progressive-disclosure` — the base layer assembly format and `context.yml` schema must be stable before Path B resolution in `context.yml` can be tested end-to-end; stories can be parallelised but need a synchronisation point at schema agreement
- **Downstream:** `p1.3-assurance-agent-ci-gate` — CI gate stories benefit from having a defined surface type in their context; however P1.3 can proceed without this story if needed
- **Downstream:** `p1.7-standards-model-phase1` — the `standards/index.yml` routing table uses surface type as a routing key; the surface type vocabulary defined here is the input to the routing table

## Acceptance Criteria

**AC1:** Given the `execute(surface, context) → result` interface definition exists in the platform repository, when a git-native surface adapter is invoked with `surface: "git-native"` and a valid `context` object, then it returns a `result` object conforming to the defined interface contract — with at minimum: `status` (pass/fail/error), `surface` (echoed from input), `findings` (array, may be empty), and `trace` (trace identifier for the run).

**AC2:** Given `context.yml` in a consuming squad repository contains a `surface:` declaration block with `type: "git-native"` under Path B (`explicit-declaration`), when the surface adapter resolution runs, then the git-native adapter is selected — not Path A resolution, which requires an EA registry lookup not available in Phase 1.

**AC3:** Given a multi-surface `context.yml` declaration (more than one surface type listed), when Path B resolution runs, then each declared surface selects the correct adapter for that surface type — demonstrating the multi-surface declaration model works beyond a single surface.

**AC4:** Given the Phase 2 EA registry integration requirement (Path A), when the `execute` interface contract is reviewed, then a developer can add Path A resolution by implementing an EA registry lookup that returns a surface type string — without modifying the `execute` function signature or breaking any existing Path B users. `[Verification: code review]`

**AC5:** Given the surface adapter interface definition, when it is reviewed against the five non-git surface types listed as Phase 2 scope (IaC, SaaS-API, SaaS-GUI, M365-admin, manual), then none of those surface types are foreclosed by the interface shape — each could be added as a new adapter without an interface revision. `[Verification: code review]`

**AC6:** Given the `context.yml` Path B format is agreed between P1.1 (distribution/assembly) and P1.2 (surface adapter resolution), when both stories are DoD-complete, then a `context.yml` file conforming to the Path B format is valid as both an assembly configuration (P1.1) and a surface type resolution input (P1.2) — it is one shared file, not two separate files.

## Out of Scope

- IaC, SaaS-API, SaaS-GUI, M365-admin, and manual adapter implementations — Phase 2; this story delivers the interface and git-native reference only
- Path A (EA registry) surface type resolution — Phase 2; the interface must accommodate it without being foreclosed by it, but no registry lookup is implemented
- DoD criteria variant generation by surface type — the surface adapter identifies the surface type; producing surface-specific DoD criteria variants from that type is a Phase 2 behaviour of the assurance agent
- `context.yml` MCP credentials section — governed by product constraint #12; this story does not touch it

## NFRs

- **Performance:** Path B surface type resolution (reading `context.yml` and selecting an adapter) must complete within 1 second — it is an inline step in the CI gate run budget
- **Security:** The git-native adapter must not log, persist, or transmit any credential values — it reads from the secrets store reference in `context.yml` only (product constraint #12)
- **Audit:** The `result` object returned by the adapter must include the surface type and the adapter version used — enabling the trace to record which adapter implementation produced the result

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the interface contract and git-native adapter are well-understood; only the `context.yml` schema synchronisation with P1.1 introduces potential rework, and this is a bounded coordination point
