# Story: Author and review MODEL-RISK.md before non-dogfood adoption

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-4-standards-eval-compliance.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want `MODEL-RISK.md` authored and reviewed before the platform is used beyond the dogfood context,
So that any adopting team or governance function has a documented, honest account of the AI governance risks inherent in the platform — and substantiating evidence from the trace that the eight audit questions are answerable.

## Benefit Linkage

**Metric moved:** T3M1 — Trace readability for risk review; MM1 — Solo operator outer loop
**How:** T3M1 cannot be validated without `MODEL-RISK.md` defining the eight audit questions — the trace is the evidence; `MODEL-RISK.md` defines what is being evidenced. Without a validated T3M1, the claim that the trace supports non-engineering risk review is unsubstantiated. MM1 is also moved: a platform without `MODEL-RISK.md` creates a blocking lookup at DoR for any story requiring a governance sign-off ("what is the model risk profile for this run?").

## Architecture Constraints

- `MODEL-RISK.md` must be substantiated by real trace evidence from a live inner loop story run — not by self-assertion. The document is not DoD-complete until at least one real trace exists that answers all eight audit questions. This is the T3M1 dependency: `MODEL-RISK.md` and the trace co-validate each other.
- The eight audit questions are defined in the platform operating model (§9.8) and referenced in the T3M1 metric definition — they are fixed for Phase 1. Any revision to the eight questions requires a decision log entry before `MODEL-RISK.md` is updated.

## Dependencies

- **Upstream:** `p1.3-assurance-agent-ci-gate` — the trace that serves as T3M1 evidence is produced by the CI gate; `MODEL-RISK.md` cannot be fully validated until a real CI-triggered trace exists
- **Upstream:** `p1.7-standards-model-phase1` — `MODEL-RISK.md` must reference whether standards were injected correctly into the evidence trace; P1.7 must be DoD-complete before P1.8's trace evidence is valid
- **Downstream:** Nothing in Phase 1 is blocked by P1.8, but no non-dogfood adoption of the platform is permitted until P1.8 is DoD-complete — this is a hard adoption gate, not a pipeline dependency

## Acceptance Criteria

**AC1:** Given `MODEL-RISK.md` exists in the platform repository, when it is reviewed, then it documents at minimum the following risks from the discovery artefact (Assumptions and Risks section): three-agent independence is procedural not structural; hash verification is an audit signal not tamper prevention; anti-overfitting gate is self-assessed; token budget may constrain delivery pace; Copilot Agent mode inter-session limitation — with each risk having a stated mitigation or explicit acceptance rationale.

**AC2:** Given `MODEL-RISK.md` exists, when a non-engineering reviewer reads it alongside the eight audit question definitions from the platform operating model (§9.8), then the reviewer can identify — for each audit question — which field in the trace file answers it, without asking for engineering assistance. The mapping from audit question to trace field must be explicit in `MODEL-RISK.md`.

**AC3:** Given a real inner loop story trace produced by P1.3 after P1.7 is DoD-complete, when the platform maintainer acting as stand-in risk reviewer evaluates the trace against all eight questions, then all eight questions are answerable from the trace file alone — and this result is recorded in `MODEL-RISK.md` as the T3M1 acceptance test record, including: story ID used as evidence, trace file path, each question's answer (Y/N), and verdict (pass/fail).

**AC4:** Given `MODEL-RISK.md` is authored and the trace evidence record is complete, when a human reviewer signs off on the document, then a sign-off record is appended to `MODEL-RISK.md` containing: reviewer name, review date, verdict (approved for adoption / approved with conditions / not approved), and any conditions if applicable. A `MODEL-RISK.md` without a completed sign-off record is not DoD-complete.

**AC5:** Given `MODEL-RISK.md` sign-off is complete, when the onboarding documentation for the platform is reviewed, then `MODEL-RISK.md` is listed as a required pre-read in that documentation — not an optional reference — so that any squad tech lead encountering the platform for the first time is directed to it before running any inner loop stories.

## Out of Scope

- Automated `MODEL-RISK.md` generation or update from gate run data — Phase 2; the document is human-authored at Phase 1, informed by real trace evidence
- Formal external audit of `MODEL-RISK.md` — the platform is in dogfood context; Hamish acts as stand-in for all reviewer roles at Phase 1. Non-dogfood adoption with genuine regulatory exposure requires an independent reviewer.
- Risk quantification (likelihood × impact scoring) — `MODEL-RISK.md` documents the risk qualitatively with mitigations; formal quantitative scoring is not in scope
- Legal or compliance sign-off — out of scope for Phase 1 dogfood context; required before production adoption in a regulated enterprise

## NFRs

- **Security:** `MODEL-RISK.md` must not contain credentials, operation details, or security-sensitive specifics of the platform's internal architecture that would assist an adversary. Risk descriptions must be at a level appropriate for a governance audience, not an attack surface enumeration.
- **Audit:** `MODEL-RISK.md` is a living document — each substantive update (new risk identified, evidence record updated, sign-off appended) must be committed with a traceable commit message. The git history of `MODEL-RISK.md` is the audit trail of how the risk profile evolved.
