## Observations and /levelup candidates

**Deferred AC3/AC4 cascade:** AC4's dependency on AC3 creates a cascade where both are resolved by the same Phase 2 trigger (first real gate run). This is clean design — the evidence and sign-off are a single operation. The platform maintainer should execute both simultaneously: run the story through the gate, record the T3M1 evidence, complete the sign-off. Recommend adding this as an explicit Phase 2 onboarding action item.

**Phase 2 acceptance test bundle:** Across p1.3, p1.7, and p1.8, seven deferred ACs are all unblocked by the same trigger — the first real Phase 2 inner loop PR through the P1.3 gate. A single gate run resolves: p1.3-AC1 (CI log evidence), p1.3-AC4 (automated test run in live CI), p1.7-AC4 (standardsInjected array populated), p1.7-AC5 (hash recompute), p1.8-AC3 (T3M1 record), p1.8-AC4 (sign-off), and M3 metric (signal on-track). This should be captured as a Phase 2 kickoff checklist item. **High-value /levelup candidate.**

**U4 test design:** The `u4-signoff-verdict-not-pre-populated` test enforces that the verdict cell cannot be pre-populated by a developer. This is an effective anti-fraud pattern — prevents implementation PRs from including a fake sign-off. The sign-off action survives the test suite because the real sign-off will be recorded in a separate commit. This pattern is generalizable to any human-gated field in a platform document. **Candidate /levelup pattern.**

**T3M1 metric signal:** Deferred to Phase 2. The metric definition is sound; the measurement opportunity arrives with the first real Phase 2 gate run.
