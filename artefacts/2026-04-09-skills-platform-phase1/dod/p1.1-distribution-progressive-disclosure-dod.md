## DoD Observations

1. **M1 acceptance test record completion at DoD:** The benefit-metric.md M1 record had `distributionMechanism` and `squadMergeActionRequired` pre-filled at implementation time; `changeReference`, `elapsedTime`, and `changePresentInAssembledContext` were left blank for DoD completion. This is the intended flow per the DoR contract — DoD populated all five fields. Pattern: fields requiring live cycle observation are placed at DoD time, structural fields placed at implementation time. This pattern should be documented for future metric-carrying stories. **Candidate /levelup finding.**

2. **`platform-version: abc1234def5678` is a placeholder hash:** The assembled fixture header shows an example hash rather than the actual git commit SHA of the skills-repo at assembly time. The NR audit test accepts any non-empty non-bracket value, so it passes, but a reviewer cannot trace back to the exact platform state. The assembly script should substitute the real `git rev-parse HEAD` value at assembly time. **Candidate /levelup finding.**

3. **Self-referential dogfood nuance:** This repo is simultaneously the platform repo and the consuming squad repo. AC1's "squad repo has no copied SKILL.md files" held because the assembled file references skills by description (not SKILL.md content), with the pull mechanism reading from the platform reference. In a real multi-repo deployment, the consuming squad would have only the assembled file + the assembly script, pointing at the skills-repo remote. The current setup validates the mechanism; multi-repo validation is Phase 2.
