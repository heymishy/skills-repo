## What will be built

- `workspace/suite.json` — initial living eval suite populated with at least one outcome-oriented scenario per core skill currently in the skills platform; each scenario has the five required fields (`taskId`, `description`, `skillSurfaceCombination`, `expectedOutcome`, `failurePatternGuarded`) plus `reviewNote` (non-null on every scenario at initial delivery)
- Schema validator: validates required fields present and non-null, `taskId` unique within suite, `reviewNote` non-null (audit check)
- Suite parser: reads `workspace/suite.json`; classifies scenarios as active or retired; a scenario is retired only if both `retiredAt` and `retiredReason` are populated — any scenario missing `retiredReason` is flagged as a silent-removal policy violation
- Outcome-oriented gate evaluation: given a gate run output, evaluates each active scenario's `expectedOutcome` string for presence/absence in the output; records `pass` or `fail` result per scenario without checking the procedure used to produce the output

## What will NOT be built

- Automated scenario generation from failure logs — Phase 2
- Domain-tier and squad-tier suite extensions — Phase 2
- Performance benchmarking scenarios — Phase 2
- Cross-skill dependency scenarios — Phase 2
- Suite visualisation or dashboard — Phase 3
- M4 controlled acceptance test (AC5) — dependency-blocked on p1.4 DoD; scheduled as a follow-on acceptance test after p1.4 merges

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Schema validation: all five required fields present and non-null; taskId unique; at least one scenario per core skill | Unit (U1–U3) + Integration (I1) |
| AC2 | Gate evaluation against mock output: pass case (expectedOutcome present) + fail case (expectedOutcome absent); result recorded in gate output | Integration (I2, I3) |
| AC3a | PR merge policy: reviewNote must be populated before merge is permitted — enforced by merge policy and verifiable by inspection of reviewNote field content | Merge policy inspection |
| AC3b | reviewNote field populated and non-null; audit checker returns gap finding for null reviewNote | Unit (U4, U5) + NFR (N3) |
| AC4 | Active scenarios cannot be silently skipped; retirement requires retiredAt + retiredReason; missing retiredReason flags as policy violation | Unit (U6, U7) + Integration (I4) |
| AC5 | Controlled M4 test: add failing scenario → gate returns blocked → remove scenario → gate returns pass | Acceptance test (post p1.4 DoD) |
| AC6 | Each scenario in the initial suite passes the outcome-orientation diagnostic at DoD time | Manual authoring review at DoD |

## Assumptions

- Core skills for initial suite population are the skills already present in `.github/skills/` at time of P1.6 implementation
- Mock gate run output fixture is a synthetic string containing known `expectedOutcome` values — no live CI run required for I2/I3
- AC5 acceptance test does not block P1.6 DoD for the suite structure itself; it is a follow-on measurement after p1.4 merges
- `reviewNote` for the initial scenario set is authored by the platform maintainer during P1.6 implementation — self-review in solo operator context
