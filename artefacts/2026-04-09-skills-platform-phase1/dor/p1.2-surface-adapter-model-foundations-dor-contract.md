## What will be built

- The `execute(surface, context) → result` interface definition in the platform repository — no type-specific branching in the `execute` body; interface shape supports Path A extension without signature change
- A working (not stub) git-native adapter: implementation of `execute("git-native", context)` returning a conforming result object with fields: `status` (pass/fail/error), `surface` (echoed), `findings` (array), `trace` (non-empty identifier)
- Path B surface type resolution: reads `context.yml` for `surface.path: "explicit-declaration"`, selects the matching registered adapter; short-circuits before triggering any Path A EA registry lookup; resolves within ≤1 second
- A shared `context.yml` Path B format coordinated with P1.1 — one file valid as both an assembly configuration input (P1.1) and a surface type resolution input (P1.2)

## What will NOT be built

- IaC, SaaS-API, SaaS-GUI, M365-admin, manual adapter implementations — Phase 2
- Path A (EA registry) surface resolution — interface must accommodate it without implementing it
- DoD criteria variant generation by surface type — Phase 2 assurance agent behaviour
- `context.yml` MCP credentials section — not touched (product constraint #12)

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Call `execute("git-native", context)`; assert `status`, `surface` (echoed), `findings` (array), `trace` (non-empty) all present and correct type | Unit (×2) + Integration |
| AC2 | Run Path B resolution against `context.yml` fixture with `type: "git-native"`; assert git-native adapter selected; assert EA registry lookup not triggered (spy/stub on Path A call site) | Unit (×2) |
| AC3 | Multi-surface `context.yml` fixture (git-native + one other); run resolution; assert each surface maps to its correct adapter | Unit + Integration |
| AC4 | Code review: inspect `execute` body for absence of surface-type branching; confirm Path A can be added via a resolver without changing the function signature | Code review at DoD (design review step) |
| AC5 | Code review: inspect interface definition; confirm no narrowing that forecloses IaC, SaaS-API, SaaS-GUI, M365-admin, manual as future adapter types | Code review at DoD (design review step) |
| AC6 | Integration test: same `context.yml` Path B fixture passes both P1.1 assembly validation and P1.2 Path B resolution; manual walk-through | Integration + Manual |

## Assumptions

- The `context.yml` schema synchronisation with P1.1 is a bounded coordination point — both stories agree on format before AC6 test is run; schema agreement must happen at implementation start
- Synthetic `context` objects are sufficient for unit/integration tests — no live git clone required
- Code review verification for AC4/AC5 is performed at DoD by the same operator who authored the interface; in solo operator context this is self-review documented in the DoD artefact
