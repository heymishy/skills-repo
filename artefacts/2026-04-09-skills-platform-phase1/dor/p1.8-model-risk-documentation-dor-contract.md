## What will be built

`MODEL-RISK.md` authored in the platform repository, containing:

1. **Risk register** — all 5 named risks from the discovery artefact (three-agent independence is procedural not structural; hash verification is an audit signal not tamper prevention; anti-overfitting gate is self-assessed; token budget may constrain delivery pace; Copilot Agent mode inter-session limitation), each with a stated mitigation or explicit acceptance rationale. No bare/undocumented risk positions.

2. **Audit question mapping table** — all 8 questions from the platform operating model (§9.8) mapped to specific named trace fields. Each row: question → trace field name → how to locate it. No absent or TBD entries. Readable by a non-engineering reviewer without engineering assistance.

3. **T3M1 acceptance test record section** — structured as a clearly demarcated template block at delivery time. Contains: story ID used as evidence (placeholder), trace file path (placeholder), each question's answer (Y/N) (placeholder), overall verdict (placeholder). Filled with real values at DoD time after P1.3 + P1.7 are both DoD-complete and a real inner loop trace exists.

4. **Sign-off record block** — clearly labelled section at or near the end of MODEL-RISK.md. Contains: reviewer name, review date, verdict field, conditions field. Pre-populated with field labels only; verdict is recorded by the human reviewer at DoD time as a manual action.

**Platform onboarding documentation** — updated to list MODEL-RISK.md as a required pre-read (not an optional reference), positioned before the first inner loop instruction. If no onboarding documentation exists, a minimal ONBOARDING.md is created.

---

## What will NOT be built

- Automated MODEL-RISK.md generation or update from gate run data — Phase 2
- Formal external audit or independent reviewer sign-off — Phase 1 dogfood exclusion; Hamish acts as stand-in reviewer for all reviewer roles
- Quantitative risk scoring (likelihood × impact) — qualitative descriptions with mitigations only; no scoring matrices
- Legal or compliance sign-off process — explicitly out of scope for Phase 1
- Filling the T3M1 acceptance test record with real trace evidence — that is a DoD-time action requiring P1.3 + P1.7 to be DoD-complete first

## Assumptions

- Hamish acts as stand-in reviewer for all reviewer roles in Phase 1 dogfood context (no external reviewer available)
- "Onboarding documentation" is either an existing README.md or a new ONBOARDING.md — exact file confirmed during implementation; if ambiguous, add a PR comment and await operator direction
- The 8 audit questions from the platform operating model (§9.8) are referenced in the discovery artefact and benefit-metric T3M1 definition — they are fixed for Phase 1; the coding agent sources them from those artefacts and does not modify them
- AC3 tests (I1, I2) are dependency-gated and will not pass until P1.3 + P1.7 are DoD-complete and a real inner loop trace exists — this is expected and does not block P1.8 PR open
- "Substantive update" in the Audit NFR means: new risk added, evidence record updated, sign-off appended — cosmetic formatting changes do not require a specially-named commit
