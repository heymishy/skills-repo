## Story: Replace scrape-first skill sessions with model-first chat architecture

**Epic reference:** artefacts/2026-05-05-web-ui-model-first-chat/epics/mfc-epic-1.md
**Discovery reference:** artefacts/2026-05-05-web-ui-model-first-chat/discovery.md
**Benefit-metric reference:** artefacts/2026-05-05-web-ui-model-first-chat/benefit-metric.md

## User Story

As a **pipeline operator using the web UI to run a structured skill session**,
I want the **model to read the full SKILL.md as its system prompt and drive the conversation adaptively**,
So that **the questions asked and the artefact produced are governed by the skill instructions — matching what GitHub Copilot Chat produces in VS Code**.

## Benefit Linkage

**Metric moved:** M1 — Discovery artefact template conformance rate (0% → 100%)
**How:** The scrape-first approach produces wrong sections from operational SKILL.md headings. Replacing it with a model-first approach — where the model reads the full SKILL.md as a system prompt — enables the model to follow the skill's output instructions and produce an artefact whose sections match the template. M1 cannot move without this change.

## Architecture Constraints

- **D37 / injectable adapter rule (ADR-009):** `_skillTurnExecutor` retains its injectable adapter pattern. The two removed adapters (`_nextQuestionExecutor`, `_sectionDraftExecutor`) are exported as no-ops for backward compatibility with any external callers but are not wired in production.
- **No new npm dependencies** — implementation must use Node built-ins only.
- **No Express** — raw `http.createServer` pattern only; no router framework.
- **`req.session.accessToken` canonical** — the token passed to `_skillTurnExecutor` must come from `req.session.accessToken`, never `req.session.token`.
- **`skill-turn-executor.js` signature change** — old signature `(skillContent, priorQA, currentAnswer, token)` is replaced with `(systemPrompt, history, currentInput, token)` where `history = [{role: 'user'|'assistant', content: string}]`. This is a breaking change to all callers; all callers are in this codebase and are updated atomically.
- **Artefact-first rule** — this story artefact must be committed alongside or before the implementation.
- Read `.github/architecture-guardrails.md` before implementing.

## Dependencies

- **Upstream:** wuce.26 (merged), dsq.1–dsq.4 (merged) — these stories' implementations are replaced by this story. Their test files are rewritten.
- **Downstream:** Any future dsq stories should build on the model-first session store, not the scrape-first one.

## Acceptance Criteria

**AC1:** Given a user starts a skill session via `POST /api/skills/:name/sessions` (HTML form), when the session is created, then the user is redirected 303 to `/skills/:name/sessions/:id/chat` (not `/next`).

**AC2:** Given a `GET /skills/:name/sessions/:id/chat` request with a valid authenticated session, when the page renders, then the response is a single-page HTML UI containing: (a) a scrollable `#chat-messages` area showing the full conversation history as alternating assistant/user bubbles, (b) a fixed `#chat-form` at the bottom with a `<textarea id="chat-input">` and a submit button, and (c) client-side JavaScript that POSTs to `/api/skills/:name/sessions/:id/turn` on submit and appends the response to `#chat-messages`.

**AC3:** Given the chat page loads for a brand-new session (no prior turns), when it renders, the `#chat-messages` area shows the model's opening message (the result of an initial `_skillTurnExecutor` call with `history=[]` and `currentInput='Begin the session.'`), generated by reading the full system prompt (copilot-instructions.md + SKILL.md + product context).

**AC4:** Given a user submits an answer via the `POST /api/skills/:name/sessions/:id/turn` endpoint, when the handler runs, then: (a) the answer is appended to `session.turns` as `{role: 'user', content: sanitisedAnswer}`, (b) a single call is made to `_skillTurnExecutor(systemPrompt, session.turns, userInput, token)`, (c) the response is appended to `session.turns` as `{role: 'assistant', content: modelResponse}`, and (d) a JSON response `{done: false, response: modelResponse}` is returned. No second or third model call is made.

**AC5:** Given the model response from `_skillTurnExecutor` contains the text `---ARTEFACT-START---` followed by content followed by `---ARTEFACT-END---`, when the turn handler parses this, then: (a) `session.artefactContent` is set to the extracted content, (b) `session.done` is set to `true`, (c) the JSON response includes `{done: true, artefactContent: ...}`, and (d) a `---SLUG---` line in the model response (if present) is parsed to derive `session.artefactPath` as `artefacts/[slug]/[skillName].md`.

**AC6:** Given `session.done === true`, when `GET /skills/:name/sessions/:id/commit-preview` renders, then the preview page shows `session.artefactContent` (the model-synthesised artefact) and `session.artefactPath` (derived from the model-provided slug), not the old scrape-first content.

**AC7:** Given `buildSystemPrompt(skillName, sessionPath)` is called, when executed, then it returns a string containing: the full contents of `.github/copilot-instructions.md`, the full contents of `.github/skills/[skillName]/SKILL.md`, the contents of all four product context files (mission.md, tech-stack.md, constraints.md, roadmap.md), and a web UI protocol section instructing the model to use `---ARTEFACT-START---` / `---ARTEFACT-END---` markers and a `---SLUG---` line.

**AC8:** Given the new `_skillTurnExecutor` signature `(systemPrompt, history, currentInput, token)`, when called, then it builds a messages array as `[{role:'system', content:systemPrompt}, ...history, {role:'user', content:currentInput}]` and sends this to the Copilot Chat Completions API.

**AC9:** Given `setNextQuestionExecutorAdapter` and `setSectionDraftExecutorAdapter` are still exported from `src/web-ui/routes/skills.js` for backward compatibility, when called, they accept a function without throwing — but the adapters they set are not invoked in any code path of the model-first session flow.

**AC10:** Given npm test is run after this story's implementation is merged, then 0 tests fail. The 7 rewritten test files each pass their full suites.

## Out of Scope

- Streaming responses (SSE or chunked transfer) — deferred to a future story.
- Saving `session.turns` to disk between server restarts.
- Changing the authentication or session-cookie mechanism.
- Changing any JSON API routes (`/api/skills`, `POST /api/skills/:name/sessions` JSON path, `/api/skills/:name/sessions/:id/answers`, `/api/skills/:name/sessions/:id/state`, `/api/skills/:name/sessions/:id/commit` JSON path, `/api/skills/:name/sessions/:id/resume`).
- The `/complete` page (`htmlGetCompletePage`) — retained as-is or wired to `session.done` for future use.
- Mobile-responsive CSS polish.
- Changing the commit flow (`handleGetCommitPreviewHtml`, `handlePostCommitHtml`, `handleGetResultHtml`) beyond what is needed to read `session.artefactContent` / `session.artefactPath`.

## NFRs

- **Security:** System prompt must never be echoed in the HTTP response. `session.systemPrompt` must not appear in any client-visible response. The token is never logged. Path traversal on `skillName` is blocked by existing `_checkSkillName` guard.
- **Resilience:** If `_skillTurnExecutor` throws (network error, timeout, API error), the turn handler returns `{done: false, response: ''}` — the user sees an empty assistant bubble, not an error page. The session remains active.
- **Performance:** The `WUCE_TURN_TIMEOUT_MS` env var (default 30 000 ms) governs the API call timeout. No change to the timeout mechanism.
