# Decisions: Mock Gateway Fixtures Deploy Fix

## DECISION — Narrow `.dockerignore`/`Dockerfile` fixture-directory copy, not a blanket `tests/` ship or a fixture relocation (2026-07-23)

**Context:** `artefacts/2026-07-23-e2e-core-journey-coverage/decisions.md`'s a3 FINDING entry (2026-07-23) documented that real `wuce-staging` has `MOCK_LLM_GATEWAY=true` genuinely active (confirmed via `flyctl ssh console --app wuce-staging -C "printenv MOCK_LLM_GATEWAY"` → `true`) but the deployed container has no `/app/tests` directory at all (`flyctl ssh console --app wuce-staging -C "ls -la /app/tests"` → "No such file or directory"), so the mock gateway's fixture lookup (`src/web-ui/modules/mock-llm-gateway.js`'s `FIXTURE_DIR`) always throws "No fixture found," silently swallowed by `htmlSubmitTurn`'s surrounding try/catch, producing an empty `{"done":false,"response":"","usage":null}` response instead of the intended deterministic mock content. This blocks a3's AC3 and will block every other model-turn-driven story in the feature.

**Decision:** Fix the Docker packaging gap with the narrowest correct change: (1) narrow `.dockerignore`'s blanket `tests/` exclusion with step-wise negation patterns so only `tests/e2e/fixtures/llm-gateway/` survives into the build context, and (2) add one explicit `COPY` instruction in the `Dockerfile`'s `production` stage for that exact directory, landing at the path `FIXTURE_DIR` already expects. Do not ship the whole `tests/` tree, and do not relocate fixture files into `src/`.

**Rationale:**
1. **Read the original Docker-deployment story first, rather than assuming `tests/` was excluded by oversight.** `artefacts/archived/2026-05-02-web-ui-copilot-execution-layer/stories/wuce.4-docker-deployment.md`'s AC6 explicitly requires "the final stage contains only production runtime dependencies — no dev dependencies, build tools, or source files not required at runtime — and the final image size is documented in the build output," and `.dockerignore`'s `tests/` line carries the comment "Test files — not required at runtime (AC6: production-only final image)." This is a deliberate, well-reasoned exclusion, not an oversight — `tests/` is 5.3 MB (`du -sh tests/`), almost all of it the full unit/integration suite and non-fixture E2E specs, genuinely not needed at runtime.
2. **The one thing actually needed at runtime is 20 KB, not 5.3 MB.** `tests/e2e/fixtures/llm-gateway/` (`du -sh` → 20 KB, 16 files) is data (canned JSON responses), not code or test logic — no different in kind from the `skills/` and `product/` "runtime data directories" the Dockerfile's own comment already describes and already ships via direct `COPY` in the production stage. Shipping the whole `tests/` tree to satisfy this one dependency would reintroduce exactly the image-bloat/attack-surface problem AC6 was written to prevent, for a directory 265x larger than what's actually required.
3. **Considered and rejected: relocate fixtures under `src/`** so the existing `COPY src/` lines cover them automatically, avoiding any Dockerfile/`.dockerignore` change. Rejected because it blurs the repo's `src/` (runtime code) vs `tests/` (test-only assets) boundary for a directory that is conceptually a test fixture set (read by `tests/e2e/*.spec.js` files and this mock gateway alike), and would require updating every fixture-authoring/inventory reference (`inventoryFixtures()`, the 7-stage fixture-matrix tests, any future story adding fixtures) to a new location — treating a packaging problem as a reason to reorganise source layout. The narrower fix (teach the Docker pipeline about one small, clearly-scoped runtime dependency) is more targeted and leaves the code's own directory conventions untouched.
4. **`.dockerignore` negation requires step-wise un-excluding, not a single bare `!` line**, because a `!`-negation cannot re-include a path whose ancestor directory is itself fully excluded by an earlier pattern — the documented dockerignore/gitignore idiom for "exclude everything except one nested path" requires each intermediate directory level to be explicitly re-included in sequence. Verified this requirement directly against the real deployed container (`flyctl ssh console`) as the authoritative check, not just static reasoning, since no dockerignore-parsing library (e.g. `ignore`) is installed in this repo to lean on for a pre-verified confirmation.

**Consequence:** The mock gateway's fixture set now reaches the deployed container without reversing the original AC6 image-size/attack-surface discipline. `mock-llm-gateway.js`, `fly.staging.toml`, and all Fly secrets remain untouched — the defect was entirely in what the Docker build shipped, not in application code or configuration.

**Source:** `mgfd-s1` investigation, 2026-07-23; corroborating investigation: `Dockerfile`, `.dockerignore`, `src/web-ui/modules/mock-llm-gateway.js`, `artefacts/archived/2026-05-02-web-ui-copilot-execution-layer/stories/wuce.4-docker-deployment.md`, PR #557, `artefacts/2026-07-23-e2e-core-journey-coverage/decisions.md`'s a3 FINDING entry.

## FINDING + FIX — a second, distinct root cause was hiding behind the first: the mock-gateway adapter was never wired outside `NODE_ENV=test`, so real staging still returned an empty turn response even after the Dockerfile fix landed (2026-07-23)

**Context:** After deploying the Dockerfile/`.dockerignore` fix above and confirming via `flyctl ssh console --app wuce-staging -C "ls -la /app/tests/e2e/fixtures/llm-gateway"` that all 16 fixture files now exist in the real container (previously: `/app/tests` did not exist at all), this story's own AC4 required driving a real turn end-to-end to prove the fix. Reproducing the exact repro steps from the original FINDING (`e2e-test-admin` identity, real `POST /api/skills/discovery/sessions/:id/turn`), the turn **still** returned `{"done":false,"response":"","usage":null}` — the same symptom, but the fixture-file gap was independently confirmed closed, so this had to be a second, different defect.

**Finding:** `src/web-ui/server.js` only ever called `mock-llm-gateway.js`'s `wireDefaultMockGatewayClient()` from inside a block gated by `if (process.env.NODE_ENV === 'test')` (confirmed: this block spans lines 1072–1354 in the pre-fix file, and the wiring call was nested inside it, added by `bri-s3.2`'s own comment: "nothing in server.js ever called `wireDefaultMockGatewayClient()`... Wiring the built-in fixture-file-backed client here (test mode only) is what actually makes the @mocked gateway usable end-to-end"). But `mock-llm-gateway.js`'s own documented activation contract (`isMockGatewayEnabled()`) is explicitly **NODE_ENV=test OR MOCK_LLM_GATEWAY=true** (hard-blocked only when `NODE_ENV=production`) — real `wuce-staging` runs with `NODE_ENV=staging` (confirmed via `flyctl ssh console -C "printenv NODE_ENV"` → `staging`) and `MOCK_LLM_GATEWAY=true`, which correctly makes `isMockGatewayEnabled()` return `true`, but the wiring call site never ran because `NODE_ENV !== 'test'` on staging. So `getMockResponse()` still threw `"Adapter not wired: mockGatewayClient"` for every real turn, silently swallowed by `htmlSubmitTurn`'s surrounding try/catch into the same empty response — a second, distinct defect masquerading as the same symptom as the first.

**Decision:** Moved the wiring call out of the `NODE_ENV==='test'`-only block to a new, unconditional top-level block in `server.js` (immediately before the existing `NODE_ENV==='test'` test-infrastructure block), gated only on `isMockGatewayEnabled()` itself — matching the adapter's own documented contract exactly, rather than a narrower one that happened to also satisfy it in test mode. No change to `isMockGatewayEnabled()`'s own logic (still hard-blocks `NODE_ENV=production` regardless of `MOCK_LLM_GATEWAY`).

**Rationale:**
1. This is squarely inside this story's own stated goal ("a real turn against a mock-gateway-enabled session on real staging returns the configured fixture's text") — without this fix, AC4/AC5 could not be verified at all regardless of the Dockerfile fix, so it was not a separable, independently-schedulable follow-up in the way the credits-upsert admin-UI gap (below) is.
2. The fix is a one-line relocation of an existing, already-safe call (`wireDefaultMockGatewayClient()` only wires a client that only ever activates when `isMockGatewayEnabled()` is true, which itself already hard-blocks production) — no new adapter, no new activation logic, no schema/config change.
3. Verified directly: re-running the exact same real-turn repro against redeployed `wuce-staging` after this fix now returns real fixture text for both `/discovery` (`"Understood — I'll run a full discovery now...`, with the full canned artefact) and `/ideate` (`"Let's map out the opportunity space for your idea...` with its `---CANVAS-JSON:` marker) instead of an empty response.
4. `tests/check-bri-s3.1-mock-llm-gateway.js` (20/20) and the full `npm test` regression suite (36 pre-existing failures, identical set to `tests/known-baseline-failures.json`, 0 new) both confirm no regression from moving the wiring call.

**Consequence:** The mock-LLM-gateway now genuinely activates and serves fixture content on real `wuce-staging`, closing both halves of the original structural gap (missing fixture files AND missing wiring). Re-running a3's own Playwright spec (`tests/e2e/a3-product-feature-ideate-canvas.spec.js`) against real staging confirms AC1/AC2 still pass, and AC3 no longer skips for this reason — it now skips for the entirely separate, already-documented `getValidTenantIds()` admin-UI credits gap (`artefacts/2026-07-23-credits-upsert-fix/decisions.md`'s GAP entry), confirmed via direct invocation of `topUpTestTenantCredits()` returning the exact same "unknown tenantId" 400 reason recorded there. That gap is out of scope for this story (a pre-existing, separately-tracked, unresolved admin-endpoint validation gap) and is not fixed here.

**Source:** `mgfd-s1` implementation, 2026-07-23; corroborating investigation: `src/web-ui/server.js` (lines 1072–1354 pre-fix), `src/web-ui/modules/mock-llm-gateway.js`, real `flyctl ssh console` inspection, real redeployed-staging turn verification, `artefacts/2026-07-23-credits-upsert-fix/decisions.md`'s GAP entry.
