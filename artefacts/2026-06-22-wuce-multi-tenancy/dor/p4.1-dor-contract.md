# DoR Contract — p4.1: Prompt-cache key scoping + per-tenant rate-limit isolation

**Date:** 2026-06-24
**Story:** p4.1
**Feature:** wuce-multi-tenancy

---

## What will be built

1. `src/web-ui/adapters/cache-key.js` (NEW) — exports `buildCacheKey(session)`: returns `'${tenantId}-${sessionId}'` when tenantId present; falls back to `sessionId` alone
2. `src/modules/skill-turn-executor.js` (MODIFY) — `wrapSystemPromptForCaching` prepends `<!-- cache-scope: ${buildCacheKey(session)} -->\n` when tenantId is present, ensuring distinct content hashes per tenant
3. `src/web-ui/middleware/rate-limiter.js` (MODIFY) — exports `buildRateLimitKey(req)`; `createRateLimiter` uses tenantId-or-IP as the counter key instead of userId
4. `tests/check-p4.1-tenant-cache-ratelimit.js` (NEW) — 7 unit tests covering AC1–AC7

## What will NOT be built

- No changes to journey.js, session-store.js, repo-root.js, journey-access.js, auth.js, or server.js
- No Redis-backed rate limiting (in-process Map only — Phase 3 waived per W1 RISK-ACCEPT)
- No per-tenant rate-limit quota configuration (same limit for all tenants; only namespace changes)
- No changes to the Anthropic cache_control mechanism itself (only the system prompt content changes)
- No HTTP-level integration tests (all tests are pure-function unit tests)

## How each AC will be verified

| AC | Test approach | Type |
|----|--------------|------|
| AC1 — cache keys distinct for same sessionId across tenants | `buildCacheKey({tenantId:'org-a',sessionId:'abc'}) !== buildCacheKey({tenantId:'org-b',sessionId:'abc'})` | Unit |
| AC2 — key format `${tenantId}-${sessionId}` | `strictEqual(buildCacheKey({tenantId:'my-org',sessionId:'xyz123'}), 'my-org-xyz123')` | Unit |
| AC3 — falls back to sessionId when tenantId absent (null, undefined, absent) | Three cases: no tenantId, tenantId:null, tenantId:undefined → all return sessionId | Unit |
| AC4 — rate-limit counters isolated | Increment org-a counter 90×; assert org-b counter is 0 and org-b request not rate-limited | Unit |
| AC5 — rate-limit key includes tenantId | `buildRateLimitKey({session:{tenantId:'my-org'},...}).includes('my-org')` | Unit |
| AC6 — rate-limit key falls back to IP | `buildRateLimitKey({session:{},...}).includes('1.2.3.4')` | Unit |
| AC7 — suite passes | `node tests/check-p4.1-tenant-cache-ratelimit.js` → exit 0 | Integration |

## Assumptions

- `wrapSystemPromptForCaching` in `skill-turn-executor.js` accepts a session object (or has access to one) — confirmed from current implementation
- `req.ip` or `req.connection.remoteAddress` is available in `rate-limiter.js` middleware — confirmed from current middleware pattern
- The `<!-- cache-scope: ... -->` comment prefix does not break any downstream prompt parsing — HTML comment in system prompt is inert
- RISK-ACCEPT W1: in-process Map counters reset on server restart; acceptable for current deployment profile
