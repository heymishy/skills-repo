# DoR Contract — s5.1: Beta monitoring signals

**Feature:** wuce-multi-tenancy
**Story:** s5.1
**Date:** 2026-06-29

---

## What will be built

Two new artefacts (already created as part of the beta sprint implementation): `scripts/check-beta-health.sh` and `artefacts/2026-06-22-wuce-multi-tenancy/reference/beta-monitoring-runbook.md`. The operator will run `check-beta-health.sh` against the live deployment to verify AC5, verify log patterns exist in source via grep (AC1–AC3), and confirm the runbook contains all three failure mode sections (AC4). No `src/` code changes.

## What will NOT be built

- Full APM / Grafana / Datadog integration
- Automated alerting / PagerDuty
- Per-tenant usage dashboards
- Live infra failure simulation in CI (one-time manual beta setup only)

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 — Billing gate log pattern in fly logs | `grep -n "Journey limit reached" src/web-ui/modules/journey-store.js` — at least one match | Integration (code inspection) |
| AC2 — PG write error log pattern in fly logs | `grep -n "[journey-store] PG write error:" src/web-ui/modules/journey-store.js` — at least one match | Integration (code inspection) |
| AC3 — Redis write error log pattern in fly logs | `grep -n "[session] Redis write error:" src/web-ui/middleware/session.js` — at least one match | Integration (code inspection) |
| AC4 — Monitoring runbook exists with all 3 failure modes | Open `reference/beta-monitoring-runbook.md`; confirm 3 sections with grep queries, log format, and remediation steps | Manual (artefact check) |
| AC5 — check-beta-health.sh runs and reports PASS/WARN/FAIL | `APP=<name> bash scripts/check-beta-health.sh` — 4 check sections printed, summary shows PASS/WARN counts, exits 0 | Integration (live deployment) |

## Assumptions

- s4.1 (production deployment) is complete before running AC5.
- `scripts/check-beta-health.sh` and `reference/beta-monitoring-runbook.md` are already created and merged.
- Fly.io CLI is authenticated in the operator's local environment.
- No PG or Redis errors exist in recent logs during AC5 run (fresh deployment).
