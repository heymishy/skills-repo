## Operator Execution Instructions

**Story:** s4.2 — Tenant onboarding configuration
**Proceed:** Yes — all hard blocks pass, no warnings.
**Oversight:** Low — proceed directly.
**Prerequisite:** s4.1 (production deployment complete and healthy).

**Execution order:**
1. Set `TENANT_ORG_ALLOWLIST`: `fly secrets set TENANT_ORG_ALLOWLIST=<org1>,<org2> --app <app>` — list only the GitHub org logins of invited beta orgs
2. Set `WUCE_TENANT_ROOT_BASE`: `fly secrets set WUCE_TENANT_ROOT_BASE=/app/tenants --app <app>` (or preferred path)
3. Create tenant directories: `fly ssh console --app <app> -C "mkdir -p /app/tenants/<org1> /app/tenants/<org2>"`
4. Verify directories: `fly ssh console --app <app> -C "ls /app/tenants"` — each org listed
5. Test allowlisted login (AC1): log in with GitHub account from allowlisted org; confirm redirect to /dashboard; confirm `[auth] login` log line with correct tenantId
6. Test rejected login (AC2): log in with non-allowlisted account in private browser window; confirm error page; confirm message "You are not a member of an authorised organisation." (no org names disclosed)
7. Test journey isolation (AC4): create journey as allowlisted user; confirm artefact under `$WUCE_TENANT_ROOT_BASE/<tenantId>/artefacts/`
8. Verify cap (AC5): `fly secrets list` shows MAX_JOURNEYS_PER_TENANT; optionally test cap boundary
9. Test solo-mode (AC6): start locally without TENANT_ORG_ALLOWLIST; confirm login uses GitHub username as tenantId

**Verification script:** artefacts/2026-06-22-wuce-multi-tenancy/verification-scripts/s4.2-verification.md (6 scenarios)

**Key constraints:**
- NEVER commit `TENANT_ORG_ALLOWLIST` value to git (NFR-sec from nfr-profile.md — NFR-sec-allowlist-disclosure)
- The 403 rejection message must not expose org names (NFR-sec-allowlist-disclosure) — test this explicitly in AC2
- Tenant directories must be created before inviting beta users (Decision 3: no auto-creation)
- Log out / use private browser window between Scenarios 1 and 2 to avoid session bleed in testing
