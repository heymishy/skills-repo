# DoR Contract — s3.1: Neon Postgres provisioning and schema activation

**Feature:** wuce-multi-tenancy
**Story:** s3.1
**Date:** 2026-06-29

---

## What will be built

The operator will: (1) create a Neon free-tier Postgres project via console.neon.tech and retrieve the `DATABASE_URL`; (2) run `node scripts/migrate-schema-pg.js` to create the `journeys` table and `journeys_tenant_id_idx` index; (3) run `node scripts/smoke-test-pg.js` to confirm round-trip write/read/delete with `accessToken` strip verification; (4) set `DATABASE_URL` as a Fly.io secret via `fly secrets set`.

## What will NOT be built

- Automated Neon provisioning via Terraform or CLI scripts
- HA or read replicas (single free-tier instance per Decision 9)
- Automated schema migration on `fly deploy` (operator-run once)
- Migration of existing disk-based journeys to Postgres (out of scope)

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 — Neon project exists and DATABASE_URL accessible | Operator retrieves URL from Neon dashboard; format validated by smoke test | Manual |
| AC2 — Schema migration exits 0 with confirmation message | `node scripts/migrate-schema-pg.js` run; exit code and stdout checked | Manual |
| AC3 — journeys table and index exist | Neon SQL editor: `\dt journeys` + `\di journeys_tenant_id_idx` | Manual |
| AC4 — Round-trip smoke test passes | `DATABASE_URL=<url> node scripts/smoke-test-pg.js` — exits 0, confirms accessToken absent | Integration |
| AC5 — DATABASE_URL in Fly.io secrets only | `fly secrets list` output checked; git grep confirms no committed value | Manual |

## Assumptions

- Neon free tier is available and sufficient for beta (≤3 GiB, cold starts acceptable per Decision 9).
- The `scripts/migrate-schema-pg.js` and `scripts/smoke-test-pg.js` scripts are already merged (PR #419).
- `DATABASE_URL` will be of the form `postgresql://user:password@host/dbname?sslmode=require`.
