# DoR Contract — s4.1: Production deployment and end-to-end smoke test

**Feature:** wuce-multi-tenancy
**Story:** s4.1
**Date:** 2026-06-29

---

## What will be built

The operator will: (1) set all 7 required Fly.io secrets; (2) run `fly deploy` to deploy the full multi-tenancy stack; (3) run `bash scripts/smoke-test-production.sh` to verify health endpoint, clean startup logs, a real journey persisting to Neon with correct tenant_id, session surviving machine restart via Redis, and billing gate firing at cap. No `src/` code changes — all code is already merged.

## What will NOT be built

- Load testing or concurrent-user testing
- Automated CI/CD redeploy pipeline (manual `fly deploy` for beta)
- Rollback procedure beyond machine restart
- Schema validation or migration tooling (migration was run in s3.1)

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 — All 7 secrets set in Fly.io | `fly secrets list` shows all 7 names; values not committed to git | Manual |
| AC2 — App starts cleanly | `fly logs` grep for PG/Redis startup errors returns empty | Manual |
| AC3 — Health check passes | `curl -s -o /dev/null -w "%{http_code}" https://<app>.fly.dev/health` → 200 | Manual |
| AC4 — Journey persists to Neon | Create journey via UI; Neon SQL confirms row with correct journey_id and tenant_id | Manual (E2E) |
| AC5 — Session survives restart | `GET /dashboard` 200 → `fly machine restart` → `GET /dashboard` still 200 | Manual (E2E) |
| AC6 — Billing gate fires at cap | Set MAX_JOURNEYS_PER_TENANT=1; attempt second journey → HTTP 402 | Manual (E2E) |

## Assumptions

- s3.1 (Neon provisioned) and s3.2 (Upstash provisioned) are complete before this story runs.
- s4.2 (TENANT_ORG_ALLOWLIST configured) can run in parallel or immediately after s4.1.
- The Fly.io app already exists (previously deployed); this story sets secrets and redeploys.
- GitHub OAuth credentials (GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET) are already configured on the Fly.io app.
