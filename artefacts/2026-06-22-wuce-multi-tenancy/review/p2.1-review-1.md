### 5-M2 — MEDIUM | Category E: Relationship to existing `setRepoRoot()` / `_repoRoot` injectable undefined

**Location:** `artefacts/2026-06-22-wuce-multi-tenancy/stories/p2.1.md` — the story introduces `getRepoRoot(req)` (a tenant-aware, request-scoped resolver) but does not specify its relationship to the existing static injectable already in `src/web-ui/routes/journey.js`:
- `var _repoRoot = null;` (line 14)
- `function setRepoRoot(root) { _repoRoot = root; }` (line 59)

Without specifying whether `getRepoRoot(req)` replaces, wraps, or coexists with the existing injectable, there are two competing interfaces for the same concept. An implementer could produce divergent outcomes:
- Replace `_repoRoot` entirely (correct for multi-tenant) — but this breaks any caller that previously called `setRepoRoot()` (e.g. server.js wiring, test helpers)
- Make `getRepoRoot(req)` fall back to `_repoRoot` when `session.tenantId` is absent (consistent with AC2's fallback logic) — but this must be stated explicitly
- Leave both in place (ambiguous — maintenance burden)

The AC2 fallback spec ("falls back to the existing static resolution logic") does not clarify whether "static resolution" means `_repoRoot` or the env-var fallback chain. This is an architectural gap that the DoR must resolve before implementation begins.

**Recommended action:** Add a sentence to the architecture constraints section clarifying one of: (a) `getRepoRoot(req)` replaces `_repoRoot` and `setRepoRoot()` — the static injectable is removed; callers must be updated, or (b) `getRepoRoot(req)` falls back to `_repoRoot` when `session.tenantId` is absent — the injectable remains for single-tenant wiring. The recommended path is (a) — `setRepoRoot()` can remain for test-helper use only (wiring to a temp dir), with `getRepoRoot(req)` as the sole production code path. Non-blocking for proceeding to /test-plan if documented in the DoR contract.

### 5-L2 — LOW | Category C: AC4 slugification omits the 48-character length cap

**Location:** `artefacts/2026-06-22-wuce-multi-tenancy/stories/p2.1.md`, AC4.

AC4 specifies: "slugified (lowercased, non-alphanumeric characters replaced with `-`, stripped of leading/trailing `-`)". The 48-character length cap is present in the architecture constraints section but not in AC4 itself. An AC should be independently testable without cross-referencing the architecture section. **Recommended action:** Add "and truncated to ≤48 characters" to AC4's Then clause. Non-blocking — the spec is not wrong, just split across two sections.
