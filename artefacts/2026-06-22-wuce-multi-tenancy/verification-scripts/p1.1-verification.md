### Scenario 1: `read:org` scope added to OAuth URL when TENANT_ORG_ALLOWLIST is set

**Covers:** AC1

**Steps:**
1. Set `TENANT_ORG_ALLOWLIST=org-a` in the test environment.
2. Call the function that constructs the GitHub OAuth authorize URL.
3. Inspect the returned URL string.

**Expected outcome:**
> The scope parameter in the URL contains `read:org` alongside the existing `repo,read:user` scopes.
> Example: `scope=repo%2Cread%3Auser%2Cread%3Aorg` or `scope=repo,read:user,read:org`.
> When `TENANT_ORG_ALLOWLIST` is absent, `read:org` does NOT appear in the scope.

### Scenario 2: Single-page org list — first allowlist match wins by allowlist order

**Covers:** AC2, AC5

**Steps:**
1. Configure `TENANT_ORG_ALLOWLIST=org-a,org-b`.
2. Mock the org-fetch function to return `[{ login: 'org-b' }, { login: 'org-a' }]` (org-b first in response).
3. Run the OAuth callback tenant resolution function.
4. Check `session.tenantId`.

**Expected outcome:**
> `session.tenantId === 'org-a'` — first in allowlist order, even though org-b appeared first in the orgs response.

**Result:** [ ] Pass  [ ] Fail

### Scenario 3: Multi-page pagination — match on page 2 is found

**Covers:** AC3

**Steps:**
1. Configure allowlist `org-a`.
2. Mock the org-fetch function to simulate two pages: page 1 returns `[{ login: 'org-b' }]` with a next-page signal; page 2 returns `[{ login: 'org-a' }]`.
3. Run the org resolution.
4. Check `session.tenantId`.

**Expected outcome:**
> All pages are fetched. `session.tenantId === 'org-a'` (the match from page 2).
> If only page 1 were fetched, no match would be found. The test confirms full pagination.

**Result:** [ ] Pass  [ ] Fail

### Scenario 4: Zero match — session not created, 4xx returned, allowlist not disclosed

**Covers:** AC4

**Steps:**
1. Configure allowlist `org-a,org-b`.
2. Mock org-fetch to return `[{ login: 'org-x' }]` — no match.
3. Run the OAuth callback through to the response.
4. Inspect the response status and body.

**Expected outcome:**
> HTTP 4xx response (400 or 403).
> Session is not created (`req.session.tenantId` is not set, session not established).
> Response body contains a message indicating the user is not a member of an allowed organisation.
> Response body does NOT contain `org-a` or `org-b` — allowlist contents are not disclosed.

**Result:** [ ] Pass  [ ] Fail
**Notes:**

### Scenario 5: TENANT_ORG_ALLOWLIST absent — backward-compatible single-user mode

> Org-fetch function is NOT called.
> `session.tenantId` is not set on the session.
