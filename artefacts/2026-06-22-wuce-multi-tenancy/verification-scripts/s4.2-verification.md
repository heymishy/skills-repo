## Setup

**Before you start:**
1. s4.1 (production deployment smoke test) must be complete — the app is live.
2. Have two GitHub accounts ready: one that IS a member of an allowlisted org, one that IS NOT.
3. Set `APP=<your-fly-app-name>` in your shell.
4. Know your intended `WUCE_TENANT_ROOT_BASE` path (e.g. `/app/tenants`).

**Reset between scenarios:** Log out of the app between Scenarios 1 and 2 (clear cookies or use a private browser window for each).

### Scenario 1: Allowlisted user logs in with correct tenantId (AC1)

**Covers:** AC1

**Steps:**
1. Set the allowlist secret: `fly secrets set TENANT_ORG_ALLOWLIST=<org1>,<org2> --app $APP`
2. Open `https://$APP.fly.dev` in a browser.
3. Log in with a GitHub account that IS a member of one of the allowlisted orgs.
4. After login, check the Fly.io logs: `fly logs --app $APP | grep "\[auth\] login"`

**Expected outcome:**
> You are redirected to `/dashboard` after login — no error page.
> The log line shows: `[auth] login { userId: <number>, tenantId: "<allowlisted-org-login>", timestamp: <timestamp> }`
> The `tenantId` value matches the org login (e.g. `my-org`), not the GitHub username.

### Scenario 2: Non-allowlisted user is rejected at login (AC2)

**Covers:** AC2

**Steps:**
1. Open `https://$APP.fly.dev` in a private browser window (or log out first).
2. Log in with a GitHub account that is NOT a member of any allowlisted org.
3. Complete the GitHub OAuth flow.

**Expected outcome:**
> After completing OAuth, you receive an error page (not redirected to dashboard).
> The page shows exactly the message: "You are not a member of an authorised organisation."
> The message does NOT contain the names of any orgs from your TENANT_ORG_ALLOWLIST.
> No session cookie is set (you cannot access `/dashboard`).

### Scenario 6: Solo-mode (no allowlist) works for operator account (AC6)

**Covers:** AC6

**Steps:**
1. Start the app locally WITHOUT setting `TENANT_ORG_ALLOWLIST`:
   ```bash
   # bash — load .env except TENANT_ORG_ALLOWLIST, then start server
   unset TENANT_ORG_ALLOWLIST
   export $(grep -v '^#' .env | grep -v TENANT_ORG_ALLOWLIST | xargs)
   node src/web-ui/server.js
   ```
   Or in PowerShell:
   ```powershell
   $env:TENANT_ORG_ALLOWLIST = $null
   Get-Content .env | Where-Object { $_ -notmatch '^#' -and $_ -ne '' -and $_ -notmatch 'TENANT_ORG_ALLOWLIST' } | ForEach-Object { $k,$v = $_ -split '=',2; Set-Item "env:$k" $v }
   node src/web-ui/server.js
   ```
2. Open `http://localhost:3000` and log in with your GitHub account (any account — no org check).
3. After login, note what appears as your tenant scope (your GitHub username should appear in journey paths).

**Expected outcome:**
> You are redirected to `/dashboard` after login. No 403 is shown.
> Journey paths use your GitHub username (not an org login) as the tenant identifier.
> No error about allowlist configuration appears.
