# Story s3.2 — Upstash Redis provisioning and session persistence activation

**Feature:** wuce-multi-tenancy
**Epic:** sprint-3-infrastructure-provisioning
**Status:** Definition
**Complexity:** 1
**Scope stability:** Stable

## User story

As the operator,
I want the Upstash Redis instance provisioned and session persistence credentials configured,
So that OAuth sessions (p3.2) survive server restarts and beta users are not silently logged out on every redeploy.

## Metric linkage

- **Beta readiness decision #2** (confirmed 2026-06-29): session persistence required before beta — this story activates it.

## Acceptance criteria

**AC1** — Upstash Redis database exists and REST credentials are available
Given the operator has created a Redis database at console.upstash.com (free tier, single-region),
When the REST API credentials are retrieved from the Upstash dashboard,
Then `UPSTASH_REDIS_REST_URL` (of the form `https://*.upstash.io`) and `UPSTASH_REDIS_REST_TOKEN` are available.

**AC2** — Round-trip smoke test passes
Given `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are set,
When `node scripts/smoke-test-redis.js` is run,
Then the script: (1) writes a test key `session:smoke-test` with TTL 60, (2) reads it back, (3) deletes it, and (4) exits 0 with "Upstash smoke test PASSED".

**AC3** — `accessToken` is confirmed absent from Redis writes
Given the smoke test runs with a payload containing `{ accessToken: 'should-not-persist', login: 'test-user' }`,
When the smoke test reads the written value back from Redis,
Then `accessToken` is absent from the stored value and `login` is present.

**AC4** — TTL of 86400 seconds is confirmed on session keys
Given the smoke test writes a session key via the `session-redis` adapter,
When the TTL is checked immediately after write (`client.ttl('session:smoke-test')`),
Then the TTL value is between 86390 and 86400 (within 10 seconds of the target).

**AC5** — Credentials stored as Fly.io secrets (not committed to repo)
Given the Fly.io app is the deployment target,
When `fly secrets list` is run,
Then `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` appear in the output and neither value appears in any committed file in the repo.

## Out of scope

- Multi-region Upstash read replicas (single-region default per Decision 9)
- Session encryption beyond Upstash's default TLS
- Per-tenant session TTL

## Dependencies

- `@upstash/redis` npm package (installed in PR #419)
- `src/web-ui/adapters/session-redis.js` (merged in PR #419)

## Implementation touchpoints

- `scripts/smoke-test-redis.js` (new): standalone script — writes a session key, reads back, checks accessToken stripped, deletes key, exits 0 on pass
- Upstash dashboard: database creation, REST URL + token retrieval
- Fly.io CLI: `fly secrets set UPSTASH_REDIS_REST_URL=<url> UPSTASH_REDIS_REST_TOKEN=<token>`

## Test

Run: `UPSTASH_REDIS_REST_URL=<url> UPSTASH_REDIS_REST_TOKEN=<token> node scripts/smoke-test-redis.js`
