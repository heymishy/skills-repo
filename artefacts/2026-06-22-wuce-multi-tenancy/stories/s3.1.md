# Story s3.1 — Neon Postgres provisioning and schema activation

**Feature:** wuce-multi-tenancy
**Epic:** sprint-3-infrastructure-provisioning
**Status:** Definition
**Complexity:** 1
**Scope stability:** Stable

## User story

As the operator,
I want the Neon free-tier Postgres database provisioned and the journeys schema created,
So that journey persistence (p3.1) is active in production and journeys survive server restarts.

## Metric linkage

- **Beta readiness decision #2** (confirmed 2026-06-29): journey persistence required before beta — this story activates it.

## Acceptance criteria

**AC1** — Neon project exists and DATABASE_URL is accessible
Given the operator has created a Neon project at console.neon.tech (free tier),
When the connection string is retrieved from the Neon dashboard,
Then a valid `DATABASE_URL` of the form `postgresql://user:password@host/dbname?sslmode=require` is available.

**AC2** — Schema migration runs successfully
Given `DATABASE_URL` is set in the local environment,
When `node scripts/migrate-schema-pg.js` is run,
Then exit code is 0 and the output confirms "Schema created (or already exists): journeys table + tenant_id index".

**AC3** — `journeys` table and index exist in Neon
Given the migration ran successfully (AC2),
When the Neon SQL editor (or psql) runs `\dt journeys` and `\di journeys_tenant_id_idx`,
Then both the table and index are confirmed present.

**AC4** — Round-trip smoke test passes
Given `DATABASE_URL` is set,
When `node scripts/smoke-test-pg.js` is run,
Then the script: (1) inserts a test journey row, (2) reads it back by `journey_id`, (3) deletes it, and (4) exits 0 with "Neon smoke test PASSED".

**AC5** — `DATABASE_URL` is stored as a Fly.io secret (not committed to repo)
Given the Fly.io app is the deployment target,
When `fly secrets list` is run,
Then `DATABASE_URL` appears in the output and the value does NOT appear in any committed file in the repo.

## Out of scope

- HA / read replicas (single Neon free-tier instance per Decision 9)
- Migrating existing disk journeys to Postgres (separate operator step; disk files remain as fallback for 30 days)
- Automated schema migration on deploy (schema is operator-run once; the app does not auto-migrate)

## Dependencies

- `scripts/smoke-test-pg.js` (new): standalone script — inserts one row, reads it back, deletes it, exits 0 on pass
- Fly.io dashboard / CLI: `fly secrets set DATABASE_URL=<value>`
- Neon dashboard: project creation, connection string retrieval

## Test

Verification: `node scripts/smoke-test-pg.js` (requires real `DATABASE_URL` env var — skips gracefully with "SKIP: DATABASE_URL not set" when absent).
Run: `DATABASE_URL=<neon-url> node scripts/smoke-test-pg.js`
