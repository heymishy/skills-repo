## User story

As the operator,
I want the full multi-tenancy stack deployed to production with all required env vars set and verified by a post-deploy smoke test,
So that the beta can open and I have confidence that persistence, auth, and the billing gate are all live.

## Metric linkage

- **M1** (Authorization coverage rate, benefit-metric.md §M1): AC3 (health check 200) and AC6 (billing gate 402) provide the first production signal that the authorization guard and billing gate are live. M1 can only be verified in production once this story completes.
- **M2** (Cross-tenant journey data leakage prevention, benefit-metric.md §M2): AC4 (journey persists to Neon with correct tenant_id) and AC5 (session survives restart via Redis) confirm the persistence isolation stack is live — the precondition for M2 to be measurable.

## Acceptance criteria

**AC1** — All required env vars are set as Fly.io secrets before redeploy
Given the operator runs `fly secrets list`,
Then the following are all present: `DATABASE_URL`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`, `TENANT_ORG_ALLOWLIST`, `MAX_JOURNEYS_PER_TENANT`, `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET`.
None of these values appear in any committed file in the repo.

**AC2** — App starts cleanly after redeploy
Given `fly deploy` completes,
When `fly logs` is inspected for the startup sequence,
Then no `[journey-store] PG write error` or `[server] loadAllFromPg failed` or `[session] Redis load error` messages appear within the first 30 seconds of startup.

**AC3** — Health check passes
Given the deployed app is running,
When `GET https://<app>.fly.dev/health` is called,
Then HTTP 200 is returned.

**AC4** — Journey created via the UI persists to Neon
Given an authenticated user creates a new journey via `POST /journey`,
When the Neon SQL editor runs `SELECT journey_id, tenant_id FROM journeys ORDER BY created_at DESC LIMIT 1`,
Then a row appears with the expected `journey_id` and `tenant_id` values.

**AC5** — Session survives simulated restart
Given an authenticated user has an active session (confirmed by `GET /dashboard` returning 200),
When `fly machine restart` is run to force a process restart,
Then `GET /dashboard` still returns 200 within 30 seconds (session was reloaded from Redis on startup).

**AC6** — Billing gate fires at cap
Given `MAX_JOURNEYS_PER_TENANT` is set to a test cap (e.g. 1) and the tenant already has 1 journey,
When the authenticated user attempts to create a second journey via `POST /journey`,
Then the response is HTTP 402 with the "Journey limit reached" message visible.
After confirming, the cap is reset to the intended beta value.

## Out of scope

- Load testing or concurrent-user testing at this stage
- Automated redeploy CI/CD pipeline (manual `fly deploy` for beta)
- Rollback procedure (restart suffices at this scale)

## Dependencies

- s3.1 (Neon provisioned) and s3.2 (Upstash provisioned) must be complete
- s4.2 (tenant onboarding config, specifically TENANT_ORG_ALLOWLIST) must be complete

## Implementation touchpoints

- `scripts/smoke-test-production.sh` (new): shell script documenting and scripting the post-deploy smoke test sequence (ACs 2–6)
- Fly.io CLI: `fly secrets set`, `fly deploy`, `fly logs`, `fly machine restart`
- Neon SQL editor: SELECT verification (AC4)
