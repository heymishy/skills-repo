# Story p5.1 — tenantId validation + adversarial path-traversal test suite

**Feature:** wuce-multi-tenancy
**Epic:** phase-5-security-hardening
**Status:** Definition
**Complexity:** 3
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want every path-traversal guard that uses a tenant-derived `repoRoot` to be validated against an adversarial `tenantId` injection test suite,
So that T3-M2 (path-traversal guard validity under variable repoRoot) is closed: a maliciously crafted `tenantId` cannot escape the tenant directory, regardless of encoding or traversal technique.

## Metric linkage

- **T3-M2** (Path-traversal guard validity under variable tenantId-derived repoRoot): this story is the measurement and closure mechanism — the adversarial suite produces the binary PASS/FAIL signal

## Acceptance criteria

**AC1** — Slugification rejects path separators (forward and backward slash)
Given `slugifyTenantId('../../etc/passwd')` is called,
When the function runs,
Then the output contains neither `/` nor `\` and does not produce a path that exits `WUCE_TENANT_ROOT_BASE` when appended.

**AC2** — Slugification rejects URL-encoded traversal sequences
Given `slugifyTenantId('%2e%2e%2f%2e%2e%2fetc%2fpasswd')` is called,
When the function runs,
Then the output is a safe slug — the URL-encoded sequence is not decoded and then used in path construction (decode-then-slugify, not slugify-then-decode).

**AC3** — Slugification rejects null bytes
Given `slugifyTenantId('org-a\x00../../etc')` is called,
When the function runs,
Then the null byte and everything after it is stripped, or the entire input is rejected, producing a safe slug.

**AC4** — Slugification rejects overlong inputs
Given `slugifyTenantId('a'.repeat(300))` is called (300-character string),
When the function runs,
Then the output is truncated to ≤48 characters (consistent with the `_slugify()` length cap already in `journey.js`).

**AC5** — Slugification rejects Unicode homoglyphs and non-ASCII characters
Given `slugifyTenantId('оrg-а')` is called (Cyrillic 'о' and 'а' that visually resemble ASCII),
When the function runs,
Then all non-ASCII characters are stripped or replaced with `-`, producing an ASCII-only slug.

**AC6** — Path-traversal guard fires on all adversarial slugs
Given `slugifyTenantId()` produces any output for any of the adversarial inputs in AC1–AC5,
When that slug is used to construct a repoRoot via `path.resolve(WUCE_TENANT_ROOT_BASE + '/' + slug)`,
Then `path.resolve(derivedPath).startsWith(path.resolve(WUCE_TENANT_ROOT_BASE) + path.sep)` returns `true` for every adversarial input (the derived path is always inside the base, never outside).

**AC7** — HTTP 400 on traversal attempt (guard fires in route handler)
Given a request arrives with a `session.tenantId` that (before slugification) contains a traversal sequence,
When the route handler calls `getRepoRoot(req)` and performs the path-traversal guard assertion,
Then the handler returns HTTP 400 and no file is written to disk.

**AC8** — Audit coverage: every path-derived-from-tenantId write point is identified and tested
Given the Phase 2 implementation (p2.1) introduced N distinct disk write operations whose path is derived from `tenantId`,
When the adversarial test suite runs,
Then each of the N write points has at least one adversarial test case that asserts the guard fires correctly — no write point is left unaudited.

**AC9** — Existing traversal guards for static repoRoot are not regressed
Given the pre-Phase-2 path-traversal guards (which used a static `repoRoot`),
When the adversarial test suite runs against them with a static repoRoot,
Then all existing guard assertions still pass — no regression.

**AC10** — Test suite passes
Given `node tests/check-p5.1-adversarial-traversal.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- New application features — this story is test and validation only; no new user-visible behaviour
- Phase 3 Postgres/Redis write path audit — those write paths don't exist until Phase 3 ships; if Phase 5 runs before Phase 3, only the filesystem and disk-session write paths from Phase 2 are in scope
- Penetration testing by an external party — this is an automated code-level adversarial suite, not a black-box pentest

## Dependencies

- **p2.1** must be complete — `getRepoRoot(req)` must be parameterised by `tenantId` before adversarial testing of that path is meaningful
- **p2.2** may run in parallel with p5.1 (they test different dimensions of the same code: p2.2 tests correctness, p5.1 tests adversarial inputs) but p2.1 is the shared prerequisite for both

## Architecture constraints

- Path-traversal guard standard (CLAUDE.md ougl): the guard is `path.resolve(derived).startsWith(path.resolve(base) + path.sep)` — both sides must use `path.resolve()` to handle platform-specific separators; `path.sep` (not a hardcoded `/`) is the correct separator sentinel
- Raw `tenantId` values (pre-slugification) must never appear in production log output — the adversarial test suite should assert that no unsafe value is logged (AC7)
- `slugifyTenantId()` should be a standalone exported function testable independently of the full request-handling stack — not inlined in `getRepoRoot()`
- Test file: `tests/check-p5.1-adversarial-traversal.js`, plain Node.js `assert`

## NFRs

- All adversarial test inputs in AC1–AC5 are ASCII or UTF-8 strings resolvable at test time — no filesystem writes required for the slugification tests (pure function tests)
- The 8 disk write points identified in AC8 are enumerated explicitly in the test file, not discovered dynamically at test runtime — the audit coverage is static and reviewable
