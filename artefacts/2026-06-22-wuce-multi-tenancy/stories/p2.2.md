# Story p2.2 — Tenant-isolation test suite

**Feature:** wuce-multi-tenancy
**Epic:** phase-2-storage-isolation
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want an automated tenant-isolation test suite that runs two distinct tenant fixtures and asserts zero cross-read at both the HTTP and filesystem layers,
So that M2 (cross-tenant journey data leakage prevention) has a repeatable, CI-runnable evidence artefact that confirms the Phase 2 isolation guarantee holds.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): this story is the M2 measurement mechanism — it is the test suite that produces the binary PASS/FAIL signal for the "zero cross-tenant reads" target

## Acceptance criteria

**AC1** — Cross-tenant HTTP read returns 404 (all 9 KNOWN BUG routes)
Given a test fixture creates two tenant sessions — `tenant-a` (login `alice`, tenantId `org-a`) and `tenant-b` (login `bob`, tenantId `org-b`) — and creates one journey per tenant,
When a `tenant-b` session calls any of the 9 previously-KNOWN-BUG route handlers for `tenant-a`'s journey,
Then all return HTTP 404 (the authorization guard + `isSameTenant()` both fire; the test asserts each of the 9 routes individually).

**AC2** — Mutating routes return 403, not 404, for cross-tenant calls
Given the same two-tenant fixture,
When a `tenant-b` session calls the two mutating routes (`recommit`, `stage-commit`) for `tenant-a`'s journey,
Then both return HTTP 403.

**AC3** — No filesystem cross-bleed (tenant-a files)
Given p2.1 is complete and both tenants have written artefact files under `WUCE_TENANT_ROOT_BASE/org-a/` and `WUCE_TENANT_ROOT_BASE/org-b/` respectively,
When the test performs a recursive directory scan of `WUCE_TENANT_ROOT_BASE/org-a/`,
Then zero files belonging to `org-b` are found in that directory tree (and vice versa).

**AC4** — No session store cross-bleed
Given both tenants have active disk sessions under `SESSION_STORE_PATH/org-a/` and `SESSION_STORE_PATH/org-b/`,
When the test queries the session store using `org-b`'s namespaced lookup path for a session ID that belongs to `org-a`,
Then no session is returned.

**AC5** — Same-tenant access is not broken
Given the two-tenant fixture,
When `tenant-a`'s session calls the same 9 routes for `tenant-a`'s own journey,
Then all return the expected successful responses (no regression for same-tenant callers).

**AC6** — Test suite passes
Given `node tests/check-p2.2-tenant-isolation.js` is run after implementation,
Then all assertions pass and exit code is 0.

**AC7** — Pre-existing test suite does not regress
Given the full test suite `node tests/check-*.js` is run after p2.2 merges,
Then all tests that were passing before p2.2 still pass and exit code is 0.

## Out of scope

- Concurrent-request stress testing (two tenants interleaved under load) — Phase 3 (p3.3)
- Adversarial `tenantId` injection (e.g. `../../etc/passwd` as tenantId) — Phase 5 (p5.1)
- Postgres/Redis adapter isolation — Phase 3 (the in-memory journey store is not the isolation boundary being tested here)
- Performance benchmarking — this is a correctness test, not a load test

## Dependencies

- **p2.1** must be complete — filesystem namespacing must be in place before the filesystem scan assertions (AC3, AC4) are meaningful

## Architecture constraints

- Test file: `tests/check-p2.2-tenant-isolation.js`, plain Node.js `assert`, no test runner
- The `makeReq` fixture must accept `tenantId` on the session object: `makeReq({ session: { accessToken, userId, login, tenantId } })`
- Test must assert on actual filesystem state (AC3) using `fs.readdirSync` or equivalent — HTTP response assertions alone are not sufficient to prove filesystem isolation
- Test must create real files in a temporary directory (e.g. `os.tmpdir()`) and clean up after itself — not use pre-seeded fixture files on disk
- The 9 KNOWN BUG routes must be tested individually by name (not by loop without naming) — each route assertion must be independently readable in the test output

## NFRs

- Test suite must complete in under 10 seconds (all synchronous or fast async file ops)
- Test suite must clean up any temporary directories it creates (no persistent test artefacts left on the developer's machine after a run)
- Data classification of test data: test-only, ephemeral — not Internal/Confidential; test fixtures use synthetic values only
