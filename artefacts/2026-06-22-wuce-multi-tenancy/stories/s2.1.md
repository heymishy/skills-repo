# Story s2.1 — Pre-flight billing gate: per-tenant journey creation cap

**Feature:** wuce-multi-tenancy
**Epic:** sprint-2-preflight-gate
**Status:** Complete
**Complexity:** 1
**Scope stability:** Stable

## User story

As an operator running a beta deployment,
I want journey creation to be blocked once a tenant has reached a configurable cap,
So that individual beta users cannot inadvertently drive up AI costs beyond a per-user budget without me noticing.

## Metric linkage

- **Beta readiness decision #1** (confirmed 2026-06-29): pre-flight billing gate required before beta. Cap is enforced at journey-creation time; AI cost per journey is bounded by the number of stages × average tokens.

## Acceptance criteria

**AC1** — Journey creation allowed below cap
Given `MAX_JOURNEYS_PER_TENANT=5` is set and a tenant has 4 existing journeys,
When that tenant calls `POST /journey`,
Then the journey is created normally (no gate fires).

**AC2** — Journey creation blocked at cap
Given `MAX_JOURNEYS_PER_TENANT=5` is set and a tenant has 5 existing journeys,
When that tenant calls `POST /journey`,
Then the handler returns HTTP 402 with a user-facing HTML error message.

**AC3** — No env var = no blocking
Given `MAX_JOURNEYS_PER_TENANT` is not set,
When any tenant calls `POST /journey` regardless of journey count,
Then the journey is created normally (unlimited).

**AC4** — Per-tenant override via `tenant-caps.json`
Given a file `{repoRoot}/tenant-caps.json` exists with `{ "alice": 2 }` and `MAX_JOURNEYS_PER_TENANT=10`,
When tenant `alice` has 2 existing journeys and calls `POST /journey`,
Then the per-tenant cap of 2 is applied and the request is blocked (override beats global).

**AC5** — Cap is per-tenantId (tenants do not share a counter)
Given `MAX_JOURNEYS_PER_TENANT=1` is set, tenant `alice` has 1 journey and tenant `bob` has 0,
When tenant `bob` calls `POST /journey`,
Then bob's journey is created (bob's count is 0, below cap); alice's cap does not affect bob.

**AC6** — Unauthenticated `POST /journey` still redirects (existing behaviour preserved)
Given no `accessToken` in session,
When `POST /journey` is called,
Then the handler returns 302 (unchanged from pre-s2.1 behaviour).

## Implementation touchpoints

- `src/web-ui/modules/tenant-plan.js` (new): `checkJourneyCap(tenantId, currentCount, repoRoot)` — reads `MAX_JOURNEYS_PER_TENANT` env var; falls back to `{repoRoot}/tenant-caps.json` for per-tenant overrides; injectable `setCapReader(fn)` for tests
- `src/web-ui/routes/journey.js`: `handlePostJourney` — cap check injected after auth guard, before journey creation; returns 402 HTML if blocked
- `tests/check-s2.1-preflight-gate.js`: verification script for all 6 ACs

## Schema note

`tenant-caps.json` is a flat `{ [tenantId]: number }` map. Keys are `session.tenantId` values (= `user.login` after s0.2). Created manually by the operator when a per-user override is needed; the platform does not create or modify this file at runtime.
