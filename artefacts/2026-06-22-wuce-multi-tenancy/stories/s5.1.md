# Story s5.1 — Beta monitoring signals

**Feature:** wuce-multi-tenancy
**Epic:** sprint-5-beta-assessment
**Discovery:** artefacts/2026-06-22-wuce-multi-tenancy/discovery.md
**Benefit-metric:** artefacts/2026-06-22-wuce-multi-tenancy/benefit-metric.md
**Status:** Definition
**Complexity:** 1
**Scope stability:** Stable

## User story

As the operator,
I want a lightweight monitoring script and a documented log-query runbook so that I can detect and respond to the three most likely beta failure modes — billing gate fires, persistence errors, and session expiry — without building full APM infrastructure.

## Metric linkage

- **M1** (Authorization coverage rate): monitoring confirms the guard is live by surfacing any 403/404 anomalies.
- **M2** (Cross-tenant leakage prevention): monitoring detects unexpected cross-tenant access attempts via log pattern matching.

## Acceptance criteria

**AC1** — Billing gate fires are visible in Fly.io logs
Given `MAX_JOURNEYS_PER_TENANT` is set and a tenant reaches the cap,
When `fly logs --app <app> | grep "Journey limit reached"` is run,
Then a log line appears for each blocked `POST /journey` request with the tenant's login and the cap value.

**AC2** — Persistence write errors are visible in Fly.io logs
Given the PG adapter's fire-and-forget error handler logs via `console.error('[journey-store] PG write error:', ...)`,
When a Postgres connection failure is simulated (e.g. by temporarily revoking the DATABASE_URL),
Then `fly logs | grep "PG write error"` captures the error within 10 seconds.

**AC3** — Redis write errors are visible in Fly.io logs
Given the Redis adapter's fire-and-forget error handler logs via `console.error('[session] Redis write error:', ...)`,
When an Upstash connection failure is simulated,
Then `fly logs | grep "Redis write error"` captures the error.

**AC4** — Monitoring runbook exists as an artefact
Given this story is complete,
When the operator opens `artefacts/2026-06-22-wuce-multi-tenancy/reference/beta-monitoring-runbook.md`,
Then the file documents: the three failure modes (billing gate, PG error, Redis error), the exact `fly logs | grep` query for each, the expected log format, and the remediation step for each.

**AC5** — `scripts/check-beta-health.sh` runs and reports status
Given the operator has Fly.io CLI configured and the app is running,
When `bash scripts/check-beta-health.sh` is run,
Then the script: (1) calls `GET /health` and asserts 200, (2) checks `fly logs` for error patterns in the last 5 minutes, (3) prints a PASS/WARN/FAIL summary per check.

## Out of scope

- Full APM / Grafana / Datadog integration (not required at beta scale of 5–10 users)
- Automated alerting / PagerDuty
- Per-tenant usage dashboards

## Dependencies

- s4.1 (production deployment) — must be complete; monitoring only makes sense against a live deployment

## Implementation touchpoints

- `scripts/check-beta-health.sh` (new): health check script implementing AC5
- `artefacts/2026-06-22-wuce-multi-tenancy/reference/beta-monitoring-runbook.md` (new): operator runbook for AC4
- No src/ changes required — monitoring relies on existing `console.error` log lines already in place

## Architecture Constraints

None identified — checked against `.github/architecture-guardrails.md`. This story creates two new artefacts (`scripts/check-beta-health.sh` and the monitoring runbook) but makes no changes to governed code paths (`src/`, SKILL.md, dashboard logic, or pipeline-state schema). The health check script uses existing `fly logs` and `curl` — no new service dependencies.

## NFRs

- **NFR-sec-no-credentials-in-output** (nfr-profile.md §Security): `scripts/check-beta-health.sh` must not echo or log credential variable values (`$UPSTASH_REDIS_REST_TOKEN`, `$DATABASE_URL`, etc.). Test plan NFR-SEC-1 verifies this by code inspection.
- **NFR-PERF-health-check** (nfr-profile.md §Performance): `GET /health` check must complete within 10 seconds (`curl --max-time 10`). Test plan NFR-PERF-1 verifies this by code inspection.
- **NFR-PORT-bash3** (nfr-profile.md §Backward compatibility): `check-beta-health.sh` must run on bash 3+ (macOS default). No bash 4+ features (`declare -A`, `mapfile`). Test plan NFR-PORT-1 verifies by code inspection.

## Test

Verification: `bash scripts/check-beta-health.sh` against the live Fly.io deployment. AC1–AC3 are verified by simulating each error condition once during initial beta setup and confirming the log lines appear.
