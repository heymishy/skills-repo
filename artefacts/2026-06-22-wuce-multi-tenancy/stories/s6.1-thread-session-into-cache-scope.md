# Story s6.1 — Thread session into prompt-cache scoping (activate Decision 8)

**Feature:** wuce-multi-tenancy
**Epic:** phase-4-performance-isolation
**Discovery:** artefacts/2026-06-22-wuce-multi-tenancy/discovery.md
**Benefit-metric:** artefacts/2026-06-22-wuce-multi-tenancy/benefit-metric.md
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## Background — fix-forward, not new scope

p4.1 built `buildCacheKey()` (`src/web-ui/adapters/cache-key.js`) and wired it into `_anthropicSystem()` (`src/modules/skill-turn-executor.js`), which prepends `<!-- cache-scope: ${tenantId}-${sessionId} -->` as the first line of the Anthropic system prompt when a `session` object with a `tenantId` is passed in. Decision 8 (`artefacts/2026-06-22-wuce-multi-tenancy/decisions.md`, 2026-06-24) RISK-ACCEPTed shipping p4.1 with this scoping logic built but inert: "session threading into the call stack is deferred; the scoping logic is implemented and ready to activate when session is passed through." Today, no caller of `_anthropicSystem()` passes a `session` argument, so the `if (session && session.tenantId)` branch never fires in production — every tenant's system prompt is byte-identical when the underlying skill content matches, which is exactly the cross-tenant Anthropic prompt-cache collision Decision 8 flagged as a real (if unlikely) risk. This story activates the existing guard by threading `tenantId` + `sessionId` through the real production call chain. No new scoping algorithm is introduced — `buildCacheKey()` and `_anthropicSystem()` are unchanged.

## User story

As a solo founder / operator running `wuce` for multiple tenants,
I want the prompt-cache scope guard that p4.1 already built to actually receive the real tenant and session identifiers on every live skill-turn request,
So that M2 (cross-tenant journey data leakage prevention) is upheld in production, not only in `buildCacheKey()`'s own unit tests — closing the gap RISK-ACCEPTed in Decision 8.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): an inert cache-scope guard provides zero protection — the risk Decision 8 accepted as "unlikely but possible" remains fully live in production until `session` reaches `_anthropicSystem()` on real requests.

## Traced call chain (read before implementing)

The production skill-session architecture is the **model-first chat flow** (`docs/web-ui-skill-session-as-built.md` — this document explicitly supersedes the older per-question wizard flow). Two entry points reach `_anthropicSystem()`:

1. **Non-streaming turn:** `handlePostTurnHtml(req, res)` (`src/web-ui/routes/skills.js` ~L3672) → `htmlSubmitTurn(skillName, sessionId, rawAnswer, token)` (~L1932) → `_skillTurnExecutor(...)` (wired in `server.js` L162 directly to `src/modules/skill-turn-executor.js`'s `skillTurnExecutor(systemPrompt, history, currentInput, token, meta)`, L604) → `_callAnthropic(systemPrompt, history, currentInput, maxTokens, timeoutMs)` (L66) → `_anthropicSystem(systemPrompt)` (L48, called with no `session` argument today).
2. **Streaming turn (SSE):** `handlePostTurnStreamHtml(req, res)` (~L3768) → `_skillTurnExecutorStream(...)` (~L3915, wired to `skillTurnExecutorStream`, L640) → `_callAnthropicStream(...)` (L157) → `_anthropicSystem(systemPrompt)` (same call, session omitted).

`req.session.tenantId` is already available at both entry points (both handlers already read it a few lines later, for PostHog analytics and credit deduction — see `_guardTenantId` / `_streamGuardTenantId` in the current code). `sessionId` (the html-session id, `req.params.id`) is already a parameter of both `htmlSubmitTurn` and `handlePostTurnStreamHtml`. Neither value currently reaches `_anthropicSystem()` — that is the entire gap.

**Out of scope — explicitly not touched:** `htmlRecordAnswer` / `handlePostAnswerHtml` (~L4307–4455), the older per-question wizard flow. `docs/web-ui-skill-session-as-built.md` states this architecture "has been superseded" and is "not a target for implementation." Threading session into a superseded code path would add risk with no production benefit.

## Acceptance criteria

**AC1** — Two tenants sending an identical system prompt produce two different cache-scope values at the Anthropic request boundary (non-streaming path)
Given `handlePostTurnHtml` is exercised twice with the same `session.systemPrompt` but `req.session.tenantId = 'org-a'` for the first call and `req.session.tenantId = 'org-b'` for the second (same `sessionId` reused across both calls to isolate the tenant variable),
When the request reaches the Anthropic API boundary (intercepted via a mocked `https.request`, per the existing `check-wuce26-per-answer-model-response.js` T9 pattern — no live Anthropic call),
Then the two captured `system[0].text` values differ, and each contains its own tenant's `<!-- cache-scope: org-a-<sessionId> -->` / `<!-- cache-scope: org-b-<sessionId> -->` line — proving the scope string is genuinely tenant-differentiated by the time it reaches the boundary, not merely that a parameter was accepted.

**AC2** — Cache-scope value format matches `buildCacheKey()` exactly (non-streaming path)
Given `req.session.tenantId = 'my-org'` and a known `sessionId`,
When the captured request body's `system[0].text` is inspected,
Then its first line is exactly `<!-- cache-scope: my-org-<sessionId> -->` (tenantId, dash, sessionId — matching `buildCacheKey()`'s existing, unchanged format).

**AC3** — Streaming path (SSE) threads the same tenant+session scoping
Given `handlePostTurnStreamHtml` is exercised twice with the same `session.systemPrompt` but two different `req.session.tenantId` values (same `sessionId` reused),
When the request reaches the Anthropic API boundary (mocked `https.request`, streaming variant),
Then the two captured `system[0].text` values differ, each carrying its own tenant's cache-scope line — the streaming path is not left behind while only the non-streaming path is fixed.

**AC4** — No tenantId (single-tenant / pre-Phase-1 session) — unchanged fallback behaviour
Given `req.session.tenantId` is absent (`undefined` or `null`),
When either the non-streaming or streaming path reaches the Anthropic boundary,
Then `system[0].text` is the raw `systemPrompt` with no `cache-scope` comment line prepended — identical to current (pre-fix) behaviour. This is a regression guard: activating the guard for multi-tenant sessions must not alter single-tenant behaviour.

**AC5** — `buildCacheKey()` and `_anthropicSystem()` are unchanged
Given this story only threads existing parameters through the call stack,
When `src/web-ui/adapters/cache-key.js` and the `_anthropicSystem()` function body in `src/modules/skill-turn-executor.js` are diffed against their pre-story versions,
Then neither has a body change — only new/extended parameters on the intermediate functions in the call chain (`htmlSubmitTurn`, `skillTurnExecutor`, `skillTurnExecutorStream`, `_callAnthropic`, `_callAnthropicStream`) and their call sites.

**AC6** — Existing test suites for the touched files still pass unmodified
Given `tests/check-wuce26-per-answer-model-response.js`, `tests/check-p4.1-tenant-cache-ratelimit.js`, `tests/check-mfc1-model-first-chat-session.js`, `tests/check-bri-s3.1-mock-llm-gateway.js`, and `tests/check-pla-s2-posthog-wiring.js` all call the functions this story modifies (`htmlSubmitTurn`, `skillTurnExecutor`, `skillTurnExecutorStream`) with their pre-existing (shorter) argument lists,
When each is run after this story's changes,
Then all pass with no test-file edits required — new parameters must be additive/optional, never required.

**AC7** — Test suite passes
Given `node tests/check-s6.1-cache-scope-session-threading.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Any change to `buildCacheKey()`'s algorithm or `_anthropicSystem()`'s comment format (Decision 8's design is accepted as-is; only activation is in scope)
- The legacy per-question wizard flow (`htmlRecordAnswer` / `handlePostAnswerHtml`) — superseded per `docs/web-ui-skill-session-as-built.md`
- The Copilot provider path (`_callCopilot` / `_callCopilotStream`) — Anthropic's prompt-caching mechanism is what Decision 8 addresses; Copilot has no equivalent cache_control mechanism in this codebase
- Rate-limiting (`rate-limiter.js`) — already tenant-scoped by p4.1; unaffected by this story
- Any change to the mock LLM gateway routing (`meta.stage` / `bri-s3.1`) — when the mock gateway is enabled, `_callAnthropic` is bypassed entirely and this story's change is a no-op, which is correct (no real Anthropic cache to protect in test mode)

## Architecture constraints

- `req.session.accessToken` is the canonical field name — unchanged; this story reads `req.session.tenantId`, a separate, already-established session field (used elsewhere in the same handlers for PostHog/credits)
- New parameters on existing functions must be additive (appended, optional) so all six existing test files listed in AC6 continue to pass without modification (D37-adjacent discipline: do not silently break existing call sites)
- Test file: `tests/check-s6.1-cache-scope-session-threading.js`, plain Node.js `assert`, matching the `https.request`-mock pattern already established in `tests/check-wuce26-per-answer-model-response.js` T9/T10 — no live Anthropic API call in any test

## NFRs

- No new I/O — threading two already-in-memory string values (`tenantId`, `sessionId`) through existing function calls; negligible latency impact
- No new npm dependencies (matches repo-wide `docs/web-ui-skill-session-as-built.md` constraint: Node built-ins only)
