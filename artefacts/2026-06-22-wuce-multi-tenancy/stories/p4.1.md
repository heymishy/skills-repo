# Story p4.1 — Prompt-cache key scoping + per-tenant rate-limit isolation

**Feature:** wuce-multi-tenancy
**Epic:** phase-4-performance-isolation
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want prompt-cache keys scoped to `${tenantId}-${sessionId}` and per-tenant API rate-limit isolation,
So that M2 (cross-tenant journey data leakage prevention) is upheld at the inference layer: one tenant's cached context cannot bleed into another tenant's session, and one high-volume tenant cannot exhaust the API limit for all other tenants.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): a prompt-cache key collision between two tenants is a data leakage vector — prior session content from tenant A could be returned as a cache hit to tenant B if keys are not tenant-scoped

## Acceptance criteria

**AC1** — Prompt-cache key is tenant-scoped
Given prompt caching is enabled (`WUCE_ENABLE_PROMPT_CACHE=1`) and two sessions exist — one for `tenantId = 'org-a'` with `sessionId = 'abc'` and one for `tenantId = 'org-b'` with `sessionId = 'abc'` (same sessionId, different tenants),
When prompt-cache keys are generated for both sessions,
Then the keys are distinct — `'org-a-abc'` and `'org-b-abc'` — so a cache hit for tenant A's session cannot be served as a hit for tenant B's identically-IDed session.

**AC2** — Prompt-cache key format: `${tenantId}-${sessionId}`
Given a session with `tenantId = 'my-org'` and `sessionId = 'xyz123'`,
When the prompt-cache key is constructed,
Then the key is `'my-org-xyz123'` (tenantId first, then sessionId, separated by `-`).

**AC3** — Prompt-cache key falls back to sessionId only when tenantId is absent
Given a session with no `tenantId` (single-user mode or pre-Phase-1 session),
When the prompt-cache key is constructed,
Then the key falls back to `sessionId` alone (backward compatible — no change for single-tenant deployments).

**AC4** — Per-tenant rate-limit: requests from tenant A do not count against tenant B's limit
Given `src/web-ui/middleware/rate-limiter.js` uses a rate-limit key to track request counts,
When two tenants make concurrent requests,
Then the rate-limit counter for `tenantId = 'org-a'` is tracked separately from the counter for `tenantId = 'org-b'` — one tenant hitting the rate limit does not affect the other.

**AC5** — Rate-limit key includes tenantId when available
Given a request with `session.tenantId = 'my-org'`,
When the rate-limiter computes the limit key,
Then the key includes `'my-org'` (e.g. `'my-org-<ip>'` or `'my-org'` alone — the exact format is implementation choice, but it must include tenantId).

**AC6** — Rate-limit falls back to IP-based key when tenantId is absent
Given a request with no `session.tenantId`,
When the rate-limiter computes the limit key,
Then the key falls back to IP address (backward compatible with pre-Phase-1 deployments).

**AC7** — Test suite passes
Given `node tests/check-p4.1-tenant-cache-ratelimit.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Changes to the prompt-cache backend (still uses the Anthropic/GHCP `cache_control` content-block mechanism — only the key scoping is changed)
- Per-tenant rate-limit quota configuration (the limit value is the same for all tenants — only the counter namespace changes)
- Redis-backed distributed rate limiting (in-process Map is sufficient for Phase 4; Phase 3 Redis adds distributed correctness)

## Dependencies

- **p3.1 and p3.2** (Phase 3 Postgres + Redis) must be complete before p4.1 is implemented — prompt-cache session persistence is more meaningful once sessions survive restarts; and the rate-limit counter should use Redis (from Phase 3) for correctness across restarts. This story is defined now because the scope is stable, but implementation must not begin until Phase 3 is done.

## Architecture constraints

- `req.session.accessToken` is the canonical field name — unchanged
- The prompt-cache key construction must be in a single function or injectable so it can be unit-tested without HTTP fixtures (AC1–AC3 are pure-function tests)
- The rate-limiter key must be extractable from the rate-limiter middleware for unit testing (AC4–AC6)
- Test file: `tests/check-p4.1-tenant-cache-ratelimit.js`, plain Node.js `assert`

## NFRs

- Key construction is synchronous — no I/O, negligible latency impact
- The change to the rate-limit key namespace means existing in-process rate-limit counters reset on deploy — acceptable; rate-limit counters are transient state
