# Story p3.1 — Postgres journey persistence adapter (Neon)

**Feature:** wuce-multi-tenancy
**Epic:** phase-3-persistence
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want journey data to survive server restarts by being persisted to a managed Postgres database,
So that beta users do not lose their in-progress journeys when the server is redeployed or crashes.

## Metric linkage

- **Beta readiness decision #2** (confirmed 2026-06-29): journey persistence required before beta — disk files are lost on container redeploy; Postgres (Neon free tier, per Decision 9) is the target store.

## Acceptance criteria

**AC1** — Journey created via `POST /journey` is persisted to Postgres
Given `DATABASE_URL` is set and the PG adapter is active,
When a tenant creates a journey,
Then a row is inserted into the `journeys` table with `journey_id`, `tenant_id`, `owner_id`, `feature_slug`, `created_at`, and `data` (JSONB for `completedStages`, `sessions`, `activeSkill`).

**AC2** — Journey survives simulated server restart
Given a journey was created and persisted to Postgres,
When the in-memory store is cleared and `loadAllFromPg()` is called (simulating restart),
Then `getJourney(journeyId)` returns the same journey with the same fields.

**AC3** — Journey fields updated via `setJourneyFields` are written through to Postgres
Given a journey exists in Postgres,
When `setJourneyFields(journeyId, { completedStages, activeSkill })` is called,
Then the `data` JSONB column in the corresponding Postgres row is updated to reflect the new values.

**AC4** — `listJourneys()` returns all journeys across tenants (for cap counting)
Given multiple tenants have journeys in Postgres,
When `listJourneys()` is called (no tenantId filter),
Then all journeys are returned (same semantics as the in-memory implementation — filtering by tenantId is the caller's responsibility).

**AC5** — `accessToken` is never written to Postgres
Given a session object attached to a journey contains `accessToken`,
When any journey field update is written to Postgres,
Then the `accessToken` field is absent from the persisted `data` column — it is stripped before write.

**AC6** — PG adapter is not activated when `DATABASE_URL` is absent
Given `DATABASE_URL` is not set,
When the server starts,
Then the disk adapter remains active (graceful fallback — no PG connection attempted, no crash).

**AC7** — Test suite passes
Given `node tests/check-p3.1-pg-journey-adapter.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Migrations run at deploy time (schema is created by a one-off `scripts/migrate-schema-pg.js` script run by the operator; the adapter does not auto-create tables)
- One-off migration of existing disk journeys to Postgres (separate operator script)
- Postgres connection pooling beyond `pg.Pool` defaults (adequate at beta scale)
- HA / read replicas (single Neon instance, see Decision 9)

## Dependencies

- Decision 9 (Neon free tier selected) — recorded in `decisions.md`
- `pg` npm package must be installed

## Implementation touchpoints

- `package.json`: add `pg` as prod dependency
- `src/web-ui/adapters/journey-store-pg.js` (new): async PG adapter implementing `createJourney`, `getJourney`, `listJourneys`, `setJourneyFields`, `setActiveSession`, `loadAllFromPg`
- `src/web-ui/modules/journey-store.js`: add `setPgAdapter(adapter)` and `loadAllFromPg()` alongside existing disk path; add `setPgAdapterForTesting(adapter)` injectable for tests
- `src/web-ui/server.js`: when `DATABASE_URL` is set, call `setPgAdapter` and `loadAllFromPg()` at startup instead of disk path
- `scripts/migrate-schema-pg.js` (new): one-time CREATE TABLE IF NOT EXISTS script
- `tests/check-p3.1-pg-journey-adapter.js`: verification script for all 7 ACs using injectable stub adapter (no real PG connection in tests)

## Architecture constraints

- The in-memory Map in `journey-store.js` remains the runtime read path — PG is write-through on create/update, and source-of-truth at restart
- The existing synchronous `getJourney` / `listJourneys` interface is preserved — the async PG writes happen via fire-and-forget (with error logging) so the HTTP response path is not blocked
- `accessToken` must be stripped before any JSONB write to PG — use a `_sanitiseForPg(journey)` helper that deletes `accessToken` from any nested session object
- The adapter must be injectable via `setPgAdapterForTesting(stub)` — stub defaults must throw per D37: `throw new Error('PG adapter not wired. Call setPgAdapter() before use.')`
- Schema: one `journeys` table — `journey_id VARCHAR PRIMARY KEY`, `tenant_id VARCHAR`, `owner_id VARCHAR`, `feature_slug VARCHAR`, `created_at TIMESTAMPTZ DEFAULT NOW()`, `data JSONB`

## NFRs

- PG write errors must be logged via `pino` but must not crash the request — the in-memory store is the fallback for reads within the same session
- At beta scale (< 100 journeys) `loadAllFromPg` is a full-table scan — acceptable; pagination not required until scale dictates
