# Story p2.1 — Tenant-parameterised repoRoot + session store namespacing + featureSlug collision guard

**Feature:** wuce-multi-tenancy
**Epic:** phase-2-storage-isolation
**Status:** Definition
**Complexity:** 3
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want artefact files written to a tenant-namespaced directory under `WUCE_TENANT_ROOT_BASE/${tenantId}/` and session files written to `SESSION_STORE_PATH/${tenantId}/`,
So that M2 (cross-tenant journey data leakage prevention) is met at the filesystem level: a scan of one tenant's directory reveals zero files from another tenant.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): this story establishes the filesystem isolation boundary; p2.2 verifies it

## Acceptance criteria

**AC1** — getRepoRoot returns tenant-scoped path when WUCE_TENANT_ROOT_BASE is set
Given `WUCE_TENANT_ROOT_BASE` is set to `/data/tenants` and a request arrives with `session.tenantId = 'my-org'`,
When any handler calls `getRepoRoot(req)` (or the equivalent tenant-aware resolver),
Then the returned absolute path is `path.resolve('/data/tenants/my-org')`.

**AC2** — getRepoRoot fallback to static resolution when tenantId is absent
Given a request with no `session.tenantId` (single-user mode or legacy session),
When `getRepoRoot(req)` is called,
Then the returned path falls back to the existing static resolution logic (`CLAUDE_REPO_PATH || COPILOT_REPO_PATH || path.resolve(...)`) — backward compatible.

**AC3** — Path-traversal guard fires on tenant-derived path
Given a resolved repoRoot of `/data/tenants/my-org`,
When a handler computes any artefact file path within that root and the computed path resolves outside the tenant root (e.g. a crafted featureSlug of `../../etc/passwd`),
Then the path-traversal guard rejects the request with HTTP 400 and no file is written.

**AC4** — `tenantId` is slugified before use in path construction
Given `session.tenantId` is set to a value containing characters outside `[a-z0-9-]` (e.g. uppercase, spaces, path separators, or encoded sequences like `%2f`),
When `getRepoRoot(req)` constructs the tenant directory path,
Then the `tenantId` is slugified (lowercased, non-alphanumeric characters replaced with `-`, stripped of leading/trailing `-`) before being appended to `WUCE_TENANT_ROOT_BASE`.

**AC5** — Session file written to tenant-namespaced path
Given `SESSION_STORE_PATH` is set and a session is written to disk for a user with `session.tenantId = 'my-org'`,
When the session-store adapter writes the file,
Then the file is created at `SESSION_STORE_PATH/my-org/${sessionId}.json`.

**AC6** — Session file read from tenant-namespaced path
Given a session exists at `SESSION_STORE_PATH/my-org/${sessionId}.json`,
When the session-store adapter reads the session using `sessionId` for a request with `session.tenantId = 'my-org'`,
Then the correct session is returned; sessions at `SESSION_STORE_PATH/other-org/${sessionId}.json` are never returned.

**AC7** — featureSlug collision guard
Given two different tenants (`org-a` and `org-b`) each create a journey with `featureSlug = '2026-06-22-my-feature'`,
When both journeys exist in the system,
Then the journey store does not conflate them — each journey is uniquely addressable by its `journeyId` (UUID), and the disk artefacts for each are written under separate tenant directories (`/data/tenants/org-a/artefacts/...` vs `/data/tenants/org-b/artefacts/...`).

**AC8** — `accessToken` strip-before-write invariant preserved
Given a session is written to the tenant-namespaced path,
When the session file is read back from disk,
Then `accessToken` is NOT present in the file (NFR-sec-no-accesstoken-disk — strip-before-write must be preserved with the new namespaced path).

**AC9** — Test suite for p2.1 passes
Given `node tests/check-p2.1-tenant-storage.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Automated tenant directory provisioning — the operator creates `WUCE_TENANT_ROOT_BASE/my-org/` before first use; the platform reads from it but does not create it
- In-memory journey store namespacing — the in-memory `_journeys` Map in `journey-store.js` is NOT namespaced by tenant in this story (it is replaced by Postgres in Phase 3; the isolation guarantee here is at the filesystem and disk-session boundaries)
- Adversarial tenantId injection test suite — Phase 5 (p5.1) covers this; this story establishes the slugification guard but the adversarial test suite runs in p5.1
- Phase 3 Postgres/Redis migrations — out of scope for this epic

## Dependencies

- **p1.2** must be complete — `session.tenantId` must be populated on requests before repoRoot can be derived from it

## Architecture constraints

- Path-traversal guard (CLAUDE.md ougl standard): any disk write path derived from `tenantId` MUST be checked with `path.resolve(derived).startsWith(resolvedTenantRoot + path.sep)` — return HTTP 400 if the check fails; do not log the raw tenant-derived path value in production (information disclosure risk)
- `tenantId` slugification: apply the same pattern as `_slugify()` in `journey.js` (lowercase, replace non-alphanumeric with `-`, strip leading/trailing `-`, limit to 48 characters)
- `req.session.accessToken` strip-before-write: the session-store adapter must strip `accessToken` before writing, unchanged from the existing invariant — adding `tenantId` to the write path must not change this
- `WUCE_TENANT_ROOT_BASE` env var: if absent, fall back to single-tenant mode (AC2); the platform must not fail to start when `WUCE_TENANT_ROOT_BASE` is unset
- A dedicated test must cover the path-traversal case (AC3) and assert both the HTTP 400 response and that no file was written to disk — this is mandatory per the CLAUDE.md path-traversal guard standard

## NFRs

- `getRepoRoot()` must return in under 1ms (synchronous path resolution, no I/O)
- Session file writes must not block the event loop — existing async write pattern preserved
- Data classification: Internal — tenant data is not public; `tenantId` (a GitHub org login) is not sensitive but must not appear in unguarded log output
