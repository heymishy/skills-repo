# Story p0.2 — Wire guard into all KNOWN BUG routes

**Feature:** wuce-multi-tenancy
**Epic:** phase-0-authorization-guard
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want all 9 KNOWN BUG route handlers to call `requireJourneyAccess()` before returning any journey data, and both mutating routes to replace their inline ownerId checks with the same guard,
So that T3-M1 (KNOWN BUG: 9 unguarded handlers) is closed to 0 and M1 (authorization coverage rate) reaches 100%.

## Metric linkage

- **M1** (Authorization coverage rate): this story wires the guard from p0.1 into all 11 handlers → 100%
- **T3-M1** (KNOWN BUG closure): this story closes the bug — 9 open → 0 open

## Acceptance criteria

**AC1** — handleGetJourneyState: cross-user read returns 404
Given an authenticated user with `session.login = 'alice'` calls `GET /api/journey/:journeyId` where the journey has `ownerId = 'bob'`,
When the handler runs,
Then the HTTP response status is 404 and the body contains no journey state data.

**AC2** — handleGetJourney (HTML page): cross-user read returns 404 or redirect
Given an authenticated user calls `GET /journey/:journeyId` for a journey owned by a different user,
When the handler runs,
Then the response is HTTP 404 or a redirect to `/` — the full journey HTML is not returned.

**AC3** — handleGetJourneyById (HTML by slug): cross-user read returns 404 or redirect
Given an authenticated user calls `GET /journey/:id` (by feature slug) for a journey owned by a different user,
When the handler runs,
Then the response is HTTP 404 or a redirect — the journey HTML is not returned.

**AC4** — handleGetJourneyViewers: cross-user read returns 404
Given an authenticated user calls `GET /api/journey/:journeyId/viewers` for a journey owned by a different user,
When the handler runs,
Then the HTTP response status is 404.

**AC5** — handleGetJourneyStageView: cross-user read returns 404
Given an authenticated user calls any route handled by `handleGetJourneyStageView` for a journey they don't own,
When the handler runs,
Then the HTTP response status is 404.

**AC6** — handleGetStageControls: cross-user read returns 404
Given an authenticated user calls `GET /api/journey/:journeyId/stage-controls` for a journey they don't own,
When the handler runs,
Then the HTTP response status is 404.

**AC7** — handlePostSideTripClarify: cross-user call returns 404
Given an authenticated user calls `POST /api/journey/:journeyId/side-trip/clarify` for a journey they don't own,
When the handler runs,
Then the HTTP response status is 404.

**AC8** — handleDeleteSideTrip: cross-user call returns 404
Given an authenticated user calls `DELETE /api/journey/:journeyId/side-trip` for a journey they don't own,
When the handler runs,
Then the HTTP response status is 404.

**AC9** — handleGetTrace: cross-user read returns 404
Given an authenticated user calls `GET /api/journey/:journeyId/trace` for a journey they don't own,
When the handler runs,
Then the HTTP response status is 404.

**AC10** — handlePostJourneyRecommit: cross-user mutating call returns 403
Given an authenticated user calls `POST /api/journey/:journeyId/recommit` for a journey owned by a different user,
When the handler runs,
Then the HTTP response status is 403 (mutating route — caller has already observed the journey exists, 403 is appropriate rather than existence-concealing 404).

**AC11** — handlePostJourneyStageCommit: cross-user mutating call returns 403
Given an authenticated user calls the stage commit route for a journey owned by a different user,
When the handler runs,
Then the HTTP response status is 403.

**AC12** — Owner can still access their own journey — no regression
Given an authenticated user with `session.login = 'alice'` calls any of the above routes for a journey where `journey.ownerId = 'alice'`,
When the handler runs,
Then the response is identical to the current behaviour (the guard does not alter the response for the journey's owner).

**AC13** — Legacy journeys (ownerId null) are accessible to any authenticated user
Given a journey with `ownerId: null` exists (created before Phase 0),
When any authenticated user calls any of the above handlers for this journey,
Then the request proceeds without a 404 or 403 (permissive legacy behaviour as defined in p0.1 AC1).

**AC14** — Test suite passes
Given `node tests/check-p0.2-journey-guard-wiring.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Changes to `session.shape` — session stays `{ accessToken, userId, login }` (3-field); `tenantId` is added in Phase 1 (p1.2)
- Tenant-level isolation (`isSameTenant()` returns `true` throughout Phase 0 — no cross-tenant rejections yet)
- Any changes to routes outside `src/web-ui/routes/journey.js` (e.g. `skills.js`, `auth.js`)
- Changing the response body format for authorized callers — only the guard call is added; all downstream logic for valid callers is unchanged

## Dependencies

- **p0.1** (Authorization guard module must exist before wiring — `require('../middleware/journey-access')` must resolve)

## Architecture constraints

- `req.session.accessToken` is the canonical field name in all session reads — never `req.session.token`
- ADR-024: `GET /api/journey/:id` response shape contract (`turns`, `stages`, `completedStages`, `stage`, `ownerId`, `activeSkill` required fields) must be preserved for authorized callers — the guard must not alter the shape
- All 9 handlers in the KNOWN BUG table must call `requireJourneyAccess()` and call `asHttpResponse(err, policy)` to derive the HTTP status — no new ad-hoc inline checks
- The two mutating handlers (`handlePostJourneyRecommit`, `handlePostJourneyStageCommit`) must have their existing inline `ownerId` checks replaced with the `OWNER` policy guard — not supplemented
- Path-traversal guards already present in `journey.js` must not be removed or weakened
- A grep of `module.exports` in `journey.js` must be performed before implementation to confirm the exact exported function names — the DoR AC must name every exported handler that the wiring touches

## NFRs

- No performance regression for authorized callers — the guard is a synchronous check with no I/O; latency impact is negligible
- Test file: `tests/check-p0.2-journey-guard-wiring.js` using plain Node.js `assert`, `freshRequire()` for module isolation, and `makeReq({ session: { accessToken, userId, login } })` as the session fixture (3-field shape — do not pre-empt Phase 1)
