# Story p0.1 — Authorization guard module

**Feature:** wuce-multi-tenancy
**Epic:** phase-0-authorization-guard
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want a reusable authorization guard module (`journey-access.js`) that throws typed errors for unauthenticated, forbidden, and not-found cases,
So that all journey route handlers can enforce ownership access using a single tested function and M1 (authorization coverage rate) can reach 100% — eliminating the ad-hoc inline checks that produced the KNOWN BUG.

## Metric linkage

- **M1** (Authorization coverage rate): p0.1 creates the guard that p0.2 wires — both stories together close M1 from 18% → 100%
- **T3-M1** (KNOWN BUG closure): p0.1 is the prerequisite; the bug is closed when p0.2 wires this module

## Acceptance criteria

**AC1** — Legacy journey passthrough
Given `requireJourneyAccess(journey, session, 'TENANT')` is called with `journey.ownerId === null` or `journey.ownerId === undefined` (legacy journeys created before Phase 0),
When the function runs,
Then no error is thrown (permissive legacy behaviour — null ownerId = readable by any authenticated user).

**AC2** — Owner access granted
Given `requireJourneyAccess(journey, session, 'TENANT')` is called with `journey.ownerId` set to a non-null value that exactly matches `session.login`,
When the function runs,
Then no error is thrown.

**AC3** — Non-owner access denied (TENANT policy)
Given `requireJourneyAccess(journey, session, 'TENANT')` is called with `journey.ownerId` set to a value that does NOT match `session.login`,
When the function runs,
Then a typed error object `{ code: 'FORBIDDEN' }` is thrown.

**AC4** — Unauthenticated access denied
Given `requireJourneyAccess(journey, session, 'TENANT')` is called where `session` is `null`, `undefined`, or has no `accessToken` property,
When the function runs,
Then a typed error object `{ code: 'UNAUTHENTICATED' }` is thrown.

**AC5** — Non-existent journey
Given `requireJourneyAccess(null, session, 'TENANT')` is called where `journey` is `null` or `undefined`,
When the function runs,
Then a typed error object `{ code: 'NOT_FOUND' }` is thrown.

**AC6** — `asHttpResponse` TENANT policy maps FORBIDDEN → 404 (existence leak prevention)
Given `asHttpResponse({ code: 'FORBIDDEN' }, 'TENANT')` is called,
When the function runs,
Then it returns the integer `404`.

**AC7** — `asHttpResponse` OWNER policy maps FORBIDDEN → 403
Given `asHttpResponse({ code: 'FORBIDDEN' }, 'OWNER')` is called,
When the function runs,
Then it returns the integer `403`.

**AC8** — `asHttpResponse` maps UNAUTHENTICATED → 401
Given `asHttpResponse({ code: 'UNAUTHENTICATED' }, anyPolicy)` is called,
When the function runs,
Then it returns the integer `401`.

**AC9** — `asHttpResponse` maps NOT_FOUND → 404
Given `asHttpResponse({ code: 'NOT_FOUND' }, anyPolicy)` is called,
When the function runs,
Then it returns the integer `404`.

**AC10** — `isSameTenant()` Phase 0 passthrough
Given `isSameTenant(journey, session)` is called where neither `journey.tenantId` nor `session.tenantId` is set (Phase 0 — no tenant concept yet),
When the function runs,
Then it returns `true` (no tenant boundary enforced until Phase 1).

**AC11** — Module is directly requireable with no side effects
Given `require('./middleware/journey-access')`,
When the module loads,
Then it exports `{ requireJourneyAccess, isOwner, isSameTenant, asHttpResponse }` and has no side effects (no process.env reads at require time, no I/O).

## Out of scope

- Wiring this module into any route handler — that is p0.2
- Tenant-level enforcement (isSameTenant returning false) — Phase 1 (p1.2) updates isSameTenant to enforce a real boundary once tenantId exists on both sides
- HTTP response writing — `asHttpResponse` returns a status integer, not an http.ServerResponse interaction; callers write the response

## Dependencies

None. This story has no upstream story dependency.

## Architecture constraints

- `req.session.accessToken` is the canonical field name — the module receives `session` as a plain object; it MUST check `session.accessToken` (not `session.token`) when testing authentication state (AC4)
- No injectable adapter is required for this module — it is a pure utility with no I/O, no injectable defaults needed
- File location: `src/web-ui/middleware/journey-access.js` — matches the path referenced throughout the discovery and benefit-metric artefacts
- Test file: `tests/check-p0.1-journey-access.js`, following the `tests/check-[story-id]-description.js` naming convention with plain Node.js `assert` (no test runner)

## NFRs

- No I/O, no async operations — all functions are synchronous
- Module must load in under 5ms (measured by requiring it in the test file)
- No external npm dependencies
