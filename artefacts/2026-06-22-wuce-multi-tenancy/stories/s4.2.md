## User story

As the operator,
I want `TENANT_ORG_ALLOWLIST` configured with the beta org(s), tenant directory trees provisioned, and `tenant-caps.json` set with appropriate per-user caps,
So that the first beta users can log in, are correctly assigned their tenantId, and have their journey data isolated to their own directory tree.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): this story provisions the directory isolation boundary that M2 measures — without it, multi-tenancy is code-complete but inactive.

## Acceptance criteria

**AC1** — `TENANT_ORG_ALLOWLIST` contains the intended beta org(s)
Given `fly secrets list` shows `TENANT_ORG_ALLOWLIST`,
When a user from one of the allowlisted GitHub orgs completes the OAuth flow,
Then `req.session.tenantId` equals their org login (confirmed in Fly.io logs: `[auth] login { userId: ..., timestamp: ... }`) and they are redirected to `/dashboard`.

**AC2** — A user from a non-allowlisted org is rejected at login
Given `TENANT_ORG_ALLOWLIST` is set,
When a user who belongs to zero allowlisted orgs completes the OAuth callback,
Then they receive HTTP 403 with "You are not a member of an authorised organisation." and no session is created.

**AC3** — Tenant root directories exist for all invited orgs
Given `WUCE_TENANT_ROOT_BASE` is set in the deployment environment,
When `fly ssh console -C "ls $WUCE_TENANT_ROOT_BASE"` is run,
Then a subdirectory exists for each allowlisted org login (created manually by the operator per Decision 3).

**AC4** — First beta user can create a journey and it is stored under their tenant tree
Given AC1 and AC3 are satisfied and the user creates a journey via `POST /journey`,
When `fly ssh console -C "ls $WUCE_TENANT_ROOT_BASE/<tenantId>/artefacts/"` is run,
Then the journey's feature artefact directory is present under the correct tenant path.

**AC5** — Per-user journey cap is configured appropriately for beta
Given the operator has created `{repoRoot}/tenant-caps.json` (or set `MAX_JOURNEYS_PER_TENANT`) with the intended beta cap,
When `node scripts/smoke-test-pg.js` or a manual check confirms the cap is active,
Then the cap value matches the intended beta limit (operator-specified, e.g. 5 journeys per tenant).

**AC6** — Solo-mode (no TENANT_ORG_ALLOWLIST) still works for the operator's own account
Given a second deployment or local dev run has `TENANT_ORG_ALLOWLIST` unset,
When the operator logs in,
Then `req.session.tenantId` equals `req.session.login` (GitHub username) per s0.2 fallback — no org check is performed.

## Out of scope

- Self-service tenant registration (all onboarding is operator-manual at beta scale)
- Automated tenant directory creation (Decision 3: operator-manual)
- Per-tenant feature flag configuration beyond journey cap

## Implementation touchpoints

- `fly secrets set TENANT_ORG_ALLOWLIST=<org1>,<org2>` — operator CLI step
- `fly secrets set WUCE_TENANT_ROOT_BASE=/app/tenants` (or equivalent) — operator CLI step
- `fly ssh console` — to create tenant root directories manually
- `tenant-caps.json` — operator creates at `{repoRoot}/tenant-caps.json` with per-tenant limits
- `scripts/smoke-test-production.sh` — extended to include AC1–AC4 verification steps

## Architecture Constraints

None identified — checked against `.github/architecture-guardrails.md`. This story makes no changes to governed code paths (no `src/`, no SKILL.md, no dashboard logic). All actions are operator CLI steps (fly secrets set, fly ssh console, creating local tenant-caps.json).

## NFRs

- **NFR-sec-allowlist-disclosure** (nfr-profile.md §Security): The 403 rejection message must not expose `TENANT_ORG_ALLOWLIST` contents to the rejected user. AC2 verifies the rejection message is exactly "You are not a member of an authorised organisation." with no org names disclosed.
- **Allowlist not committed** (nfr-profile.md §Security): `TENANT_ORG_ALLOWLIST` value must appear only as a Fly.io secret, never in any committed file. Verified by `git grep` check in test plan NFR-SEC-2.

## Test

Verification: AC1 and AC2 are verified by the operator completing real OAuth login flows with both allowlisted and non-allowlisted GitHub accounts. AC3–AC4 are verified via `fly ssh console`. AC5 via cap smoke test. All steps captured in `scripts/smoke-test-production.sh`.
