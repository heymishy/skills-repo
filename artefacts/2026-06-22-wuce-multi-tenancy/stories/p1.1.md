# Story p1.1 — OAuth scope + tenant resolution

**Feature:** wuce-multi-tenancy
**Epic:** phase-1-tenant-identity
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable (pending OQ4/OQ5 from discovery — pagination and multi-org disambiguation; both are addressed in ACs below)

## User story

As a solo founder / operator,
I want the OAuth flow to resolve a `tenantId` from the authenticating user's GitHub org memberships against a configured allowlist,
So that M2 (cross-tenant journey data leakage prevention) moves toward its target by establishing the identity boundary that `isSameTenant()` will enforce in p1.2.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): p1.1 establishes the identity that makes tenant isolation meaningful — without a resolved `tenantId` on the session, no boundary can be enforced

## Acceptance criteria

**AC1** — `read:org` scope included in OAuth request
Given `TENANT_ORG_ALLOWLIST` is set to a non-empty value,
When the OAuth authorize URL is constructed in `oauth-adapter.js`,
Then the scope string includes `read:org` alongside the existing `repo,read:user` scopes.

**AC2** — Single-page org resolution
Given a user authenticates and `GET /user/orgs` returns orgs within a single page (≤30 results),
When the OAuth callback handler resolves `tenantId`,
Then the first org login in `TENANT_ORG_ALLOWLIST` order that also appears in the orgs response is set as `req.session.tenantId`.

**AC3** — Multi-page org pagination
Given a user authenticates and their org list spans more than one page (`GET /user/orgs` returns a `Link: rel="next"` header),
When the OAuth callback handler resolves `tenantId`,
Then all pages are fetched before the allowlist match is evaluated — the match is not performed on the first page alone.

**AC4** — Zero-match rejection
Given a user authenticates and their org list contains no org login that appears in `TENANT_ORG_ALLOWLIST`,
When the OAuth callback handler resolves `tenantId`,
Then the login is rejected: the session is not created, and the user receives a clear error response (HTTP 4xx with a message explaining that they are not a member of an allowed organisation — not a silent redirect to `/`).

**AC5** — Multi-match: first allowlist match wins
Given a user authenticates and their org list contains two orgs that both appear in `TENANT_ORG_ALLOWLIST` (e.g. allowlist is `['org-a','org-b']` and user belongs to both),
When the callback resolves `tenantId`,
Then `req.session.tenantId` is set to `org-a` (first in allowlist order, not first in the `/user/orgs` response order).

**AC6** — `TENANT_ORG_ALLOWLIST` absent: backward-compatible single-user mode
Given `TENANT_ORG_ALLOWLIST` is not set (env var absent or empty string),
When the OAuth callback runs,
Then no org fetch is performed, `req.session.tenantId` is not set, and the rest of the login flow proceeds as before Phase 1 (backward compatible with single-user deployments).

**AC7** — Org-fetch function is injectable
Given the org-fetch function is implemented as an injectable adapter (`let _fetchOrgs = defaultFetchFn; function setFetchOrgs(fn) { _fetchOrgs = fn; }`),
When the default `_fetchOrgs` is called before `setFetchOrgs()` has been invoked,
Then it throws `Error('Adapter not wired: fetchOrgs. Call setFetchOrgs() with a real implementation before use.')` — not return null or empty.

**AC8** — Test suite passes
Given `node tests/check-p1.1-oauth-tenant-resolution.js` is run after implementation,
Then all assertions pass and exit code is 0.

**AC9** — `setFetchOrgs()` wired in server.js with a real GitHub API implementation
Given the OAuth callback module exports `setFetchOrgs()`,
When `src/web-ui/server.js` is started,
Then `setFetchOrgs()` is called with a real implementation that makes a `GET /user/orgs` call to the GitHub API using the OAuth access token — not a stub, not undefined. The wiring call must appear in `server.js` before the OAuth routes are registered.

## Out of scope

- Writing `tenantId` to the journey object — that is p1.2
- Updating `isSameTenant()` to enforce a real boundary — that is p1.2
- Updating test session fixtures to 4-field shape — that is p1.2
- Multi-org membership disambiguation beyond "first match in allowlist order" (discovery OQ5 — deferred)
- GHE token URL fix (discovery discrepancy 4 — explicitly out of scope for this feature)

## Dependencies

- **p0.2** must be complete (Phase 0 authorization guard wired) before Phase 1 begins — if tenantId is populated but the guard isn't wired, the identity exists but the enforcement is missing

## Architecture constraints

- `req.session.accessToken` is the canonical field name — the accessToken received at OAuth callback must be stored under this exact field name, unchanged
- Injectable adapter rule (D37): the org-fetch function must be injectable with a stub-throws default (AC7) — stub must throw, not return empty/null
- The production wiring of `setFetchOrgs()` with a real GitHub API implementation must be done in `server.js` (or equivalent wiring module) — not inline in the callback handler
- `TENANT_ORG_ALLOWLIST` is parsed as a comma-separated string of org logins: `process.env.TENANT_ORG_ALLOWLIST.split(',').map(s => s.trim()).filter(Boolean)`
- Test file: `tests/check-p1.1-oauth-tenant-resolution.js`, plain Node.js `assert`, injectable org-fetch adapter used to mock responses without real GitHub API calls

## NFRs

- Org pagination fetch must not block the event loop with synchronous waits — use `async/await` or Promise chaining
- Total org fetch (all pages) must complete within 5 seconds on a normal GitHub API connection — log a warning if it exceeds 3 seconds
- The zero-match error message must not expose the contents of `TENANT_ORG_ALLOWLIST` to the end user (potential information disclosure)
