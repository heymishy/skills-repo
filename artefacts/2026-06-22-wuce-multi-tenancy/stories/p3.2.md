# Story p3.2 — Redis session persistence adapter (Upstash)

**Feature:** wuce-multi-tenancy
**Epic:** phase-3-persistence
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want OAuth sessions to survive server restarts by being backed by Upstash Redis,
So that beta users are not silently logged out every time the server redeploys.

## Metric linkage

- **Beta readiness decision #2** (confirmed 2026-06-29): session persistence required before beta — the in-memory Map in `session.js` is wiped on every server restart; Upstash Redis (per Decision 9) is the target store.

## Acceptance criteria

**AC1** — Session is written to Redis on creation
Given `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are set and the Redis adapter is active,
When `createSession(id, data)` is called,
Then a key `session:{id}` is written to Redis with TTL = 24 hours, and the value is the session data with `accessToken` stripped.

**AC2** — `accessToken` is never written to Redis
Given a session contains `session.accessToken`,
When the session is persisted to Redis,
Then the `accessToken` field is absent from the Redis value — it is stripped before write.

**AC3** — Session survives simulated server restart
Given a session was written to Redis,
When the in-memory Map is cleared and `loadSessionsFromRedis()` is called (simulating restart),
Then `getSession(id)` returns the session data (minus `accessToken`, which is never persisted).

**AC4** — Session is deleted from Redis on `destroySession`
Given a session exists in both in-memory Map and Redis,
When `destroySession(id)` is called (e.g. on logout),
Then the key `session:{id}` is removed from Redis.

**AC5** — Redis adapter is not activated when env vars are absent
Given `UPSTASH_REDIS_REST_URL` is not set,
When the server starts,
Then the in-memory-only path remains active (graceful fallback — no Redis connection attempted, no crash).

**AC6** — `getSession` returns session from memory if present (fast path)
Given a session exists in the in-memory Map,
When `getSession(id)` is called,
Then the in-memory Map is returned without a Redis round-trip (memory is the primary read path; Redis is only consulted on memory-miss after restart).

**AC7** — Test suite passes
Given `node tests/check-p3.2-redis-session-adapter.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- HTML skill sessions in `session-store.js` — this story covers OAuth sessions in `session.js` only; the disk-backed skill session store is a separate concern
- Multi-region Upstash read replicas (single region, default config)
- Session encryption at rest beyond Upstash's default TLS (not required for beta)
- Per-tenant session TTL configuration

## Dependencies

- Decision 9 (Upstash Redis selected) — recorded in `decisions.md`
- `@upstash/redis` npm package must be installed

## Implementation touchpoints

- `package.json`: add `@upstash/redis` as prod dependency
- `src/web-ui/adapters/session-redis.js` (new): async Redis adapter wrapping `@upstash/redis` — `writeSession(id, data)`, `deleteSession(id)`, `loadAllSessions()` (returns array of `{id, data}`)
- `src/web-ui/middleware/session.js`: extend `createSession` to call Redis write (fire-and-forget); extend `destroySession` to call Redis delete; add `loadSessionsFromRedis(adapter)` for startup; add `setRedisAdapterForTesting(adapter)` injectable
- `src/web-ui/server.js`: when `UPSTASH_REDIS_REST_URL` is set, call `setRedisAdapter` and `loadSessionsFromRedis()` at startup
- `tests/check-p3.2-redis-session-adapter.js`: verification script for all 7 ACs using injectable stub adapter

## Architecture constraints

- The in-memory Map in `session.js` remains the runtime read path (no Redis read on every `getSession` during a live request)
- `accessToken` must be stripped before any Redis write — use a `_sanitiseForRedis(data)` helper that deletes `accessToken` from the data object before serialising
- After restart, sessions loaded from Redis will not have `accessToken` — callers must re-authenticate if they require the token. This is acceptable: GitHub OAuth allows token re-issue; the user experience is a re-login prompt, not data loss.
- The adapter must be injectable via `setRedisAdapterForTesting(stub)` — stub defaults must throw per D37: `throw new Error('Redis adapter not wired. Call setRedisAdapter() before use.')`
- Redis key format: `session:{sessionId}` — avoids collision with any other Upstash keys

## NFRs

- Redis write errors must be logged via `pino` but must not block session creation — the in-memory Map is updated first, Redis write is async
- TTL for Redis keys: 86400 seconds (24 hours) — same as session cookie maxAge
- `loadSessionsFromRedis` must complete before the server starts accepting requests — await it in the startup sequence in `server.js`
