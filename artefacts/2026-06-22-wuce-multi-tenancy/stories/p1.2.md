# Story p1.2 ‚ÄĒ Session shape + journey tenantId + isSameTenant enforcement

**Feature:** wuce-multi-tenancy
**Epic:** phase-1-tenant-identity
**Status:** Definition
**Complexity:** 2
**Scope stability:** Stable (pending OQ6 from discovery ‚ÄĒ fixture audit scope; addressed in AC5)

## User story

As a solo founder / operator,
I want `tenantId` stored on both the session and the journey at create time, and `isSameTenant()` to enforce the real tenant boundary,
So that M2 (cross-tenant journey data leakage prevention) is enforced at the application layer: a user in org A receives a 404 when calling any journey route for a journey created by org B ‚ÄĒ the authorization guard now has a real tenant boundary to check.

## Metric linkage

- **M2** (Cross-tenant journey data leakage prevention): this story makes the in-memory isolation guarantee real ‚ÄĒ `isSameTenant()` now returns `false` when tenantIds differ

## Acceptance criteria

**AC1** ‚ÄĒ Journey created with `tenantId` from session
Given a user with `session.tenantId = 'my-org'` creates a new journey,
When `createJourney()` is called,
Then `journey.tenantId` is set to `'my-org'` on the new journey object (and persisted to disk via the journey disk adapter if active).

**AC2** ‚ÄĒ `isSameTenant()` enforces real boundary when both sides have tenantId
Given `isSameTenant(journey, session)` is called where `journey.tenantId = 'org-a'` and `session.tenantId = 'org-b'`,
When the function runs,
Then it returns `false`.

**AC3** ‚ÄĒ `isSameTenant()` matching tenantIds returns true
Given `isSameTenant(journey, session)` is called where `journey.tenantId = 'org-a'` and `session.tenantId = 'org-a'`,
When the function runs,
Then it returns `true`.

**AC4** ‚ÄĒ `isSameTenant()` Phase 0 passthrough preserved (null/undefined tenantId on either side)
Given `isSameTenant(journey, session)` is called where either `journey.tenantId` or `session.tenantId` is `null` or `undefined`,
When the function runs,
Then it returns `true` (permissive legacy behaviour ‚ÄĒ Phase 0 contract is preserved; journeys created before Phase 1 and single-user-mode sessions are not broken).

**AC5** ‚ÄĒ Existing test session fixtures updated from 3-field to 4-field shape where required
Given the complete list of test files in `tests/` that use `makeReq({ session: { accessToken, userId, login } })` (3-field shape),
When p1.2 is implemented and a test handler requires `session.tenantId` to be present for its Phase 1 logic to be exercised,
Then only those specific tests are updated to 4-field shape `{ accessToken, userId, login, tenantId }` ‚ÄĒ tests for handlers that do not read `session.tenantId` are left at 3-field shape and must continue to pass unchanged.

**AC6** ‚ÄĒ Cross-tenant read returns 404 after p1.2
Given a user with `session.login = 'alice'` and `session.tenantId = 'org-a'` calls `GET /api/journey/:journeyId` where the journey has `ownerId = 'bob'` and `tenantId = 'org-b'`,
When the handler runs,
Then the HTTP response status is 404 (guard fires on tenant mismatch via `isSameTenant()` returning `false` ‚Üí `FORBIDDEN` ‚Üí 404 via TENANT policy).

**AC7** ‚ÄĒ Pre-existing test suite does not regress
Given the full test suite `node tests/check-*.js` is run after p1.2 merges,
Then all tests that were passing before p1.2 still pass and exit code is 0.

**AC8** ‚ÄĒ Test suite for p1.2 passes
Given `node tests/check-p1.2-tenant-session-journey.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Storage namespacing (repoRoot parameterisation, SESSION_STORE_PATH namespacing) ‚ÄĒ Phase 2 (p2.1)
- Updating ALL test files to 4-field shape globally ‚ÄĒ only files where the handler under test reads `session.tenantId` are updated (AC5)
- Phase 3 Postgres/Redis persistence ‚ÄĒ `journey.tenantId` is stored in the in-memory store and written to disk via the existing journey disk adapter; no schema migration required for the in-memory path

## Dependencies

- **p1.1** must be complete ‚ÄĒ `session.tenantId` must be resolvable at OAuth callback before it can be written to journeys

## Architecture constraints

- `req.session.accessToken` is the canonical field name ‚ÄĒ `tenantId` is added as a 4th field alongside `accessToken`, `userId`, `login`; no existing field is renamed
- The session fixture pattern `makeReq({ session: { ... } })` is preserved ‚ÄĒ only the session object fields change; `makeReq` itself is not restructured
- `isSameTenant()` in `src/web-ui/middleware/journey-access.js` is updated in place ‚ÄĒ it is already exported; no new export is needed
- `journey.tenantId` is stored on the journey object at `createJourney()` time ‚ÄĒ the field is optional on pre-Phase-1 journeys (null/undefined is acceptable and triggers the permissive passthrough in AC4)
- Test file: `tests/check-p1.2-tenant-session-journey.js`, plain Node.js `assert`

## NFRs

- No change to session cookie shape or HttpOnly/SameSite attributes ‚ÄĒ only the in-memory session object gains `tenantId`
- `accessToken` must not be written to disk by the session store adapter ‚ÄĒ the strip-before-write invariant (NFR-sec-no-accesstoken-disk) must be preserved when tenantId is added to the disk session format
