# Story p3.3 — Restart-survival and concurrent-session tests

**Feature:** wuce-multi-tenancy
**Epic:** phase-3-persistence
**Status:** Definition
**Complexity:** 1
**Scope stability:** Stable

## User story

As a solo founder / operator,
I want an automated test suite that proves journeys and sessions survive a simulated server restart and that concurrent multi-tenant sessions do not interfere with each other,
So that I have a repeatable CI signal that the Phase 3 persistence layer is correct before going to beta.

## Metric linkage

- **Beta readiness decision #2** (confirmed 2026-06-29): this story is the evidence artefact for persistence correctness — it is the test suite that produces the binary PASS/FAIL signal before beta sign-off.

## Acceptance criteria

**AC1** — Journey persists across simulated restart (in-memory cleared, reloaded from PG stub)
Given a journey was created via `createJourney` and the PG stub recorded the write,
When the in-memory Map is cleared and `loadAllFromPg()` is called with the PG stub's recorded data,
Then `getJourney(journeyId)` returns the journey with all original fields intact.

**AC2** — Journey field update persists across simulated restart
Given `setJourneyFields(journeyId, { completedStages: ['discovery'] })` was called,
When the in-memory Map is cleared and `loadAllFromPg()` replays the PG stub's recorded data,
Then the reloaded journey has `completedStages: ['discovery']`.

**AC3** — Session persists across simulated restart (in-memory cleared, reloaded from Redis stub)
Given a session was created with `tenantId = 'org-a'` and the Redis stub recorded the write,
When the in-memory Map is cleared and `loadSessionsFromRedis()` is called with the Redis stub's data,
Then `getSession(sessionId)` returns the session with `tenantId = 'org-a'`.

**AC4** — `accessToken` is absent from both PG and Redis stub records
Given sessions and journeys were created with `accessToken: 'secret'` in the session data,
When the PG stub's recorded journey writes and the Redis stub's recorded session writes are inspected,
Then `accessToken` is absent from all persisted records.

**AC5** — Two concurrent tenant sessions do not share journey data
Given `tenant-a` and `tenant-b` each create a journey concurrently (interleaved via async calls),
When `listJourneys()` is called and filtered by each tenantId,
Then `tenant-a`'s filter returns only `tenant-a`'s journey; `tenant-b`'s filter returns only `tenant-b`'s journey.

**AC6** — Session TTL is set correctly on Redis stub writes
Given a session is created via `createSession`,
When the Redis stub's recorded write is inspected,
Then the TTL value is 86400 (24 hours) and the key format is `session:{sessionId}`.

**AC7** — Test suite passes
Given `node tests/check-p3.3-persistence-survival.js` is run after implementation,
Then all assertions pass and exit code is 0.

## Out of scope

- Real PG or Redis connections — tests use injectable stub adapters only
- Load testing or benchmarking (adequate coverage at correctness level)
- Failure injection (PG write failure, Redis connection drop) — separate resilience story if needed

## Dependencies

- **p3.1** and **p3.2** must be complete — this test suite exercises the injectable interfaces introduced by those stories

## Implementation touchpoints

- `tests/check-p3.3-persistence-survival.js` (new): verification script for all 7 ACs using stub adapters injected via `setPgAdapterForTesting` (from p3.1) and `setRedisAdapterForTesting` (from p3.2)

## Architecture constraints

- No real database connections — stub adapters must be in-memory maps that record calls
- Test must clear the in-memory Map by calling an exported `_clearForTesting()` function (added to `journey-store.js` in p3.1 and to `session.js` in p3.2) — it must not reach into module internals
- All ACs must be independently readable in test output (no opaque loop assertions)
- Test file must complete in under 5 seconds

## NFRs

- Test is self-contained — no env vars required, no network access, no disk I/O
