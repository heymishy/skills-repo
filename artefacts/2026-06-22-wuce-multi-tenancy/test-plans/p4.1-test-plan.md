## Unit Tests

### buildCacheKey — returns tenant-scoped key for two sessions with same sessionId

- **Verifies:** AC1 (safety-critical case)
- **Precondition:** Two session objects: `{ tenantId: 'org-a', sessionId: 'abc' }` and `{ tenantId: 'org-b', sessionId: 'abc' }`
- **Action:** `const keyA = buildCacheKey({ tenantId: 'org-a', sessionId: 'abc' }); const keyB = buildCacheKey({ tenantId: 'org-b', sessionId: 'abc' }); assert.notStrictEqual(keyA, keyB);`
- **Expected result:** `keyA !== keyB` — a cache hit for org-a cannot be served to org-b's identically-IDed session
- **Edge case:** Yes — this is the primary safety case (cross-tenant cache bleed prevention)

### buildCacheKey — format is `${tenantId}-${sessionId}`

- **Verifies:** AC2
- **Precondition:** Session `{ tenantId: 'my-org', sessionId: 'xyz123' }`
- **Action:** `assert.strictEqual(buildCacheKey({ tenantId: 'my-org', sessionId: 'xyz123' }), 'my-org-xyz123')`
- **Expected result:** `'my-org-xyz123'`
- **Edge case:** No

### buildCacheKey — falls back to sessionId when tenantId absent

- **Verifies:** AC3
- **Precondition:** Session `{ sessionId: 'xyz123' }` (no tenantId)
- **Action:** `assert.strictEqual(buildCacheKey({ sessionId: 'xyz123' }), 'xyz123')`
- **Expected result:** `'xyz123'` — backward compatible with single-tenant deployments
- **Edge case:** Also test `tenantId: null` and `tenantId: undefined` — both must fall back to sessionId

### Rate-limit counters are isolated per tenant

- **Verifies:** AC4
- **Precondition:** In-process Map rate-limiter; `org-a` and `org-b` rate-limit keys
- **Action:** Increment `org-a`'s counter 100 times; read `org-b`'s counter
- **Expected result:** `org-b`'s counter is 0 — org-a hitting the rate limit does not consume org-b's quota
- **Edge case:** No

### buildRateLimitKey — includes tenantId when available

- **Verifies:** AC5
- **Precondition:** `req = makeReq({ session: { tenantId: 'my-org' }, ip: '1.2.3.4' })`
- **Action:** `const key = buildRateLimitKey(req); assert.ok(key.includes('my-org'));`
- **Expected result:** Key contains `'my-org'`
- **Edge case:** No

### buildRateLimitKey — falls back to IP when tenantId absent

- **Verifies:** AC6
- **Precondition:** `req = makeReq({ session: {}, ip: '1.2.3.4' })`
- **Action:** `const key = buildRateLimitKey(req); assert.ok(key.includes('1.2.3.4'));`
- **Expected result:** Key is IP-based — backward compatible with pre-Phase-1 deployments
- **Edge case:** Also test `tenantId: null` and `tenantId: undefined` — both must use IP

### buildRateLimitKey — tenantId takes precedence over IP when both present

- **Verifies:** AC5 (disambiguation)
- **Precondition:** `req = makeReq({ session: { tenantId: 'my-org' }, ip: '1.2.3.4' })`
- **Action:** `const key = buildRateLimitKey(req);`
- **Expected result:** Key includes `'my-org'` (tenant namespace); IP may also be included but tenantId is the primary discriminator
- **Edge case:** No

## NFR Tests

### buildCacheKey is synchronous

- **NFR addressed:** Performance — "key construction is synchronous — no I/O, negligible latency impact"
- **Measurement method:** Assert `!(buildCacheKey(session) instanceof Promise)`
- **Pass threshold:** Returns a string synchronously
- **Tool:** Plain `assert`

### Rate-limit counter namespace change noted (in-process Map reset on deploy)

- **NFR addressed:** Operational — "existing in-process rate-limit counters reset on deploy — acceptable"
- **Measurement method:** Document-only test: verify the test file includes a comment confirming this is expected behaviour and that no persisted counter state is assumed
- **Pass threshold:** Comment present in test file; no test assumes pre-existing counter state survives restart
- **Tool:** Code review at DoR

## Out of Scope for This Test Plan

- Prompt-cache backend changes — AC out-of-scope; only the key scoping is changed
- Per-tenant rate-limit quota configuration — rate limit value is the same for all tenants; only the counter namespace changes
- Redis-backed distributed rate limiting — Phase 3; in-process Map is sufficient for Phase 4
- HTTP-level rate-limit response (429 Too Many Requests) — the counter isolation is the testable unit here; HTTP integration depends on Phase 3 Redis
