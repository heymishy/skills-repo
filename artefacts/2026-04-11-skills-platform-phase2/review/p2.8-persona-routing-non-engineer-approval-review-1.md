## Category C — AC Quality

**F-1 [MEDIUM]:** AC2 states "non-engineer sign-off recorded in [configured channel] tool with timestamp" but does not specify the mechanism by which the sign-off event is detected and written back to the pipeline. Specifically: is the sign-off detected by (a) a manual operator command (e.g. `/record-sign-off [story-slug]`)? (b) a webhook from the configured tool? (c) polling? (d) a structured comment format the pipeline watches for? For a C2/Unstable story, this mechanism ambiguity is material — two engineers could build radically different solutions, one of which integrates correctly and one of which does not. The AC behaviourally specifies the outcome (sign-off recorded with timestamp) but leaves the trigger entirely open. Add a note to Architecture Constraints or create a SCOPE-APPROVAL-01 assumption in decisions.md that declares the expected trigger mechanism for Phase 2.

## Category D — Completeness

## Category E — Architecture Compliance

**F-2 [LOW]:** AC1 describes channel hint routing and the channel hint may appear as a rendered element in pipeline-viz.html (e.g. a "Pending approval via Jira" indicator on the story card). If this is the case, MC-A11Y-01 (keyboard-accessible) and MC-A11Y-02 (colour not sole indicator) apply to the channel hint rendering. Architecture Constraints mention MC-CONSIST-02 for channel_hints schema sync but do not reference accessibility guardrails. If no viz element is added in this story, clarify that the channel hint is external-tool-only (no viz component) and this finding can be closed. Otherwise add MC-A11Y-01/02 to Architecture Constraints.
