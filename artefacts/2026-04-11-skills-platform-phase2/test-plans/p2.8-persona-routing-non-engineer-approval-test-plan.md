`dorApprover` must store a role identifier or username (e.g. GitHub username, Jira account ID) — not a full name or email address. Test fixtures enforce this: no PII in any fixture.

## Unit Tests

### Channel hint routing: notification payload contains story slug, squad ID, approval action description, fleet registry reference

- **Verifies:** AC1
- **Precondition:** context.yml fixture with `channel_hints.approval: { type: "github-issue", issueUrl: "https://example/issue/1" }`; story slug available
- **Action:** Call the channel routing notification function with story slug and squad ID
- **Expected result:** Notification payload object has `storySlug` (string), `squadId` (string), `approvalActionDescription` (non-empty string), and `fleetRegistryRef` (non-null reference to the squad's `pipeline-state.json` path via fleet registry)
- **Edge case:** No

### Channel hint routing: notification target URL is read from context.yml (not hardcoded)

- **Verifies:** AC1, AC6
- **Precondition:** Two context.yml fixtures with different `channel_hints.approval.issueUrl` values
- **Action:** Call routing function for each context; capture the target URL in each notification
- **Expected result:** Target URL in notification matches each context's `approval.issueUrl` respectively — no shared hardcoded URL
- **Edge case:** No

### Channel hint routing: approval_channel type value from context.yml selects adapter (github-issue selects github-issue adapter)

- **Verifies:** AC1, AC6
- **Precondition:** context.yml with `approval_channel: github-issue`
- **Action:** Call the channel adapter selector; inspect adapter name
- **Expected result:** `github-issue` adapter selected; not a fallback/manual adapter
- **Edge case:** No

### pipeline-state.json write: dorStatus set to "signed-off" on approval trigger

- **Verifies:** AC3
- **Precondition:** Stub `/approve-dor` event payload received; pipeline-state.json fixture loaded
- **Action:** Process the stub event (simulate GitHub Actions workflow trigger); inspect written pipeline-state.json
- **Expected result:** `dorStatus: "signed-off"` written to the relevant story's pipeline-state.json entry — same field as a VS Code sign-off (ADR-002 compliance)
- **Edge case:** No

### pipeline-state.json write: dorApprover stores username, not full name or email

- **Verifies:** AC3 (Security/PII constraint)
- **Precondition:** Stub event payload with `actorUsername: "hamish-dev"` and `actorEmail: "hamish@example.com"` in the payload
- **Action:** Process event; inspect `dorApprover` in written pipeline-state.json
- **Expected result:** `dorApprover: "hamish-dev"` (username); no email address or full name written to pipeline-state.json
- **Edge case:** No

### pipeline-state.json write: dorChannel records the approval interface used

- **Verifies:** AC3 (auditability)
- **Precondition:** context.yml with `approval_channel: github-issue`; sign-off trigger fires
- **Action:** Inspect `dorChannel` in written pipeline-state.json after sign-off
- **Expected result:** `dorChannel: "github-issue"` written; field is non-null and reflects the configured channel type
- **Edge case:** No

### IDE channel regression: VS Code notification emitted when both channels configured

- **Verifies:** AC4
- **Precondition:** context.yml with `channel_hints.ide: vscode` AND `channel_hints.approval: { type: "github-issue", ... }`
- **Action:** Call the notification dispatch function; inspect dispatch call log
- **Expected result:** Both IDE (VS Code) notification AND approval channel (github-issue) notification are dispatched; neither suppresses the other
- **Edge case:** No

### IDE channel regression: IDE notification is not suppressed when approval_channel is set

- **Verifies:** AC4 (non-regression)
- **Precondition:** context.yml has both `channel_hints.ide: vscode` and an approval channel
- **Action:** Call dispatch function; count notifications emitted
- **Expected result:** Exactly 2 notifications dispatch calls made (one per channel); IDE notification is present; no errors
- **Edge case:** No

### Unconfigured approval channel: configuration warning logged to output

- **Verifies:** AC5
- **Precondition:** context.yml with no `channel_hints.approval` key
- **Action:** Call the notification dispatch function; capture output log
- **Expected result:** Log output contains "No approval channel configured — DoR sign-off defaulting to IDE." (or equivalent per AC5 wording); no exception thrown
- **Edge case:** No

### Unconfigured approval channel: IDE notification still sent (no error, no blocked state)

- **Verifies:** AC5 (no blocking)
- **Precondition:** As above
- **Action:** Call dispatch function; inspect dispatch calls and exit state
- **Expected result:** IDE notification dispatched; function exits without error; process does not enter an error state; sign-off is not blocked
- **Edge case:** No

### Context.yml single source of truth: routing target differs when context differs (no shared constant)

- **Verifies:** AC6 (no hardcoded values)
- **Precondition:** Two fixtures: context A with `channel_hints.approval.issueUrl: "https://a.example/1"`, context B with `channel_hints.approval.issueUrl: "https://b.example/2"`
- **Action:** Run routing for each context; compare notification target URLs
- **Expected result:** Target URLs differ and exactly match each context's value; no constant URL appears in any case
- **Edge case:** No

### Hardcoded value scan: no channel URL, workspace ID, or project key in skill SKILL.md or viz JS

- **Verifies:** AC6 (static check)
- **Precondition:** Implementation committed; SKILL.md for persona routing in `.github/skills/`; viz JS file
- **Action:** Scan SKILL.md and viz JS source for any `https://` URLs, Jira project keys, Slack workspace IDs, or Teams channel IDs that are constants rather than template variables or config reads
- **Expected result:** No hardcoded channel targets found; all channel references are config reads from context.yml
- **Edge case:** No

## NFR Tests

### Security: dorApprover field in pipeline-state.json for all test fixtures contains no PII

- **NFR addressed:** Security (no PII in pipeline-state)
- **Measurement method:** Inspect `dorApprover` values written by all AC3 tests; verify none match email-address pattern (`@`-containing string) or full-name pattern; only username format permitted
- **Pass threshold:** No `dorApprover` value contains `@` or a space (which would indicate full name); all values are username-format strings
- **Tool:** String pattern check in test script

### Consistency: channel_hints.approval schema values match approval_channel adapter enum

- **NFR addressed:** Consistency (MC-CONSIST-02)
- **Measurement method:** Read the `approval_channel` enum from `pipeline-state.schema.json` (or context schema); verify `channel_hints.approval.type` valid values match the adapter enum values (`github-issue`, `jira-transition`, `confluence-comment`, `slack-reaction`, `manual`)
- **Pass threshold:** Schema enum values and adapter enum values are identical sets
- **Tool:** Schema comparison in test script

### Auditability: dorChannel field cannot be overwritten by a second sign-off attempt

- **NFR addressed:** Auditability (dorChannel permanent record)
- **Measurement method:** Write `dorChannel: "github-issue"` via sign-off; attempt a second write via a different channel simulation (`manual`); read pipeline-state.json
- **Pass threshold:** `dorChannel` retains the original value `"github-issue"` — second write is rejected or a new sign-off creates a new record rather than overwriting
- **Tool:** Two-write sequence test in integration script

## Out of Scope for This Test Plan

- Full automated webhook pipeline from GitHub Issues to pipeline-state.json (GitHub Actions live test) — Phase 2 uses stub event processing.
- Bi-directional sync between pipeline-state.json and non-engineer tools.
- Multi-approver quorum logic.
- Microsoft Teams bot or Slack app OAuth registration.
- Channel integrations other than github-issue (jira-transition, confluence-comment, slack-reaction are Phase 3 — verified via stub/adapter interface test only, not full integration tests).
