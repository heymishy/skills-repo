## Unit Tests

### SaaS-GUI adapter: execute() returns object with status, surface, findings, trace fields

- **Precondition:** `src/surface-adapter/adapters/saas-gui.js` implemented
- **Action:** Call `execute("saas-gui", minimalContext)`
- **Expected result:** Returned object has `status` (string), `surface: "saas-gui"` (exact value), `findings` (array), `trace` (non-null)
- **Edge case:** No

### SaaS-GUI adapter: result.status is one of "pass", "fail", "error"

- **Verifies:** AC1 (P1.2 interface constraint)
- **Precondition:** As above
- **Action:** Inspect `result.status`
- **Expected result:** Value is exactly one of the three permitted strings
- **Edge case:** No

### Manual adapter: execute() result contains resultPattern: "checklist"

- **Verifies:** AC3
- **Precondition:** Manual adapter implemented; `standards/manual/POLICY.md` stub present
- **Action:** Call `execute("manual", minimalContext)`
- **Expected result:** `result.resultPattern === "checklist"`
- **Edge case:** No

### Manual adapter: findings array contains checklist item objects (item + status structure)

- **Verifies:** AC3
- **Precondition:** As above; context includes at least one checklist item
- **Action:** Inspect `result.findings[0]`
- **Expected result:** Finding object has `item` (string) and `status` (one of: `"complete"`, `"incomplete"`) — not code-diff or static-analysis fields
- **Edge case:** No

### Manual adapter: findings do NOT contain code-diff or static-analysis finding fields

- **Verifies:** AC3 (negative check)
- **Precondition:** Manual adapter result
- **Action:** Inspect finding objects for diff-analysis fields (e.g. `file`, `line`, `rule`, `severity` in a static-analysis sense)
- **Expected result:** No finding object contains diff-analysis-specific fields; only `item` and `status` are present
- **Edge case:** No

### Manual POLICY.md floor: contains no criterion with diff-analysis or code-review keywords

- **Verifies:** AC5
- **Precondition:** `standards/manual/POLICY.md` stub committed
- **Action:** Read the POLICY.md floor content; scan for keywords: "diff", "code review", "pull request", "lint", "static analysis", "source code"
- **Expected result:** No diff-analysis or code-review criteria present; only checklist-type governance criteria (e.g. "Change request approved", "Evidence file attached")
- **Edge case:** No — this is a hard requirement, failing this is a test failure per AC5

### Manual POLICY.md floor: contains at least one checklist-type governance criterion

- **Verifies:** AC5 (affirmative)
- **Precondition:** `standards/manual/POLICY.md` stub committed
- **Action:** Read the POLICY.md floor content; check for at least one criterion in checklist format
- **Expected result:** At least one criterion exists; floor is not empty; criteria are checklist-pattern
- **Edge case:** No

### standards/index.yml routing lookup: saas-gui and m365-admin surface types resolve to their POLICY.md files

- **Verifies:** AC4
- **Precondition:** `standards/index.yml` updated; stub POLICY.md files present
- **Action:** Resolve routing for `surface: "saas-gui"` and `surface: "m365-admin"` via the adapter routing function
- **Expected result:** Both lookups return a non-null path pointing to the respective POLICY.md stub file; the file at that path exists
- **Edge case:** No

### Unknown surface type: execute() returns error result with error field naming the surface type

- **Verifies:** AC6
- **Precondition:** All 5 adapters registered; `surface: "unknown-type"` not in registry
- **Action:** Call `execute("unknown-type", minimalContext)` on the adapter framework
- **Expected result:** `{ status: "error", surface: "unknown-type", findings: [], trace: "<identifier>", error: "unknown surface type: unknown-type" }` — structure matches exactly; no exception thrown
- **Edge case:** Yes — boundary/guard case

### Unknown surface type: status is "error", findings is empty array (not null)

- **Verifies:** AC6 (structure check)
- **Precondition:** As above
- **Action:** Inspect `result.status` and `result.findings`
- **Expected result:** `result.status === "error"`; `result.findings` is an empty array `[]`, not `null` or `undefined`
- **Edge case:** No

## NFR Tests

### Performance: execute() for each new adapter completes in ≤5s

- **NFR addressed:** Performance
- **Measurement method:** Timer wrapper around each execute() call for SaaS-GUI, M365-admin, and manual adapters
- **Pass threshold:** ≤ 5000ms per call
- **Tool:** Node.js timing assertion

### Security: no external network calls made during execute() for SaaS-GUI and M365-admin adapters (MC-SEC-03)

- **NFR addressed:** Security
- **Measurement method:** Stub DNS/HTTP at network layer; verify both adapters complete without attempting any network request; check stdout/stderr for credential values
- **Pass threshold:** No network calls attempted; no credential values in output
- **Tool:** In-test network stub + stdout capture

### Auditability: resultPattern: "checklist" field present in pipeline-state.schema.json before manual adapter writes it

- **NFR addressed:** Auditability / Schema-first (ADR-003)
- **Measurement method:** Read `pipeline-state.schema.json`; confirm `resultPattern` field with `"checklist"` as a valid enum value is declared; this file must be committed in the same commit as the manual adapter
- **Pass threshold:** Field present in schema; adapter commit does not pre-date schema commit
- **Tool:** File read assertion in test script
