## Contract Proposal

### What will be built

- **Bitbucket Cloud YAML syntax validator**: A validation function that parses `bitbucket-pipelines.yml` — accepts well-formed files and rejects files with YAML syntax errors, returning a typed error distinguishing syntax errors from pipeline-shape errors.
- **Bitbucket Cloud pipeline-shape validator**: A check that detects whether the governance assurance gate step is present in `bitbucket-pipelines.yml`. Fails with a named failure output ("assurance gate step missing") for syntactically valid files that are missing the required step — no false passes.
- **Bitbucket DC Docker auth topology tests** (three variants): (a) app-password auth — no 401/403; (b) OAuth consumer auth — token obtained; (c) SSH key auth — no SSH handshake failure. Each test confirms the respective auth mechanism resolves against a local Bitbucket DC Docker instance. All auth secrets injected via CI secrets — never committed.
- **Cloud/DC isolation**: Cloud and DC validation paths are independently executable with no shared runtime state. No cross-contamination in either direction.
- **Docker Compose deliverable** (AC7): `tests/fixtures/docker-compose.bitbucket-dc.yml` committed as a reproducible local Bitbucket DC Docker environment. Scaffolds all three auth topologies. No credentials in the file — all secret values referenced as environment variables. Any developer with Docker installed can `docker compose up` and run DC topology tests locally.

### What will NOT be built

- Bitbucket Pipelines native enterprise environment (non-Docker DC) — Docker DC is the Phase 2 path
- Jenkins CI validation — not in Phase 2 scope
- GitHub Actions regression testing — inherited from Phase 1 (P1.3); no re-validation
- Automated Bitbucket CI setup provisioning for new squads
