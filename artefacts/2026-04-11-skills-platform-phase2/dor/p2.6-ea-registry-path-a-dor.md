## Contract Proposal

### What will be built

- **Path A HTTP GET query** in `src/surface-adapter/resolver.js`: When `context.yml` contains `registry_source: ea-registry`, `registry_url`, and `app_id`, the resolver performs an HTTPS GET to the EA registry retrieving the application YAML entry. It reads `technology.hosting` to derive the skills-platform surface type via mapping table (`saas` → `saas-api`, `cloud` → `iac`, `on-prem` → `manual`). The `owner` field is read for team/squad identification metadata.
- **context.yml `adapter_override` support**: If `adapter_override` is set in `context.yml` (non-null, non-empty), it takes precedence over the registry-derived type — enabling `saas-gui` and `m365-admin` resolution without a registry lookup.
- **Path A fallback mechanism**: On registry unavailability (timeout ≤3s, HTTP 5xx, DNS failure), the resolver falls back to an explicit `surface:` declaration in `context.yml` if present, or returns an explicit error result (no silent git-native default).
- **Path B unchanged**: `context.yml` with `surface:` only is entirely unaffected — no HTTP call attempted, no `app_id` required.
- **Both paths present guard**: Warning logged when both `registry_source` and `surface:` are declared; Path B wins.

### What will NOT be built

- Writing to the EA registry (read-only) — squads register via the EA registry PR process
- EA registry response caching (Phase 3 performance optimisation)
- EA registry authentication/bearer token (Phase 2 assumes public read endpoint) — flagged as Phase 3 NFR
- Handling unknown `technology.hosting` values beyond `saas`, `cloud`, `on-prem`, `hybrid` — these are treated as errors (falls through to Path B or explicit error)
- M365-admin or SaaS-GUI resolver logic — those are specified via `context.yml` `adapter_override`, not derived from registry fields
