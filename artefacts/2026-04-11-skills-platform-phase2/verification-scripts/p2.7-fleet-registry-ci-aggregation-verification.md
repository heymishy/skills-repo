# AC Verification Script: Fleet registry files and CI-aggregated fleet-state.json (p2.7)

**Story reference:** artefacts/2026-04-11-skills-platform-phase2/stories/p2.7-fleet-registry-ci-aggregation.md
**Date:** 2026-04-11

---

## Overview

Fleet registry uses per-squad `fleet/squads/{id}.json` files aggregated by a CI job into `fleet-state.json`. The viz fleet panel reads `fleet-state.json` and renders squad cards. Key guards: squad files must be validated at commit/CI; aggregation must degrade gracefully for unreachable squads; viz must convey health via text/icon in addition to colour (MC-A11Y-02).

---

## Scenarios

### Scenario 1 — Squad file validation: missing required field rejected with named error

**AC:** AC1

**Steps:**
1. Create a `fleet/squads/test-squad.json` with the `squadId` field removed.
2. Commit the file or run the validation check manually.
3. Observe the validation output.

**Expected result:** Validation fails. Error message includes "squadId" as the missing field name. File is rejected (commit blocked or CI check fails). A valid file with all four fields (`squadId`, `repoUrl`, `pipelineStateUrl`, `registeredAt`) passes without error.

**Pass/Fail:** Pass if incomplete file is rejected with field-specific error; valid file passes.

---

### Scenario 2 — CI aggregation: fleet-state.json written with one entry per squad

**AC:** AC2, AC3

**Steps:**
1. Create two squad fixture files (e.g. `squad-a.json`, `squad-b.json`) with `registry_mode: publishing` and valid `pipelineStateUrl` values pointing to stub pipeline-state.json files.
2. Run the CI aggregation job.
3. Read `fleet-state.json`.

**Expected result:** `fleet-state.json` has a `squads` array with exactly 2 entries. Each entry contains: `squadId` (matching the squad file), `stage` (from the fetched pipeline-state.json), `health` (green/amber/red), `updatedAt` (ISO datetime from the aggregation run), `sourceUrl` (matching `pipelineStateUrl` from squad file).

**Pass/Fail:** Pass if all five fields present in each entry with correct values.

---

### Scenario 3 — registry_mode filtering: publishing included, none excluded

**AC:** AC4

**Steps:**
1. Create three squad fixture files: one with `registry_mode: publishing`, one with `registry_mode: none`, one with no `registry_mode` key.
2. Run CI aggregation.
3. Check which squads appear in `fleet-state.json`.

**Expected result:** Only the `registry_mode: publishing` squad appears in `squads[]`. The `none` squad and the squad with no `registry_mode` are absent. Aggregation completes without error.

**Pass/Fail:** Pass if exactly the `publishing` squad is included.

---

### Scenario 4 — Fleet viz panel renders ≥2 squad cards

**AC:** AC5

**Steps:**
1. Ensure `fleet-state.json` has ≥2 squad entries.
2. Open the pipeline-viz fleet panel in a browser (or run `check-viz-behaviour.js`).
3. Inspect the rendered squad cards.

**Expected result:** ≥2 squad cards visible. Each card displays: `squadId`, current `stage`, `health` indicator, and `updatedAt`. No manual data entry was required after the CI job ran.

**Pass/Fail:** Pass if ≥2 cards rendered with all four data fields.

---

### Scenario 5 — Fleet viz panel health indicator uses text/icon alongside colour (MC-A11Y-02)

**AC:** AC5 (accessibility guard)

**Steps:**
1. Render the fleet panel with at least one squad in amber health and one in red health.
2. Inspect each squad card's health display.
3. Check that health state is conveyed by a text label or icon — not by colour class alone.

**Expected result:** Each card's health section contains a visible text label (e.g. "Amber", "Warning", "Healthy") or an icon element alongside the colour indicator. Removing CSS colour classes from the element does not make the health state invisible or ambiguous.

**Pass/Fail:** Pass if health state is identifiable without colour.

---

### Scenario 6 — Graceful degradation: unreachable squad gets health:"unknown"; job continues

**AC:** AC6

**Steps:**
1. Create three squad fixture files: one where `pipelineStateUrl` points to an HTTP 404 endpoint, two with valid stub responses.
2. Run CI aggregation.
3. Inspect `fleet-state.json` and aggregation exit code.

**Expected result:** `fleet-state.json` has 3 entries. The unreachable squad has `health: "unknown"` and a non-empty `error` field. The two reachable squads have correct live state entries. Aggregation job exit code = 0 (job did not abort).

**Pass/Fail:** Pass if all three conditions met: unknown entry exists, valid entries exist, exit code = 0.

---

### Scenario 7 — npm test passes: check-viz-behaviour.js fleet panel cases green

**AC:** AC5, AC6

**Steps:**
1. Run `npm test`.
2. Review the `[check-viz-behaviour]` output section for fleet panel test cases.

**Expected result:** Fleet panel test cases pass (≥2 cards rendered, A11Y text/icon present). Suite exits 0.

**Pass/Fail:** Pass if `[check-viz-behaviour]` fleet panel cases green and overall exit code = 0.
