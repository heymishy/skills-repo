## Scenario 1: queryTraces filters return only matching traces

**AC covered:** AC1

**Pre-conditions:**
- `src/improvement-agent/trace-interface.js` committed
- At least 5 synthetic trace fixtures in workspace/traces/

**Steps:**
1. Call `queryTraces({ storySlug: "story-a" })` with 5 fixtures (3 for story-a, 2 for story-b)
2. Count results
3. Repeat with `patternLabel` filter
4. Repeat with `dateRange` filter

**Expected outcome:** Each call returns only the matching subset. No traces from outside the filter criteria appear in results.

**Pass/Fail criteria:**
- PASS: All filter types work; only matching traces returned
- FAIL: Extra traces returned; any filter returns all traces regardless of filter value

## Scenario 2: Failure signal detection triggers diff proposal on 3+ occurrences

**AC covered:** AC2

**Pre-conditions:**
- 10 synthetic trace fixtures for one surface type, ≥3 with same kebab-case pattern label in the failurePattern field

**Steps:**
1. Run improvement agent failure signal detection
2. Check `workspace/proposals/` for a new `*-failure-proposal.md` file
3. Open the proposal file; verify it references the 3 trace IDs
4. Run with only 2 matching occurrences; confirm no proposal written

**Expected outcome:**
- With 3 occurrences: proposal file written; trace IDs referenced as evidence
- With 2 occurrences: no proposal file written

**Pass/Fail criteria:**
- PASS: Threshold boundary observed exactly
- FAIL: Proposal written at <3 occurrences (false positive); not written at ≥3 (false negative)

## Scenario 3: Staleness signal detection triggers staleness proposal after 90-day inactivity

**AC covered:** AC3

**Pre-conditions:**
- Synthetic trace fixture with `createdAt` ≥90 days ago for a given surface type
- `improvement_agent.staleness_window_days: 90` set in context.yml (or default)

**Steps:**
1. Run improvement agent staleness scan
2. Check workspace/proposals/ for a `*-staleness-proposal.md` file
3. Open the file; verify it names the surface type, last activity date, and a recommendation (defer or remove)

**Expected outcome:** Staleness proposal file present; contains surface type, last activity date, recommendation.

**Pass/Fail criteria:**
- PASS: File present with required content
- FAIL: No file; file missing any required element; staleness scan errors

## Scenario 4: Anti-overfitting gate blocks proposals that remove currently-passing checks

**AC covered:** AC4

**Pre-conditions:**
- Proposal would remove or weaken a MUST statement from a SKILL.md
- Current 10-trace window shows that check as PASS in all traces

**Steps:**
1. Run improvement agent; trigger a proposal that would remove a passing check
2. Check workspace/proposals/ — confirm no `*-failure-proposal.md` written for this check
3. Confirm an `*-overfitting-warning.md` file is written instead
4. Read the warning; verify it contains: check label, trace count, "Challenger pre-check required" note

**Expected outcome:** Proposal blocked; overfitting-warning file created with all required elements.

**Pass/Fail criteria:**
- PASS: Proposal blocked; warning file present with correct content
- FAIL: Proposal written despite removing a passing check; no warning file written

## Scenario 5: Proposal file contains all 6 required fields

**AC covered:** AC5

**Pre-conditions:**
- Any proposal file generated by Scenario 2 or 3

**Steps:**
1. Open a generated proposal file
2. Check for: `evidence`, `proposed_diff`, `confidence`, `anti_overfitting_gate`, `status`, `created_at`
3. Verify `status` equals `pending_review`
4. Verify `created_at` is a valid ISO datetime
5. Verify `confidence` is one of `high`, `medium`, `low` with a one-line rationale string

**Expected outcome:** All 6 fields present; values are correctly typed and populated.

**Pass/Fail criteria:**
- PASS: All 6 fields present; correctly typed
- FAIL: Any field missing; `status` != `pending_review`; `confidence` not in permitted set; `created_at` not ISO format

## Scenario 6: Integration — real Phase 2 delivery traces trigger failure proposal (EXECUTION GATE)

> ⛔ **BLOCKED until Phase 2 inner loop gate satisfied.** Skip this scenario until gate condition above is met.

**AC covered:** AC2 (integration, M3 sub-condition evidence)

**Pre-conditions:**
- Phase 2 inner loop gate satisfied (≥1 Phase 2 story merged, trace written)
- ≥3 Phase 2 delivery traces contain the same failure pattern label within a 10-story sliding window
- ASSUMPTION-P2.11-01 confirmed (P1.6 trace field name validated from actual trace files)

**Steps:**
1. Confirm gate condition: `ls workspace/traces/` returns ≥1 file from a Phase 2 inner loop session
2. Confirm ASSUMPTION-P2.11-01: open a real trace file; identify the field carrying the failure pattern label; confirm it is `failurePattern` (string) or `failurePatterns[]` (array)
3. Run improvement agent failure signal detection against real traces
4. Open the generated proposal file
5. Confirm proposal references real trace IDs (not synthetic)

**Expected outcome:** Diff proposal written from real delivery signal; real trace IDs in evidence; M3 sub-condition (2) satisfied.

**Pass/Fail criteria:**
- PASS: Proposal written; references real trace IDs; M3 sub-condition (2) observable in proposal file
- FAIL: Proposal produced from synthetic data; traces not from Phase 2 inner loop sessions; ASSUMPTION-P2.11-01 confirmed wrong (field name mismatch — stop and raise DoR finding)

**AC covered:** AC6

**Steps:**
