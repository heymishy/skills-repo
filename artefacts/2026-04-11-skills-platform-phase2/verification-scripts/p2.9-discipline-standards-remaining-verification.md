# AC Verification Script: Eight remaining discipline standards and three pilot domain standards (p2.9)

**Story reference:** p2.9-discipline-standards-remaining
**Template:** artefacts templates/ac-verification-script.md
**Date:** 2026-04-11

---

## Purpose

Plain-language human-review checklist. Run before implementation begins (confirm failures expected) and as a smoke check after the PR merges. Complements the technical test plan — does not replace it.

---

## Scenario 1: Eight new discipline entries appear in standards/index.yml

**AC covered:** AC1

**Pre-conditions:**
- Implementation PR merged or under review

**Steps:**
1. Open `standards/index.yml`
2. Find the `disciplines:` key
3. Count total entries
4. List the slugs present

**Expected outcome:** Exactly 11 discipline entries total. The 8 new slugs are present: `data`, `ux`, `security-extended`, `regulatory`, `infrastructure`, `devops`, `product`, `ml-ai`. The original 3 (`software-engineering`, `security-engineering`, `quality-assurance`) present with unchanged paths.

**Pass/Fail criteria:**
- PASS: All 11 slugs present; original 3 unchanged
- FAIL: Fewer than 11; any original entry missing or path changed

## Scenario 2: Each new discipline has core.md and POLICY.md at the declared path

**AC covered:** AC2

**Pre-conditions:**
- All 8 new discipline file pairs committed
- Implementation branch checked out

**Steps:**
1. For each of the 8 new discipline slugs, read the path from `standards/index.yml`
2. Navigate to the declared path; verify `core.md` exists
3. Navigate to the declared path; verify `POLICY.md` exists
4. Open one core.md and one POLICY.md and check for required headings: `## Overview`, `## Outcomes`, `## Requirements`, `## Out of Scope for this standard`
5. Check that at least one MUST/SHOULD/MAY statement appears and describes an outcome

**Expected outcome:** All 16 files exist; all required headings present; outcome-oriented language used in requirement statements.

**Pass/Fail criteria:**
- PASS: All files exist; all headings present; at least one outcome-oriented requirement per file
- FAIL: Any file missing; any heading absent; no outcome statement

## Scenario 3: domains: key has ≥3 pilot domain entries with POLICY.md files

**AC covered:** AC3

**Pre-conditions:**
- `standards/index.yml` updated with `domains:` key

**Steps:**
1. Open `standards/index.yml`; find the `domains:` key
2. Count entries under `domains:`
3. For each domain entry, check if a `POLICY.md` path is declared
4. Verify each declared POLICY.md file exists on disk

**Expected outcome:** ≥3 domain entries; each has a `POLICY.md` path declared; all POLICY.md files exist.

**Pass/Fail criteria:**
- PASS: ≥3 domains; all POLICY.md files exist
- FAIL: Fewer than 3 domains; any POLICY.md missing; `domains:` key absent

## Scenario 5: No hardcoded tool product names in new standard files

**AC covered:** AC6

**Pre-conditions:**
- All new standard files committed

**Steps:**
1. Open 3 of the new standard files (e.g. `standards/devops/POLICY.md`, `standards/ml-ai/POLICY.md`, `standards/regulatory/POLICY.md`)
2. Search for tool product names: "Terraform", "Jenkins", "Jira", "GitHub", "Azure DevOps", "Datadog", "Splunk", "Sonar"
3. Check whether any appear as hard assumptions rather than placeholders

**Expected outcome:** No tool name appears as a literal assumption; any tool reference uses `[FILL IN: <description>]` or equivalent placeholder syntax.

**Pass/Fail criteria:**
- PASS: No hardcoded tool names found; all references use placeholders
- FAIL: Any tool product name appears as a literal assumption (not a placeholder)

## Scenario 6: Existing standards tests remain passing after extension (A6 check)

**AC covered:** AC5, AC1 (non-breaking)

**Pre-conditions:**
- Baseline npm test run available before extension
- Extension PR merged or on branch

**Steps:**
1. Run `npm test` on the extension branch
2. Compare `[check-standards-model]` pass/fail status for `software-engineering`, `security-engineering`, `quality-assurance` against the baseline
3. Confirm no test file modifications were required for existing entries

**Expected outcome:** All 3 original discipline routing tests pass unchanged; no test file edits were needed; additive-only change confirmed.

**Pass/Fail criteria:**
- PASS: All 3 existing tests pass; no test changes needed
- FAIL: Any of the 3 fails; test modifications were required to keep them passing
