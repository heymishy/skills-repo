## Purpose

Plain-language human-review checklist. Run before implementation begins (to confirm expected failures) and as a smoke check after the PR merges. Complements the technical test plan — does not replace it.

## Prerequisite note

Scenarios 3–5 and 7 require Docker to be running and the `tests/fixtures/docker-compose.bitbucket-dc.yml` file to be committed (AC7 deliverable). If Docker is unavailable, mark those scenarios as `[SKIPPED — PREREQ-DOCKER]` and verify them in CI.

## Scenario 1: Bitbucket Cloud YAML syntax validation passes for a well-formed pipeline

**AC covered:** AC1

**Pre-conditions:**
- Implementation branch checked out
- A well-formed `bitbucket-pipelines.yml` fixture with the assurance gate step is available

**Steps:**
1. Run the YAML syntax validator against the well-formed fixture
2. Inspect the output

**Expected outcome:** No parse error; validator reports valid. No assurance-gate-missing error.

**Pass/Fail criteria:**
- PASS: Valid result returned; no error
- FAIL: Error returned; validator always-passes (does not differentiate)

---

## Scenario 2: Bitbucket Cloud pipeline-shape test detects missing assurance gate step

**AC covered:** AC2

**Pre-conditions:**
- A `bitbucket-pipelines.yml` fixture that is valid YAML but does NOT contain the assurance gate step

**Steps:**
1. Run the pipeline-shape check against the deficient fixture
2. Inspect the output

**Expected outcome:** Shape check fails with a named error identifying the missing assurance gate step. Does NOT report pass.

**Pass/Fail criteria:**
- PASS: Named failure reported identifying the missing step
- FAIL: Passes (false pass); fails with a different error that does not name the assurance gate

## Scenario 3: Bitbucket DC Docker — app-password auth topology test (Docker-gated)

**AC covered:** AC3

**Pre-conditions:**
- Docker running
- `docker compose -f tests/fixtures/docker-compose.bitbucket-dc.yml up -d` executed successfully
- CI secrets `APP_PASSWORD` and `BB_USER` available or equivalent local env vars set

**Steps:**
1. Run the DC app-password auth topology test
2. Inspect the HTTP response or auth result

**Expected outcome:** Auth topology test passes; response status is not 401 or 403; test reports topology as valid.

**Pass/Fail criteria:**
- PASS: No 401/403; auth topology confirmed
- FAIL: 401/403 returned; test errors at host-not-found (external network dependency revealed)

## Scenario 4: Bitbucket DC Docker — OAuth consumer auth topology test (Docker-gated)

**AC covered:** AC4

**Pre-conditions:**
- Docker running; DC instance up (as above)
- CI secrets `OAUTH_KEY` and `OAUTH_SECRET` available

**Steps:**
1. Run the DC OAuth consumer auth topology test
2. Check whether a token was obtained

**Expected outcome:** Token obtained; no auth error.

**Pass/Fail criteria:**
- PASS: Token obtained; test reports OAuth topology valid
- FAIL: Token not obtained; OAuth error returned; external network dependency revealed

## Scenario 5: Bitbucket DC Docker — SSH key auth topology test (Docker-gated)

**AC covered:** AC5

**Pre-conditions:**
- Docker running; DC instance up
- CI secret `SSH_PRIVATE_KEY` available

**Steps:**
1. Run the DC SSH key auth topology test
2. Check SSH handshake result

**Expected outcome:** SSH handshake completes without ConnReset, AuthFail, or PermissionDenied error; topology test reports valid.

**Pass/Fail criteria:**
- PASS: No SSH handshake failure; topology confirmed
- FAIL: SSH failure error returned; external SSH dependency revealed

## Scenario 6: Cloud and DC paths run independently without cross-contamination

**AC covered:** AC6

**Pre-conditions:**
- Implementation committed
- Test harness can run individual paths

**Steps:**
1. Run Cloud path only — DC credentials/env vars intentionally absent
2. Confirm Cloud path completes cleanly; inspect for DC auth calls
3. Run DC path only — Cloud credentials/env vars absent
4. Confirm DC path completes cleanly; inspect for Cloud API calls

**Expected outcome:** Each path runs cleanly without the other's credentials. No cross-contamination errors (e.g. Cloud path does not fail because DC auth is absent; DC path does not fail because Cloud token is absent).

**Pass/Fail criteria:**
- PASS: Both paths independently clean; zero cross-calls
- FAIL: Either path requires the other's credentials to run; cross-calls detected

---

## Scenario 7: docker-compose.bitbucket-dc.yml exists, reproduces DC locally, no external network required (Docker-gated)

**AC covered:** AC7

**Pre-conditions:**
- Docker running (no internet access ideally, or firewall rule blocking external DC)
- `tests/fixtures/docker-compose.bitbucket-dc.yml` committed

**Steps:**
1. Run `docker compose -f tests/fixtures/docker-compose.bitbucket-dc.yml up -d`
2. Confirm DC services start without connection errors
3. Run DC auth tests against local DC instance
4. Confirm tests reach auth topology evaluation stage (not network-resolution failure)
5. Run `docker compose -f tests/fixtures/docker-compose.bitbucket-dc.yml down`
6. Confirm no credential literals appear in the docker-compose.yml file (only env-var references)

**Expected outcome:** DC instance starts. Auth tests execute locally. No "host not found" or "connection refused to external host" errors. No credential literals in the file.

**Pass/Fail criteria:**
- PASS: All steps complete; no external network dependency; no committed credentials
- FAIL: Any DC test fails at DNS resolution for external host; credential literal found in compose file; `docker compose up` fails
