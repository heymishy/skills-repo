## Scenario 1: Challenger spec generation produces two required files

**AC covered:** AC1

**Pre-conditions:**
- A proposal file with `status: pending_review` exists in workspace/proposals/

**Steps:**
1. Run the challenger pre-check preparation step for the proposal
2. Check workspace/proposals/ for `[proposal-id]-challenger-spec.md`
3. Check workspace/proposals/ for `[proposal-id]-proposed-skill.md`
4. Open challenger-spec.md; confirm it contains User Story, ACs, and Done condition sections
5. Open proposed-skill.md; compare structure to the target SKILL.md

**Expected outcome:** Both files written. challenger-spec.md has all three required sections. proposed-skill.md has the same section structure as the target SKILL.md with only the proposed section content changed. No credentials, org names, or personal data in either file.

**Pass/Fail criteria:**
- PASS: Both files written; all required sections present; no sensitive data
- FAIL: Either file missing; User Story or Done condition section absent; proposed-skill.md has different heading structure

## Scenario 2: Pre-check result recording — all 5 required fields present

**AC covered:** AC2

**Pre-conditions:**
- Challenger spec generated (Scenario 1 complete)
- A completed challenger-result.md fixture is available with verdict, session_summary, traces_produced, reviewer, reviewed_at

**Steps:**
1. Record a pre-check result to `[proposal-id]-challenger-result.md`
2. Open the file; verify all 5 fields are present
3. Verify `reviewer` is a named human identity (not `github-actions`, `copilot`, or any CI runner name)
4. Remove one field and re-run the improvement agent; confirm it flags the incomplete result file

**Expected outcome:** Complete result passes validation. Incomplete result is flagged with the name of the missing field.

**Pass/Fail criteria:**
- PASS: Complete result validated; incomplete result flagged by name; CI job ID rejected as reviewer
- FAIL: Incomplete result silently accepted; reviewer field accepts CI job ID

## Scenario 3: Proposal acceptance — SKILL.md updated, status changed, single commit (GATE APPLIES)

> ⛔ **Requires real proposal from p2.11 + p2.11 DoD-complete.**

**AC covered:** AC3

**Pre-conditions:**
- Phase 2 traces available; p2.11 DoD-complete
- A real proposal with `status: pending_review`; challenger-result.md with `verdict: pass`

**Steps:**
1. Record a passing pre-check result
2. Accept the proposal via the workflow
3. Open the target SKILL.md; confirm the proposed section is now live
4. Open the proposal file; confirm `status: accepted`, `accepted_at`, and `accepted_by` present
5. Run `git log --name-only -1`; confirm the commit message contains the proposal ID and both files appear in the same commit diff

**Expected outcome:** SKILL.md updated. Proposal status = accepted. Commit message references proposal ID. Both changes in one commit.

**Pass/Fail criteria:**
- PASS: All five confirmations above pass
- FAIL: SKILL.md not updated; status not accepted; commit message missing proposal ID; two commits instead of one

## Scenario 4: Proposal rejection — SKILL.md unchanged, mandatory rationale enforced

**AC covered:** AC4

**Pre-conditions:**
- A real proposal with `status: pending_review`

**Steps:**
1. Attempt to reject the proposal WITHOUT providing a rationale (empty string or absent field)
2. Confirm the workflow blocks the rejection with an error message
3. Reject again WITH a rationale string
4. Open the target SKILL.md; confirm it is unchanged from pre-rejection state
5. Read the proposal file; confirm `status: rejected`, `rejected_at`, `rejected_by`, `rejection_rationale` all present

**Expected outcome:**
- Empty rationale attempt: blocked by workflow with named error
- Valid rejection: SKILL.md unchanged; all three rejection fields present; no SKILL.md commit

**Pass/Fail criteria:**
- PASS: Both rejection paths behave correctly
- FAIL: Empty rationale accepted; SKILL.md changed on rejection; any of the three fields missing

## Scenario 5: Proposal deferral — deferred_until and deferral_reason set; deferred proposal suppressed

**AC covered:** AC5

**Pre-conditions:**
- A proposal with `status: pending_review`

**Steps:**
1. Defer the proposal with `deferred_until` = a future date + `deferral_reason` = "untimely — Phase 3 scope"
2. Open the proposal; confirm `status: deferred`, `deferred_until`, `deferral_reason` all present
3. Run improvement agent with current date < `deferred_until`; confirm proposal NOT re-surfaced
4. SKILL.md must be unchanged

**Expected outcome:** Proposal deferred with both fields. Not re-surfaced before deferred_until. SKILL.md unchanged.

**Pass/Fail criteria:**
- PASS: All three fields present; proposal suppressed before due date; SKILL.md unchanged
- FAIL: Deferred_until absent; proposal re-surfaced immediately; SKILL.md changed

**Pre-conditions:**

**Pass/Fail criteria:**

- FAIL: Any proposal targets the improvement-agent SKILL.md itself
