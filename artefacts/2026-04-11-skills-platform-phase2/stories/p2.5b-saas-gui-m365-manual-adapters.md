# Story: SaaS-GUI, M365-admin, and manual surface adapters with POLICY.md floor variants (p2.5b)

**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

> **Scope note:** p2.5 was split into p2.5a (IaC + SaaS-API) and p2.5b (SaaS-GUI + M365-admin + manual) per operator pre-authorisation. Decision recorded in artefacts/2026-04-11-skills-platform-phase2/decisions.md SCOPE-01.

## User Story

As a **platform maintainer**,
I want SaaS-GUI, M365-admin, and manual surface adapters implemented with their respective POLICY.md floor variants,
So that squads delivering UI-driven SaaS changes, Microsoft 365 admin configurations, and fully manual processes can run assurance checks against surface-appropriate governance standards — including checklist-based results for processes where no code diff exists.

## Benefit Linkage

**Metric moved:** M2 — Non-git-native surface adapter assurance verdict (partial delivery — SaaS-GUI, M365-admin, manual completes the full five-adapter set); M1 — Second squad outer loop unassisted
**How:** Together with p2.5a, this story delivers all five non-git-native adapters. When p2.5a and p2.5b are both merged, the M2 acceptance test becomes runnable for any of the five surface types. M1: SaaS-GUI, admin-panel, and manual-process squads currently cannot adopt the platform; this story removes those adoption blockers.

## Architecture Constraints

- SaaS-GUI and M365-admin adapters implement the same `execute(surface, context) → result` interface as p2.5a — interface signature unchanged
- The manual adapter's result object includes `resultPattern: "checklist"` as a declared field alongside the standard interface fields — this must be added to `pipeline-state.schema.json` in the same commit (ADR-003, MC-CORRECT-02)
- Manual adapter POLICY.md floor must contain checklist-only governance criteria — it must NOT contain any code-review or diff-analysis criteria (those would fail every manual process by definition); the floor is reviewed against this constraint at /review before DoR
- `standards/index.yml` must be updated for all three new surface types in the same commit as adapter registration (p1.7 pattern)
- MC-SEC-03: SaaS-GUI and M365-admin adapters read governance criteria against declared context only — they do not authenticate to, query, or screenshot any live SaaS or M365 tenant; evidence is provided by the human operator in `context.yml` or a referenced evidence file

## Dependencies

- **Upstream:** p1.2 (Phase 1, DoD-complete) — P1.2 interface contract
- **Upstream:** p2.5a — both stories register adapters into the same adapter registry; p2.5b should ideally be sequenced after p2.5a to avoid merge conflicts on the registry file, or the merge order must be agreed before both PRs are raised
- **Downstream:** p2.6 — EA registry Path A resolver requires all five adapter types registered; p2.6 depends on both p2.5a and p2.5b being DoD-complete

## Acceptance Criteria

**AC1 (SaaS-GUI adapter — result interface conformance):** Given `context.yml` declares `surface: saas-gui`, when `execute("saas-gui", context)` is called, then the SaaS-GUI adapter returns a result conforming to the P1.2 interface: `{ status: "pass" | "fail" | "error", surface: "saas-gui", findings: [...], trace: "<identifier>" }`.

**AC2 (M365-admin adapter — result interface conformance):** Given `context.yml` declares `surface: m365-admin`, when `execute("m365-admin", context)` is called, then the M365-admin adapter returns a result conforming to the P1.2 interface with `surface: "m365-admin"`.

**AC3 (manual adapter — checklist result pattern):** Given `context.yml` declares `surface: manual`, when `execute("manual", context)` is called, then the manual adapter returns a result with `resultPattern: "checklist"` alongside the standard interface fields — findings in the result represent checklist item completions (e.g. `{ item: "Change request approved", status: "complete" | "incomplete" }`), not code-diff or static-analysis findings.

**AC4 (SaaS-GUI and M365-admin POLICY.md floors):** Given `standards/index.yml` routing lookups with `surface: "saas-gui"` and `surface: "m365-admin"` respectively, when the POLICY.md floors are resolved, then surface-specific POLICY.md floor files exist for both surface types and are returned by the routing lookup — each containing at least one surface-appropriate governance criterion (SaaS-GUI: e.g. change-review approval, screenshot evidence reference; M365-admin: e.g. admin audit log reference, change ticket linkage).

**AC5 (manual POLICY.md floor — checklist-only criteria):** Given a `standards/index.yml` routing lookup with `surface: "manual"`, when the POLICY.md floor is resolved, then the manual adapter floor contains only checklist-type governance criteria — no code-review criteria, no diff-analysis criteria, and no criteria that assume the existence of a source code repository. Presence of any diff-analysis criterion in the manual floor is a test failure.

**AC6 (unknown surface type — error result, not silent failure):** Given all five surface adapter types (iac, saas-api, saas-gui, m365-admin, manual — registered across p2.5a and p2.5b) are in the adapter registry, when `execute()` is called with an unrecognised surface type (e.g. `surface: "unknown-type"`), then the adapter framework returns `{ status: "error", surface: "<declared type>", findings: [], trace: "<identifier>", error: "unknown surface type: <declared type>" }` — not a runtime exception, not a silent fall-through to git-native, and not a null result.

## Out of Scope

- IaC and SaaS-API adapter implementations — those are p2.5a
- Live authentication to any SaaS platform or M365 tenant — the adapters evaluate governance criteria against operator-declared context; they do not make authenticated calls to any external platform (MC-SEC-03)
- Screen-capture tooling or browser automation for SaaS-GUI evidence collection — evidence is operator-declared; the adapter evaluates the declaration, not the live screen
- M365 Graph API integration — the M365-admin adapter evaluates governance criteria (change approval, audit log reference) based on operator-declared context fields, not by calling Graph API

## NFRs

- **Performance:** Each adapter's `execute()` call must complete within 5 seconds (same as p2.5a budget)
- **Security:** No external network calls from inside `execute()` (MC-SEC-03); no credential values logged or persisted (MC-SEC-02)
- **Auditability:** Each adapter result's `trace` field must include adapter name and version; `resultPattern: "checklist"` for the manual adapter must appear in `pipeline-state.schema.json` before the adapter writes it (ADR-003)

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — interface contract defined by p1.2; pattern defined by p2.5a; main unknowns are the SaaS-GUI and M365-admin POLICY.md floor criteria definitions and the `resultPattern: "checklist"` schema addition
