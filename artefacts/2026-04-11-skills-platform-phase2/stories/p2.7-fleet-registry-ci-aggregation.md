## User Story

As a **platform maintainer**,
I want per-squad fleet registry files aggregated by a CI job into a single `fleet-state.json`,
So that I can see all squad states in the pipeline-viz fleet panel without querying each squad repo individually or making manual edits to a shared file.

## Benefit Linkage

**Metric moved:** M4 â€” Fleet observability: â‰Ą2 squad states visible without manual entry
**How:** The CI aggregation job is the mechanism that makes fleet observability automatic. Once â‰Ą2 squads have registered fleet files and the CI job runs, `fleet-state.json` is written with their live states and the viz renders them â€” satisfying all three M4 sub-conditions (CI-written file, viz rendering, no manual maintainer edits to `fleet-state.json`).

## Architecture Constraints

- ADR-003: `fleet/squads/{id}.json` schema and `fleet-state.json` schema must be defined in `pipeline-state.schema.json` (or a co-located schema file) before the CI job reads or writes them â€” schema-first rule applies to fleet files
- ADR-004: `registry_mode` and any fleet-related config (squad ID format, aggregation job settings) must live in `context.yml` â€” no hardcoded values in the CI job script or viz
- MC-CORRECT-02: any new fields in `fleet-state.json` or any `pipeline-state.json` extension for fleet status must exist in the schema before use
- MC-SEC-02: `fleet/squads/{id}.json` must not contain credentials, access tokens, or personal data â€” only public squad metadata and repo paths
- PAT-05: CI job reads aggregation config from `context.yml`, not from hardcoded constants
- A4 (discovery assumption): fleet files use `fleet/squads/{id}.json` per-squad pattern â€” a single shared `fleet-registry.json` is explicitly excluded as a merge-conflict sink

## Dependencies

- **Upstream:** None â€” first E3 story; no dependency on other Phase 2 stories (fleet registry is a new capability not gated by prior Phase 2 stories)
- **Downstream:** p2.8 (persona routing) depends on the fleet registry to identify the correct squad `pipeline-state.json` when an approval comes from outside the squad's VS Code environment

## Acceptance Criteria

**AC1 (squad file creation and validation):** Given a squad operator creates `fleet/squads/{id}.json` with the required fields (`squadId`, `repoUrl`, `pipelineStateUrl`, `registeredAt`), when the file is committed to the repo, then a pre-commit or CI validation check confirms the file conforms to the fleet squad schema â€” any file missing required fields is rejected with a descriptive error naming the missing field.

**AC2 (CI aggregation execution):** Given â‰Ą1 squad files exist in `fleet/squads/`, when the CI aggregation job runs (triggered on schedule or on push), then it reads each squad file, fetches or reads the referenced `pipeline-state.json` at `pipelineStateUrl`, and writes the combined state to `fleet-state.json` in a top-level `squads[]` array â€” one entry per squad.

**AC3 (fleet-state.json structure):** Given the CI aggregation job completes, when `fleet-state.json` is read, then each squad entry contains: `squadId`, `stage` (current feature stage from the squad's `pipeline-state.json`), `health` (green/amber/red), `updatedAt` (ISO datetime of the aggregation run), and `sourceUrl` (the `pipelineStateUrl` that was read) â€” matching the squad's live `pipeline-state.json` at the time the job ran.

**AC4 (registry_mode in context.yml):** Given a squad's `context.yml` has `registry_mode: publishing`, when the CI aggregation job runs, then that squad's state is included in `fleet-state.json`. If `registry_mode` is `none` or absent, the squad is excluded from the aggregation output and its `fleet/squads/{id}.json` file, if present, is ignored by the CI job.

**AC5 (portfolio-viz fleet panel rendering):** Given `fleet-state.json` exists with â‰Ą2 squad entries, when the pipeline-viz fleet panel is opened in a browser, then it displays â‰Ą2 squad cards â€” each showing `squadId`, current `stage`, `health` indicator, and `updatedAt` â€” without any manual data entry by the platform maintainer after the CI job ran.

**AC6 (graceful degradation):** Given a squad file in `fleet/squads/` references a `pipelineStateUrl` that is unreachable at aggregation time (network failure, private repo, 404), when the CI job runs, then the job continues processing remaining squads and records the unreachable squad with `health: "unknown"` and an `error` field describing the failure â€” the aggregation job does not abort, and reachable squads appear in `fleet-state.json` correctly.

## Out of Scope

- Cross-repo CI job triggering or federated aggregation â€” the CI job runs in the platform repo and reads squad files from within the same repo (or accessible URLs). Federated cross-organisation aggregation is Phase 3.
- Automated squad file creation at squad onboarding â€” squads create their own `fleet/squads/{id}.json` manually using the template. Automated provisioning is out of scope.
- Real-time fleet state updates â€” `fleet-state.json` is updated on CI job trigger (scheduled or push), not in real time. Sub-minute fleet refresh is out of scope.
- Fleet-state historical trending or archiving â€” each CI run overwrites the previous `fleet-state.json`. Time-series fleet state is Phase 3.

## NFRs

- **Performance:** CI aggregation job completes in under 60 seconds for â‰¤20 registered squads under normal network conditions.
- **Security:** No credentials, access tokens, or personal data in `fleet/squads/{id}.json` or `fleet-state.json` â€” squad files reference public or internally accessible URLs only.
- **Reliability:** Aggregation job failure for one squad must not block the job overall â€” partial fleet state is preferred over no fleet state (AC6).
- **Consistency:** `fleet-state.json` schema must remain backwards compatible as new fields are added â€” existing consumers (viz) must not break when optional new fields appear.
