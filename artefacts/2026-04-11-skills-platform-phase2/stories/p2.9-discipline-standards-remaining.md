# Story: Eight remaining discipline standards and three pilot domain standards

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-4-standards-ci-validation.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As an **engineer working in a non-software-engineering discipline (data, UX, security-extended, regulatory, infrastructure, DevOps, product, or ML/AI)**,
I want a POLICY.md floor standard for my discipline registered in `standards/index.yml`,
So that stories in my discipline have a governance baseline and the assurance gate can apply discipline-appropriate quality checks rather than defaulting to software-engineering standards.

## Benefit Linkage

**Metric moved:** MM5 — Flow findings conversion rate; MM1 — Solo operator outer loop Phase 2 calibration
**How:** Adding the 8 remaining discipline standards resolves one of the five Phase 1 flow findings (the "only 3 of 11 disciplines have standards" gap documented at Phase 1 /levelup). This is a direct contribution to MM5 (≥50% of Phase 1 flow findings actioned within 2 features). For MM1, squads in non-software-engineering disciplines currently hit a blocking external lookup when trying to find their POLICY.md floor — these standards eliminate that lookup.

## Architecture Constraints

- ADR-003: new entries in `standards/index.yml` must define the file paths before any skill or CI script reads or writes them — schema-first applies to the index file itself (the index is the schema contract for the standards routing)
- ADR-004: no hardcoded tool names, org labels, or vendor-specific product names in any new standard file — all tool references must be expressed as `[FILL IN: <description>]` placeholders so teams can swap tooling without editing the standard
- AP-02: anti-pattern of hardcoded org names applies equally to standard files as it does to skill files — enforced at /review Category E
- A6 (discovery assumption, must confirm): `standards/index.yml` v1.0 schema is extensible without breaking changes when 8 new discipline entries and a `domains:` key for domain-tier standards are added — the existing `software-engineering`, `security-engineering`, and `quality-assurance` entries must route correctly after the extension; this assumption must be validated before this story proceeds past /review
- P1.7 structural pattern: all new `core.md` and `POLICY.md` files must follow the heading structure established in the three Phase 1 discipline standards — `/review` Category E checks for structural conformance

## Dependencies

- **Upstream:** None — no dependency on other Phase 2 stories; all three Phase 1 standards already exist and provide the established pattern
- **Downstream:** p2.5a and p2.5b (surface adapter stories) consumed POLICY.md floors for IaC, SaaS-API, SaaS-GUI, M365-admin, and manual surfaces — those are surface-type variants, not discipline standards. No direct dependency but discipline standards inform the assurance gate quality floor for any story in a new discipline.

## Acceptance Criteria

**AC1 (standards/index.yml extension):** Given the current `standards/index.yml` has 3 discipline entries (software-engineering, security-engineering, quality-assurance) at version 1.0, when this story is delivered, then `standards/index.yml` is extended with 8 new discipline entries (`data`, `ux`, `security-extended`, `regulatory`, `infrastructure`, `devops`, `product`, `ml-ai`) under a `disciplines:` key — preserving the existing 3 entries unchanged and confirming A6 (no breaking change to existing routing).

**AC2 (discipline standard file pairs):** Given each new discipline entry is added to `standards/index.yml`, when the standard files are created, then each discipline has a `core.md` and a `POLICY.md` at the declared path — both files conform to the P1.7 structural pattern (required headings: Overview, Outcomes, Requirements, MUST/SHOULD/MAY statements, Out of Scope for this standard) and all MUST/SHOULD/MAY statements are outcome-oriented (describe what result is enabled/prevented, not just what process is required).

**AC3 (pilot domain standards):** Given the discovery specifies at least 3 pilot domain standards at domain tier, when this story is delivered, then `standards/index.yml` contains a `domains:` key alongside `disciplines:` with at least 3 pilot domain entries — each domain entry has at minimum a `POLICY.md` floor at the declared path, with core.md optional at pilot stage. Domain names are chosen by the platform maintainer during story implementation and recorded in decisions.md.

**AC4 (routing correctness):** Given any new discipline slug (e.g. `data`, `ux`, `regulatory`), when a skill or CI script performs a `standards/index.yml` lookup for that discipline, then the lookup returns the correct `core.md` and `POLICY.md` paths — confirmed by a routing smoke test that reads each new entry and resolves the file at the declared path (file must exist).

**AC5 (A6 assumption confirmed — no breaking change):** Given the existing routing tests for `software-engineering`, `security-engineering`, and `quality-assurance` pass on the current `standards/index.yml`, when the 8 new discipline entries and `domains:` key are added, then all existing routing tests continue to pass without modification — the extension is additive and non-breaking.

**AC6 (no hardcoded tool names):** Given any new standard file (core.md or POLICY.md), when the file is read by /review Category E, then no hardcoded tool product names, vendor URLs, or organisation labels appear in the file body — all tool references use `[FILL IN: <description>]` placeholder syntax, consistent with ADR-004.

## Out of Scope

- Detailed ML/AI safety frameworks, model cards, fairness evaluation tooling, or bias assessment methodologies — the `ml-ai` discipline standard covers the POLICY.md floor pattern only. Specialist ML safety frameworks are a separate workstream.
- Automated routing of a story to a discipline standard based on story metadata — the routing remains manual (operator selects discipline in context.yml). Automated discipline inference is Phase 3.
- Backfilling Phase 1 stories with discipline-standard references — new Phase 2 stories forward only.
- Validation that every team using the platform has configured a discipline in context.yml — adoption checking is out of scope for this story.

## NFRs

- **Consistency:** All new standard files must use the same POLICY.md format as the three Phase 1 standards (heading structure, MUST/SHOULD/MAY statement format, outcome-oriented phrasing) — no structural variation across standards.
- **Extensibility:** `standards/index.yml` must remain machine-readable after extension — no YAML syntax that breaks existing parsers; CI validation of `index.yml` syntax must pass.
- **Maintainability:** Each standard file is a standalone document — it must not import or reference another standard's content. Standards may note relationships in an "Adjacent standards" section but must not require reading another file to be interpreted.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the pattern is established from P1.7; the 8 discipline names and 3 pilot domains are confirmed in the discovery; the main work is authoring 16–22 new Markdown files following the established template; A6 assumption validation is the only significant risk
