# Story: Improvement agent — challenger pre-check, proposal review workflow, and improvement-agent SKILL.md

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-5-self-improving-harness.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the improvement agent to produce a structured challenger spec and proposed SKILL.md for each diff proposal so that I can run a targeted pre-check session, and I want a clear proposal review workflow that records outcomes (accept/reject/defer) and applies accepted changes,
So that the harness can safely improve itself — with every change to a skill file backed by evidence, a pre-check result, and a human decision record.

## Benefit Linkage

**Metric moved:** M3 — Improvement agent: first SKILL.md diff proposal from real failure; MM3 — Improvement loop: first proposal reviewed within one feature cycle
**How:** This story delivers the remaining M3 sub-conditions: (3) challenger pre-check result recorded alongside the proposal; (4) proposal reviewed and actioned. MM3 requires the proposal to be reviewed within one feature cycle — the proposal review workflow (AC3–AC5) and the `improvement-agent` SKILL.md (AC6) are the enablers. Together with p2.11, this story closes the full M3/MM3 evidence chain.

## Architecture Constraints

- **Sequencing constraint — requires real Phase 2 delivery traces:** Same as p2.11. This story must not be dispatched to the inner loop until at least one Phase 2 inner loop story batch is merged and its trace is written. The challenger spec (AC1) is only meaningful when the diff proposal it pre-checks was generated from real delivery signal.
- A3 (discovery assumption — confirmed in scope): Challenger pre-check is human-assisted in Phase 2. One Copilot session cannot spawn another. The improvement agent writes the challenger spec and proposed SKILL.md to `workspace/proposals/`; the platform maintainer manually runs a dev agent session + assurance session and records the result. Full automation is explicitly Phase 3.
- ADR-004: any org-specific or tool-specific content in the challenger spec must use `[FILL IN: <context>]` placeholders — the spec template must be reusable across teams without editing
- ADR-003: if new fields are added to `pipeline-state.json` to track accepted skill changes (e.g. `skillChangeId`, `proposalRef`), they must be schema-first
- MC-SEC-02: challenger spec and proposed SKILL.md files must not contain credentials, personal data, or hardcoded team/org identifiers
- The `improvement-agent` skill must not self-modify — AC6 (the improvement-agent SKILL.md) is created by human-authored content in this story, not generated by running the improvement agent on itself. The agent governs its own procedure but does not write its own SKILL.md as a proposal.

## Acceptance Criteria

**AC1 (challenger spec generation):** Given a diff proposal exists in `workspace/proposals/` with `status: pending_review`, when the improvement agent runs the challenger pre-check preparation step, then it writes two files: (a) `workspace/proposals/[proposal-id]-challenger-spec.md` — a synthetic story spec including User Story, ACs that the proposed SKILL.md change must satisfy, and a clear done condition; (b) `workspace/proposals/[proposal-id]-proposed-skill.md` — the full content of the target SKILL.md with the diff applied, identical in structure to the existing SKILL.md but with the proposed section replaced.

**AC2 (pre-check result recording):** Given a maintainer manually runs a dev agent session and an assurance session against the `[proposal-id]-challenger-spec.md`, when they record the result, then the result is written to `workspace/proposals/[proposal-id]-challenger-result.md` with the following fields: `verdict` (pass/fail), `session_summary` (one paragraph describing what the dev session produced and whether the assurance session passed), `traces_produced` (list of any new trace file paths, or `none`), `reviewer` (operator identity), `reviewed_at` (ISO datetime). Incomplete result files (missing any required field) are flagged by the next improvement agent run.

**AC3 (proposal acceptance):** Given a challenger pre-check result is recorded as `verdict: pass` and the maintainer accepts the proposal, when the acceptance is actioned, then: (a) the target SKILL.md is updated with the content from `[proposal-id]-proposed-skill.md`; (b) the proposal's `status` is updated to `accepted` with `accepted_at` and `accepted_by`; (c) the change is committed with a commit message referencing the proposal ID (e.g. `improvement: apply [proposal-id] — [one-line description]`). The SKILL.md update and proposal status update are made in the same commit.

**AC4 (proposal rejection):** Given a maintainer rejects a proposal (pre-check verdict `fail`, OR maintainer review decision against the change), when the rejection is recorded, then: (a) the target SKILL.md is unchanged; (b) the proposal's `status` is updated to `rejected` with `rejected_at`, `rejected_by`, and `rejection_rationale` (mandatory — not optional); (c) no commit is made to skill files. A rejected proposal with no rationale is not a valid rejection — the workflow must enforce rationale is provided.

**AC5 (proposal deferral):** Given a maintainer defers the proposal (change is valid but untimely), when the deferral is recorded, then: (a) the target SKILL.md is unchanged; (b) the proposal's `status` is updated to `deferred` with `deferred_until` (ISO date) and `deferral_reason`; (c) the improvement agent does not re-surface this proposal before `deferred_until` unless a new matching failure pattern with higher severity emerges (in which case a new proposal is created, referencing the deferred one).

**AC6 (improvement-agent SKILL.md):** Given the file `.github/skills/improvement-agent/SKILL.md` is created by this story, when `/workflow` is run, then the skill is discoverable and its trigger phrases (`/improvement-agent`, `review proposals`, `challenger pre-check`, `generate proposals`) activate it. The SKILL.md covers, in order: (1) trace interface query and filter parameter selection; (2) failure signal detection and proposal generation (p2.11 behaviour); (3) staleness signal detection (p2.11 behaviour); (4) anti-overfitting gate (p2.11 behaviour); (5) challenger pre-check preparation (this story AC1); (6) pre-check result recording (this story AC2); (7) proposal review workflow (this story AC3–AC5); (8) state.json proposals block update.

## Out of Scope

- Fully automated challenger execution — the pre-check requires a human-initiated dev agent session and assurance session in Phase 2. One agent session auto-spawning another is Phase 3 (A3).
- Bulk proposal processing — proposals are reviewed one at a time in Phase 2. Batch processing UI is out of scope.
- Rollback of an accepted SKILL.md change via the improvement agent — rollback is done via git revert. The improvement agent does not manage git history.
- Improvement agent applying proposed changes to templates (`.github/templates/*.md`) — scope is limited to `.github/skills/*/SKILL.md` files. Template changes require a separate pipeline evolution story.

## NFRs

- **Human oversight:** Phase 2 challenger pre-check is human-assisted without exception — no automated test runner is a substitute for a human running the dev agent session and recording the result. The `challenger-result.md` `reviewer` field must be a named human identity, not a CI job ID.
- **Traceability:** Every accepted SKILL.md change must have a corresponding proposal ID in the commit message — no skill file changes without a traceable proposal evidence chain.
- **Immutability of evidence:** Once a `challenger-result.md` file is written, it must not be modified — only the outer proposal's `status` transitions via the proposal review workflow. The pre-check result is a permanent record.

## Complexity Rating

**Rating:** 3
**Scope stability:** Unstable — the challenger spec template format and the proposal review workflow UI (command-line vs skill prompt vs structured file) are the highest-ambiguity areas; the sequencing constraint on real Phase 2 delivery traces is an external dependency on inner loop progress
