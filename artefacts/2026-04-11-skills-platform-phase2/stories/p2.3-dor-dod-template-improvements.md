# Story: DoR and DoD template improvements — cross-story schema dependency check, AC verification prompt, and NFR guardrail enforcement (D8/D9/B1-enforce)

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-1-pipeline-evolution-foundation.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the /definition-of-ready skill to enforce cross-story schema dependency declarations and NFR profile presence, and the DoD template to include an explicit AC verification prompt,
So that stories with cross-story schema dependencies are caught before coding begins, stories with NFRs cannot reach DoR sign-off without a feature NFR profile, and DoD includes explicit AC-to-code tracing rather than relying on implicit CI pass.

## Benefit Linkage

**Metric moved:** MM5 — Flow findings conversion rate; MM1 — Solo operator outer loop Phase 2 calibration
**How:** D8, D9, and B1-enforce are three of the five Phase 1 flow findings being actioned in Phase 2. Delivering all three actions MM5 (≥50% conversion of logged findings). They also close outer loop gaps that caused rework in Phase 1 DoR runs (missing NFR profile block surfaced late; cross-story schema dependencies not declared; DoD AC verification was implicit).

## Architecture Constraints

- D8: modifies `.github/skills/definition-of-ready/SKILL.md` — adds a new check in the H-block sequence after H8 (or as H8-ext); the check reads the story's Dependencies block and the story's declared schema fields
- D9: modifies `.github/templates/definition-of-done.md` — adds a "Verification method" field per AC in the AC Coverage section; this is a template content change, not a script change
- B1-enforce: modifies `.github/skills/definition-of-ready/SKILL.md` — adds an NFR profile presence check to the H-NFR guardrail block; reads `artefacts/[feature]/nfr-profile.md` existence
- MC-CONSIST-02 (governance sync): any new H-check added to the DoR skill that is referenced by check-governance-sync.js (gate criteria) must be reflected in the sync script in the same commit; confirm with the maintainer whether D8/B1-enforce changes are in scope for governance-sync tracking
- ADR-003: schema fields referenced by the D8 cross-story check must exist in `pipeline-state.schema.json` — if D8 introduces a new `schemaDepends` field to the DoR contract, add it to the schema in the same commit

## Dependencies

- **Upstream:** p2.1 — D8's cross-story schema check validates dependency declarations that D1 (p2.1) requires /definition to produce; if p2.1 is DoD-complete first, Phase 2 stories entering DoR will already have well-formed dependency blocks
- **Downstream:** All Phase 2 stories from p2.4 onwards benefit from D8/D9/B1-enforce at their DoR runs; p2.6 in particular has a cross-story schema dependency on p2.5a/b that D8 will validate

## Acceptance Criteria

**AC1 (D8 — cross-story schema dependency field in DoR contract):** Given the /definition-of-ready skill is producing a DoR contract for a story whose Dependencies block lists at least one upstream story (not "None"), when the DoR contract template is filled, then the operator is prompted to declare which `pipeline-state.json` schema fields this story expects the upstream story to have written — producing a `schemaDepends: [field1, field2]` entry in the DoR contract before sign-off.

**AC2 (D8 — schema presence hard block):** Given a DoR contract contains a non-empty `schemaDepends` list, when the DoR H-checks run, then each field in `schemaDepends` is checked against `pipeline-state.schema.json` — if any declared field is absent from the schema, the check fires as a hard block with: "H8-ext FAIL: [field] declared in schemaDepends is not present in pipeline-state.schema.json. Add the field to the schema or correct the schemaDepends declaration before sign-off." Evaluation scenario name: `s-h8-ext-blocks-story-missing-nfr-guardrail-entry` (governance-sync guard: this gate must appear in the suite alongside H-NFR-profile).

**AC3 (D8 — no false block for stories with no dependencies):** Given a story's Dependencies block declares "None" (no upstream stories), when the DoR H-checks run, then the D8 schema dependency check passes with "H8-ext: no upstream dependencies declared — schema check not required" and does not block sign-off.

**AC4 (D9 — AC verification prompt in DoD template):** Given the DoD template at `.github/templates/definition-of-done.md` is used by /definition-of-done for a merged story, when the AC Coverage section is reached, then each AC entry in the template includes a "Verification method" field prompting the operator to record how the AC was verified: `Verification method: [automated test slug / code review / manual run]` — not a free-text field; one of these three values is expected.

**AC5 (B1-enforce — NFR profile presence hard block):** Given a story declares at least one NFR (NFR section is not "None" or blank), when /definition-of-ready runs its H-checks, then an NFR profile presence check fires — if `artefacts/[feature]/nfr-profile.md` does not exist, the check fails as a hard block: "H-NFR-profile FAIL: story declares NFRs but no feature NFR profile exists at artefacts/[feature]/nfr-profile.md. Create the NFR profile (run /definition Step 7) before sign-off." Evaluation scenario name: `s-b1-enforce-rejects-story-nfr-section-absent-pipeline-state` (governance-sync guard: this gate must appear in suite alongside H8-ext).

**AC6 (B1-enforce — no block for stories without NFRs):** Given a story's NFR section is "None" or blank, when /definition-of-ready runs, then the NFR profile presence check does not fire — stories without NFRs are not blocked by the absence of an NFR profile.

## Out of Scope

- Automated schema diffing between story versions (checking if schema fields changed across review passes) — D8 is a declaration-and-presence check, not a schema diff tool
- Changes to the DoR hard block numbering or RISK-ACCEPT framework — D8 adds H8-ext as a new check; it does not renumber existing checks
- Changes to the DoD template sections other than adding the Verification method field per AC — D9 is additive only
- Automated AC-to-test-slug mapping (linking each AC to a specific test file) — the Verification method is operator-declared; automation is a Phase 3 improvement candidate

## NFRs

- **Consistency:** The D8 schema presence check must produce identical results regardless of the order H-checks run — it reads a static file (`pipeline-state.schema.json`) and must not have session-state dependencies
- **Backward compatibility:** D9 DoD template change must not break existing DoD artefacts for Phase 1 stories — the new Verification method field is additive; Phase 1 DoD artefacts already written are not required to be backfilled

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — three bounded changes to two SKILL.md files and one template file; the H-check pattern and template format are well-established from Phase 1; only the MC-CONSIST-02 governance-sync interaction has any ambiguity
