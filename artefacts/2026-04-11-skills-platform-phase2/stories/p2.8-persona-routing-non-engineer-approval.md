# Story: Persona routing and non-engineer approval interface

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-3-fleet-observability-personas.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **non-engineer approver (PM, risk lead, or compliance lead)**,
I want to complete a DoR sign-off action from Jira, Confluence, Slack, or Teams without opening VS Code,
So that I can participate in the delivery approval process using my existing tooling, without requiring engineering environment access.

## Benefit Linkage

**Metric moved:** M5 — Non-engineer approval interface: DoR sign-off completed outside VS Code
**How:** This story delivers the channel routing and approval interface wiring that is the pre-condition for M5. Once configured, a non-engineer approver can take the approval action in their native tool, the action is recorded, and `pipeline-state.json` is updated with `dorStatus: signed-off`. All three M5 sub-conditions (approval in non-engineer interface, `pipeline-state.json` updated, VS Code not opened) become satisfiable.

## Architecture Constraints

- ADR-004: all channel routing targets (Jira project, Confluence space, Slack workspace, Teams channel) must be read from `context.yml` — no hardcoded URLs, workspace IDs, or team names in skill instruction text or viz JS constants
- ADR-002: DoR sign-off must write `dorStatus: signed-off` as the evidence field in `pipeline-state.json` — the approval action via a non-engineer interface must result in exactly the same evidence field write as a VS Code sign-off; stage-proxy alone is not sufficient
- MC-CONSIST-02: any channel hint additions to the viz must align with the approval channel routing defined in this story's `context.yml` schema additions — gate criteria and channel config must stay synchronised
- ADR-003: any new fields added to `pipeline-state.json` to record the approver identity or channel source must be schema-first — defined in `pipeline-state.schema.json` before the skill or integration writes them
- **Sequencing constraint — fleet registry dependency:** The persona routing approval update requires the fleet registry (`fleet/squads/{id}.json`) from p2.7 to locate the correct squad `pipeline-state.json` when the sign-off event arrives from outside the squad's VS Code environment. p2.7 must be DoD-complete before the p2.8 end-to-end integration path can be tested.

## Dependencies

- **Upstream:** p2.7 (fleet-registry-ci-aggregation) must be DoD-complete — the persona routing integration uses the fleet registry to identify the target squad `pipeline-state.json` for a non-engineer sign-off event arriving outside VS Code
- **Downstream:** None within Phase 2 — this story completes the persona routing capability

## Acceptance Criteria

**AC1 (channel hint routing):** Given `context.yml` has a `channel_hints.approval` entry specifying a Jira project, Confluence space, Slack workspace, or Teams channel, when a story reaches the `definition-of-ready` stage, then the pipeline emits an approval-stage notification to the configured channel — the notification contains the story slug, squad ID, approval action description, and a reference to the relevant `pipeline-state.json` location (via fleet registry).

**AC2 (non-engineer sign-off action):** Given a non-engineer approver receives the approval notification in their configured tool, when they take the approval action (Jira status transition, Confluence page comment, Slack thread reply, or Teams message reaction as configured), then the action is recorded in that tool with a timestamp and the approver's identity — the record is independently verifiable by the platform maintainer without accessing VS Code.

**AC3 (pipeline-state.json update):** Given a sign-off action is recorded in the non-engineer interface, when the platform maintainer (or CI integration, if configured) processes the record, then `pipeline-state.json` for the relevant squad is updated with `dorStatus: "signed-off"`, `dorApprover` (identity from the interface record), and `dorChannel` (the interface used) — `dorStatus` is the canonical evidence field per ADR-002 and must be written regardless of channel.

**AC4 (IDE channel regression guard):** Given `context.yml` has `channel_hints.ide: vscode` (the default), when a story reaches definition-of-ready, then the IDE channel notification is still emitted — the addition of persona routing does not suppress or override the existing VS Code notification path. Both channels receive the notification when both are configured.

**AC5 (unconfigured approval channel):** Given `context.yml` has no `channel_hints.approval` entry, when a story reaches definition-of-ready, then the system falls back to IDE-only notification and emits a configuration warning (logged to session output or `state.json`): "No approval channel configured — DoR sign-off defaulting to IDE. Set channel_hints.approval in context.yml to enable non-engineer approval." No error is thrown; no approval action is blocked.

**AC6 (context.yml as single source of truth):** Given any persona channel routing is active, when the routing resolves a channel target, then the target (URL, project key, workspace ID, team name) is read exclusively from `context.yml` — no hardcoded values appear in the skill instruction text, viz JS, or CI integration script. Swapping from Jira to Confluence requires only a `context.yml` update, with no skill or code edits (ADR-004).

## Out of Scope

- Full automated CI integration between the non-engineer interface and `pipeline-state.json` update — in Phase 2, the maintainer may process the sign-off manually based on the interface record. Full automation pipeline (webhook → CI → state write) is a Phase 3 capability.
- Bi-directional sync between `pipeline-state.json` and the non-engineer interface (e.g. Jira issue reflecting full story state) — Phase 2 is one-way: approval intent captured in the interface, state written to `pipeline-state.json`.
- Multi-approver workflows or approval quorum logic — Phase 2 supports a single approver per story sign-off. Group approval is out of scope.
- Microsoft Teams bot or Slack app development — Phase 2 uses the native messaging interface (reactions, replies, comments). Bot/app development requiring OAuth app registration is Phase 3.

## NFRs

- **Security:** No personal data (full names, email addresses) stored in `pipeline-state.json` — `dorApprover` stores a role identifier or username, not full PII. Approver identity is auditable but not a PII store.
- **Consistency:** Any change to the `channel_hints` schema in `context.yml` must be reflected simultaneously in the skill instruction text and the schema definition — MC-CONSIST-02.
- **Auditability:** The `dorChannel` field written to `pipeline-state.json` is a permanent record of how the sign-off was obtained — it must not be overwritten by a subsequent sign-off attempt.
