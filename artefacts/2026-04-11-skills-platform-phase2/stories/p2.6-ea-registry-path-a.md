# Story: EA registry Path A surface type resolution in surface adapter resolver

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-2-full-surface-adapter-model.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the surface adapter resolver to support Path A (EA registry lookup) for surface type resolution — querying the EA registry by application identifier to retrieve the registered surface type — in addition to Path B (explicit `context.yml` declaration),
So that squads registered in the EA registry can resolve their surface type automatically without maintaining a separate explicit `surface:` declaration in `context.yml`, and so that the registry is the single source of truth for surface type in registry-native organisations.

## Benefit Linkage

**Metric moved:** M2 — Non-git-native surface adapter assurance verdict; M1 — Second squad outer loop unassisted
**How:** Path A removes a declaration-maintenance burden for squads that are already registered in the EA registry — they do not need to also declare their surface type in `context.yml`. For registry-native organisations (A5 assumption: ea_registry_authoritative: true), this reduces the outer loop setup friction for a new squad to zero for surface type resolution. M2: Path A extends the resolver without breaking Path B, meaning the first non-git-native assurance verdict can be produced by either path.

## Architecture Constraints

- ADR-003 (schema-first): the EA registry response schema — specifically the field name carrying the surface type — must be documented in a schema file, YAML comment, or code comment before the resolver reads it; the field name assumed is `surfaceType` per decisions.md ASSUMPTION-01; this must be confirmed at DoR H1
- **ASSUMPTION-02 — Provisional EA registry stub contract (DoR H1 gate):** For Phase 2 implementation, the resolver is implemented and tested against a stub EA registry contract. The stub response contains exactly three named fields: `surfaceType` (string — the registered surface type for the application), `teamId` (string — the squad identifier in the EA registry), and `adapterOverride` (string or null — optional explicit adapter override; null means use the surfaceType default). This contract is provisional and scoped to a 3-field stub. When the real EA registry schema is confirmed (post-Phase 2 engagement with the EA registry repo owner), the contract must be revisited — any additional fields returned by the registry should be ignored or validated and the stub test fixtures updated. This assumption is recorded in decisions.md ASSUMPTION-02.
- ADR-004: `registry_url`, `registry_source`, and `app_id` (application identifier for registry lookup) are `context.yml` fields — not hardcoded in the resolver; the resolver reads them from config
- MC-SEC-03: the resolver may make a network call to the EA registry — this is the only permitted external call; it must use HTTPS; credentials for authenticated registry access (if required) are referenced by name from `context.yml` secrets reference, not embedded as values
- MC-SEC-02: the resolver must not log the full EA registry response body — only the resolved surface type (a single string) is logged to the run output
- A5 (discovery assumption): Path A is additive; Path B squads are entirely unaffected — a Path B `context.yml` must produce identical resolver behaviour before and after this story is merged

## Dependencies

- **Upstream:** p2.5a AND p2.5b — both must be DoD-complete before Path A is meaningful; the resolver needs all five non-git-native adapter types registered to handle any EA registry response returning one of those surface types; Path A with an incomplete adapter registry produces a routing error, not a governance failure
- **Downstream:** None within Phase 2

## Acceptance Criteria

**AC1 (Path A — EA registry query):** Given `context.yml` is configured for Path A with `registry_source: ea-registry`, `registry_url: <URL>`, and `app_id: <identifier>`, when the surface adapter resolver runs, then it performs an HTTP GET to the EA registry URL with the `app_id` as a query parameter — the request is made, not skipped.

**AC2 (Path A — resolved surface type drives adapter selection):** Given the EA registry stub returns a JSON response containing the three provisional contract fields (`surfaceType` string, `teamId` string, `adapterOverride` string|null), when the resolver processes the response, then the resolved `surfaceType` value is used for adapter selection — identical outcome to Path B explicit declaration with the same surface type. If `adapterOverride` is non-null, it takes precedence over `surfaceType` for adapter selection. If `adapterOverride` is null, `surfaceType` is used.

**AC3 (Path A — unavailability fallback to Path B):** Given the EA registry is unavailable (network timeout, HTTP 5xx, DNS failure), when the resolver runs under Path A configuration, then: (a) if `context.yml` also contains a Path B `surface:` declaration, the resolver falls back to Path B and logs "EA registry unavailable — falling back to explicit surface declaration" in the run output; (b) if no Path B declaration exists, the resolver returns `{ status: "error", error: "EA registry unavailable and no fallback surface declaration found" }` — it does not silently default to git-native.

**AC4 (Path B — unchanged behaviour, no Path A interference):** Given `context.yml` uses Path B (contains a `surface:` explicit declaration, no `registry_source` or `registry_url` fields), when the resolver runs, then Path A is not attempted — no EA registry request is made, no `app_id` is required, and the resolver outcome is identical to pre-p2.6 behaviour.

**AC5 (Both paths present — Path B precedence with warning):** Given `context.yml` contains both `registry_source: ea-registry` (Path A) AND an explicit `surface:` declaration (Path B), when the resolver runs, then Path B takes precedence, Path A is not queried, and the run log contains: "Both Path A (registry_source) and Path B (surface:) declared — Path B (explicit) takes precedence. Remove one to resolve."

## Out of Scope

- Writing a Path A registration tool or UI for adding squads to the EA registry — squads are registered in the EA registry repo (`https://github.com/heymishy/ea-registry`) by its own contribution process; this story reads from it, does not write to it
- Caching EA registry responses — Phase 2 makes a live call per gate run; caching is a performance optimisation for Phase 3 when fleet scale makes live calls expensive
- Handling EA registry responses returning surface types not yet in the adapter registry (e.g. a type registered in EA but with no adapter in Phase 2) — the resolver treats an unknown surface type as an error (AC6 from p2.5b applies)
- EA registry authentication (bearer token, OAuth) — Phase 2 assumes the EA registry endpoint is publicly readable for the pilot; authenticated access is flagged as a Phase 3 consideration in the NFR section

## NFRs

- **Performance:** EA registry HTTP call must have a timeout of ≤3 seconds — the resolver blocks the CI gate and must not extend total gate time beyond its 10-minute budget (P1.3 NFR)
- **Security:** HTTPS only for EA registry calls (MC-SEC-03); resolver must not log the full registry response body, only the resolved surface type string (MC-SEC-02); credentials for authenticated registry endpoints referenced by secret name only, never as values
- **Resilience:** Registry unavailability must not silently degrade to git-native (AC3) — the fallback to Path B or explicit error ensures a misconfigured or drifted registry does not produce invisible false passes
