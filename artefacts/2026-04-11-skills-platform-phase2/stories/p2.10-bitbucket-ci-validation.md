# Story: Bitbucket CI validation — Cloud pipeline-shape tests and DC Docker auth topology

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-4-standards-ci-validation.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **squad running Bitbucket Cloud or Bitbucket Data Center pipelines**,
I want the governance CI gate validated for Bitbucket pipelines — syntax and shape checks for Cloud, and auth topology validation for DC Docker,
So that I can adopt the assurance gate without being on GitHub Actions, and the platform can certify that the gate works correctly on Bitbucket CI infrastructure.

## Benefit Linkage

**Metric moved:** M1 — Second squad onboarding full outer loop unassisted; M2 — Non-git-native surface adapter passing assurance verdict (prerequisite)
**How:** A Bitbucket-hosted squad is currently blocked from adopting the CI gate (P1.3) because the gate has only been validated on GitHub Actions (A1 — assumption in discovery). This story removes that blocker for Bitbucket Cloud and DC squads. Without Bitbucket CI validation, a second squad on Bitbucket would hit a blocking external lookup at onboarding (M1 failure). M2 is a partial dependency: non-git-native surface adapters (p2.5a/b) running on a Bitbucket CI pipeline require this story to produce a passing assurance verdict on the Bitbucket platform.

## Architecture Constraints

- A1 (discovery assumption — this story resolves it): GitHub Actions was the only validated CI surface at phase start; Bitbucket Cloud and Bitbucket DC must be explicitly validated and the assumption updated in decisions.md when ACs are met
- A2 (discovery assumption — this story confirms): Bitbucket Cloud (YAML syntax + pipeline-shape) and Bitbucket DC Docker (auth topology) are distinct validation tiers; ACs must be split cleanly between Cloud and DC — a Cloud validation run must not attempt DC auth tests, and vice versa
- ADR-004: any Bitbucket-specific configuration (org, repo slug, app password reference, OAuth client ID) must be read from `context.yml` — no hardcoded Bitbucket credentials or org identifiers in the CI gate script or validator
- MC-CORRECT-02: if new fields are added to `pipeline-state.json` to record which CI platform a gate ran on, those fields must exist in `pipeline-state.schema.json` first (ADR-003)
- MC-SEC-02: app passwords, OAuth tokens, and SSH private keys used in DC Docker tests must be stored in test fixtures only — never committed to the repo; test secrets follow the CI secret injection pattern

## Dependencies

- **Upstream:** P1.3 (assurance-agent-ci-gate, Phase 1) must be DoD-complete — this story validates Bitbucket support for an already-working gate, not a gate that is still being built. P1.3 is confirmed merged.
- **Downstream:** p2.5a / p2.5b (non-git-native adapters) — Bitbucket CI validation unblocks the full M2 evidence path for squads running non-git-native adapters on Bitbucket infrastructure

## Acceptance Criteria

**AC1 (Bitbucket Cloud — YAML syntax validation):** Given a well-formed `bitbucket-pipelines.yml` containing the governance assurance gate step, when the CI gate validator runs against Bitbucket Cloud (free trial account), then the YAML syntax check passes without error and the pipeline definition is accepted by the Bitbucket Cloud parser.

**AC2 (Bitbucket Cloud — pipeline-shape test — missing step detection):** Given a `bitbucket-pipelines.yml` that is syntactically valid but is missing the required governance assurance gate step, when the pipeline-shape validator runs, then the missing step is detected and the validator reports a failure naming the missing step — it does not produce a false pass.

**AC3 (Bitbucket DC Docker — app password auth topology):** Given a Bitbucket DC Docker environment configured with an app password credential, when the auth topology test runs, then the test confirms the app password credential resolves correctly via the Bitbucket DC auth mechanism and the pipeline can authenticate to start a run — test completes without 401 or 403.

**AC4 (Bitbucket DC Docker — OAuth auth topology):** Given a Bitbucket DC Docker environment with an OAuth consumer configured, when the auth topology test runs, then the test confirms the OAuth flow completes (client credentials or authorisation code as applicable) and returns an authenticated session token — no auth failure.

**AC5 (Bitbucket DC Docker — SSH key auth topology):** Given a Bitbucket DC Docker environment with SSH key auth configured, when the auth topology test runs, then the test confirms SSH key resolution succeeds for the pipeline runner and repo checkout is authenticated — test completes without SSH handshake failure.

**AC6 (Cloud/DC isolation — no cross-contamination):** Given a Bitbucket Cloud validation run is executed, when the validator processes the Cloud pipeline definition, then no DC-specific operations are attempted (no app password calls, no OAuth flows, no SSH auth tests); and given a DC Docker validation run is executed, then no Cloud-only YAML-parsing API calls are made against the Cloud endpoint. Cloud and DC validation paths are independently executable and do not share runtime state.

**AC7 (Docker Compose deliverable — Reproducibility NFR):** Given the DC Docker auth topology tests (AC3–AC5) require a local Bitbucket DC instance, when this story is merged, then a Docker Compose configuration file (`tests/fixtures/docker-compose.bitbucket-dc.yml`) is committed to the repository — scaffolding the Bitbucket DC Docker instance with the three auth topologies (app password, OAuth, SSH key) configured. Any developer with Docker installed must be able to run `docker compose -f tests/fixtures/docker-compose.bitbucket-dc.yml up` and reproduce the DC auth topology tests without a network-accessible Bitbucket DC instance.

## Out of Scope

- Bitbucket Pipelines native enterprise environment (non-Docker Bitbucket DC) — discovered in decisions.md 2026-04-11 ASSUMPTION: Docker DC is the Phase 2 validation path; a shared enterprise DC environment is deferred until confirmed available.
- Jenkins CI validation — Jenkins is not explicitly in scope for Phase 2. Bitbucket Cloud and DC Docker are the two named targets.
- GitHub Actions regression testing — GitHub Actions validation is inherited from Phase 1 (P1.3); this story does not modify or re-validate GitHub Actions behaviour.
- Automated Bitbucket CI setup provisioning for new squads — squads configure their own Bitbucket pipeline using the provided template. Automated Bitbucket project provisioning is out of scope.

## NFRs

- **Security:** App passwords, OAuth tokens, and SSH private keys used in DC Docker tests must be injected via CI secret variables — never committed to the repository. Test fixture files must reference secret variable names, not literal credential values.
- **Isolation:** Cloud and DC test paths must be independently executable with no shared runtime state — a failure in the DC auth tests must not affect the Cloud YAML validation result, and vice versa.
- **Reproducibility:** Docker DC test environment must be reproducible from the committed Docker Compose / container configuration — any developer with Docker installed must be able to run the DC auth topology tests locally without a network-accessible Bitbucket DC instance.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — A1 and A2 from discovery establish the exact scope boundary (Cloud syntax/shape + DC Docker auth); the three DC auth topology variants (app password, OAuth, SSH) are discrete and well-understood; the main execution risk is Docker DC environment setup time, not scope ambiguity
