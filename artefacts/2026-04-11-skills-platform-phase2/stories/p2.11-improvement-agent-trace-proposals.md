# Story: Improvement agent — queryable trace interface, failure and staleness detection, and diff proposals

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-5-self-improving-harness.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the improvement agent to read delivery traces through a queryable interface, identify failure and staleness patterns, and produce targeted SKILL.md diff proposals in `workspace/proposals/`,
So that the platform can systematically close its own quality gaps based on observed delivery signal — rather than relying on manual learnings capture that is dependent on an individual operator noticing patterns across sessions.

## Benefit Linkage

**Metric moved:** M3 — Improvement agent: first SKILL.md diff proposal from real failure; MM3 — Improvement loop: first proposal reviewed within one feature cycle
**How:** This story delivers the trace ingestion, pattern detection, and proposal generation mechanism that is the direct pre-condition for M3. Once a real Phase 2 delivery failure pattern accumulates in the trace archive and this story is operational, the agent produces a diff proposal referencing specific trace IDs — satisfying M3 sub-conditions (1), (2). MM3 sub-condition (first proposal reviewed within one feature cycle) depends on both this story and the proposal review capability in p2.12.

## Architecture Constraints

- **Sequencing constraint — requires real Phase 2 delivery traces:** The improvement agent cannot produce M3/MM3 evidence from Phase 1 traces or synthetic test traces. This story must not be dispatched to the inner loop until at least one Phase 2 story has been merged and its trace written to the trace archive. Record in DoR instructions block when this story reaches /definition-of-ready.
- **Queryable trace interface — implementation vehicle:** The trace interface is implemented as a JavaScript module (`src/improvement-agent/trace-interface.js`) exposing a `queryTraces(filters)` function — callable from both CLI (via a thin wrapper script) and from the improvement agent SKILL.md procedure. This is the canonical entry point for all trace data access in the improvement agent; no other module reads trace files directly.
- **ASSUMPTION-P2.11-01 — P1.6 trace archive format (DoR H1 gate):** The trace interface reads trace files in the format established by P1.6 (`workspace/traces/` YAML or JSON). The trace record field carrying the failure pattern label must be confirmed at /definition-of-ready H1 before coding begins — specifically: the field name (assumed `failurePattern` or `failurePatterns[]` in the P1.6 trace schema), the type (string or string array), and whether the label is author-written or agent-extracted. If the P1.6 trace record does not contain a label field matching the kebab-case format required by AC2, the clustering algorithm cannot be implemented as written. Resolution of this assumption must be recorded in decisions.md before the DoR sign-off block is produced.
- ADR-004: the trace interface filter configuration (failure-pattern threshold count, staleness window in days) must be read from `context.yml` — no hardcoded threshold constants in the agent SKILL.md or trace interface implementation
- ADR-003: any new fields written to `workspace/state.json` (the `proposals` block) or to `pipeline-state.json` tracking proposal generation must be schema-first — define in the relevant schema before the agent writes them
- MC-SEC-02: trace data read by the agent must not contain credentials or personal data — if traces contain any such fields, the agent must redact before processing
- AP-02: proposal files must not contain hardcoded org names, squad names, or personal identifiers — only skill slugs, trace IDs, and pattern labels

## Dependencies

- **Upstream:** P1.6 (living-eval-regression-suite, Phase 1) must be DoD-complete — provides the trace archive format and the eval scenario pattern that the trace interface reads. P1.6 is confirmed merged.
- **Upstream (sequencing):** At least one Phase 2 inner loop story batch must be merged and its trace written before M3 evidence is possible — this is a sequencing gate, not a code dependency.
- **Downstream:** p2.12 (challenger pre-check and proposal review) consumes proposals written by this story — p2.12 must be DoD-complete for the full M3/MM3 evidence chain to close

## Acceptance Criteria

**AC1 (queryable trace interface — filtering):** Given traces exist in the workspace trace archive, when the improvement agent queries the trace interface with filter parameters (surface type, ISO date range, failure pattern keyword), then the interface returns only traces matching all specified filters — traces outside the surface type, date range, or not containing the failure pattern keyword are excluded from the result set.

**AC2 (failure signal detection — proposal generation):** Given ≥3 traces for the same surface type contain the same failure pattern (identified by a pattern label appearing 3+ times within a 10-story sliding window of the most recent traces for that surface type), when the improvement agent analyses the filtered trace set, then it writes a diff proposal to `workspace/proposals/YYYY-MM-DD-[skill-slug]-failure-proposal.md` — the proposal file references the specific trace IDs as evidence and describes the proposed change as a before/after diff to the target SKILL.md section. Pattern label format: kebab-case failure category string (e.g. `missing-test-coverage`, `ac-not-testable`, `ac-ambiguous`); no spaces, no uppercase. Matching algorithm: exact string match on label value (case-insensitive); the sliding window is the 10 most recent traces per surface type by `createdAt` timestamp regardless of calendar date. Evaluation scenario name: `s-improvement-agent-failure-clustering`.

**AC3 (staleness signal detection — staleness proposal):** Given a surface type has no trace activity for ≥90 days (or the value set in `context.yml` `improvement_agent.staleness_window_days`), when the improvement agent scans the trace archive, then it writes a staleness proposal to `workspace/proposals/YYYY-MM-DD-[skill-slug]-staleness-proposal.md` — the proposal notes the inactive surface, the last activity date, and suggests either explicit deferral or removal of the surface type entry, with rationale.

**AC4 (anti-overfitting gate):** Given a generated proposal would remove or weaken a check (skip a MUST statement, reduce a threshold, or remove an AC from a skill), and that check has passed on every trace within the filter window for the same surface type, when the improvement agent applies the anti-overfitting gate, then the proposal is blocked from being written — instead, a warning is recorded in `workspace/proposals/YYYY-MM-DD-[skill-slug]-overfitting-warning.md`: "Proposed removal affects a currently-passing check ([check label], last N traces all pass). Challenger pre-check required to proceed."

**AC5 (proposal file structure):** Given any proposal file is written to `workspace/proposals/`, when the file is read, then it contains all of: (a) `evidence` — trace IDs or surface/date range that triggered the proposal; (b) `proposed_diff` — before/after content or full replacement for the target SKILL.md section; (c) `confidence` — `high`, `medium`, or `low` with a one-line rationale; (d) `anti_overfitting_gate` — `passed` or `blocked` with result summary; (e) `status: pending_review`; (f) `created_at` — ISO datetime.

**AC6 (state.json proposals block):** Given at least one proposal file is written during a session, when the session ends, then `workspace/state.json` is updated with a `proposals` block listing each proposal file path, `created_at`, and `status` — enabling the next session to discover all pending proposals without scanning `workspace/proposals/` manually.

## Out of Scope

- Fully automated application of accepted proposals to SKILL.md files without human review — the agent generates proposals; challenger pre-check and human acceptance are in p2.12.
- Cross-team or cross-repository trace ingestion — the trace interface reads from the local workspace trace archive only. Cross-repo federation is Phase 3.
- Automatic re-triggering of the improvement agent on each new trace write — agent is run on-demand or on a scheduled CI job. Event-driven trigger is Phase 3.
- ML-based failure clustering beyond pattern label matching — the failure signal detection uses lexical pattern matching on trace content, not statistical clustering or embedding similarity.

## NFRs

- **Idempotency:** Running the improvement agent twice against the same trace set and the same filters produces the same proposals — no duplicate proposal files are written on re-run (deduplicated by skill slug + pattern label + date).
- **Privacy:** If any trace file contains personal data or credentials (e.g. operator names in trace metadata), the agent redacts those fields before processing and does not include them in proposal files.
- **Auditability:** Proposal files are append-only after creation — status transitions (`pending_review` → `accepted` / `rejected` / `deferred`) are recorded by adding a `status_history` entry, not by overwriting the original content.
