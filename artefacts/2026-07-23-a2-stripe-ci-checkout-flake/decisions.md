# Decisions: a2-stripe-ci-checkout-flake (a2ccf-s1)

## DECISION — serialize the Scenario A CI job's real-staging specs (`--workers=1`) rather than relax timeouts or touch application code

**Date:** 2026-07-23

**Context:** After the serlb-s1 rate-limiter fix (PR #564, merged) resolved the previous 429-cascade blocker, the real failing CI run for PR #563 (https://github.com/heymishy/skills-repo/actions/runs/29981769208/job/89124881273) showed 6 passed, 2 skipped, 3 failed — all 3 failures in `tests/e2e/a2-stripe-test-mode-plan-selection.spec.js`, all downstream of the same interaction never completing:
- AC1: `expect(planState.plan).toBe('paid')` — received `'trial'`. `pollPlanState` polled 10 times over ~15s and the state never changed.
- AC2: `page.waitForURL(/wuce-staging\.fly\.dev/, { timeout: 30000 })` — timed out; the browser never navigated back to the app after the submit click.
- AC3: `expect(page.getByText(/declined/i)).toBeVisible({ timeout: 15000 })` — element never appeared.

Critically, this exact spec had passed 3/3 earlier this session, both in a manual local run and in the run recorded in PR #563's own description ("all 3 pass"). Something specific to the real GitHub Actions CI invocation — not the application code, not real staging itself — was different.

## Investigation log

**1. Read the workflow file at the PR branch's actual HEAD** (`git cat-file -p` on the blob at `origin/a5-ci-gate-scenario-a-blocking`, since the local worktree was at master post-#564 and did not yet have the `scenario-a-staging-e2e` job). Confirmed the exact failing command: `npx playwright test tests/e2e/a1-staging-auth-stub.spec.js tests/e2e/a2-stripe-test-mode-plan-selection.spec.js tests/e2e/a3-product-feature-ideate-canvas.spec.js tests/e2e/a4-ideate-session-resume.spec.js` — no `--workers` flag. `playwright.config.js` has no `workers` field either, so Playwright falls back to its CPU-based default. The real failing run's log shows `Running 11 tests using 2 workers`, consistent with `ubuntu-latest`'s small vCPU count.

**2. Checked test file signatures to identify which specs actually drive a browser.** `a1-staging-auth-stub.spec.js`'s all 3 tests destructure only `{ request }` — no `page` fixture, no real browser page is driven; this spec is pure HTTP/API traffic and finishes fast. `a2` uses `{ page, request }`/`{ page }` for all 3 tests (real Stripe Checkout interaction). `a3` has 29 `page.*` call sites (canvas rendering, product creation UI); `a4` has 13 (SSE-stream session resume, closing/reopening a real browser context). This means: with 2 workers and 4 files queued, `a1` (fast, API-only) frees its worker almost immediately, and that freed worker then picks up `a3` or `a4` — both heavy, real-browser specs — likely while `a2` (also browser-heavy) is still mid-flight in the other worker. This is the concrete mechanism for genuine concurrent browser automation against the same small CI runner.

**3. Ruled out rate limiting as this run's cause, with direct evidence, not assumption.** `gh secret list --repo heymishy/skills-repo` shows `E2E_STAGING_AUTH_STUB_SECRET` was set at `2026-07-23T05:11:16Z` — before the failing run started at `05:18:29Z`. The failing run's log contains zero HTTP 429 errors anywhere in `a2`'s 3 failures (all are assertion mismatches or Playwright timeouts, never an HTTP-level error). serlb-s1's bypass was genuinely active for this run.

**4. Attempted to download the CI run's own trace/screenshot artifacts for direct evidence of page state at the point of failure.** `gh api repos/heymishy/skills-repo/actions/runs/29981769208/artifacts` returned `{"total_count":0,"artifacts":[]}` — no artifacts were ever uploaded, despite the job failing. Investigated why: `gh run view --job 89124881273` shows the "Upload Playwright traces and screenshots on failure" step's own status as skipped (`-`), not run, despite the preceding "Run Scenario A E2E..." step genuinely failing (exit code 1, no `continue-on-error` on this job by design). Root cause of THIS sub-defect: GitHub Actions implicitly ANDs any custom `if:` expression that does not itself reference `success()`/`failure()`/`always()`/`cancelled()` with `success()`. The upload step's condition (`steps.scenario-a-config.outputs.scenario_a_enabled == 'true' && steps.scenario-a-run.outcome == 'failure'`) has no such reference, so once the run step failed, the implicit `success()` was false and the upload step was silently skipped — the opposite of its intended "upload only on failure" behavior. This meant no trace/screenshot evidence was recoverable from the actual failing run; addressed as AC6/a bundled secondary fix in this story (`always() &&` prepended to the condition) so future failures of this job are diagnosable.

**5. Reproduced locally, in a controlled comparison, using the identical command and identical real staging app.** Confirmed via `git diff HEAD origin/a5-ci-gate-scenario-a-blocking -- tests/e2e/ src/` that the local worktree's `tests/e2e/` and `src/` trees were byte-identical to the PR branch's (PR #563 only touches workflow/config/a new check script, not application or spec code) — so a local run using this worktree is a valid proxy for the CI run's own code, modulo the runner environment itself.
   - **Run A — `--workers=1` (forced serial), full 4-file command:** `Running 11 tests using 1 worker`. Result: 6 passed, 4 skipped, 1 failed — the 1 failure was `a4`'s NFR-Security test, hitting `HTTP 429 "Too many attempts"` at `fixtures/staging-auth.js:148` (this local run does not have `E2E_STAGING_AUTH_STUB_SECRET` set — that CI-only secret bypass was unavailable to this local investigation — so hitting the real 10-per-5-min per-IP limit on the LAST spec in the queue, after `a1`+`a2`+`a3` had already each signed up 1-3 tenants, is expected and unrelated to this story). **All 3 of `a2`'s tests (AC1, AC2, AC3) passed cleanly** in this run.
   - **Run B — default worker count (unspecified, matching CI's own invocation), full 4-file command, run immediately after Run A:** `Running 11 tests using 2 workers`. Result: 9 failed, 2 skipped — but every one of the 9 failures in this run was `HTTP 429`, because this run started well within the same 5-minute window as Run A's ~8 real signups against the same real per-IP limiter, from the same local IP, with no bypass secret available locally. This run is **confounded** by the two runs' shared rate-limit budget and does NOT by itself reproduce the CI failure signature (CI's failures were never 429s) — it is not used as positive evidence for the concurrency hypothesis; only Run A's clean, unconfounded pass under `--workers=1` is used as evidence.
   - **Interpretation:** the CI environment's own constraint (small vCPU count, default-computed worker count) could not be faithfully reproduced on this session's own (8-core) machine — running 2 workers locally does not recreate the same CPU-contention profile as a 2-vCPU GitHub Actions runner, because this environment has ample spare CPU. This means local runs cannot, by themselves, conclusively prove or disprove the CPU-contention mechanism; they can only show that removing concurrency (Run A) does not break anything and does produce a clean `a2` pass. **The real, load-bearing verification for this story is therefore explicitly deferred to a real GitHub Actions run of the fix (AC4/E2E3)** — a local pass alone, on a machine that cannot recreate the CI runner's resource constraints, does not conclusively prove the fix, exactly as the task's own investigation brief anticipated.

**6. Hypotheses considered and their status:**
   - **Concurrency/parallel-worker interference (CPU contention from concurrent real-browser automation on a small runner):** supported by (a) the mechanism identified in step 2 (a1 frees its worker fast, a3/a4 are browser-heavy and likely run concurrently with a2), (b) a2's failure signature (silent no-op on our own click/interaction, not a network/auth error) being consistent with a client-side timing race rather than a server-side or auth problem, and (c) `--workers=1` producing a clean a2 pass locally with no other change. **This is the hypothesis the fix targets.** Not proven with a byte-for-byte identical CPU-constrained reproduction (not achievable in this session's own, more capable, environment) — the real CI run of the fix (AC4) is the conclusive test.
   - **Headless-specific Stripe Checkout behavior (viewport/user-agent differences):** `playwright.config.js`'s `use.headless: true` applies identically whether run locally or in CI — no CI-specific headless/headed or viewport override exists anywhere in the config. Ruled out — there is no config-level difference to explain a CI-only failure.
   - **Timeout-too-tight for CI network latency:** ruled out per Architecture Constraints in the story — AC1's plan state never changed at all across 10 polls (not a marginal miss), and the same spec completed in 33.5s total against the same real staging app earlier this session, well inside AC2/AC3's 30s/15s budgets.
   - **Stripe test-mode rate limiting or bot detection of GitHub Actions IP ranges:** not fully ruled out in isolation — no trace/screenshot was recoverable (see step 4) to directly observe the Stripe Checkout page's state at the moment of failure, and this session cannot directly test from a GitHub Actions IP. However, the concurrency-removal fix is identical regardless of whether the trigger is CPU contention specifically or another concurrency-coupled variable (e.g. if Stripe's own client-side JS behaves differently under system load, or if two near-simultaneous checkout sessions from the same runner exhibit some interaction), so resolving this ambiguity further was not required to select and ship the fix. If the real CI run of this fix (AC4) still fails, that would be evidence favoring this alternative hypothesis and would need separate, further investigation — reported honestly rather than assumed away.

**Decision:** Add `--workers=1` to the `scenario-a-staging-e2e` job's `npx playwright test ...` command only (`.github/workflows/e2e.yml`), not to `playwright.config.js` globally. This keeps the fix's blast radius to exactly the one job with the concurrency problem, leaves the pre-existing 29 local-mocked specs' parallel-worker speed untouched, and does not touch any application code or spec assertion.

**Rationale for not blanket-increasing timeouts as a first resort:** the story's own investigation (step 3 of the Architecture Constraints' "ruled out" list) shows AC1's failure is a complete non-occurrence of the server-side effect (plan state never transitions even once across repeated polling), not a marginal timing miss — a longer timeout would not fix a webhook that never fires because the checkout was never actually submitted. Treating this as "just raise the timeout" would have been treating a symptom, not the cause, and — per this repo's own established culture (evidenced by every fix-forward this session) — would very likely resurface as the same or a differently-shaped flake later.

**Bundled secondary fix (AC6):** `always() &&` prepended to the "Upload Playwright traces and screenshots on failure" step's `if:` condition in the same job, fixing the GitHub-Actions-implicit-`success()` defect discovered in step 4 above, so any future failure of this exact job produces real diagnostic artifacts instead of silently uploading nothing.

## FINDING — local reproduction cannot faithfully recreate the CI runner's CPU-constrained profile

**Date:** 2026-07-23

This session's own sandbox has 8 logical CPUs (`node -e "console.log(require('os').cpus().length)"`), versus GitHub Actions' `ubuntu-latest` small runner (the failing run's own log, `Running 11 tests using 2 workers`, implies Playwright computed a default of 2 workers from a small vCPU count — Playwright's undocumented-but-observed default is roughly half the detected CPU count). Running the same `--workers=2` command locally does not recreate the same resource-contention profile, because this environment has ample spare CPU regardless of concurrent browser automation. This is why local reproduction attempts (see Investigation log, step 5) could show a clean `--workers=1` pass but could not conclusively demonstrate the failure under `--workers=2` locally (that attempt was additionally confounded by this session's own rate-limit budget exhaustion from back-to-back local runs without the CI-only bypass secret). **The real, conclusive verification for this story is a real GitHub Actions run of the fix** — recorded as AC4 in the story and as a mandatory Coding Agent Instruction step, not treated as optional given this constraint.
