# Decisions: a2-stripe-ci-checkout-flake (a2ccf-s1)

## DECISION — serialize the Scenario A CI job's real-staging specs (`--workers=1`) rather than relax timeouts or touch application code

**Date:** 2026-07-23

**Context:** After the serlb-s1 rate-limiter fix (PR #564, merged) resolved the previous 429-cascade blocker, the real failing CI run for PR #563 (https://github.com/heymishy/skills-repo/actions/runs/29981769208/job/89124881273) showed 6 passed, 2 skipped, 3 failed — all 3 failures in `tests/e2e/a2-stripe-test-mode-plan-selection.spec.js`, all downstream of the same interaction never completing:
- AC1: `expect(planState.plan).toBe('paid')` — received `'trial'`. `pollPlanState` polled 10 times over ~15s and the state never changed.
- AC2: `page.waitForURL(/wuce-staging\.fly\.dev/, { timeout: 30000 })` — timed out; the browser never navigated back to the app after the submit click.
- AC3: `expect(page.getByText(/declined/i)).toBeVisible({ timeout: 15000 })` — element never appeared.

Critically, this exact spec had passed 3/3 earlier this session, both in a manual local run and in the run recorded in PR #563's own description ("all 3 pass"). Something specific to the real GitHub Actions CI invocation — not the application code, not real staging itself — was different.

## Investigation log

**1. Read the workflow file at the PR branch's actual HEAD** (`git cat-file -p` on the blob at `origin/a5-ci-gate-scenario-a-blocking`, since the local worktree was at master post-#564 and did not yet have the `scenario-a-staging-e2e` job). Confirmed the exact failing command: `npx playwright test tests/e2e/a1-staging-auth-stub.spec.js tests/e2e/a2-stripe-test-mode-plan-selection.spec.js tests/e2e/a3-product-feature-ideate-canvas.spec.js tests/e2e/a4-ideate-session-resume.spec.js` — no `--workers` flag. `playwright.config.js` has no `workers` field either, so Playwright falls back to its CPU-based default. The real failing run's log shows `Running 11 tests using 2 workers`, consistent with `ubuntu-latest`'s small vCPU count.

**2. Checked test file signatures to identify which specs actually drive a browser.** `a1-staging-auth-stub.spec.js`'s all 3 tests destructure only `{ request }` — no `page` fixture, no real browser page is driven; this spec is pure HTTP/API traffic and finishes fast. `a2` uses `{ page, request }`/`{ page }` for all 3 tests (real Stripe Checkout interaction). `a3` has 29 `page.*` call sites (canvas rendering, product creation UI); `a4` has 13 (SSE-stream session resume, closing/reopening a real browser context). This means: with 2 workers and 4 files queued, `a1` (fast, API-only) frees its worker almost immediately, and that freed worker then picks up `a3` or `a4` — both heavy, real-browser specs — likely while `a2` (also browser-heavy) is still mid-flight in the other worker. This is the concrete mechanism for genuine concurrent browser automation against the same small CI runner.

**3. Ruled out rate limiting as this run's cause, with direct evidence, not assumption.** `gh secret list --repo heymishy/skills-repo` shows `E2E_STAGING_AUTH_STUB_SECRET` was set at `2026-07-23T05:11:16Z` — before the failing run started at `05:18:29Z`. The failing run's log contains zero HTTP 429 errors anywhere in `a2`'s 3 failures (all are assertion mismatches or Playwright timeouts, never an HTTP-level error). serlb-s1's bypass was genuinely active for this run.

**4. Attempted to download the CI run's own trace/screenshot artifacts for direct evidence of page state at the point of failure.** `gh api repos/heymishy/skills-repo/actions/runs/29981769208/artifacts` returned `{"total_count":0,"artifacts":[]}` — no artifacts were ever uploaded, despite the job failing. Investigated why: `gh run view --job 89124881273` shows the "Upload Playwright traces and screenshots on failure" step's own status as skipped (`-`), not run, despite the preceding "Run Scenario A E2E..." step genuinely failing (exit code 1, no `continue-on-error` on this job by design). Root cause of THIS sub-defect: GitHub Actions implicitly ANDs any custom `if:` expression that does not itself reference `success()`/`failure()`/`always()`/`cancelled()` with `success()`. The upload step's condition (`steps.scenario-a-config.outputs.scenario_a_enabled == 'true' && steps.scenario-a-run.outcome == 'failure'`) has no such reference, so once the run step failed, the implicit `success()` was false and the upload step was silently skipped — the opposite of its intended "upload only on failure" behavior. This meant no trace/screenshot evidence was recoverable from the actual failing run; addressed as AC6/a bundled secondary fix in this story (`always() &&` prepended to the condition) so future failures of this job are diagnosable.

**5. Reproduced locally, in a controlled comparison, using the identical command and identical real staging app.** Confirmed via `git diff HEAD origin/a5-ci-gate-scenario-a-blocking -- tests/e2e/ src/` that the local worktree's `tests/e2e/` and `src/` trees were byte-identical to the PR branch's (PR #563 only touches workflow/config/a new check script, not application or spec code) — so a local run using this worktree is a valid proxy for the CI run's own code, modulo the runner environment itself.
   - **Run A — `--workers=1` (forced serial), full 4-file command:** `Running 11 tests using 1 worker`. Result: 6 passed, 4 skipped, 1 failed — the 1 failure was `a4`'s NFR-Security test, hitting `HTTP 429 "Too many attempts"` at `fixtures/staging-auth.js:148` (this local run does not have `E2E_STAGING_AUTH_STUB_SECRET` set — that CI-only secret bypass was unavailable to this local investigation — so hitting the real 10-per-5-min per-IP limit on the LAST spec in the queue, after `a1`+`a2`+`a3` had already each signed up 1-3 tenants, is expected and unrelated to this story). **All 3 of `a2`'s tests (AC1, AC2, AC3) passed cleanly** in this run.
   - **Run B — default worker count (unspecified, matching CI's own invocation), full 4-file command, run immediately after Run A:** `Running 11 tests using 2 workers`. Result: 9 failed, 2 skipped — but every one of the 9 failures in this run was `HTTP 429`, because this run started well within the same 5-minute window as Run A's ~8 real signups against the same real per-IP limiter, from the same local IP, with no bypass secret available locally. This run is **confounded** by the two runs' shared rate-limit budget and does NOT by itself reproduce the CI failure signature (CI's failures were never 429s) — it is not used as positive evidence for the concurrency hypothesis; only Run A's clean, unconfounded pass under `--workers=1` is used as evidence.
   - **Interpretation:** the CI environment's own constraint (small vCPU count, default-computed worker count) could not be faithfully reproduced on this session's own (8-core) machine — running 2 workers locally does not recreate the same CPU-contention profile as a 2-vCPU GitHub Actions runner, because this environment has ample spare CPU. This means local runs cannot, by themselves, conclusively prove or disprove the CPU-contention mechanism; they can only show that removing concurrency (Run A) does not break anything and does produce a clean `a2` pass. **The real, load-bearing verification for this story is therefore explicitly deferred to a real GitHub Actions run of the fix (AC4/E2E3)** — a local pass alone, on a machine that cannot recreate the CI runner's resource constraints, does not conclusively prove the fix, exactly as the task's own investigation brief anticipated.

**6. Hypotheses considered and their status:**
   - **Concurrency/parallel-worker interference (CPU contention from concurrent real-browser automation on a small runner):** supported by (a) the mechanism identified in step 2 (a1 frees its worker fast, a3/a4 are browser-heavy and likely run concurrently with a2), (b) a2's failure signature (silent no-op on our own click/interaction, not a network/auth error) being consistent with a client-side timing race rather than a server-side or auth problem, and (c) `--workers=1` producing a clean a2 pass locally with no other change. **This is the hypothesis the fix targets.** Not proven with a byte-for-byte identical CPU-constrained reproduction (not achievable in this session's own, more capable, environment) — the real CI run of the fix (AC4) is the conclusive test.
   - **Headless-specific Stripe Checkout behavior (viewport/user-agent differences):** `playwright.config.js`'s `use.headless: true` applies identically whether run locally or in CI — no CI-specific headless/headed or viewport override exists anywhere in the config. Ruled out — there is no config-level difference to explain a CI-only failure.
   - **Timeout-too-tight for CI network latency:** ruled out per Architecture Constraints in the story — AC1's plan state never changed at all across 10 polls (not a marginal miss), and the same spec completed in 33.5s total against the same real staging app earlier this session, well inside AC2/AC3's 30s/15s budgets.
   - **Stripe test-mode rate limiting or bot detection of GitHub Actions IP ranges:** not fully ruled out in isolation — no trace/screenshot was recoverable (see step 4) to directly observe the Stripe Checkout page's state at the moment of failure, and this session cannot directly test from a GitHub Actions IP. However, the concurrency-removal fix is identical regardless of whether the trigger is CPU contention specifically or another concurrency-coupled variable (e.g. if Stripe's own client-side JS behaves differently under system load, or if two near-simultaneous checkout sessions from the same runner exhibit some interaction), so resolving this ambiguity further was not required to select and ship the fix. If the real CI run of this fix (AC4) still fails, that would be evidence favoring this alternative hypothesis and would need separate, further investigation — reported honestly rather than assumed away.

**Decision:** Add `--workers=1` to the `scenario-a-staging-e2e` job's `npx playwright test ...` command only (`.github/workflows/e2e.yml`), not to `playwright.config.js` globally. This keeps the fix's blast radius to exactly the one job with the concurrency problem, leaves the pre-existing 29 local-mocked specs' parallel-worker speed untouched, and does not touch any application code or spec assertion.

**Rationale for not blanket-increasing timeouts as a first resort:** the story's own investigation (step 3 of the Architecture Constraints' "ruled out" list) shows AC1's failure is a complete non-occurrence of the server-side effect (plan state never transitions even once across repeated polling), not a marginal timing miss — a longer timeout would not fix a webhook that never fires because the checkout was never actually submitted. Treating this as "just raise the timeout" would have been treating a symptom, not the cause, and — per this repo's own established culture (evidenced by every fix-forward this session) — would very likely resurface as the same or a differently-shaped flake later.

**Bundled secondary fix (AC6):** `always() &&` prepended to the "Upload Playwright traces and screenshots on failure" step's `if:` condition in the same job, fixing the GitHub-Actions-implicit-`success()` defect discovered in step 4 above, so any future failure of this exact job produces real diagnostic artifacts instead of silently uploading nothing.

## FINDING — local reproduction cannot faithfully recreate the CI runner's CPU-constrained profile

**Date:** 2026-07-23

This session's own sandbox has 8 logical CPUs (`node -e "console.log(require('os').cpus().length)"`), versus GitHub Actions' `ubuntu-latest` small runner (the failing run's own log, `Running 11 tests using 2 workers`, implies Playwright computed a default of 2 workers from a small vCPU count — Playwright's undocumented-but-observed default is roughly half the detected CPU count). Running the same `--workers=2` command locally does not recreate the same resource-contention profile, because this environment has ample spare CPU regardless of concurrent browser automation. This is why local reproduction attempts (see Investigation log, step 5) could show a clean `--workers=1` pass but could not conclusively demonstrate the failure under `--workers=2` locally (that attempt was additionally confounded by this session's own rate-limit budget exhaustion from back-to-back local runs without the CI-only bypass secret). **The real, conclusive verification for this story is a real GitHub Actions run of the fix** — recorded as AC4 in the story and as a mandatory Coding Agent Instruction step, not treated as optional given this constraint.

## FINDING — real CI trace evidence identifies Stripe's own hCaptcha bot-detection as the true, final root cause

**Date:** 2026-07-23

**Context:** After the `--workers=1` fix (above) was pushed, the real CI run of PR #565 still failed on AC2 and AC3 of `a2-stripe-test-mode-plan-selection.spec.js` (AC1 passed cleanly). The `--trace on` diagnostic flag added to the `scenario-a-staging-e2e` job (see the run step's own comment history in `.github/workflows/e2e.yml`) was used specifically to capture visual/network evidence of the page state at the moment of failure, since the concurrency fix had ruled out CPU contention as AC2/AC3's remaining cause.

**Evidence gathered:** `gh api repos/heymishy/skills-repo/actions/runs/29984917608/artifacts` returned artifact `scenario-a-e2e-traces-29984917608`, downloaded and inspected directly (real trace, not simulated). The network trace for AC2 and AC3 both show:
- Every request to `checkout.stripe.com` and its supporting assets succeeding with HTTP 200, through page load and card-field population, exactly as the local (non-CI) run does.
- A request to `api.hcaptcha.com/getcaptcha/24ed0064-...` — Stripe's hosted Checkout page embeds hCaptcha's invisible challenge widget (`hcaptcha-invisible`) as part of Stripe's own bot/fraud-detection layer on the Checkout submit action.
- After that request, complete network silence for approximately 20 seconds — no further requests of any kind — until the test's own timeout fires (AC2's 30s `page.waitForURL` timeout; AC3's 15s `getByText(/declined/i)` timeout). The Checkout page is stuck waiting on a CAPTCHA token that an automated/headless Playwright browser cannot legitimately obtain; the submit click never completes on Stripe's side.

**This conclusively identifies the final root cause of AC2/AC3's continued CI failure**, distinct from (and downstream of) the CPU-contention issue the `--workers=1` fix genuinely resolved: even with the concurrency issue removed, hCaptcha independently blocks the automated interaction. This is a real, third-party (Stripe/hCaptcha) bot-detection constraint — not a defect in this repo's application code, not a defect in the test's assertions or timeouts, and not something a worker-count or timeout adjustment can address, because the page never receives a legitimate CAPTCHA token for headless/automated traffic to proceed past, regardless of how much time is given.

**Hypothesis from the earlier investigation log, resolved:** step 6 of the investigation log above listed "Stripe test-mode rate limiting or bot detection of GitHub Actions IP ranges" as "not fully ruled out in isolation" pending trace evidence. This finding confirms that hypothesis (in the more general form of hCaptcha-based bot detection, not specifically IP-range-based) as the correct, final explanation.

**Operator's decision:** Drop AC2 and AC3 from the CI-blocking gate. AC1 (the real-to-Stripe server-to-server webhook check via `GET /billing/plan-state` — no browser, no CAPTCHA interaction, already passes reliably in CI) remains the CI-blocking signal for this story. AC2/AC3 are reclassified as manual-verification-only, following this repo's own established "External-dependency" gap-type convention (see `templates/test-plan.md` and `a5-ci-gate-scenario-a-blocking`'s own test plan, which already handles its own External-dependency gap for real branch-protection-blocking behaviour — same pattern, different underlying cause here). AC2/AC3 are NOT weakened, deleted, or worked around — they remain real, valid requirements, verifiable by a human running the verification script by hand (a human completing a real Stripe test-mode checkout is not subject to the same invisible-CAPTCHA gate that headless/automated traffic is).

**Explicitly rejected as an approach:** attempting to bypass, defeat, or spoof past Stripe's CAPTCHA (e.g. seeking a CAPTCHA-solving service, or misrepresenting automated traffic as non-automated) was considered and rejected. Stripe's bot-detection is a legitimate anti-fraud control; the correct response to a real anti-bot measure is to stop automating that specific interaction, not to defeat it.

**Implementation carried out on this decision:**
- `tests/e2e/a2-stripe-test-mode-plan-selection.spec.js`: added `test.skip(process.env.CI === 'true', <reason>)` to the AC2 and AC3 test bodies only (not AC1). Not skipped for local/interactive runs.
- `artefacts/2026-07-23-e2e-core-journey-coverage/stories/a2-stripe-test-mode-plan-selection.md`: added a "Coding Agent Instructions — AC2/AC3 CI classification" note recording the reclassification and reason, without touching the ACs themselves.
- `artefacts/2026-07-23-e2e-core-journey-coverage/test-plans/a2-stripe-test-mode-plan-selection-test-plan.md`: AC2/AC3 gap type changed to External-dependency in the AC Coverage table; a new Coverage gaps row and Test Gaps and Risks row added with the reason and handling.
- `artefacts/2026-07-23-e2e-core-journey-coverage/verification-scripts/a2-stripe-test-mode-plan-selection-verification.md`: added a note that Scenarios 2/3 are now the only verification path for AC2/AC3 (no CI-automated equivalent).
- `artefacts/2026-07-23-e2e-core-journey-coverage/stories/a5-ci-gate-scenario-a-blocking.md` and its DoR (`artefacts/2026-07-23-e2e-core-journey-coverage/dor/a5-ci-gate-scenario-a-blocking-dor.md`): added clarifying notes that the gate's own job-level pass/fail semantics are unaffected — "the job passes/fails" already meant "every test the job actually runs passes/fails," and A2's AC1 remains the CI-automated signal contributing to that outcome for A2. No re-review of a5's DoR sign-off was triggered.
- `.github/workflows/e2e.yml`: reverted the temporary `--trace on` diagnostic flag on the `scenario-a-staging-e2e` job's run step, now that the diagnostic question it existed to answer has been conclusively resolved. Judgement: removed rather than left permanently on, since AC2/AC3 no longer run in this job at all (skipped) and leaving trace capture on indefinitely would add artifact-upload overhead to every CI run with no further diagnostic value for this job; a future genuinely new failure can have `--trace on` re-added at that time.

## FINDING — AC1 also fails in every real CI run observed so far; the "AC1 passes reliably" premise does not hold

**Date:** 2026-07-23

**Context:** After implementing the AC2/AC3 CI-skip and pushing it, the real CI run of PR #565 (run 29986106359, then again after a `'use strict'` ordering fix on run 29986188689) still failed the `scenario-a-staging-e2e` job — but the failing test in BOTH runs was **AC1**, not AC2/AC3 (which correctly skipped, confirmed by "4 skipped" in each run's summary). This was unexpected and is reported honestly rather than assumed away.

**Evidence:**
- Run 29986106359 (`Scenario A E2E (staging)`, 2026-07-23T06:46:32Z): `AC1: successful test-mode checkout activates the tenant plan` — `Expected: "paid"`, `Received: "trial"`. 1 failed, 4 skipped, 6 passed.
- Run 29986188689 (same job, 2026-07-23T06:48:05Z, on the `'use strict'`-fixed commit): identical failure — same AC1 test, same `"paid"`/`"trial"` mismatch, same 1 failed/4 skipped/6 passed breakdown.
- The uploaded diagnostic artifact (`scenario-a-e2e-traces-29986188689`, `error-context.md`) for the second run's AC1 failure shows the page snapshot **still on Stripe's hosted Checkout page** ("Subscribe to Starter", $10.00/month, card fields, Pay button) at the moment the assertion failed — i.e. the same "the browser never actually completed the Checkout submission" signature previously attributed to AC2/AC3's hCaptcha stall, not a webhook-delivery-delay symptom (which would show the browser having moved on from the Checkout page already).
- **Re-examining run 29984917608 itself** (the run this session's task briefing cited as evidence that "AC1 already passes reliably in CI, only AC2/AC3 fail"): its own failure log (re-checked directly, not from memory) shows **all three tests — AC1, AC2, AND AC3 — failed**, not just AC2/AC3 (`3 failed` / `2 skipped` / `6 passed`, listing all three AC1/AC2/AC3 test names under "failed"). The trace evidence gathered from this run conclusively identified hCaptcha as AC2/AC3's blocker, but the run itself does not actually demonstrate AC1 passing — that premise appears to have been an incorrect summary carried into this session's task framing, not something borne out by the run's own data.
- **Checking the one other real test-executing run on this branch** (29982995610, 2026-07-23T05:44:59Z, before `--trace on` was added): same pattern — all three (AC1, AC2, AC3) failed, AC1 with the identical `"trial"` received value.

**This means: across all 4 real CI runs on this branch that reached actual test execution (29982995610, 29984917608, 29986106359, 29986188689), AC1 has failed every single time**, with a 0/4 real-CI pass rate, not the "passes reliably" premise this task's implementation plan was built on. The most likely explanation, given the identical failure signature (browser still on Stripe's Checkout page, payment never actually submitted) to AC2/AC3's confirmed hCaptcha finding, is that Stripe's bot-detection is not selectively targeting AC2/AC3's specific assertions — it is stalling the underlying Checkout-page submit interaction itself, which AC1 also depends on (AC1 differs from AC2/AC3 only in how it verifies the *outcome* — a fresh re-login against the webhook-driven plan-state, rather than tracking the browser's redirect — but it still requires the same real browser click-to-pay submission to actually succeed on Stripe's side first).

**Not independently confirmed via network trace** (this branch's `--trace on` diagnostic flag was deliberately reverted as part of this session's own work, once the AC2/AC3 diagnosis was believed complete — see the revert note in `.github/workflows/e2e.yml`'s run-step comment). The `error-context.md` page-snapshot evidence (page stuck on Checkout) is consistent with the hCaptcha hypothesis but is not itself a definitive network-level confirmation for AC1 specifically, unlike the network-trace evidence gathered for AC2/AC3.

**Status: unresolved, requires an operator decision.** This session's assigned task was scoped specifically to reclassifying AC2/AC3 as manual-verification-only while treating AC1 as the retained CI-blocking signal — a premise this new evidence does not support. Extending the same reclassification to AC1, or pursuing a different fix (e.g. re-enabling `--trace on` for one more diagnostic run to get network-level confirmation for AC1, or replacing AC1's real-browser-driven Checkout interaction with a Stripe test-mode API-level trigger that does not depend on hosted Checkout's UI at all) is a scope decision beyond what this session was authorized to implement unilaterally, since it touches AC1 (out of this session's explicit mandate) and/or the story's fundamental automation approach. Not acted on further in this session — reported here for the operator's decision rather than guessed at.

## DECISION — corrected: extend the CI-only skip to AC1 as well; A2 contributes zero automated CI-blocking signal

**Date:** 2026-07-23

**Context:** The previous "FINDING" entry above (2026-07-23, "AC1 also fails in every real CI run observed") reported, but did not act on, evidence that AC1 failed in all 4 real CI runs on this branch (29982995610, 29984917608, 29986106359, 29986188689), contradicting the premise that AC1 "verifies plan-state via a fresh re-login/server-side webhook, never re-driving the browser through Checkout, so it's unaffected by Stripe's CAPTCHA." That prior session treated extending the reclassification to AC1 as outside its authorized scope and left the matter for an operator decision. This entry records that operator decision and the direct evidence that confirms it.

**Independent verification, direct trace analysis (operator, this session):** The operator downloaded and directly examined real CI run 29984917608's own trace artifact (`scenario-a-e2e-traces-29984917608`, PR #565) — specifically AC1's own browser-page network log, `1-trace.network` (not the API-only `0-trace.network`, which the earlier hCaptcha finding for AC2/AC3 had relied on). AC1's own test body (`tests/e2e/a2-stripe-test-mode-plan-selection.spec.js`, the `AC1` test, then around line 146) calls the SAME `goToCheckoutAndFillCard(page, SUCCESS_CARD)` helper and clicks the SAME `button[type="submit"]` as AC2/AC3, before its webhook-based `GET /billing/plan-state` assertion runs. The `1-trace.network` file shows: full checkout page load, the exact same invisible-hCaptcha challenge sequence as AC2/AC3 (`hcaptcha-invisible`, `api.hcaptcha.com/getcaptcha/24ed0064-...`), then the interaction stalls — no further meaningful network activity. The checkout submission itself never completes, so no webhook ever fires, and `GET /billing/plan-state` correctly never flips to `"paid"` because the payment genuinely never completed on Stripe's side.

**Corrected conclusion:** All 3 of A2's ACs (AC1, AC2, AC3) are blocked by Stripe's own bot-detection on the checkout submission itself — not just the post-checkout browser-redirect step AC2/AC3 depend on. AC1 differs from AC2/AC3 only in how it verifies the *outcome* of a successful checkout (fresh re-login against the webhook-driven plan-state, rather than tracking the browser's own post-checkout redirect); it still requires the same real-browser Checkout submission to succeed first, and that submission is exactly what Stripe's hCaptcha blocks for all three ACs alike. There is no way to automate past a real CAPTCHA challenge from CI; this is a genuine, unfixable-by-us third-party constraint (Stripe/hCaptcha targeting GitHub Actions' traffic specifically — this exact same flow passes reliably from interactive/local environments, confirmed multiple times across this investigation).

**This corrects the earlier "AC2/AC3-only" fix** (see the "FINDING — real CI trace evidence identifies Stripe's own hCaptcha bot-detection as the true, final root cause" entry above), which was based on the premise that AC1 was unaffected. That premise was independently verified wrong using the direct trace evidence described above.

**Final decision:** Reclassify ALL THREE of A2's ACs as manual-verification-only in CI. A2 (the whole `a2-stripe-test-mode-plan-selection` story) contributes ZERO automated CI-blocking signal going forward — Scenario A's CI-blocking gate (`a5-ci-gate-scenario-a-blocking`, PR #563) becomes A1 + A3 + A4 only. Billing/plan-activation correctness is verified via the story's existing manual verification script instead. None of AC1/AC2/AC3 are weakened, deleted, or worked around — they remain real, valid requirements, verifiable by a human running the verification script by hand (a human completing a real Stripe test-mode checkout is not subject to the same invisible-CAPTCHA gate that headless/automated traffic is).

**Explicitly rejected as an approach (reaffirmed):** attempting to bypass, defeat, or spoof past Stripe's CAPTCHA remains rejected, for the same reason recorded in the earlier AC2/AC3 finding — Stripe's bot-detection is a legitimate anti-fraud control.
