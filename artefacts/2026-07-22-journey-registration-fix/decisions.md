# Decisions: Journey Registration Fix

## FIX ‚Äî "New feature" journeys were never registered in the shared in-memory store (2026-07-22)

**Context:** Operator reported "Journey not found" at `/api/journey/:id/gate-confirm` after creating a product, clicking "New feature," completing discovery, and saving the artefact. Direct inspection of the real `journeys` row on `wuce-staging` confirmed it existed in Postgres but with an empty `data: {}` ‚Äî the row was written via a raw, direct `INSERT INTO journeys` in `handlePostProductFeature` (`routes/products.js`, added by jrf-s1), completely bypassing `modules/journey-store.js`'s `createJourney()`. This meant the journey was never added to the shared in-memory `_journeys` Map every other journey-store operation reads from ‚Äî `getJourney`, `setActiveSession`, `completeStage` all silently miss it. `setActiveSession`'s own `if (!journey) return;` guard meant the discovery session's `activeSessionId`/`activeSkill` were never recorded anywhere, in memory or Postgres.

**Distinct from srf-s1's session bug:** this is NOT a redeploy-timing/cache-loss issue. The journey was never registered in the in-memory store to begin with ‚Äî any subsequent request, deploy or no deploy, would 404 the same way. Every journey created via "New feature" on the product page has been broken past discovery since jrf-s1 shipped.

**Decision:** Rewrite `handlePostProductFeature` to register through `journey-store.js`'s `createJourney()` ‚Üí `setJourneyFields()` ‚Äî the exact same pattern `handlePostJourney` (`routes/journey.js`) already uses correctly, and which jrf-s1's own code comment claimed to follow but didn't actually implement. Extend `journey-store-pg.js`'s `saveJourney()` to also write `product_id` into its own real column (not just the JSONB `data` blob), since `handleGetProductView`'s journeys-listing query filters directly on that column ‚Äî a gap the old raw INSERT covered (it did set `product_id` via its own SQL) that the shared `createJourney`/`saveJourney` path didn't otherwise close.

**Test methodology finding:** jrf-s1's own test file (`tests/check-jrf-s1-new-feature-redirect.js`) contains a hand-copied reimplementation of `handlePostProductFeature` inside the test file itself, rather than importing and calling the real exported function ‚Äî a mock-shape-divergence pattern this repo's own CLAUDE.md already names as a recurring risk (tir-s5/tir-s8). This is very likely why the bug went uncaught: the test validated a hand-written stand-in that happened to do the right thing, not the actual shipped code. `/improve` candidate: any future test for a `routes/*.js` handler must import and call the real exported function, never re-implement its logic locally.

**Verification:** New test file `tests/check-jrf-s2-register-product-feature-journeys.js` ‚Äî 6/6 passing, covering AC1-AC5, all calling the real exported `handlePostProductFeature`/`handlePostGateConfirm` directly (not reimplementations). Existing regression suites: `check-jrf-s1-new-feature-redirect.js` (5/5), `check-cdg4-gate-confirm-validation.js` (10/10), `check-psh-s4-navigation.js` (found and fixed a genuine regression ‚Äî its own T3 test asserted the OLD raw-INSERT mechanism; updated to verify the new, correct in-memory-registration mechanism instead ‚Äî 6/6 after the fix). Full 362-file suite: 37 failed, identical to the documented baseline ‚Äî zero net regressions.

Deployed to `wuce-staging`, clean boot confirmed via logs.

**Known, accepted gap:** The operator's own already-broken journey (`58e606e9-...`) and the 2 earlier placeholder journeys (`new-feature-aa349dd1`, `new-feature-f3765c1a`) discovered earlier this session remain permanently un-registerable ‚Äî their skill sessions no longer exist to resume. Not retroactively repaired; the operator will need to start a fresh "New feature" journey after this fix ships.
