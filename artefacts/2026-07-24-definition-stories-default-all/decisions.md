# Decisions: Default to all stories from /definition when starting the per-story review sequence (dsda-s1)

## Server-side extractor co-located in journey.js, not a shared module

**Date:** 2026-07-24
**Context:** The story's Architecture Constraints require the new server-side story-ID extractor to mirror the client-side `parseDefinitionArtefact`'s regex patterns exactly, without introducing a second, independently-drifting parser. Out of Scope explicitly excludes refactoring `parseDefinitionArtefact` into a shared client+server module.
**Decision:** `extractStoryIdsFromDefinitionArtefact` is added directly in `src/web-ui/routes/journey.js`, immediately above `handleGetStories`, with a comment cross-referencing the exact client-side function it mirrors. It supports only the two documented formats (H1 epic/story headers, flat-story fallback) — not the third, older nested-`###`-header fallback branch `parseDefinitionArtefact` also has, since that legacy format isn't named in this story's Architecture Constraints or test plan.
**Rationale:** A shared module is a larger refactor explicitly out of scope for this bounded fix. Co-locating the new function next to its only caller, with an explicit code-comment cross-reference to the client-side twin, satisfies the "don't silently drift" constraint without expanding scope. Cross-checked directly (not just by inspection): the new server-side function was run against the same fixtures as the real client-side `parseDefinitionArtefact` (extracted and evaluated from its literal source in `journey.js`), confirming identical story-ID output for both supported formats.

## AC3's E2E test uses the existing definition.success.json fixture as-is, not a modified one

**Date:** 2026-07-24
**Context:** The DoR contract calls for a real-browser Playwright test of AC3 (the manual edit affordance remains reachable). The shared mock-LLM fixture `tests/e2e/fixtures/llm-gateway/definition.success.json` (used by `bri-s3.2` and other specs) uses the third, unsupported nested-header format, so a full mocked-LLM walkthrough using this fixture yields an empty auto-populated textarea (AC4's fallback path), not a genuinely pre-filled one.
**Decision:** Left `definition.success.json` untouched. `tests/e2e/dsda-s1-default-all-stories.spec.js` drives a real mocked journey to `/journey/:id/stories` and proves the textarea is visible, genuinely editable (`toBeEditable()`), and that a typed-in value is what proceeds — this is a legitimate real-browser proof of AC3's substance (the edit affordance is reachable and authoritative) even though this particular run's starting point is an empty rather than pre-filled textarea. AC1's true auto-population (a real pre-filled value, not empty) is covered at the integration level in `tests/check-dsda-s1-default-all-stories.js` against fixture content authored directly in the two currently-supported formats.
**Rationale:** Modifying a fixture shared across multiple other stories' E2E specs (`bri-s3.2` and others) to serve this one story's test is a disproportionate, avoidable risk for a fix this narrow — the safer path is a test-file-local fixture for the integration tests, and accepting that the one shared-fixture-driven E2E run exercises AC4's fallback path alongside AC3's edit-affordance proof. Both are honestly documented in the spec file's own header comment.
