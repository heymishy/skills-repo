## User Story

As a **web UI operator running a structured skill session**,
I want the **next question to be generated by the model based on my prior answers**,
So that **I am asked questions that are relevant to what I have said rather than a fixed list that ignores my context**.

## Benefit Linkage

**Metric moved:** M2 â€” Model question adaptation rate (> 70% of generated questions show visible adaptation)
**How:** This story is the direct delivery mechanism for metric M2 â€” it introduces the model call that generates each next question, replacing the static mechanically-next item. Every question served after answer submission is a model-generated question unless the API call fails, at which point the static fallback fires silently.

**Secondary metric:** M1 â€” Fallback invisibility rate â€” the silent fallback implemented in this story is the primary control for that metric.

## Architecture Constraints

- **D37 / injectable adapter rule (ADR-009):** A new injectable adapter `_nextQuestionExecutor` must be introduced following the exact same pattern as `_skillTurnExecutor` in `src/web-ui/routes/skills.js`. The default stub must throw: `throw new Error('Adapter not wired: _nextQuestionExecutor. Call setNextQuestionExecutorAdapter() with a real implementation before use.')`. A corresponding `setNextQuestionExecutorAdapter(fn)` export is required. The production wiring must be explicit in the server entry point and verified by a test.
- **No new npm dependencies** â€” implementation must use Node built-ins only.
- **No Express** â€” raw `http.createServer` pattern only; no router framework.
- **`req.session.accessToken` canonical** â€” the token passed to `_nextQuestionExecutor` must come from `req.session.accessToken`, never `req.session.token`.
- **Static list retained as fallback** â€” `session.questions` (static list) remains the source of truth for question count and progress display. `dynamicQuestions` are per-turn substitutions only, not additions.

## Dependencies

- **Upstream:** wuce.26 (merged) â€” `_skillTurnExecutor` adapter pattern, `htmlRecordAnswer`, `registerHtmlSession` with product context loading. All in production on master.
- **Downstream:** dsq.2 (section confirmation loop) depends on the session store structure introduced here.

## Acceptance Criteria

**AC1:** Given a web UI skill session is in progress and the operator submits an answer, when `htmlRecordAnswer` completes the existing model-response call, then a second model call is made to `_nextQuestionExecutor` with a prompt that includes the full SKILL.md content, the full conversation history (all prior Q&A pairs plus the just-recorded answer), and the instruction "Given the skill instructions and the conversation so far, what is the single best next question to ask the operator?".

**AC2:** Given the `_nextQuestionExecutor` call returns a non-empty string, when `htmlGetNextQuestion` is called for the next index, then the returned question text is the model-generated string (not the static list item at that index), and `session.dynamicQuestions[answerIndex]` holds the generated value.

**AC3:** Given the `_nextQuestionExecutor` call throws or returns an empty/null response, when `htmlGetNextQuestion` is called for the next index, then the returned question text is the static list item at that index (silent fallback) â€” no error is surfaced to the operator, and the session continues normally.

**AC4:** Given a session where some dynamic questions were generated and some fell back to static, when `htmlGetNextQuestion` is called for any index, then `questionIndex` and `totalQuestions` in the returned object always reflect the static list count â€” the dynamic substitution does not change the reported question count or progress.

**AC5:** Given a new injectable adapter `setNextQuestionExecutorAdapter(fn)` is exported from `src/web-ui/routes/skills.js`, when the adapter is wired in `src/web-ui/server.js` (or equivalent wiring module) to `skillsAdapter.nextQuestionExecutor`, then a test confirms the wiring is present and the default stub is not in use in production.

**AC6:** Given the default stub for `_nextQuestionExecutor` is invoked (i.e. production wiring has not been applied), when any code path calls the stub, then it throws with message `'Adapter not wired: _nextQuestionExecutor. Call setNextQuestionExecutorAdapter() with a real implementation before use.'` â€” it does not return null or an empty string silently.

**AC7:** Given all 14 wuce.26 tests are passing on master before this story is implemented, when this story's implementation is merged, then all 14 tests continue to pass with no regressions.

## Out of Scope

- Streaming or SSE delivery of the generated question â€” deferred to a later story
- Changing the question count or progress indicator based on dynamic content â€” static list length governs these throughout
- Model-generated questions for the very first question of a session â€” dynamic generation starts from question 2 (there is no prior answer to reason from at index 0)
- Persisting `dynamicQuestions` to disk between server restarts â€” in-memory session store only, same as existing session data

## NFRs

- **Performance:** The `_nextQuestionExecutor` call must have a timeout of â‰¤ 10 000 ms. If it does not complete within that window, the static fallback fires silently â€” the operator is not left waiting beyond the existing answer-submission response time plus 10 s.
- **Security:** The token passed to `_nextQuestionExecutor` must be `req.session.accessToken` â€” never logged, never included in error messages surfaced to the operator.
- **Resilience:** Any exception thrown by `_nextQuestionExecutor` (network error, API error, JSON parse error) must be caught at the call site and treated as a fallback trigger â€” the exception must not propagate to the HTTP response handler.
