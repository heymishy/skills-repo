# DoR Contract: Define lockfile schema and implement `pin` and `verify` commands

**Story:** `artefacts/2026-04-24-platform-onboarding-distribution/stories/p11.4-lockfile-schema-pin-verify.md`
**Date approved:** 2026-04-28

---

## What will be built

The `pin()` and `verify()` stub functions in `src/enforcement/cli-adapter.js` will be replaced with real implementations. `pin()` walks `.github/skills/` recursively, computes SHA-256 for each `SKILL.md` using Node.js `crypto`, and writes a lockfile to `.github/skills/skill-lockfile.json` (path per ADR-016). `verify()` reads the lockfile, recomputes hashes, and returns a structured result object with `passed` and `mismatches`. A JSON Schema document for the lockfile format will be created at `docs/skill-lockfile.schema.json`. Both implementations use Node.js built-ins only — no new npm dependencies.

## What will NOT be built

- Remote fetch of skills from an upstream URL — this is p11.5
- CLI command routing or argument parsing beyond what exists — functions are called directly
- WS4.3 hash verification integration — the `sha256` field in the schema satisfies AC5's schema-level interoperability signal; WS4.3 itself is out of scope
- Any UI, dashboard, or human-readable output formatting

## Assumptions

1. SHA-256 is computed over the UTF-8 file contents of each SKILL.md (not metadata such as filename or timestamp).
2. The lockfile is overwritten entirely on each `pin()` call — it is not incrementally updated.
3. The `docs/skill-lockfile.schema.json` file is a standalone JSON document; it is not imported by the Node.js implementation (it is a documentation/tooling artefact).
4. `verify()` only checks files that appear in the lockfile; new SKILL.md files added after `pin()` was last run are NOT treated as violations (no "missing entry" check) — this matches the MVP ACs.
