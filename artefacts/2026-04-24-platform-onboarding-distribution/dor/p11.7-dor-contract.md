# DoR Contract: Extend `/start` with brownfield routing logic

**Story:** `artefacts/2026-04-24-platform-onboarding-distribution/stories/p11.7-start-brownfield-routing.md`
**Date approved:** 2026-04-28

---

## What will be built

`.github/skills/start/SKILL.md` will be extended with a brownfield routing section containing three detection entries in priority order (C > A > B): Entry C (DoD artefacts found → retrospective template), Entry A (in-flight stories in pipeline-state.json → `/tdd`), and Entry B (src/ directory with code but no pipeline state → `/reverse-engineer`). The first matching entry wins. A `<!-- brownfield-routing -->` marker will be added inside the brownfield routing section. The `brownfield-routing` string will be added to the start skill's required strings in `check-skill-contracts.js`.

## What will NOT be built

- Full codebase analysis or file inventory — the three entry signals are the only detection mechanism
- Cross-entry overlap handling beyond first-match-wins priority
- Any change to the greenfield orientation path established by p11.6

## Assumptions

1. "DoD artefacts in artefacts/" (Entry C signal) means one or more `.md` files in an `artefacts/[feature]/dod/` subdirectory.
2. "In-flight stories" (Entry A signal) means `pipeline-state.json` exists and contains at least one story with `stage` equal to `"implementation"`, `"branch-setup"`, or `"test-plan"`.
3. The brownfield routing section is appended after the greenfield orientation section established by p11.6. The single-turn-constraint marker from p11.6 must remain present.
4. "First match wins" means the skill checks C first; if C matches, it routes there without checking A or B. This is implemented in instruction text (AI agent follows priority). The test asserts the priority order is explicitly stated in the SKILL.md.
