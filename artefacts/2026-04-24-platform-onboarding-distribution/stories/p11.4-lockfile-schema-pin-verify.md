## Story: Define lockfile schema and implement `pin` and `verify` in `cli-adapter.js`

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-onboarding-lockfile.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **platform consumer**,
I want to run `pin` to record the current SHA-256 hash of every skill file in my repository and `verify` to check whether any skill has drifted since I pinned,
So that I can detect unauthorised or unintended changes to the skills that govern my delivery pipeline (M4 — lockfile pin/verify round-trip integrity), and so that the lockfile schema I produce is forward-compatible with WS4.3 hash verification.

## Benefit Linkage

**Metric moved:** M4 — Lockfile pin/verify round-trip integrity
**How:** Currently, `pin()` and `verify()` in `cli-adapter.js` are stubs that return `{ status: 'ok' }` regardless of actual file state. This story replaces them with real implementations: `pin()` writes a JSON lockfile with per-skill SHA-256 hashes; `verify()` reads that lockfile and computes current hashes, returning a deterministic pass/fail with the names of any drifted files. M4's minimum validation signal (deterministic, non-flaky result across two runs on the same machine) is achievable only with real implementations.

## Architecture Constraints

- No new npm dependencies — use Node.js built-in `crypto` (`createHash('sha256')`) and `fs` modules only. This is a mandatory constraint established by the WS0.6 pattern (ADR constraint in discovery).
- Lockfile schema must define a `schemaVersion` field (string, semver format e.g. `"1.0.0"`) to enable forward-compatible migration. WS4.3 (Phase 5 spec integrity workstream) will read the `sha256` field per skill entry. Do NOT include verbatim instruction text in the lockfile — G19 is intentionally deferred to Phase 6 (see `artefacts/phase5-6-roadmap.md` G19 entry).
- Lockfile file path: `.github/skills/skill-lockfile.json` — this path is a design decision that must be documented in `decisions.md` as it constrains WS4.3 integration.
- The lockfile schema JSON document must be saved as `docs/skill-lockfile.schema.json` in the repository root — creating a versioned schema reference that WS4.3 can read without depending on skill SKILL.md content.
- ADR-003 (schema-first): the lockfile is a separate JSON file from `pipeline-state.json` and is not validated by `pipeline-state.schema.json`. No changes to `pipeline-state.schema.json` are required for this story.

## Dependencies

- **Upstream:** None — the lockfile schema is defined from scratch in this story. p11.5 (init/fetch) depends on this story.
- **Downstream:** p11.5 (`p11.5-cli-init-fetch.md`) — `init` creates the lockfile structure defined here. p11.5 must not be started until this story is DoD-complete.

## Acceptance Criteria

**AC1:** Given a populated `.github/skills/` directory containing at least one SKILL.md file, when `pin()` is called in `src/enforcement/cli-adapter.js`, then it writes a valid JSON file to `.github/skills/skill-lockfile.json` conforming to the following schema: `{ schemaVersion: "1.0.0", pinnedAt: "<ISO-8601 timestamp>", skills: [{ skill: "<skill-name>", path: "<relative-path-from-repo-root>", sha256: "<64-char hex digest>" }] }` — one entry per SKILL.md file found recursively under `.github/skills/`.

**AC2:** Given a lockfile written by `pin()`, when `verify()` is called with no changes to any SKILL.md files since pinning, then it returns `{ status: 'pass', drifted: [], checked: <n> }` where `n` is the count of skills verified.

**AC3:** Given a lockfile written by `pin()`, when one SKILL.md file is modified (any change to content) and `verify()` is called, then it returns `{ status: 'fail', drifted: [{ skill: "<skill-name>", path: "<path>", expected: "<original-sha256>", actual: "<new-sha256>" }] }` listing only the drifted files.

**AC4:** Given `pin()` is called twice with no changes to SKILL.md files between calls, when both resulting lockfiles are compared, then the `sha256` values for each skill are identical across both runs — confirming determinism.

**AC5:** Given the lockfile schema JSON document at `docs/skill-lockfile.schema.json`, when it is read by any future WS4.3 implementation, then each entry in the `skills` array provides a `sha256` field that is the SHA-256 hex digest of the corresponding file's UTF-8 byte content at the time of pinning.

**AC6:** Given `npm test` after this change, then all existing test suites pass with no new failures. A new test file `tests/check-p11-lockfile.js` covers AC1–AC4 above.

## Out of Scope

- `upgrade` command with lockfile-diff visibility (WS0.2 — a subsequent story after the lockfile model is established).
- `init` and `fetch` command implementations — those are p11.5.
- WS4.3 hash verification integration (Phase 5 workstream) — the lockfile schema defined here is the entry condition for WS4.3, but WS4.3 integration is a separate story.
- Lockfile for non-SKILL.md files (templates, standards files) — scope is `.github/skills/*/SKILL.md` only at MVP.
- Lockfile signing or cryptographic key management — SHA-256 hash integrity only, no signature chain.

## NFRs

- **Performance:** `pin()` and `verify()` complete within 5 seconds for a repository with up to 50 skill files (the current repository has ~39 skills; headroom for growth to 50 is the target).
- **Security:** No credentials, tokens, or personal data written to the lockfile. File content is hashed; content is not stored.
- **None other identified** — checked against `.github/architecture-guardrails.md`.
