## Story: Implement `init` and `fetch` commands in `cli-adapter.js`

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-onboarding-lockfile.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **platform consumer**,
I want to run `init` to set up the skill package structure in my repository and `fetch` to retrieve the latest upstream skill content,
So that I can onboard from a single terminal command and my lockfile structure is ready for `pin` to record my initial skill hashes (M1 ŌĆö time-to-first-skill-run, as part of the guided onboarding flow that `/start` references).

## Benefit Linkage

**Metric moved:** M1 ŌĆö Time-to-first-skill-run
**How:** The `/start` concierge (p11.6) guides new consumers through `init`, `fetch`, and `pin` as the first three terminal commands in their onboarding sequence. If `init` and `fetch` are stubs, consumers who follow `/start`'s guidance will receive no meaningful feedback. Real `init` and `fetch` implementations make the guided onboarding sequence executable, not just readable ŌĆö reducing the time-to-first-skill-run by eliminating the manual setup steps that currently require consulting ONBOARDING.md.

## Architecture Constraints

- No new npm dependencies ŌĆö use Node.js built-ins only (`fs`, `path`, `child_process` for git operations). ADR constraint from discovery, consistent with WS0.6 pattern.
- `fetch` reads the configured upstream remote from `context.yml` under `skills_upstream.remote` ŌĆö per ADR-004 (context.yml is the single config source of truth). It must NOT hardcode a remote URL.
- `init` must be idempotent: running it twice on a repository that already has the lockfile structure must not overwrite or corrupt the existing lockfile.
- The lockfile structure created by `init` must conform to the schema defined in p11.4 (`schemaVersion: "1.0.0"`, empty `skills: []` array at init time ŌĆö populated by `pin` in a subsequent step).
- `check-pipeline-artefact-paths.js`: no new artefact output paths ŌĆö no PIPELINE_PATHS update needed.

## Dependencies

- **Upstream:** p11.4 (`p11.4-lockfile-schema-pin-verify.md`) ŌĆö `init` creates the lockfile stub conforming to the schema defined in p11.4. This story must not be dispatched until p11.4 is DoD-complete.
- **Downstream:** p11.6 (`p11.6-start-skill.md`) ŌĆö the `/start` SKILL.md references `init` and `fetch` in its guidance. p11.6 does not depend on this story's implementation being complete (SKILL.md can reference commands before they are implemented), but p11.5 should be DoD-complete before `/start` is validated in production.

## Acceptance Criteria

**AC1:** Given a repository with no `.github/skills/` directory, when `init()` is called in `src/enforcement/cli-adapter.js`, then it creates the `.github/skills/` directory and writes a stub lockfile at `.github/skills/skill-lockfile.json` with `{ schemaVersion: "1.0.0", pinnedAt: "<ISO-8601 timestamp>", skills: [] }`, and returns `{ status: 'ok', command: 'init', created: true }`.

**AC2:** Given a repository that has already had `init()` run (`.github/skills/skill-lockfile.json` exists), when `init()` is called again, then it returns `{ status: 'ok', command: 'init', created: false }` without modifying the existing lockfile.

**AC3:** Given a `context.yml` where `skills_upstream.remote` is `null` or absent, when `fetch()` is called, then it returns `{ status: 'not-configured', command: 'fetch', message: "skills_upstream.remote is not set in .github/context.yml. Add the remote URL and re-run fetch." }` without throwing an error.

**AC4:** Given a `context.yml` where `skills_upstream.remote` is set to a valid git remote URL, when `fetch()` is called, then it executes `git fetch <remote>` using `child_process.execSync`, and returns `{ status: 'ok', command: 'fetch', remote: '<url>' }` on success.

**AC5:** Given `npm test` after this change, then all existing test suites pass and a new test file `tests/check-p11-init-fetch.js` covers AC1ŌĆōAC4 with no external git remote required (mock or stub the `execSync` call for the fetch test).

## Out of Scope

- Actual file copy/sync of skill content after `fetch` ŌĆö the git fetch retrieves ref metadata; the consumer sync step is handled by `sync-from-upstream.sh/ps1`. Real file copy for non-git consumers is a WS0.4 concern.
- `pin` and `verify` implementation ŌĆö those are p11.4.
- `upgrade` command (WS0.2).
- Writing or modifying any files other than `src/enforcement/cli-adapter.js` and `tests/check-p11-init-fetch.js`.

## NFRs

- **Performance:** `init()` completes within 1 second. `fetch()` completion time is network-dependent; no timeout requirement at MVP.
- **Security:** `init()` does not write any credentials or tokens to the lockfile stub. The `fetch()` git remote URL is read from config, not hardcoded ŌĆö no opportunity for injection via static content.
- **None other identified** ŌĆö checked against `.github/architecture-guardrails.md`.
