## Story: Create `/start` SKILL.md — greenfield context orientation

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-onboarding-lockfile.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **new platform consumer** (any role — engineer, BA, product manager),
I want to run `/start` as my first command after cloning the repository and receive a guided, conversational orientation in a single turn,
So that I know exactly what the pipeline is, where I currently am in it, and what my next action is — without reading `ONBOARDING.md` beyond the instruction to run `/start` (M1 — time-to-first-skill-run under 2 minutes; M2 — zero support contacts; MM1 — `/start` concierge hypothesis).

## Benefit Linkage

**Metric moved:** M1 (primary), M2, MM1
**How:** The current onboarding path requires reading an 8-step `ONBOARDING.md` before the consumer knows what to do. The `/start` skill replaces that path for the orientation step: a consumer who runs `/start` receives one paragraph describing the pipeline, a state read of `workspace/state.json` and `artefacts/` to determine their current context, and a single concrete next action. If completed within a single conversational turn, M1 (under 2 minutes) is achievable and M2 (zero orientation contacts) becomes measurable.

## Architecture Constraints

- ADR-011 (artefact-first rule): this story is the required artefact chain for the creation of `.github/skills/start/SKILL.md`. The SKILL.md must be committed in the same PR as this story's test and DoR artefacts.
- SKILL.md files are Markdown only — no embedded HTML except HTML comments. No trailing whitespace. Unix line endings (`.github/architecture-guardrails.md` style guide).
- `check-skill-contracts.js`: a new contract entry must be added for the `/start` skill in the same commit that creates the SKILL.md. The contract must include at least one required structural marker string that enforces the sub-2-minute constraint design (e.g. a `<!-- single-turn-constraint -->` marker that documents the design intent). If this marker is removed, the contract fails.
- `check-pipeline-artefact-paths.js`: `/start` reads `workspace/state.json` and `artefacts/` but does not write any pipeline artefact. No PIPELINE_PATHS update required.
- The `/start` SKILL.md must conform to the established SKILL.md structure used by other skills in `.github/skills/` — YAML frontmatter (`name`, `description`, `triggers`), then instruction sections. Check an existing SKILL.md (e.g. `/workflow/SKILL.md`) for the pattern.
- Sub-2-minute constraint: the SKILL.md must be designed so that its complete output fits within a single conversational turn. It must NOT begin an interview sequence (multiple questions before providing the orientation). The first response to `/start` is the orientation output itself.

## Dependencies

- **Upstream:** None — `/start` SKILL.md is an instruction file; it does not depend on `init`/`fetch`/`pin`/`verify` being implemented to be written. It references those commands in its guidance text.
- **Downstream:** p11.7 (`p11.7-start-brownfield-routing.md`) — brownfield routing is an extension to the SKILL.md created here. p11.7 depends on this story being DoD-complete.

## Acceptance Criteria

**AC1:** Given a repository with no `workspace/state.json` and no content in `artefacts/` (greenfield — just cloned), when `/start` is invoked, then the output within a single conversational turn: (a) describes the pipeline in one sentence (what it is and why it exists), (b) identifies the current state as "new installation — no pipeline state found", and (c) names the exact next action for a greenfield consumer (e.g. "Run `/discovery` to begin your first feature, or `/workflow` to see the full pipeline map").

**AC2:** Given a repository with a `workspace/state.json` where `currentPhase` is set (e.g. `"discovery"` or `"benefit-metric"`), when `/start` is invoked, then the output names the current pipeline phase, identifies what skill was last run, and names the next skill to run — without requiring the consumer to read any other file.

**AC3:** Given the `/start` SKILL.md, when `check-skill-contracts.js` runs (via `npm test`), then the new contract entry for the `/start` skill passes — the required structural marker(s) are present in the SKILL.md.

**AC4:** Given the `/start` SKILL.md text, when it is read as a design review, then it does not contain any multi-step interview sequence before producing the orientation output — the first invocation produces the orientation directly.

**AC5:** Given the `/start` SKILL.md `triggers` block in the YAML frontmatter, when it is read, then it includes at minimum: `"start"`, `"where am I"`, `"what do I do"`, `"help me get started"`, `"I just cloned this"` — making the skill discoverable by the natural language a new consumer is likely to type.

## Out of Scope

- Brownfield context detection and routing to Entry A, B, or C — that is p11.7. This story delivers greenfield orientation only.
- CLI commands (`init`, `fetch`, `pin`, `verify`) — those are p11.4 and p11.5. The `/start` SKILL.md references these commands but does not implement them.
- Multi-turn interview or wizard flow — out of scope by design; the sub-2-minute constraint requires single-turn output.
- Teams bot adaptation of the `/start` output (WS0.7 — subsequent feature).
- Modification to `ONBOARDING.md` — that document continues to exist; `/start` does not replace it, it augments it as the guided entry point that reads current pipeline state.

## NFRs

- **None identified** — this is a SKILL.md instruction-text-only change. No performance, security, or accessibility NFRs apply to SKILL.md content.
- The sub-2-minute constraint is a design constraint on the SKILL.md content structure (single-turn output), captured in AC4. It is not an NFR in the traditional sense.
