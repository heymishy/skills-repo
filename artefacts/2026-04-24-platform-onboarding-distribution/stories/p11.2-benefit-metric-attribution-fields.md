## Story: Add required attribution fields to `/benefit-metric` template and SKILL.md

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-governance-attribution.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **platform maintainer**,
I want the `/benefit-metric` template to include required `metric owner` and `reviewers` attribution fields, and the `/benefit-metric` SKILL.md to prompt the operator to populate these at the completion step,
So that every benefit-metric artefact produced after this change names who is accountable for measuring and reporting each metric — completing the attribution model for M3 and ensuring that metric ownership is structurally assigned, not left implicit.

## Benefit Linkage

**Metric moved:** M3 — Non-engineering outer loop attribution rate
**How:** The benefit-metric artefact is the second outer loop document that requires non-engineering sign-off to be meaningful. If only the discovery artefact has attribution fields, a single engineer can produce the discovery + the benefit metric unilaterally and H-GOV at DoR will only catch the discovery gap. Adding metric owner and reviewer fields to the benefit-metric template closes this second gap and ensures the attribution model covers the full outer loop.

## Architecture Constraints

- ADR-011 (artefact-first rule): this story is the required artefact chain for the change to `.github/skills/benefit-metric/SKILL.md` and `.github/templates/benefit-metric.md`. Both must be changed in the same PR as this story's test and DoR artefacts.
- SKILL.md and template files are Markdown only — no embedded HTML except HTML comments. No trailing whitespace. Unix line endings.
- The existing benefit-metric contract in `check-skill-contracts.js` must continue to pass after this change. If a new marker is added for the attribution prompt, update the contract in the same commit.
- `check-pipeline-artefact-paths.js`: no new output paths introduced — no update needed.

## Dependencies

- **Upstream:** None — parallel to p11.1. Can be delivered in any order relative to p11.1.
- **Downstream:** p11.3 (H-GOV hard block) may be extended to check benefit-metric attribution as well; this story provides the fields that enable that extension. p11.3 does not block on this story but benefits from it.

## Acceptance Criteria

**AC1:** Given the updated `.github/templates/benefit-metric.md`, when a new benefit-metric artefact is created from the template, then the `Metric owner` field in the document header section contains an instructional comment specifying that this is a named person with their role, and that a non-engineering role is expected for metrics that measure user outcomes rather than technical outputs.

**AC2:** Given the updated `.github/templates/benefit-metric.md`, when a new benefit-metric artefact is created, then it contains a `Reviewers` field in the document header listing who reviewed the metrics before they were marked active — with an instructional comment indicating at least one reviewer should be from outside the engineering team for Tier 1 product metrics.

**AC3:** Given the updated `.github/skills/benefit-metric/SKILL.md`, when the skill reaches its completion step (the `Benefit metric artefact complete ✅` block), then the output explicitly prompts the operator to confirm the metric owner and reviewers are populated before marking the artefact as active.

**AC4:** Given `npm test` after this change, then `check-skill-contracts.js` passes for the benefit-metric skill with no new failures. If a new required marker is added to the SKILL.md, it must be added to the contract definition simultaneously.

## Out of Scope

- H-GOV hard block enforcement — that is p11.3.
- Changes to the `/discovery` template — that is p11.1.
- Enforcement of reviewer participation (verifying that named reviewers actually reviewed) — structural presence is the MVP.
- Backfilling attribution fields in the existing benefit-metric artefact for this feature (`benefit-metric.md` at `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`) — the template change applies to new artefacts going forward.
