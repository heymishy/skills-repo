## Story: Extend `/start` with brownfield context detection and Entry A/B/C routing

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-onboarding-lockfile.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **platform consumer with an existing codebase** (brownfield team — engineer or BA who has shipped code or produced discovery notes),
I want `/start` to detect my brownfield context and route me to the correct entry pattern (Entry A, B, or C) rather than treating me as a greenfield consumer,
So that I receive guidance appropriate to my actual starting point and do not face a "start from scratch" barrier (MM3 — brownfield routing hypothesis; M1 for brownfield consumers).

## Benefit Linkage

**Metric moved:** MM3 (primary) — brownfield routing hypothesis validation; M1 (secondary) — brownfield consumers can reach their first actionable pipeline step in under 2 minutes once routed correctly
**How:** Currently, `/start` (from p11.6) handles greenfield context only. Brownfield consumers who run `/start` receive greenfield orientation that is not relevant to their situation — they need to be told to run `/reverse-engineer` or `/tdd [story-slug]`, not `/discovery`. This story adds context detection signals and routes to the three formally documented entry patterns, making the sub-2-minute onboarding target achievable for brownfield consumers too.

## Architecture Constraints

- ADR-011 (artefact-first rule): this story's changes are extensions to `.github/skills/start/SKILL.md` created in p11.6. The extension must be in the same SKILL.md file — no separate skill file. The story is the artefact chain for the content extension.
- SKILL.md files are Markdown only — no embedded HTML except HTML comments. No trailing whitespace.
- `check-skill-contracts.js`: the `/start` contract created in p11.6 must still pass after brownfield routing is added. If new required markers are needed for the brownfield routing section, add them to the contract in the same commit.
- Sub-2-minute constraint: brownfield routing output must also fit within a single conversational turn. The routing decision must be made from observable file system signals (presence of `src/`, presence of artefacts, presence of DoD records) without requiring the consumer to answer questions before receiving their routing.
- Brownfield detection signals (all derived from file system inspection of the repository): Entry A trigger — `artefacts/[any-feature]/stories/` directory exists with at least one story file; Entry B trigger — `src/` directory exists with at least one non-test file but no pipeline artefacts in `artefacts/`; Entry C trigger — `artefacts/[any-feature]/dod/` directory exists (shipped features with DoD records). Greenfield is the fallback when none of these signals are present.

## Dependencies

- **Upstream:** p11.6 (`p11.6-start-skill.md`) — the `/start` SKILL.md must exist before this story extends it. This story must not be dispatched until p11.6 is DoD-complete.
- **Downstream:** None — this is the last story in Epic 2.

## Acceptance Criteria

**AC1:** Given a repository where `artefacts/` contains at least one `stories/` subdirectory with a story file (Entry A signal), when `/start` is invoked, then the output routes to Entry A by naming `/tdd [story-slug]` as the next action and including one sentence explaining what `/tdd` does (enforces RED-GREEN-REFACTOR for the named story).

**AC2:** Given a repository where a `src/` directory exists with at least one non-test `.js` file but `artefacts/` contains no `stories/` or `dod/` subdirectories (Entry B signal), when `/start` is invoked, then the output routes to Entry B by naming `/reverse-engineer` as the next action and including one sentence explaining what it does (extracts business rules and data contracts from existing code).

**AC3:** Given a repository where `artefacts/[feature]/dod/` exists with at least one DoD record (Entry C signal — shipped code with completed pipeline records), when `/start` is invoked, then the output routes to Entry C by naming the retrospective story template path (`.github/templates/retrospective-story.md`) as the mechanism and including one sentence on its purpose (creates a retroactive artefact chain for already-shipped features).

**AC4:** Given a repository with no `src/` directory and no `artefacts/` content (greenfield — no brownfield signals present), when `/start` is invoked with the brownfield routing active, then the output routes to the greenfield path defined in p11.6 — no false positive brownfield detection.

**AC5:** Given `npm test` after this change, then `check-skill-contracts.js` passes for the `/start` skill with the updated contract (any new required markers from the brownfield routing section are present).

## Out of Scope

- A fourth brownfield entry pattern — if no A/B/C signal is detected and it is also not a clean greenfield, `/start` falls back to `/workflow` ("Run `/workflow` to see your current pipeline state") rather than attempting to invent a route.
- Automated artefact creation or modification based on detected context — `/start` is a read-only concierge. It detects and routes; it does not create artefacts.
- Detection of partial pipeline states within a feature (e.g. "discovery exists but benefit-metric does not") — that level of navigation is handled by `/workflow`. `/start` routes to entry patterns, not mid-pipeline states.
- Multiple simultaneous brownfield signals (e.g. both `src/` and `artefacts/stories/` exist) — priority order: Entry A (stories exist) takes precedence over Entry B (code exists); Entry C (DoD exists) takes precedence over Entry A. Priority is defined in the SKILL.md text.
