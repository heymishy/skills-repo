## Story: Implement H-GOV hard block in `/definition-of-ready` SKILL.md

**Epic reference:** `artefacts/2026-04-24-platform-onboarding-distribution/epics/p11-governance-attribution.md`
**Discovery reference:** `artefacts/2026-04-24-platform-onboarding-distribution/discovery.md`
**Benefit-metric reference:** `artefacts/2026-04-24-platform-onboarding-distribution/benefit-metric.md`

## User Story

As a **platform maintainer**,
I want the `/definition-of-ready` SKILL.md to include a H-GOV hard block that fires when the parent discovery artefact has no non-engineering entry in the `approved-by` section,
So that engineer-only outer loop execution is structurally visible at the DoR gate — an engineer cannot dispatch a story to the coding agent if the discovery that scoped it has no named non-engineering sign-off (M3 minimum validation signal; MM2 H-GOV hypothesis validation).

## Benefit Linkage

**Metric moved:** M3 — Non-engineering attribution rate (minimum validation signal: H-GOV fires at least once in CI within 30 days of delivery); MM2 — H-GOV hard block hypothesis (the block firing AND a team responding by seeking resolution — not bypass — validates the hypothesis)
**How:** H-GOV is the enforcement mechanism. Without it, the attribution fields added in p11.1 and p11.2 are advisory. H-GOV converts them from advisory to authoritative by blocking the inner loop entry point when attribution is absent.

## Architecture Constraints

- ADR-011 (artefact-first rule): this story is the required artefact chain for the change to `.github/skills/definition-of-ready/SKILL.md`. The SKILL.md change must be in the same PR as this story's test and DoR artefacts.
- SKILL.md and template files are Markdown only — no embedded HTML except HTML comments. No trailing whitespace.
- The existing DoR skill contract in `check-skill-contracts.js` defines required structural markers. H-GOV must add a required marker (e.g. `<!-- h-gov-block -->`) to the SKILL.md and add it to the contract in `check-skill-contracts.js` in the same commit — to ensure no future edit silently removes the block.
- H-GOV reads the discovery artefact by path from the parent feature's artefacts folder. It does not read `pipeline-state.json` to determine attribution — it reads the artefact file directly (consistent with how H1–H9 read story content, not state fields).

## Dependencies

- **Upstream:** p11.1 (`p11.1-discovery-attribution-fields.md`) — H-GOV reads the `approved-by` section whose format is defined in p11.1. This story must not be dispatched until p11.1 is DoD-complete.
- **Upstream:** Bounded attribution model decision — "artefact-level sign-off vs section-level sign-off" must be documented as an ADR in `decisions.md` before this story is dispatched. [External: decision to be made by platform maintainer using `/decisions` skill — confirmed as required dependency 2026-04-28]
- **Downstream:** None — this story unblocks the full governance enforcement model for all future features.

## Acceptance Criteria

**AC1:** Given a story whose parent feature's discovery artefact contains an `## Approved By` section with at least one entry that includes a non-engineering role (e.g. "Product Manager", "Business Analyst", "Business Lead"), when `/definition-of-ready` evaluates that story, then H-GOV passes and the DoR run proceeds to the next hard block without a block message.

**AC2:** Given a story whose parent feature's discovery artefact contains an `## Approved By` section that is empty (the section exists but has no entries), when `/definition-of-ready` evaluates that story, then H-GOV fires as a hard block with a message that: (a) names the block as H-GOV, (b) states the missing content ("approved-by section is empty — at least one non-engineering role entry is required"), and (c) gives one concrete resolution step ("Populate the Approved By section in the discovery artefact with the name, role, and date of the non-engineering approver before re-running DoR").

**AC3:** Given a story whose parent feature's discovery artefact does not contain an `## Approved By` section at all (a pre-p11.1 artefact), when `/definition-of-ready` evaluates that story, then H-GOV fires as a hard block with a message that references the updated template and explains the field is absent (not just empty).

**AC4:** Given a story whose parent feature's discovery artefact contains an `## Approved By` section with only engineering-role entries (e.g. "Software Engineer — Alice"), when `/definition-of-ready` evaluates that story, then H-GOV fires as a hard block with a message that distinguishes between "present but engineer-only" and "absent" — naming the engineering entry found and stating that at least one non-engineering role entry is required.

**AC5:** Given the updated `check-skill-contracts.js`, when `npm test` is run, then the DoR skill contract passes with the H-GOV marker included as a required string — confirming the block text cannot be silently removed.

## Out of Scope

- H-GOV checking benefit-metric attribution (reviewer and metric owner fields from p11.2) — the MVP hard block is scoped to discovery `approved-by` only. Extending H-GOV to benefit-metric attribution is a post-MVP enhancement.
- Any change to the DoR hard block numbering (H1–H9, H-E2E, H-NFR*) — H-GOV is a named extension block alongside H9, not a renumbering of existing blocks.
- Changes to any SKILL.md other than `/definition-of-ready`.
- Section-level attribution enforcement (checking that individual ACs or sections have named reviewers) — the bounded attribution model decision scopes this to artefact-level only.
- Automated CI detection of H-GOV events — the minimum signal is the block firing during a manual DoR skill run; CI instrumentation is a Phase 5 WS5 concern.
