## Problem Statement

Platform operators running `/ideate` sessions work in a pure text interface. When the model generates output during a session — assembling opportunity clusters, surfacing assumptions, forming the artefact draft — three things are invisible:

1. **What context was loaded.** The operator cannot see which SKILL.md files, reference artefacts, or prior pipeline documents the model read at session start. If key context was missing or misread, the operator finds out only when reviewing the final artefact.

2. **What assumptions have accumulated.** Assumptions generated by the model accumulate silently inside the conversation thread. They appear only in the final Lens B output — which may be two or three lenses after the assumption first formed. The operator cannot flag, confirm, or challenge an assumption at the moment it is relevant.

3. **What the artefact draft looks like in progress.** The live artefact is not visible until the session completes. For long ideation sessions (Lenses A through B), the operator is working blind to the output shape until the very end.

The result: operators discover scoping gaps and unconfirmed assumptions only after the session is complete, requiring re-runs or creating avoidable rework in downstream /definition and /review stages. This was observed directly in two recent deliveries: 2026-05-20-cloud-platform (7 adoption barriers not visible mid-Lens-A) and 2026-05-21-ideate-web-ux itself (12 Lens B assumptions, none visible during Lenses A or D).

## Who It Affects

**Primary — Platform operator (developer / tech lead)** running `/ideate` sessions for feature discovery. This persona uses `/ideate` at the start of any new feature, typically 1–3 times per quarter per feature. The problem compounds on features where context is loaded from multiple reference files and assumptions build up across multiple lenses. The operator is also the author — they have no external view of what the model is assembling.

**Secondary — Platform operator receiving a completed ideation artefact from another session.** When a handoff occurs (e.g. operator A runs Lens A, operator B reviews Lens B), the invisible-assumption problem is worse: operator B has no record of the assumptions that formed during the lenses they weren't present for.

The personas affected are explicitly the same as the platform's primary Developer / Engineer and Tech Lead / Squad Lead personas (product/mission.md) — this feature serves the platform's own daily operators.

## Why Now

Three triggers converged:

1. **Observed delivery cost.** The 2026-05-21-ideate-web-ux feature produced 12 unconfirmed assumptions during ideation (Lens B). None were visible during Lenses A or D. This created a required spike (A1) before /discovery could proceed — a direct delivery cost from invisible assumptions.

2. **SSE infrastructure confirmed viable.** Spike A1 (completed 2026-06-03) established that the existing `handleGetChatHtml` / `handlePostTurnStreamHtml` SSE architecture supports assumption card emission as a new event type without structural changes. The last technical blocker is removed.

3. **Active usage makes the gap measurable.** `/ideate` is now in regular use (dogfooded by this pipeline, used in 2026-05-20-cloud-platform and 2026-05-21-ideate-web-ux). The downstream signal — rework rate from unconfirmed assumptions — is now traceable in pipeline-state.json delivery actuals.

## MVP Scope

Three clusters from the Lens A opportunity map (Clusters 1, 2, 4 — per Lens D strategic prioritisation):

**Cluster 1 — Context manifest panel**
At session start, before the first lens begins, the right panel shows which context was loaded: which SKILL.md was read, which reference artefacts were found, which prior pipeline artefacts (benefit-metric, stories, etc.) were included. The manifest is static — it shows the session's starting context, not a live feed.

**Cluster 2 — Assumption cards**
During a session, when the model emits an assumption marker (`---ASSUMPTION-JSON: {...}---` protocol, per ADR-018), the assumption surfaces as a card in the right panel via SSE event type `assumptionCard`. The operator can mark a card as confirmed or flagged using a `POST /api/skills/:name/sessions/:id/assumption/:cardId/confirm` endpoint (ADR-018). Card state persists in `session.assumptionCards[]` for the life of the session.

**Cluster 4 — Live artefact draft**
As the model generates the artefact (identified via `---ARTEFACT-START---...---ARTEFACT-END---` markers, already implemented in `handlePostTurnStreamHtml`), the right panel shows the draft in real time via the existing `draftChunk` SSE event. This enhances the existing `updateDraftPanel()` mechanism to coexist with the new assumption cards panel. The right panel has two named sections: `#assumption-cards` (top) and `#draft-content` (below).

**Right panel layout (ADR-018):** `display:flex; flex-direction:column`. Assumption cards section: `flex:0 0 auto; overflow-y:auto; max-height:40%`. Artefact draft section: `flex:1 1 auto; overflow-y:auto`.

The MVP delivers these three clusters as an enhancement to the existing `/skills/:name/sessions/:id/chat` route — no new routes, no new model dependencies, no changes to the CLI interaction model.

## Out of Scope

- **Session recovery / resume (Cluster 3):** Deferred to Increment 2. Requires disk persistence of session turns and assumption card state, which adds complexity and a new security surface (sensitive artefact content written to disk mid-session). In-memory-only sessions are sufficient for MVP.

- **Structured input forms (Cluster 5):** Deferred to Increment 2. Requires SKILL.md schema changes to define structured question types. SKILL.md modifications require a separate governed pipeline story (artefact-first rule, product/constraints.md Constraint 4).

- **Conditions sidebar (Cluster 6):** Deferred to Increment 2. Builds on the assumption cards infrastructure (same SSE extension pattern, `conditionItem` event type). Deferral allows MVP to validate the assumption card interaction model before adding a second card type.

- **CLI / chat window interaction model unchanged:** Operators who run `/ideate` from the VS Code chat window continue to use the existing text-only flow. The web UI is an additive surface — it does not replace or modify the chat flow.

- **Mobile / responsive layout:** The existing platform web UI has no mobile support. This feature does not add any.

- **Multi-user / collaborative sessions:** Not in scope for MVP. Sessions remain single-operator, in-memory, scoped to a single OAuth session.

## Assumptions and Risks

[ASSUMPTION] Operators prefer reviewing and acting on assumption cards during the session rather than reviewing the full assumption list in the final Lens B output — unconfirmed, requires validation with at least one real operator session observation before Increment 2 scope is locked.

[ASSUMPTION] The `/ideate` SKILL.md system prompt can be tuned to emit `---ASSUMPTION-JSON---` markers reliably and consistently during model generation — technically feasible per Spike A1, but emission consistency at scale is untested. If marker emission is unreliable (≤50% of actual assumptions surfaced), the card value proposition is significantly reduced.

**Known risk — session interruption:** Browser close or network drop loses all in-session assumption card state since sessions are in-memory only. This is a deliberate MVP constraint (Cluster 3 deferred). Operators who need persistence should commit the artefact before closing.

**Known risk — card UI attention:** If assumption cards are too visually subtle, operators may ignore them during active chat. Card design must be distinct from the artefact draft panel — this is a design constraint for /definition stories, not a scope blocker.

## Directional Success Indicators

**1. Assumption visibility mid-session**
Baseline: 0 assumptions visible mid-session today (no web UI). Target: 100% of model-emitted `assumptionCard` SSE events produce a visible card in the right panel within 500ms of emission. Measured via: automated test asserting SSE event → DOM card render pipeline; manual operator session observation.

**2. Downstream rework rate from unconfirmed assumptions**
Baseline: ~40% of ideation sessions require a subsequent re-run or downstream correction in /definition due to unconfirmed assumptions (operator-estimated from Lens D delivery signal). Target: <15% of sessions require correction after MVP ships. Measured via: delivery actuals in pipeline-state.json — stories tagged to ideation re-runs or scoping corrections identified at /review.

**3. Session completion rate**
Baseline: [UNKNOWN BASELINE] — no web UI session telemetry exists yet. Target: 75% of started `/ideate` sessions (where the operator submits at least one answer) result in a committed artefact. Measured via: `session.done = true` signal frequency in server-side session logs.

## Constraints

- **Architecture:** Web UI must extend the existing `handleGetChatHtml` / `handlePostTurnStreamHtml` SSE streaming pattern. Assumption cards use the `assumptionCard` SSE event type per ADR-018. No new model API dependencies. Uses existing `_skillTurnExecutorStream`.

- **Governed files:** The `/ideate` SKILL.md at `.github/skills/ideate/SKILL.md` is a governed file. Any modification to its content requires a separate pipeline story with full outer loop (discovery → DoR → inner loop). System prompt tuning for `---ASSUMPTION-JSON---` marker emission is a SKILL.md change — it must be scoped as a separate story within this feature, not as a side effect of the web UI stories.

- **Session persistence:** In-memory only for MVP. `session.assumptionCards[]` lives in the in-memory `_sessionStore` Map. No disk write of assumption card state. Consistent with current architecture.

- **Compliance:** `regulated: false` (context.yml). No compliance frameworks apply. No sensitive data categories identified — session content is developer-authored ideation text, not personal data.

- **SKILL.md changes must be separate stories:** Per the artefact-first rule and copilot-instructions.md Platform change policy, any story that modifies `.github/skills/ideate/SKILL.md` must have its own DoR artefact and governed implementation path. This is not a blocker for the web UI stories, but it must be included as a named story in /definition scope.
