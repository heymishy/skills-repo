# Story: Author framework concepts documentation suite for new-user onboarding

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **first-time platform adopter**,
I want to read a structured set of concept reference documents that explain every major framework building block in plain language,
So that I can understand what the platform does, why each piece exists, and how to operate it — without needing to attend a walkthrough session or interrupt the platform engineering team.

## Benefit Linkage

**Metric moved:** MM2 — Outer loop self-sufficiency: Phase 3 delivery without documentation lookups
**How:** Phase 2 user feedback identified that first-time adopters (squad tech leads, contributing engineers, external enterprise evaluators) could not self-orient from the existing `docs/ONBOARDING.md` alone — operational steps are present but the conceptual model is absent. A new user does not know what a trace is, why a watermark gate exists, or what distinguishes DoR from DoD before they attempt the workflow. This documentation gap forces blocking lookup conversations with the platform team — direct violations of the MM2 zero-blocking-lookup target. The concept files eliminate that gap for every subsequent new user, reducing blocking lookup rate at scale. This also directly supports M4 and M5 (enterprise approval channel and AGENTS.md adapter validation) since enterprise evaluators are the primary new-user persona for those metrics.

## Architecture Constraints

- Output is documentation only — `.md` files in `docs/concepts/`. No changes to skill files, gate scripts, schema files, or any file in `.github/` or `src/`.
- File naming convention must match existing `docs/` patterns: lowercase, hyphen-separated, descriptive.
- Cross-references within concept files must use relative markdown links to other files in the repo — no absolute URLs to external sites.
- `docs/ONBOARDING.md` may be updated to add a reference/callout to `docs/concepts/` — no other content changes to ONBOARDING.md are in scope.
- None of the concept files are test artefacts; they are not registered in any test suite or governance gate.

## Dependencies

- **Upstream:** None. This story reads from existing repo content for accuracy but does not depend on any other Phase 3 story completing first.
- **Downstream:** Feeds Phase 4 enterprise adoption onboarding. The concept files are referenced (by path) in the enterprise onboarding runbook if that story is scoped in Phase 4.

## Acceptance Criteria

**AC1:** Given this story is DoD-complete, when `docs/concepts/` is inspected, then it contains a `README.md` index plus individual concept files covering all of the following building blocks, each as a separate `.md` file: `outer-loop-inner-loop`, `skills-and-the-skills-pipeline`, `benefit-metrics`, `governance-traces`, `assurance-gate`, `definition-of-ready`, `definition-of-done`, `watermark-gate`. No required concept file is absent.

**AC2:** Each concept file follows a consistent four-section structure containing exactly the headings: `## What it is`, `## Why it exists`, `## How it works`, and `## What you do with it` — in that order. A fifth `## Further reading` section is optional. Any concept file missing a required heading is a DoD failure for that file.

**AC3:** Given the `docs/concepts/README.md` index is read, when a reader opens it for the first time, then it explicitly states it is written for a first-time adopter with no prior session attendance, lists every concept file in a logical recommended reading sequence (outer loop → inner loop → traces → assurance gate → DoR → DoD → watermark gate → benefit metrics), and provides a one-line plain-language description of each concept alongside its link.

**AC4:** Given `docs/ONBOARDING.md` is read, when the reader reaches the first 20 lines of the document, then a callout or section heading references `docs/concepts/README.md` as the recommended conceptual pre-read, with a direct relative markdown link, before any operational steps are presented.

**AC5:** Each concept file is self-contained: it does not assume the reader has read any other document in the repository. Any references to other files are labelled explicitly as "Optional further reading:" — not as required dependencies. A reviewer can confirm this by reading any single concept file in isolation without losing comprehension of the concept being explained.

## Out of Scope

- Skill reference documentation (SKILL.md files are not concept docs — they are agent instructions and must not be modified).
- Tutorial or walk-through content that prescribes exact commands to run — that is `docs/ONBOARDING.md` (operational) and `docs/validation-playbook.md` (verification). The concept files explain *what and why*, not *how to run*.
- Translating concept files into other formats (HTML, PDF, Confluence pages) — this is strictly `.md` output.
- Multi-language versions or translated documentation.
- Creating or updating any test in `tests/` — documentation is not test-gated in this story.

## NFRs

- **Readability:** Each concept file must be readable by a non-engineering stakeholder (business analyst, risk reviewer, product manager) — no requirement for coding knowledge to understand the concept explanation.
- **Completeness:** Every concept file must explain the concept at sufficient depth that a reader could give a correct one-paragraph description of it to a colleague without reading any other document.
- **Accuracy:** All descriptions must accurately reflect the current state of the platform (Phase 3). No aspirational claims about Phase 4 features as if they exist now.
- **Security:** No credentials, access tokens, personal data, or internal-only infrastructure names in any concept file.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the concepts are fixed; the writing is the work. Risk is in depth and accuracy, not scope change.
