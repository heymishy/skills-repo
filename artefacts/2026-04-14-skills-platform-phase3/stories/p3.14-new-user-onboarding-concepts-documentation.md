# Story: Author framework principles and primitives documentation suite for new-user onboarding

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **first-time platform adopter**,
I want to read a structured set of concept reference documents that explain every core design principle and platform primitive in plain language,
So that I understand why the platform is built the way it is — and can explain it to others — without needing a walkthrough session or interrupting the platform engineering team.

## Benefit Linkage

**Metric moved:** MM2 — Outer loop self-sufficiency: Phase 3 delivery without documentation lookups
**How:** Phase 2 user feedback identified that first-time adopters could not self-orient from `docs/ONBOARDING.md` alone — operational steps exist but the conceptual model is absent. A new user does not know what spec immutability means, why the governance brain must never branch on surface type, or what distinguishes the eval suite from a normal test suite — before they attempt the workflow. This gap forces blocking conversations with the platform team, directly violating the MM2 zero-blocking-lookup target. The principle and primitive reference files eliminate that gap for every subsequent new user at scale.

## Architecture Constraints

- Output is documentation only — `.md` files in `docs/concepts/principles/` and `docs/concepts/primitives/`. No changes to skill files, gate scripts, schema files, or any file in `.github/` or `src/`.
- The six principles documented here must match the definitions in `.github/architecture-guardrails.md §Core Platform Design Principles` exactly. If a discrepancy is found at PR review, the docs file is wrong — not the guardrails.
- The six primitives documented here must match the definitions in `.github/architecture-guardrails.md §Core Platform Primitives` exactly. Same rule applies.
- File naming convention: lowercase, hyphen-separated, descriptive; consistent with existing `docs/` patterns.
- Cross-references must use relative markdown links — no absolute external URLs.
- `docs/ONBOARDING.md` may be updated to add a callout to `docs/concepts/` — no other content changes to ONBOARDING.md are in scope.
- None of the concept files are test artefacts; they are not registered in any test suite or governance gate.

## Dependencies

- **Upstream:** None. This story reads from existing repo content for accuracy and does not depend on any other Phase 3 story completing first.
- **Downstream:** Feeds Phase 4 enterprise adoption onboarding. Concept files are referenced (by path) in any enterprise onboarding runbook scoped in Phase 4.

## Acceptance Criteria

**AC1:** Given this story is DoD-complete, when `docs/concepts/` is inspected, then it contains:
- A `principles/` subdirectory with exactly six files, one per principle: `governance-by-demonstration.md`, `the-subset-is-the-on-ramp.md`, `adapter-isolated-surface-concerns.md`, `spec-immutability.md`, `human-approval-at-every-gate.md`, `self-improving-harness.md`
- A `primitives/` subdirectory with exactly six files, one per primitive: `skill.md`, `surface-adapter.md`, `assurance-gate.md`, `pipeline-state.md`, `eval-suite.md`, `learnings-log.md`
- A `README.md` index at `docs/concepts/README.md`
No required file is absent. No file covering a concept not listed above is present.

**AC2:** Each principle file and each primitive file follows a consistent four-section structure with exactly these headings in order: `## What it is`, `## Why it exists`, `## How it works`, `## What you do with it`. A fifth `## Further reading` section with relative links is optional. Any file missing a required heading is a DoD failure for that file.

**AC3:** Given `docs/concepts/README.md` is read, when a reader opens it for the first time, then it states it is written for a first-time adopter with no prior session attendance, provides two reading sequences — one for principles (P1 → P2 → P3 → P4 → P5 → P6) and one for primitives (Skill → Surface adapter → Assurance gate → Pipeline state → Eval suite → Learnings log) — and gives a one-line plain-language description of each item alongside its link.

**AC4:** Given `docs/ONBOARDING.md` is read, when the reader reaches the first 20 lines of the document, then a callout or section heading references `docs/concepts/README.md` as the recommended conceptual pre-read with a direct relative markdown link, before any operational steps are presented.

**AC5:** The principle file for `adapter-isolated-surface-concerns.md` explicitly notes that this principle was previously labelled "Surface-agnostic by contract" in the README and explains why the name was changed — so that a reader encountering the old name in legacy artefacts understands it references the same principle.

**AC6:** Each concept file is self-contained: it does not assume the reader has read any other document in the repository. Any references to other files are labelled explicitly as "Optional further reading:" — not as required dependencies. A reviewer can confirm this by reading any single concept file in isolation without losing comprehension of the concept being explained.

## Out of Scope

- Modifying `.github/architecture-guardrails.md` — the principles and primitives definitions are already written there (done prior to this story as active guardrails). This story produces the human-readable explanations, not the enforcement definitions.
- Operational how-to content (that is `docs/ONBOARDING.md` and `docs/validation-playbook.md`). The concept files explain *what and why*, not *how to operate*.
- Skill reference documentation — SKILL.md files are agent instructions and must not be modified.
- Translating files into other formats (HTML, PDF, Confluence pages).
- Multi-language versions.
- Creating or updating any test in `tests/`.

## NFRs

- **Readability:** Each concept file must be readable by a non-engineering stakeholder (business analyst, risk reviewer, product manager) without requiring coding knowledge.
- **Accuracy:** All descriptions must match the current authoritative definitions in `.github/architecture-guardrails.md`. No aspirational Phase 4 claims stated as current capability.
- **Completeness:** Each file must contain sufficient depth that a reader could give a correct one-paragraph description of the concept to a colleague without reading any other document.
- **Security:** No credentials, tokens, personal data, or internal-only infrastructure names in any concept file.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the principles and primitives are fixed in architecture-guardrails.md; the writing is the work. Risk is accuracy and depth, not scope change.
