# Story: Add four mandatory T3M1 trace fields to schema and gate enforcement

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e2-t3m1-audit-gap-closure.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **risk reviewer / audit function**,
I want every regulated delivery trace to contain `standardsInjected`, `watermarkResult`, `stalenessFlag`, and `sessionIdentity` fields — automatically populated and enforced by the gate — so that I can answer MODEL-RISK audit questions Q2, Q5, Q6, and Q7 from the trace alone without engineering assistance.

## Benefit Linkage

**Metric moved:** M1 — T3M1 audit question coverage; CR1 — T3M1 independent validation on record
**How:** Adding these four fields closes the four specific evidence gaps that make Q2 (standardsInjected hashes verifiable), Q5 (watermark result surfaced), Q6 (stalenessFlag present), and Q7 (sessionIdentity evidences agent independence). With these fields in every regulated trace, an independent reviewer can answer all four questions from the trace file alone.

## Architecture Constraints

- ADR-003 (Schema-first): all four fields must be added to `pipeline-state.schema.json` in the same commit as any code that writes or reads them.
- MC-CORRECT-02 and MC-CORRECT-03: fields written to `pipeline-state.json` and fields read by the viz must exist in the schema.
- ADR-002 (Gates must use evidence fields): the gate must read these four fields — not merely check their presence but also validate non-null values for regulated stories.

## Dependencies

- **Upstream:** p3.1b (assurance gate substantive checks) is recommended complete first — the gate enforcement added in this story builds on the content-check pattern established there.
- **Downstream:** p3.2b (tamper-evidence registry) — the `traceHash` that p3.2b publishes is computed over the trace record that now includes these four fields. Do not start p3.2b until p3.2a defines the complete field set.

## Acceptance Criteria

**AC1:** Given `pipeline-state.schema.json`, when this story is DoD-complete, then the schema defines all four fields: `standardsInjected` (object or array — list of standards identifiers with hashes), `watermarkResult` (object — pass/fail and passRate), `stalenessFlag` (boolean), and `sessionIdentity` (object — sessionId, agentType, and startedAt timestamp) — each with a type definition and a description mapping it to its MODEL-RISK question.

**AC2:** Given a trace record for a regulated story is being written by the assurance gate, when all four fields are populated with non-null values, then the gate writes the trace and reports success — the trace file contains all four fields with their populated values.

**AC3:** Given a trace record being written for a regulated story is missing any one of the four fields (null or absent), when the assurance gate runs, then the gate exits 1 and reports which specific field is missing or null, blocking the merge.

**AC4:** Given `MODEL-RISK.md`, when this story is DoD-complete, then the file contains an entry for each of the four new trace fields documenting: field name, MODEL-RISK question it answers (Q2/Q5/Q6/Q7), what a reviewer should see in the field for a passing answer, and a reference to the schema definition.

**AC5:** Given an existing non-regulated story trace (produced before this story), when the gate runs against it, then it does not retroactively fail — the new field requirement applies only to new traces produced after this story merges. The gate must distinguish between regulated and non-regulated stories using an existing story classification field.

## Out of Scope

- The tamper-evidence registry for traceHash (T3M1 Q8) — that is p3.2b.
- Changes to how the viz renders trace data — the viz may be updated in a separate story if display of the new fields is desired.
- Backfilling existing trace files with the new fields — no retroactive trace modification.

## NFRs

- **Audit:** Each trace file containing the four new fields must be independently verifiable by re-running the gate against a fixture — the field values must be deterministic for a given input.
- **Security:** `sessionIdentity.sessionId` must not contain personally identifiable information. If the agent session token is used, only a hash or truncated identifier is stored.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — four fields are specified; schema extension and gate logic are established patterns.
