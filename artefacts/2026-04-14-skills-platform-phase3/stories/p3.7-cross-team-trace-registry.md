# Story: Establish cross-team trace registry with 24-hour aggregation SLA

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e5-cross-team-trace-registry.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer** responsible for the improvement agent's signal quality,
I want squads to commit structured JSONL trace files to a consistent path in their delivery repos, with a scheduled CI workflow aggregating them to a central registry within 24 hours, so that the improvement agent can query multi-squad delivery traces for cross-team pattern detection.

## Benefit Linkage

**Metric moved:** MM3 (improvement agent pattern quality); CR1 (independent audit can sample cross-squad traces)
**How:** Cross-team traces are the foundational input for the improvement agent's fleet-level pattern detection. Without a structured, aggregated registry, the agent can only see single-squad signal — insufficient to distinguish platform-level failures from local practice variations. CR1 also requires an auditor to sample traces across squads; this is not possible without a centralised registry.

## Architecture Constraints

- ADR-009 applies: the CI workflow that writes aggregated traces to the platform registry must use a **separate trigger and permission scope** from the regular delivery pipeline CI. It must run on a schedule or a dedicated `trace-sync` event — not on the same job that runs tests or builds delivery artefacts.
- The aggregated traces are read-only in the platform registry from each squad's perspective — squads have no write access to other squads' aggregated trace entries.
- Squad JSONL trace format must conform to the existing `trace.schema.json` — no new schema fields are introduced by this story (new fields, if needed, are an E2 concern via p3.2a).
- `getTraces(filter)` API must be implemented as a Node.js module in `src/suite-parser/` or a new `src/trace-registry/` directory — not as an inline script in a SKILL.md.

## Dependencies

- **Upstream:** p3.2a (T3M1 trace fields) should be DoD-complete before using the registry for regulated-story audits, but the registry itself can be established independently. The aggregation pipeline does not require p3.2a.
- **Downstream:** p3.13 (compliance monitoring report) has a hard dependency on this story being DoD-complete — the audit agent needs `getTraces()` to sample the registry.

## Acceptance Criteria

**AC1:** Given a squad delivery repo configured with cross-team registry participation, when the squad completes a story and commits a trace file to `workspace/traces/[storySlug].jsonl`, then the scheduled CI workflow in the platform repo aggregates that trace to `platform/traces/[squadId]/[storySlug].jsonl` within 24 hours of the commit.

**AC2:** Given the platform repo's `src/` directory, when a caller invokes `getTraces({ squadId, filter })`, then the function returns all trace entries matching the filter — by squad ID, date range, storySlug, or passRate threshold. If no traces match the filter, it returns an empty array, not an error.

**AC3:** Given a trace entry in the aggregated registry, when its `aggregatedAt` timestamp is compared to the squad's commit timestamp, then the difference is at most 24 hours — the freshness SLA is enforced by the scheduled CI cadence (at minimum: daily schedule trigger).

**AC4:** Given the aggregation CI workflow, when it runs, then it uses a separate GitHub Actions trigger (`schedule:` or `workflow_dispatch:`) and a repository-scoped token with read-only access to squad repos — not the same job or token as the delivery pipeline (ADR-009 compliance).

**AC5:** Given a squad that has not opted in to cross-team registration, when the improvement agent calls `getTraces({ squadId })`, then it receives an empty result set for that squad — no error is thrown and no partial data is returned. The absence of traces is a valid empty-registry state, not a failure.

## Out of Scope

- Real-time aggregation (sub-minute trace sync) — 24-hour SLA is the target.
- Modifying the trace schema with new fields — that is p3.2a scope.
- Aggregating artefacts other than JSONL traces — only traces are in scope.
- Enabling squads to query each other's traces directly — only the platform maintainer and the improvement agent have registry read access.

## NFRs

- **Availability:** The `getTraces()` function must handle an empty or absent trace registry gracefully — no unhandled errors when the registry has not been seeded.
- **Data classification:** Traces contain delivery metadata (story slugs, pass rates, feature IDs) — internal classification, not customer data.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — aggregation pattern is straightforward; `getTraces()` API scope is bounded.

## Definition of Ready Pre-check

None. ASSUMPTION-01 and ASSUMPTION-02 do not apply to this story.
