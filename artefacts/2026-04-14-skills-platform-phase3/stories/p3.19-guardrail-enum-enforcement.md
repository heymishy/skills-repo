# Story: Enforce schema-valid guardrail enum values in /definition-of-done and /definition-of-ready skills

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md
**Improvement source:** workspace/learnings.md — schema-valid guardrail enum entry surfaced by /improvement run 2026-04-16

## User Story

As a **platform maintainer**,
I want the /definition-of-done and /definition-of-ready skills to explicitly state the permitted enum values for guardrail `status` and `category` fields,
So that agents writing guardrail blocks produce schema-valid output that does not trigger hard CI failures on subsequent PRs.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: reduction in governance check failures per story
**How:** Invalid enum values in guardrail blocks (e.g. `pass`, `fail`, `deferred`, `no-breach`) are currently caught only at CI time by `validate-trace.sh --ci`, causing hard failures that block agent PRs and require operator intervention. By embedding the valid enum list directly in the skill instructions where guardrail blocks are written, agents will produce valid values on first generation, eliminating this class of post-generation CI failure.

## Architecture Constraints

- Modifications are confined to two files: `.github/skills/definition-of-done/SKILL.md` and `.github/skills/definition-of-ready/SKILL.md` — no code or schema changes.
- The enum reminder block must be added in the guardrails compliance update section of each file, after the `Merge by \`id\`` instruction line.
- Enum values must match exactly those validated by `validate-trace.sh --ci` — do not paraphrase or abbreviate.
- The same enum reminder block content must appear in both SKILL.md files for consistency.

## Dependencies

- **Upstream:** None.
- **Downstream:** None within Phase 3. Future DoD and DoR runs will inherit the corrected enum guidance.

## Acceptance Criteria

**AC1 — Schema-valid enum reminder block present in definition-of-done/SKILL.md**
Given `.github/skills/definition-of-done/SKILL.md` is read,
When the reader locates the guardrails compliance update section and finds the `Merge by \`id\`` instruction,
Then a `**Schema-valid guardrail values only.**` block is present immediately after that line, listing exactly: status values (`met`, `not-met`, `na`, `excepted`, `not-assessed`) and category values (`mandatory-constraint`, `adr`, `nfr`, `compliance-framework`, `pattern`, `anti-pattern`), with a note that invalid values cause hard CI failures via `validate-trace.sh --ci`.

**AC2 — Schema-valid enum reminder block present in definition-of-ready/SKILL.md**
Given `.github/skills/definition-of-ready/SKILL.md` is read,
When the reader locates the guardrails compliance update section and finds the `Merge by \`id\`` instruction,
Then a `**Schema-valid guardrail values only.**` block is present immediately after that line with the same content as AC1.

**AC3 — Both blocks list enum values that match validate-trace.sh exactly**
Given both blocks are present,
When the enum values listed in the blocks are compared against `validate-trace.sh` validation logic,
Then the status values are exactly `met`, `not-met`, `na`, `excepted`, `not-assessed` — no variants — and the category values are exactly `mandatory-constraint`, `adr`, `nfr`, `compliance-framework`, `pattern`, `anti-pattern` — no variants.

## Out of Scope

- Changing `validate-trace.sh` or the schema that defines valid enum values.
- Back-filling existing DoD/DoR artefacts that may contain invalid enum values.
- Adding automated CI checks that validate SKILL.md content.
- Changes to any other skill, governance check, or source file.

None specific to this story — SKILL.md instruction text addition only.
