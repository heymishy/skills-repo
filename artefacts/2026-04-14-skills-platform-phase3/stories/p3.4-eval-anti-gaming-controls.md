# Story: Eval suite anti-gaming controls — traceId, failurePattern, human-approval staging

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e3-platform-structural-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want new evaluation scenario entries in `workspace/suite.json` to require a `traceId` referencing a real committed trace file and a `failurePattern` field, with proposals staged for human approval before committing, so that the improvement agent cannot produce self-serving scenarios that inflate its own pass rate without real delivery evidence.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: governance failure reduction
**How:** Requiring `traceId` and `failurePattern` for new suite entries means synthetic scenarios that have no delivery evidence cannot enter the eval suite without explicit human approval and justification. This closes the path by which the improvement agent could, through repeated scenario additions, make the baseline easier to beat.

## Architecture Constraints

- p3.3 must be DoD-complete first — `platform/suite.json` is housed in the `platform-infrastructure` repository established by p3.3. This story writes to `platform/suite.json`, not to `workspace/suite.json` directly, once p3.3 is complete. `workspace/suite.json` remains valid for local development.
- The proposal staging path (`workspace/proposals/suite-additions/`) is in the delivery repository — this is intentional. Proposals live where the agent has write access; approved entries are promoted to `platform-infrastructure` by a human maintainer.
- ADR-008 (DoR touch-point contract binding at pre-merge): the validation change must not retroactively invalidate existing `workspace/suite.json` entries that lack `traceId` or `failurePattern`. Existing entries are grandfathered; the requirement applies to entries created after this story merges.

## Dependencies

- **Upstream:** p3.3 (gate structural independence) must be DoD-complete — `platform/suite.json` host location is established by that story.
- **Upstream:** p3.1c (anti-overfitting counter fix) — the anti-overfitting gate checks that this story builds on must be correct before the scenario intake pipeline is hardened.
- **Downstream:** p3.12 (squad contribution flow) — the contribution process uses the same proposal staging path. This story establishes the pattern; p3.12 formalises the broader process.

## Acceptance Criteria

**AC1:** Given a new scenario entry is being added to `workspace/suite.json` or `platform/suite.json`, when a validation script (invoked by `npm test` or a pre-commit hook) checks the entry, then the entry fails validation if either `traceId` or `failurePattern` is absent or null — the validation script exits 1 and names the missing field.

**AC2:** Given a new scenario entry includes a `traceId` value, when the validation script runs, then it verifies the `traceId` resolves to an existing committed trace file at the expected path (`workspace/traces/[traceId]` or `platform/traces/[squad]/[traceId]`) — a `traceId` that references a non-existent file causes the validation to fail.

**AC3:** Given the improvement agent generates a new scenario proposal, when the proposal is output, then it is written to `workspace/proposals/suite-additions/[proposal-id].json` — not directly to `workspace/suite.json` or `platform/suite.json`. The proposal file includes `traceId`, `failurePattern`, and a `justification` field.

**AC4:** Given a proposal exists in `workspace/proposals/suite-additions/`, when a human maintainer approves it, then the maintainer manually commits the approved entry to `platform/suite.json` in `platform-infrastructure` — no automated promotion pathway. The proposal file in `workspace/proposals/suite-additions/` may be archived (moved to a `reviewed/` subdirectory) by the maintainer after promotion.

**AC5:** Given existing entries in `workspace/suite.json` that predate this story, when the validation script runs, then those entries pass without error — existing entries without `traceId` or `failurePattern` are grandfathered and do not cause validation failures.

## Out of Scope

- Automated proposal review or scoring — proposals are staged for human review only. No automated approval pathway.
- `traceId` validation for entries in the grandfathered set (existing `workspace/suite.json` entries).
- Retroactive addition of `traceId` and `failurePattern` to existing entries — the platform maintainer may do this voluntarily but it is not required by this story.

## NFRs

- **Audit:** Each approved suite entry in `platform/suite.json` must be traceable to a PR in the `platform-infrastructure` repository — the PR is the audit record of human approval.

**Rating:** 2
