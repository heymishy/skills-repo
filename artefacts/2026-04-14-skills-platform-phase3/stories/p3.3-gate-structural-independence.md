# Story: Move gate scripts to platform-infrastructure repo and wire checksum validation

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e3-platform-structural-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want `run-assurance-gate.js` and all scripts it invokes to live in a separate `platform-infrastructure` repository to which delivery agents have no write access, with the delivery repository's CI workflow downloading the gate at a pinned immutable ref and validating its SHA-256 checksum before executing it,
So that a delivery agent — regardless of behaviour — cannot weaken the gate that governs its own delivery by modifying gate scripts through the normal PR workflow.

## Benefit Linkage

**Metric moved:** CR2 — Gate structural independence: agent cannot weaken its own gate
**How:** Moving gate scripts out of the delivery repository and validating the checksum before execution means a delivery agent's write access to the delivery repository is insufficient to modify what the gate evaluates. The vulnerability is structurally closed, not mitigated.

## Architecture Constraints

- ASSUMPTION-01 must be confirmed (separate repository creation is feasible in the target Git host) before DoR sign-off.
- ADR-009 (Separate workflow permission scopes): the CI download-and-execute workflow in the delivery repository must have no write access to the `platform-infrastructure` repo. It downloads a release artefact only.
- ADR-003 (Schema-first): any new fields written to `pipeline-state.json` by the updated gate download step (e.g. `gateScriptRef`, `gateChecksumVerified`) must be added to `pipeline-state.schema.json` in the same commit.
- The delivery repository retains a thin shim script that invokes the downloaded gate — the shim itself contains no governance logic.

## Dependencies

- **Upstream:** p3.1a, p3.1b, p3.1c must be DoD-complete. Protocol: move a sound gate into structural protection. Do not move a flawed gate.
- **Downstream:** p3.4 (eval anti-gaming controls) — `platform/suite.json` is housed in the `platform-infrastructure` repo. p3.4 must proceed after this story establishes that repo.

## Acceptance Criteria

**AC1:** Given the `platform-infrastructure` repository exists and contains the gate scripts, when a delivery agent with write access to the delivery repository attempts to open a PR modifying `run-assurance-gate.js` or any invoked check script, then the PR is modifying the shim only (which contains no governance logic) — the actual gate script is in `platform-infrastructure` and is not in the delivery repository's working tree.

**AC2:** Given the delivery repository's assurance gate CI workflow, when the workflow runs on a pull request, then it downloads the gate script from a pinned, immutable ref in `platform-infrastructure` (exact commit SHA or a published release tag — not `main` or `master` HEAD), confirmed by the workflow YAML `ref:` value being a full SHA or a version tag.

**AC3:** Given the gate script has been downloaded, when the CI workflow runs before executing the gate, then it computes the SHA-256 checksum of the downloaded file and compares it against a known-good value committed in the delivery repository — if the checksums do not match, the workflow exits 1 with a message stating "Gate script checksum mismatch — aborting" and does not execute the gate.

**AC4:** Given the full gate-download, checksum-validate, and gate-execute sequence runs on a standard PR, when the gate passes all checks, then the trace record contains `gateScriptRef` (the `platform-infrastructure` ref used) and `gateChecksumVerified: true` — both fields present in `pipeline-state.schema.json`.

**AC5:** Given the delivery team's permission model for `platform-infrastructure`, when a delivery agent's CI service identity is reviewed, then it has read-only access (or no access) to `platform-infrastructure` — no push, no create-branch, no delete — confirmed by permission documentation in the DoD artefact.

## Out of Scope

- Automated release publishing to `platform-infrastructure` — gate script releases are created manually by the platform maintainer. Automation is a Phase 4 improvement.
- Moving skill files (`.github/skills/`) to `platform-infrastructure` — only the executable gate scripts are moved. Skill files are Markdown content governed by the pipeline process.
- Migrating other CI scripts (e.g. `scripts/validate-trace.sh`) — only `run-assurance-gate.js` and the scripts it directly invokes are moved.

## NFRs

- **Security:** Gate download must use HTTPS. No personal access tokens in workflow YAML — use OIDC or a read-only GitHub App token scoped to the `platform-infrastructure` repo.
- **Audit:** The `gateScriptRef` recorded in each trace provides an immutable audit trail of which gate version evaluated each delivery.
