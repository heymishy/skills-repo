# Story: Add atomic-replace write-safety convention to /checkpoint instructions

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md
**Improvement source:** workspace/learnings.md — state.json write-path safety entry surfaced by /improvement run 2026-04-16

## User Story

As a **platform maintainer**,
I want the /checkpoint convention section of copilot-instructions.md to explicitly describe the atomic-replace write pattern for `workspace/state.json`,
So that agents do not produce a concatenated-JSON state file that breaks `check-workspace-state.js` with "Unexpected non-whitespace character after JSON".

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: reduction in governance check failures per story
**How:** The concatenated-JSON failure mode has occurred in production (observed during Phase 2/3 delivery). Embedding the atomic-replace write convention in the /checkpoint instructions — where agents are directed to perform the state write — prevents recurrence without requiring reactive debugging.

## Architecture Constraints

- Modifications are confined to `.github/copilot-instructions.md` — no code, no schema changes.
- The write-safety paragraph must be inserted in the `/checkpoint` convention section, after the `**Completion expectation:**` paragraph (the 60-second requirement).
- Language must be concrete and actionable: name the failure mode explicitly ("Unexpected non-whitespace character after JSON"), prescribe the safe write pattern (temp file → verify JSON → rename), and name the check script that catches the failure (`check-workspace-state.js`).

## Dependencies

- **Upstream:** None.
- **Downstream:** None — this is a documentation guard. The `check-workspace-state.js` script already correctly catches the failure; this story prevents agents from triggering it.

## Acceptance Criteria

**AC1 — Write-safety paragraph present in /checkpoint convention section**
Given `.github/copilot-instructions.md` is read,
When the reader locates the `/checkpoint` convention section and the `**Completion expectation:**` block,
Then a `**State write safety:**` paragraph is present immediately after the Completion expectation block, containing: a description of the failure mode ("Unexpected non-whitespace character after JSON"), the prescribed safe write pattern (write complete JSON to temp file, verify valid JSON, rename over target), and the name of the check that detects the failure (`check-workspace-state.js`).

**AC2 — Write-safety paragraph uses the correct safe-write pattern**
Given the `**State write safety:**` paragraph is present,
When the text is read,
Then it explicitly states: (a) always write the complete JSON object, never a delta or append; (b) write to a temp file first (`state.json.tmp`) if atomic replacement cannot be guaranteed; (c) verify content is valid JSON before renaming; (d) rename over the target.

**AC3 — Failure mode is named explicitly**
Given the `**State write safety:**` paragraph is present,
When the text is read,
Then it references the exact error string `"Unexpected non-whitespace character after JSON"` or an unambiguous description of the concatenated-JSON failure, and names `check-workspace-state.js` as the check that detects it.

## Out of Scope

- Changes to `check-workspace-state.js` — it already correctly detects the failure.
- Changes to any skill file or governance check.
- Adding an automated test that validates copilot-instructions.md content.

## NFRs

None specific to this story — copilot-instructions.md text addition only.

**Rating:** 1
