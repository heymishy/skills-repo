# Story: Add second-session verification sub-step to /verify-completion skill

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md
**Improvement source:** workspace/learnings.md — second-session inherited-claim risk surfaced by /improvement run 2026-04-16

## User Story

As a **platform maintainer**,
I want the /verify-completion skill to include an explicit Step 0 (Fresh-session check) that agents must run when resuming work from a previous session,
So that agents do not inherit completion claims from prior sessions without independently verifying them, preventing silent false-pass gates.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: reduction in governance check failures per story
**How:** During Phase 2/3 delivery, a new session that picked up work from a prior session inherited its "done" declaration without re-verifying. The Step 0 gate forces independent verification whenever a prior session's state is being resumed, eliminating the class of false-pass scenarios caused by session-boundary trust.

## Architecture Constraints

- Modifications are confined to `.github/skills/verify-completion/SKILL.md` — no code or schema changes.
- The new content must be inserted as a `## Step 0 — Fresh-session check` section, positioned before the existing `## Step 1 — Run the full test suite` section.
- The section must only apply when session continuity is uncertain (i.e., when resuming from a prior session or when state.json shows in-progress work started before this session).
- The existing Steps 1–4 must not be renumbered or reordered.

## Dependencies

- **Upstream:** None.
- **Downstream:** None — new defensive step; existing steps are not modified.

## Acceptance Criteria

**AC1 — Step 0 section present and positioned before Step 1**
Given `.github/skills/verify-completion/SKILL.md` is read,
When the file structure is examined,
Then a `## Step 0 — Fresh-session check` section is present, positioned immediately before `## Step 1 — Run the full test suite`.

**AC2 — Step 0 specifies the trigger condition**
Given the Step 0 section is present,
When the trigger condition text is read,
Then it explicitly states when Step 0 applies: when resuming work from a previous session, when state.json shows in-progress work, or when a prior session has made a completion claim.

**AC3 — Step 0 includes the independent verification instructions**
Given the Step 0 section is present,
When the instructions are read,
Then they direct the agent to: (1) NOT inherit the previous session's completion claim, (2) read state.json / workspace artefacts to understand what was claimed, (3) independently verify by running Step 1 fresh regardless of prior claims, and (4) if prior claim and fresh result disagree, surface the discrepancy before proceeding.

## Out of Scope

- Changes to Steps 1–4 or the Integration/State-update sections.
- Changes to any other skill file, governance check, or source file.
- Adding automated tests that validate SKILL.md content.
- Changes to state.json schema or pipeline-state.json.
