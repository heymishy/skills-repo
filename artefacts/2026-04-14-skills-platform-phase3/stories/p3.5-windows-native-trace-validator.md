# Story: Deliver validate-trace.ps1 with full parity to validate-trace.sh

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e4-windows-ci-parity-pipeline-evolution.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **squad tech lead** working in a Windows PowerShell CI environment,
I want `scripts/validate-trace.ps1` to perform the identical trace validation checks as `validate-trace.sh` — including the `--ci` flag and non-zero exit on hard failure — so that governance checks pass on Windows runners without OS-conditional workarounds.

## Benefit Linkage

**Metric moved:** MM2 — Outer loop self-sufficiency: zero blocking lookups
**How:** Windows-native trace validation eliminates the CI environment gap that currently blocks squad onboarding in PowerShell-only environments. A squad lead can run governance checks locally and in CI without workarounds or switching environments.

## Architecture Constraints

- No external PowerShell module dependencies — must run with PowerShell 5.1+ built-in cmdlets only (consistent with the no-external-deps constraint that applies to all governance scripts).
- CI parity constraint: the `--ci` flag behaviour, exit codes, and check set must be identical to `validate-trace.sh`. Any divergence is a bug, not a design decision.
- The `.sh` file is not modified by this story — the PS1 is a parallel implementation, not a replacement.

## Dependencies

- **Upstream:** None — can begin immediately.
- **Downstream:** None within Phase 3. Windows CI validation is referenced in the Phase 3 CI environment parity confirmation required for the first enterprise adopter onboarding.

## Acceptance Criteria

**AC1:** Given `scripts/validate-trace.ps1` exists, when it is run with `--ci` on a valid trace file on a Windows PowerShell 5.1+ environment, then it exits 0 and outputs a confirmation of each check performed — matching the check names and output format of `validate-trace.sh --ci` on the same input.

**AC2:** Given a trace file with a hard validation failure (e.g. missing required field), when `scripts/validate-trace.ps1 --ci` runs against it, then it exits with a non-zero exit code and prints the specific field or check that failed — matching the exit code behaviour of `validate-trace.sh --ci` on the same input.

**AC3:** Given `validate-trace.ps1`, when the check set is compared to `validate-trace.sh`, then every check present in `validate-trace.sh` is present in `validate-trace.ps1` — no check is absent or renamed. The implementation language differences (PowerShell vs bash) are the only source of code difference.

**AC4:** Given a Windows-based CI runner (or PowerShell environment), when `pwsh -File scripts/validate-trace.ps1 --ci` runs as part of a CI job, then the job completes with the same exit code as `bash scripts/validate-trace.sh --ci` would produce on Linux for the same trace input.

**AC5:** Given `validate-trace.ps1` is present, when `scripts/validate-trace.sh` is run on Linux, then its behaviour is unchanged — the PS1 file does not affect the SH file's execution.

## Out of Scope

- PowerShell rewrites of any other shell scripts in `scripts/` — only `validate-trace.sh` parity is in scope.
- Replacing `validate-trace.sh` — both files remain in the repository.
- Windows-native equivalents for `npm test` or pre-commit hooks — those already work cross-platform via Node.js.

## NFRs

- **Performance:** `validate-trace.ps1 --ci` on a standard trace file must complete within 10 seconds on any Windows PowerShell 5.1+ environment.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — output specification is the shell script itself.
