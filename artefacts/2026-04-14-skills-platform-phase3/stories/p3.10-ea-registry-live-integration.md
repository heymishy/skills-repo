# Story: Integrate EA registry blast-radius and dependency graph at discovery time

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **squad tech lead** starting a new feature discovery,
I want the `/discovery` skill to query the EA registry for blast radius and dependency context — presenting affected systems, downstream consumers, and domain ownership — so that I do not need to open a separate registry lookup mid-discovery session.

## Benefit Linkage

**Metric moved:** MM2 — Outer loop self-sufficiency: zero blocking lookups
**How:** The current Phase 2 EA registry (p2.6) is a standalone skill invoked manually. Phase 3 integration means the `/discovery` skill calls `getBlastRadius(systemId)` automatically when the discovery references a known system — reducing the "blocking lookup" count by eliminating a manual context switch.

## Architecture Constraints

- Path A (registry repo): if `context.yml` has `architecture.ea_registry_authoritative: true`, the `/discovery` skill calls `getBlastRadius()` against the EA registry API or file store defined there.
- Path B (context.yml): if `architecture.ea_registry_authoritative: false` or not set, the skill falls back to the local `context.yml` system list — this path must remain valid and Path B squads must not be broken by this change.
- The integration is read-only — `/discovery` never writes to the EA registry.
- This story builds on Phase 2 p2.6 (EA registry SKILL.md). The Phase 2 artefact must be DoD-complete (it is).

## Dependencies

- **Upstream:** Phase 2 p2.6 (EA registry) — DoD-complete. Blast radius query API or file format is already defined.
- **Downstream:** None within Phase 3. p3.13 may benefit from EA context but does not hard-depend on this story.

## Acceptance Criteria

**AC1:** Given `context.yml` with `architecture.ea_registry_authoritative: true` and a registry that contains a system entry matching the discovery subject, when `/discovery` runs, then it presents the blast-radius summary (affected systems count, downstream consumers list, domain owner) inline in the discovery session — without the operator having to invoke `/ea-registry` separately.

**AC2:** Given `context.yml` with `architecture.ea_registry_authoritative: false` (or the key absent), when `/discovery` runs, then it behaves exactly as it did before this story — no blast-radius section is surfaced and no error occurs. Path B squads are unaffected.

**AC3:** Given a discovery subject that does not match any system in the EA registry, when `/discovery` runs and queries the registry, then it surfaces "no EA registry entry found for [systemId]" and continues the discovery session without error — the absence of a registry entry is not a blocker.

**AC4:** Given the updated `/discovery` SKILL.md, when the integration call is reviewed, then it calls the EA registry in read-only mode — no write operations are performed on the registry during discovery.

**AC5:** Given both Path A and Path B, when the modified SKILL.md is reviewed, then both paths are explicitly documented with their configuration condition — a squad lead can determine which path applies to them by reading `context.yml` alone.

## Out of Scope

- Writing to the EA registry from `/discovery` — read-only only.
- Embedding the full EA registry content in the discovery artefact — only the blast-radius summary is included.
- Phase 4: EA registry contribution from `definition` and `review` skills — out of Phase 3 scope.

## NFRs

- **Graceful degradation:** Path B fallback must not produce any error or warning when `ea_registry_authoritative` is not set.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — integration pattern is well-defined; EA registry API is already established.
