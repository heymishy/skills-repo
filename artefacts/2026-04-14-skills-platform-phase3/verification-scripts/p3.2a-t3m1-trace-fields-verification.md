### Scenario 1: The four new trace fields are defined in the schema

**Covers:** AC1 (schema field definitions present)

**Steps:**
1. Open `.github/pipeline-state.schema.json` in a text editor or code viewer.
2. Search for the text `standardsInjected`.
3. Check that a definition exists with a type (object or array) and a description that mentions "Q2" or "MODEL-RISK".
4. Search for `watermarkResult`.
5. Check that a definition exists with at minimum `pass` and `passRate` sub-fields and a description mentioning "Q5".
6. Search for `stalenessFlag`.
7. Check that a definition exists with type `boolean` and a description mentioning "Q6".
8. Search for `sessionIdentity`.
9. Check that a definition exists with at minimum `sessionId`, `agentType`, and `startedAt` sub-fields and a description mentioning "Q7".

**Expected outcome:**
> All four field names are present in the schema file. Each has a type definition and a description that references its associated MODEL-RISK question (Q2 for standardsInjected, Q5 for watermarkResult, Q6 for stalenessFlag, Q7 for sessionIdentity). No field is missing or described without a MODEL-RISK question mapping.

*Note: This scenario requires the implementation to exist. If running pre-code, read through the expected outcome and confirm it describes correct behaviour.*

**Steps:**

### Scenario 4: MODEL-RISK.md contains entries for all four new trace fields

**Covers:** AC4 (documentation completeness)

**Steps:**
1. Open `docs/MODEL-RISK.md` in a text editor.
2. Search for the heading or text corresponding to each of the four new fields: `standardsInjected`, `watermarkResult`, `stalenessFlag`, `sessionIdentity`.
3. For each field entry, check the following are present:
   a. The field name.
   b. Which MODEL-RISK question it answers (Q2, Q5, Q6, or Q7).
   c. What a reviewer should see in the field for a passing answer (e.g. "a list of at least one standards identifier with a non-null hash").
   d. A reference to where the field is defined (e.g. "see `pipeline-state.schema.json`").

**Expected outcome:**
> MODEL-RISK.md has a complete entry for all four fields. Each entry names the field, names the MODEL-RISK question, describes a passing value, and points to the schema. A reviewer reading these entries can answer their respective audit questions without asking an engineer.

### Scenario 6: sessionIdentity.sessionId does not contain a personal identifier

**Covers:** Security NFR (sessionId must not contain PII)

*Note: This scenario requires the implementation to exist. Review a trace file produced after merge.*

**Steps:**
1. After the implementation is deployed, trigger a regulated story gate run (or inspect a fixture produced by `npm test`).
2. Open the resulting trace file (or the fixture output) and locate the `sessionIdentity` object.
3. Read the `sessionId` value.

**Expected outcome:**
> The `sessionId` value is a hashed or opaque identifier — not a real email address, not a user display name, and not a UUID that is directly tied to a named individual's account. Example of a passing value: `"a3f7c2e8..."` (hex hash). Example of a failing value: `"alice@example.com"` or `"Alice Smith"`.
