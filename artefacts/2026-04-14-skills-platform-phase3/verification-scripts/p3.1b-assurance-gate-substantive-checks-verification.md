## Setup

**Before you start:**
1. Open a terminal at the repo root.
2. Ensure `npm test` passes with no failures before starting manual verification.
3. For Scenario 6: have a stopwatch or note the time — you only need to confirm the gate ran, not precisely time it.

**Reset between scenarios:** Each scenario reads from fixtures — no persistent state change between scenarios.

### Scenario 1: Gate fails when workspace/state.json has missing or invalid core fields

**Covers:** AC1

**Steps:**
1. After implementation, run `npm test` and look for the test section for `check-workspace-state` or `run-assurance-gate`.
2. Confirm tests exist that verify the gate catches a state.json missing `currentPhase` and a state.json with a malformed `lastUpdated` (e.g. `"not-a-date"`).
3. Confirm those tests pass (green checkmarks).

**Expected outcome:**
> Tests for missing/invalid `currentPhase` and malformed `lastUpdated` exist and pass. The gate reports failure for both cases — not just one.

### Scenario 6: Gate completion trace shows >5ms between start and finish

**Covers:** AC6

**Steps:**
1. Run `run-assurance-gate.js` against the real repository (or the full-pass fixture): `node .github/scripts/run-assurance-gate.js` (or equivalent path)
2. Look at the output or the trace written by the gate — find the `startedAt` and `completedAt` timestamps.
3. Calculate the difference (or check the test that asserts the delta).

**Expected outcome:**
> `completedAt` is at least 5 milliseconds after `startedAt`. On any real hardware running real file reads, this threshold is easily met. The delta should be distinctly more than 1–2ms (the pre-story baseline for file-existence-only checks).

**Result:** [ ] Pass  [ ] Fail
