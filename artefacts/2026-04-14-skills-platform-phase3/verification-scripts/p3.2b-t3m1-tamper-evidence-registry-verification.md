### Scenario 2: Auditor can recompute the hash and it matches the registry

**Covers:** AC2 (hash reproducibility — independent verification)
*[post-merge only]*

**Steps:**
1. In `workspace/traces/`, open the trace file for the most recently attested story (e.g. `[story-slug]-trace.jsonl`).
2. Find the `tamperEvidence` object at the end of the last line in the file. Note the `registryRef` value (this is the attestation URL or ID).
3. Open a terminal and run: `sha256sum workspace/traces/[story-slug]-trace.jsonl` (on Linux/Mac) or `Get-FileHash workspace/traces/[story-slug]-trace.jsonl -Algorithm SHA256` (on Windows PowerShell).
4. Copy the hash value produced.
5. Open the attestation record at the `registryRef` URL (via a browser or `gh attestation verify --repo [owner]/[repo]` in the GitHub CLI).
6. Read the hash value stored in the registry.
7. Compare the two hash values.

**Expected outcome:**
> The hash value computed in step 3–4 matches the hash value stored in the registry exactly — no characters differ. This confirms the trace file has not been modified since attestation.

**Result:** [ ] Pass  [ ] Fail

### Scenario 3: The trace file contains a complete tamperEvidence object

**Covers:** AC4 (tamperEvidence fields present in trace)
*[post-merge only]*

**Steps:**
1. In `workspace/traces/`, open the trace file for the most recently attested story.
2. Read the last line (or the line containing `tamperEvidence`).
3. Check that a `tamperEvidence` object is present with the following three fields:
   a. `registryType` — should be the text `"github-artifact-attestation"`
   b. `registryRef` — should be a non-empty URL or reference string pointing to the attestation record
   c. `publishedAt` — should be an ISO 8601 date and time (e.g. `"2026-04-15T10:30:00Z"`)

**Expected outcome:**
> The trace file line contains a `tamperEvidence` object with all three required fields. None are missing or null. `registryType` is exactly `"github-artifact-attestation"`. `registryRef` is a readable URL. `publishedAt` is a recognisable date-time.

### Scenario 4 (gap — manual): Delivery team cannot modify the attestation record

**Covers:** AC3 (access control denial)
**Gap type:** Untestable-by-nature (access control enforced by GitHub platform infrastructure)

**Steps:**
1. Open `.github/workflows/attestation-publisher.yml` (or the equivalent post-merge workflow file) in a text editor.
2. Find the `permissions:` block in the workflow YAML.
3. Confirm that the workflow does NOT have `contents: write` set for the main evaluation workflow (only the publisher has write scope).
4. Confirm that the publisher workflow fires on `push` to main only — not during a PR review where a developer could influence it.
5. Read the `id-token:` permission setting — it should say `write` (required for OIDC).
6. Confirm that no step in the workflow uses a secret named `PAT`, `GITHUB_PAT`, or any personal access token variable.

**Expected outcome:**
> The publisher workflow uses OIDC (`id-token: write`) for authentication, not a personal access token. The workflow fires post-merge, not during PR review. No PAT secret is referenced. A delivery team member with standard `contents: write` access to the delivery repository cannot write to or delete the resulting Artifact Attestation record — GitHub's attestation infrastructure enforces this independently.

**Result:** [ ] Pass  [ ] Fail  [ ] Needs confirmation from platform owner
