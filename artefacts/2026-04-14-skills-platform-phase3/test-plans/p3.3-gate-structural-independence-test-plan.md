## Unit Tests

### checksum-match-allows-execution

- **Verifies:** AC3
- **Test file:** `tests/check-assurance-gate.js` (extend existing) or `src/gate-validator/checksum-validator.test.js`
- **Precondition:** A checksum validator module exists at `src/gate-validator/checksum-validator.js`
- **Action:** Call the validator with a gate script file and its correct SHA-256 hash
- **Expected result:** Validator returns success (exits 0 / resolves promise); no error thrown
- **RED step:** Module does not exist yet — test file not found or function throws

### checksum-mismatch-exits-with-abort-message

- **Verifies:** AC3
- **Test file:** `src/gate-validator/checksum-validator.test.js`
- **Precondition:** Validator module exists; a file and an incorrect expected hash are prepared
- **Action:** Call validator with a file and a hash that does NOT match the file's actual SHA-256
- **Expected result:** Validator exits 1 (or rejects with a well-typed error); the message includes the exact text "Gate script checksum mismatch — aborting"
- **Edge case:** Verify the exact string — partial match is insufficient
- **RED step:** Validator does not exist; checksum match always passes; message text differs

### checksum-mismatch-does-not-execute-gate

- **Verifies:** AC3
- **Test file:** `src/gate-validator/checksum-validator.test.js`
- **Precondition:** A mock for the gate execution function is prepared
- **Action:** Call the checksum validator with a mismatched hash, then check whether the gate execution mock was called
- **Expected result:** Gate execution mock is NOT called when checksum does not match
- **RED step:** Validator executes gate regardless of checksum result

## Integration Tests

### workflow-yaml-uses-pinned-immutable-ref

- **Verifies:** AC2
- **Test file:** `tests/check-assurance-gate.js`
- **Precondition:** The delivery repo's CI workflow file for gate download exists (e.g. `.github/workflows/assurance-gate.yml`)
- **Action:** Read the workflow YAML file(s) that perform the gate download step; parse the `ref:` value used in the checkout or download action
- **Expected result:** The `ref:` value is either a full 40-character commit SHA or a version tag matching the pattern `v\d+\.\d+` (or similar semver) — NOT the string "main", "master", or "HEAD"
- **RED step:** Workflow does not exist yet; or `ref:` resolves to branch head

### schema-contains-gate-script-ref-fields

- **Verifies:** AC4
- **Test file:** `tests/check-workspace-state.js` or `tests/check-trace-commit.js`
- **Precondition:** `pipeline-state.schema.json` exists
- **Action:** Load `pipeline-state.schema.json`; check that the schema definition contains `gateScriptRef` and `gateChecksumVerified` as defined properties — either at the top level or within the story/trace sub-object
- **Expected result:** Both field names present in schema definitions
- **RED step:** Schema lacks both fields before implementation

## Manual Tests

### shim-only-in-delivery-repo-pr-diff

- **Verifies:** AC1
- **Instruction:** Open the implementation PR on GitHub; open "Files changed"; verify that no `run-assurance-gate.js`, `check-*.js` governance check scripts (beyond the shim wrapper) are present in the delivery repository's working tree. The PR diff should show a shim file (or wrapper entry point) that invokes the downloaded gate — not the gate implementation itself. Confirm the platform-infrastructure repo has been created and contains the gate scripts.
- **Expected outcome:** Delivery repo contains only shim; gate scripts are absent or present only as references

### read-only-service-identity-documented

- **Verifies:** AC5
- **Instruction:** Review the CI job's permission block in the assurance gate workflow YAML. Confirm that `permissions:` does not include `write` for `platform-infrastructure`-scoped resources. Confirm the DoD artefact includes a statement on the service identity's access model.
- **Expected outcome:** Permission block is read-only or not-set for platform-infrastructure access; DoD documents this
