## Outcome: PROCEED

### What was found

**Step 1 — `cli-outer-loop.js` is a pure exportable module.** It exports a single `validate(artefactPath, gateName, repoRoot)` function that returns `{ exitCode, stdout, stderr }`. It has no side effects, no subprocess calls, and no external dependencies beyond `fs` and `path`. Any caller — CLI binary or web server route handler — can `require()` it directly.

**Step 2 — `pipeline-state-writer.js` already exists** (`src/web-ui/adapters/pipeline-state-writer.js`). It is a factory that returns a `pipelineStateWriter(featureSlug, storyId, stateUpdate)` function. It performs: (a) enum validation for `prStatus`, `dorStatus`, `health`; (b) read-current-state; (c) find-or-create feature and story entries; (d) atomic write via `tmp` file then `fs.renameSync`. It is already wired in `server.js` and already called in `journey.js` `handlePostGateConfirm` on successful stage completion.

**Step 3 — The current gate-confirm path does NOT run CLI validation.** `handlePostGateConfirm` writes state immediately after the session completes — it does not call `cli-outer-loop.validate()` first. This is the integration gap: state can be written for a DoR stage even if the DoR artefact fails the H1-H9 structural checks.

**Step 4 — The web UI does not use subprocess for route handlers.** `child_process` appears in `cli-adapter.js` only for `git fetch` — not a route handler pattern. `tool-executor.js` uses direct `fs` calls. `skill-executor.js` (for `handleExecuteSkill`) does not use subprocess either. The established web UI pattern for internal functionality is `require()`, not subprocess.

**Step 5 — `governance-package.js` `advanceState` is an in-memory transition validator**, not a disk writer. It checks whether a declared workflow allows a `current → next` transition. It is complementary to `pipeline-state-writer.js` (validate transition, then write) but does not replace it.

### Reasoning

The original H7.1 framing — "subprocess integration assessment" — was the wrong lens. It assumed that the `bin/skills` binary would be the integration surface for the web UI. The code investigation shows the correct integration surface is the exported modules (`cli-outer-loop.js` and `pipeline-state-writer.js`), which are already structured for `require()` import.

Subprocess invocation from a route handler would introduce: PATH resolution risk, shell injection surface (even with sanitised inputs), stderr/stdout parsing overhead, and subprocess lifecycle management. None of these are present with direct module import.

The practical Phase 2 design emerges clearly from the findings:
- Web UI gate-confirm: call `cli-outer-loop.validate()` before calling `_pipelineStateWriter` — if validation fails, reject the gate-confirm with a structured error instead of writing bad state.
- `skills advance` CLI binary: thin wrapper that calls `pipeline-state-writer` directly (for CI use, providing typed exit codes to CI harnesses).
- Chain-hash trace: `governance-package.writeTrace()` already exists and is structured for this purpose — wire it in the gate-confirm after the state write.

## If PROCEED

**Unblocked stage:** Phase 2 definition — `skills advance` command and web UI gate-confirm integration stories can now be written.

**Conditions:**
- `cli-outer-loop.validate()` must be called with the DoR artefact path that the gate-confirm session produced. The artefact must be written to disk *before* validate is called (the gate-confirm handler already does disk write before state write — order is correct per the disk canonicity rule).
- `pipeline-state-writer.js` already enforces enum validation; Phase 2 stories must not duplicate this — they extend what fields it validates, not rewrite the pattern.
- The `bin/skills advance` command is for CI use only. The web UI uses `require()`, not subprocess. This must be stated in the Phase 2 epic out-of-scope.

---

## What remains unknown

- Whether `pipeline-state-writer.js` needs schema validation beyond the three enum fields it currently checks (`prStatus`, `dorStatus`, `health`). Phase 2 may add fields (e.g. `stage`, `dorArtefact`) that need validation. This is a Phase 2 story scope question, not a feasibility blocker.
- Chain-hash trace design: how prior trace entries are read to compute the chain hash (append-only guarantee). `governance-package.writeTrace()` exists but does not implement chain hashing yet. This is Phase 2 scope.
