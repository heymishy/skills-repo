## Uncertainty addressed

Can the web UI backend integrate with the `skills validate` and `skills advance` CLI commands — and by which mechanism (subprocess vs module import) — to enforce deterministic gate checks before allowing pipeline state to advance?

---

## Options evaluated

| Option | Description | Pros | Cons | Verdict |
|--------|-------------|------|------|---------|
| A | Subprocess: web UI route handler spawns `node bin/skills validate` via `child_process.spawn`, reads exit code and stderr | Independent process boundary; same binary as CI | PATH resolution, shell injection surface, stdio parsing overhead, subprocess lifecycle management, harder to test | ✗ Ruled out |
| B | Module import: web UI `require('../enforcement/cli-outer-loop').validate(...)` directly | Zero overhead, no shell injection risk, same test patterns as rest of codebase, already how `pipeline-state-writer` is integrated | None — this is the established pattern | ✓ Recommended |
| C | Hybrid: route handler uses module import; `bin/skills advance` CLI binary (for CI) also calls modules | Best of both — module import for web UI, typed exit codes for CI harnesses | Requires two thin callers (route handler + bin entry point) for the same modules | ✓ Also recommended — this is the Phase 2 design |

---

## Recommendation

Phase 2 should use Option C (hybrid). The web UI gate-confirm handler calls `cli-outer-loop.validate()` via `require()` before calling `_pipelineStateWriter`. If validation fails, the handler returns a 422 with the specific failing check in the response — the state write does not proceed. The `bin/skills advance` entry point is a separate thin wrapper that calls `pipeline-state-writer.js` for CI use, returning typed exit codes for CI harnesses to act on.

This design means:
1. The web UI never spawns subprocesses for enforcement logic — consistent with every other internal integration in the codebase.
2. CI systems get a clean `node bin/skills advance` binary they can call in pipeline steps.
3. Both callers exercise the same underlying modules, so a test for the module tests both surfaces.

The current gate-confirm handler (`handlePostGateConfirm` in `src/web-ui/routes/journey.js`) writes state immediately after session completion with no structural validation. Phase 2 closes this gap: validate artefact → if exit 0, write state → if non-zero, reject with error details.

## Constraints confirmed

| Constraint | Source | Implication |
|------------|--------|-------------|
| `cli-outer-loop.validate()` must be called after disk write, before state write | Disk canonicity rule (ougl ADR-023) | Gate-confirm already writes artefact to disk first — order is correct. No reordering needed. |
| No subprocess in route handlers | Established web UI pattern (no `child_process` in any route file) | `bin/skills advance` is for CI only; web UI calls modules directly. |
| `pipeline-state-writer.js` atomic write pattern must not be duplicated | ADR-004 (no hardcoded paths; callers inject config) | Phase 2 `bin/skills advance` calls the same factory, not a reimplementation. |
| Enum validation already in `pipeline-state-writer.js` | Existing code (`prStatus`, `dorStatus`, `health`) | Phase 2 stories extend the validated field list; they do not rewrite the validator. |
