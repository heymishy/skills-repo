## Story: Cleanup scripts for local disk and staging-DB test-generated data

**Short-track:** operational tooling gap — no way to clean up either local stray artefact directories or real staging-database rows created by E2E test runs.

## User Story

As **Hamish King (Founder/Operator)**,
I want **two safe, dry-run-by-default cleanup scripts — one for local stray test artefact folders, one for staging-database journeys/artefacts created by E2E test runs**,
So that **repeated test runs stop accumulating disk and database clutter with no way to remove it**.

## Background / Investigation

- **Local disk:** confirmed several stray `artefacts/YYYY-MM-DD-discovery/` directories (each containing only a bare `discovery.md`) and `workspace/test-tmp-*` directories, sitting untracked, generated by prior skill-pipeline test/dev runs.
- **CI unit/integration tests:** confirmed these never touch a real database (no `DATABASE_URL` in `assurance-gate.yml`/`pr-checks.yml`) — they use in-memory stores exclusively.
- **E2E tests:** hit the real deployed `wuce-staging` app over HTTP, which has a real Postgres (`journeys`, `artefacts` tables, `src/web-ui/adapters/journey-store-pg.js`). Every E2E run against staging creates real rows there with no cleanup path.
- **Safe identification signal found:** `tests/e2e/fixtures/staging-auth.js` tags every E2E-generated identity with an `e2e-test-` prefix (GitHub stub logins: `e2e-test-gh-...`; email signups: `e2e-test-...@example.test`). `req.session.tenantId = user.login` (confirmed in `auth.js`/`auth-stub.js`) means `journeys.tenant_id` and `journeys.owner_id` both carry this exact prefix for every E2E-created journey — a safe, narrow, false-positive-free identification rule.

## Acceptance Criteria

**AC1:** Given `scripts/clean-local-test-artefacts.js` is run with no flags, When it completes, Then it prints every `artefacts/*/` directory matching a recognised test-fixture shape (a bare `discovery.md` with no other pipeline files, or a known test-slug pattern) and every `workspace/test-tmp-*` directory, and deletes nothing.

**AC2:** Given the same script is run with `--delete`, When it completes, Then it removes exactly the directories it listed in the dry-run, and nothing else — no directory outside the recognised test-fixture shape is ever touched, even with `--delete`.

**AC3:** Given `scripts/clean-e2e-staging-data.js` is run with `DATABASE_URL` set (staging's real connection string, supplied by the operator, never hardcoded) and no `--confirm` flag, When it completes, Then it prints a count and list of `journeys` rows where `tenant_id LIKE 'e2e-test-%' OR owner_id LIKE 'e2e-test-%'`, and deletes nothing.

**AC4:** Given the same script is run with `--confirm`, When it completes, Then it deletes exactly the matched `artefacts` rows (by `journey_id` FK) followed by the matched `journeys` rows, and no row outside the `e2e-test-` prefix pattern is ever touched.

**AC5:** Given `DATABASE_URL` is not set at all, When `scripts/clean-e2e-staging-data.js` is run, Then it exits with a clear error message rather than silently connecting to nothing or crashing with a raw stack trace.

## Out of Scope

- Automatically running either script in CI or on a schedule — both are manual, operator-invoked tools this session.
- Cleaning up any table other than `journeys`/`artefacts` (no other real Postgres tables exist in this schema today, confirmed via `journey-store-pg.js`).
- Any change to how E2E tests themselves tag identities (the `e2e-test-` prefix already exists and is reused, not introduced by this story).

## Benefit Linkage

Removes a real, growing operational cost (disk clutter locally, real row accumulation in a live database) with a safe, narrowly-scoped, reversible-until-confirmed mechanism.

## Complexity Rating

**Rating:** 2 — the local-disk script is straightforward; the staging-DB script touches a real, shared external system and must be conservative about what it matches.
**Scope stability:** Stable.

## NFRs

- **Security:** The staging-DB script must never accept or log the full `DATABASE_URL` value in its own output (connection strings can carry credentials). Only ever delete rows matching the exact `e2e-test-` prefix — no wildcard/blanket-delete mode.
