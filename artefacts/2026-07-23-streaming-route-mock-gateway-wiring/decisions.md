# Decisions: Streaming Route Mock Gateway Wiring

## DECISION — Simple additive options-wiring fix, not a structural streaming/non-streaming divergence (2026-07-23)

**Context:** `artefacts/2026-07-23-e2e-core-journey-coverage/decisions.md`'s a4 FINDING entry (surfaced during a4's own dispatch, referenced from PR #559's description) documented that after `mgfd-s1` (PR #558) made `MOCK_LLM_GATEWAY=true` genuinely active on `wuce-staging` — env var set, adapter unconditionally wired in `server.js`, fixture files shipped in the Docker image — a4's own dispatch found the flag STILL has zero effect on the real chat UI. This story was dispatched to investigate whether the root cause is a simple oversight or a deeper structural gap between the streaming and non-streaming turn-handling code paths, and to fix it narrowly if the former.

**Investigation finding:** Read `src/web-ui/routes/skills.js` in full for both `handlePostTurnStreamHtml` (the real streaming route — the ONLY endpoint the actual browser chat UI calls) and `htmlSubmitTurn` (the non-streaming, UI-unused path that already correctly routes through the mock gateway, per bri-s3.2). `htmlSubmitTurn` builds `_turnMeta = { stage: skillName, scenarioName: session.mockScenarioName || 'success', tenantId: tenantId || null, sessionId: sessionId }` and passes it to `skillTurnExecutor()`. `handlePostTurnStreamHtml` builds a directly analogous `_turnOptions` object — it already threads `noThinking`, `maxTokens`, `model`, and (as of s6.1) `tenantId`/`sessionId` — but the `stage`/`scenarioName` pair was simply never added. Confirmed against `src/modules/skill-turn-executor.js`: `skillTurnExecutorStream()`'s mock-gateway routing check is `if (options && options.stage && mockLlmGateway.isMockGatewayEnabled())` — structurally identical to `skillTurnExecutor()`'s own check for the non-streaming path. Both executor functions already fully and correctly support mock-gateway routing; the gap was entirely in one route handler's object construction, not in any difference between the streaming and non-streaming execution models.

**Decision:** This is a simple oversight, not a deeper structural gap. Fixed with a two-line addition to `handlePostTurnStreamHtml`'s `_turnOptions` construction, placed immediately after the existing `tenantId`/`sessionId` assignments (the s6.1 precedent for exactly this "thread an existing session field into the options object in this exact spot" shape):
```js
_turnOptions.stage = session.skillName;
_turnOptions.scenarioName = session.mockScenarioName || 'success';
```
No change to `mock-llm-gateway.js`, `skill-turn-executor.js`, `server.js`, SSE framing, or any other part of `handlePostTurnStreamHtml`.

**Rationale:**
1. **The existing `tenantId`/`sessionId` threading (s6.1) in the exact same function is direct, load-bearing precedent** that this class of fix — reading an existing session field and assigning it onto `_turnOptions` right before the executor call — is both safe and the established pattern in this codebase, not a new design decision.
2. **The executor's mock-gateway routing condition (`options.stage` truthy AND gateway enabled) is unchanged and already correct** — verified by `tests/check-bri-s3.1-mock-llm-gateway.js` and the non-streaming path's own long-standing correct behaviour. This fix supplies the one missing input the condition needs; it does not touch the condition itself.
3. **Verified via a real, end-to-end test (not a mocked executor) that the fix closes the gap completely**: with `MOCK_LLM_GATEWAY=true` and the real `wireDefaultMockGatewayClient()` (the actual production wiring path from `mgfd-s1`) wired, calling `handlePostTurnStreamHtml` now returns the real `discovery.success.json` fixture's own distinguishing text over SSE, and `https.request` (the real network boundary shared by both the Anthropic and Copilot providers) is never invoked — proving no real model call is attempted, exactly the behaviour a real browser-driven turn on staging needs.
4. **TDD discipline followed**: the new test (`tests/check-srmw-s1-streaming-mock-gateway-wiring.js`) was written and run against the pre-fix code first, confirming all 3 assertions fail for the exact reasons the FINDING describes (`options.stage`/`options.scenarioName` undefined; a real `https.request` call attempted and only prevented from succeeding by the test's own monkey-patched throw) — then the fix was applied and the same test suite confirmed GREEN.

**Consequence:** `MOCK_LLM_GATEWAY=true` now genuinely activates for real, browser-driven streaming turns on `wuce-staging` (once deployed), closing the last remaining gap in this session's mock-gateway-on-staging chain (Docker fixtures + adapter wiring from `mgfd-s1`; options wiring from this story). This unblocks re-verification of a3's/a4's currently-skipped AC3-class assertions and any future story that drives a real turn through the real chat UI with the mock gateway active.
