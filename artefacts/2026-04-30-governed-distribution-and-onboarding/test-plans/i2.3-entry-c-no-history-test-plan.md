- **Expected result:** "Entry C" appears in the file
- **Edge case:** No

### orient-entry-c-names-state-no-history-brownfield

- **Verifies:** AC1(a) — output names state "Entry C — no-history brownfield"
- **Precondition:** SKILL.md exists
- **Action:** Read file; check for "Entry C — no-history brownfield" or equivalent canonical label
- **Expected result:** Entry C state is named explicitly
- **Edge case:** No

### orient-entry-c-explains-retrospective-discovery

- **Verifies:** AC1(b) — explains /discovery as retrospective exercise to document what was built
- **Precondition:** SKILL.md exists
- **Action:** Read file; check Entry C section explains running /discovery retrospectively (not greenfield)
- **Expected result:** Entry C section describes /discovery as documenting existing work, not starting fresh
- **Edge case:** No

### orient-entry-c-surfaces-retrospective-story-template

- **Verifies:** AC1(c) — surfaces .github/templates/retrospective-story.md
- **Precondition:** SKILL.md exists
- **Action:** Read file; check Entry C section references `retrospective-story.md` template
- **Expected result:** `retrospective-story.md` (or `retrospective-story`) appears in the Entry C routing section
- **Edge case:** No

### orient-entry-c-routes-to-discovery-with-retrospective-note

- **Verifies:** AC1(d) — routes to /discovery with explicit note that this is retrospective discovery
- **Precondition:** SKILL.md exists
- **Action:** Read file; check Entry C section routes to `/discovery` AND includes the word "retrospective" in or near the routing instruction
- **Expected result:** `/discovery` is the named next skill; "retrospective" appears in the same routing context
- **Edge case:** No

### orient-entry-c-nfr-word-retrospective-in-output

- **Verifies:** NFR-retrospective-label — the word "retrospective" must appear in the routing output instruction
- **Precondition:** SKILL.md exists
- **Action:** Read file; search specifically for the literal word "retrospective" in the Entry C routing section
- **Expected result:** "retrospective" is present — it must not be omitted or replaced with synonyms
- **Edge case:** No — this is an explicit NFR requirement

### orient-entry-c-distinguishes-from-entry-b-in-signal

- **Verifies:** AC2 — routing output distinguishes Entry C from Entry B
- **Precondition:** SKILL.md exists
- **Action:** Read file; check Entry C signal description includes language about why Entry B does not apply (e.g. "no application logic files found")
- **Expected result:** Entry C signal description explicitly references the absence of application logic files (the Entry B check) as the reason Entry C was selected
- **Edge case:** No

### orient-entry-c-signal-explains-why-not-entry-b

- **Verifies:** AC2 — explains why Entry B is not applicable
- **Precondition:** SKILL.md exists
- **Action:** Read file; check for language pattern like "Entry B is not applicable" or "No application logic files found" near Entry C
- **Expected result:** Entry C section contains an explicit statement that Entry B was considered and rejected
- **Edge case:** No

### orient-entry-c-does-not-fire-on-empty-repo

- **Verifies:** AC3 — truly empty repo routes to greenfield, not Entry C
- **Precondition:** SKILL.md exists
- **Action:** Read file; check that Entry C detection requires a brownfield signal (git history, ops files) and explicitly excludes the empty-repo case
- **Expected result:** Entry C detection condition requires at least one brownfield indicator; the greenfield / empty state is handled by the earlier i1.1 AC1 routing rule, not Entry C
- **Edge case:** Yes — this is the critical boundary condition

## Integration Tests

### orient-entry-c-is-fallback-after-a-and-b-in-priority-chain

- **Verifies:** AC3 integration — Entry C is the last resort in the A → B → C priority chain; empty repo hits greenfield before reaching Entry C
- **Components involved:** orient SKILL.md (Entry A from i2.1, Entry B from i2.2, Entry C from i2.3, greenfield from i1.1)
- **Precondition:** SKILL.md exists with full A → B → C chain
- **Action:** Read file; verify the routing priority order places i1.1 AC1 (new/empty repo) before Entry C; Entry C only fires when brownfield signals are present but neither A nor B apply
- **Expected result:** The priority chain is explicit: greenfield (i1.1 AC1) → Entry A (stories) → Entry B (code) → Entry C (fallback brownfield). Entry C is never reached from a truly empty repo.
