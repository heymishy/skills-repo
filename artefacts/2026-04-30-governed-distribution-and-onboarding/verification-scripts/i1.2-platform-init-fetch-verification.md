## Setup

**Before you start:**
1. Clone the repo into a fresh working directory.
2. Confirm `npm run platform:init` and `npm run platform:fetch` are registered in `package.json`.
3. Have a throwaway test directory ready — `platform:init` writes files to the working directory.

**Reset between scenarios:** Run each scenario in a separate temporary directory.

---

## Scenarios

### Scenario 1 — `platform:init` on an empty directory (AC1)

**Command:**
```
cd /tmp/test-repo-1
npm run platform:init
```

**Expected outcome (check all):**
- `[ ]` A `.github/skills/` directory was created containing at least one SKILL.md file
- `[ ]` A `.github/templates/` directory was created
- `[ ]` A `.github/copilot-instructions.md` file was created (generated from assembled skill content)
- `[ ]` A `workspace/` directory was created
- `[ ]` Terminal output shows a summary of what was copied: the skill names installed and file count

**Pass:** All five checkboxes satisfied.
**Fail:** Any checkbox unchecked, or the command errors out.

### Scenario 2 — VS Code slash commands available after `platform:init` (AC2) — MANUAL / IDE-DEPENDENT

**Type:** Manual. This test requires a VS Code session with Copilot.

**Steps:**
1. Run `npm run platform:init` in a new directory.
2. Open that directory as a VS Code workspace.
3. Open Copilot Chat.
4. Type `/orient` — confirm it auto-suggests in the slash command picker.
5. Type `/discovery` — confirm it auto-suggests.

**Expected outcome:**
Both `/orient` and `/discovery` appear as valid slash commands.

**Pass:** Both commands visible.
**Fail:** Either command not found. (Note: this may require restarting VS Code after init for the extension to pick up the new instructions file.)

**Gap note:** This AC cannot be automated — it depends on VS Code extension state. Acceptable as a manual check pre-launch.

### Scenario 3 — `platform:init` is non-destructive; `--force` overwrites (AC3)

**Step A — Non-destructive default:**
```
cd /tmp/test-repo-2
# Create a pre-existing file that platform:init would normally write
mkdir -p .github/skills/discover
echo "EXISTING CONTENT" > .github/skills/discover/SKILL.md
npm run platform:init
```

**Expected outcome (Step A):**
- `[ ]` The pre-existing `SKILL.md` was NOT overwritten — `cat .github/skills/discover/SKILL.md` still shows `EXISTING CONTENT`
- `[ ]` Terminal output states that existing files were skipped, not overwritten

**Step B — Force overwrite:**
```
npm run platform:init --force
```

**Expected outcome (Step B):**
- `[ ]` The previously-existing `SKILL.md` has been overwritten with the current platform content
- `[ ]` Terminal output mentions forced overwrite

**Pass:** Both Step A and Step B pass.
**Fail:** Step A overwrites (destructive), or Step B does not overwrite (--force has no effect).

### Scenario 4 — `platform:fetch` writes timestamp to `workspace/` (AC4)

**Command:**
```
cd /tmp/test-repo-3
npm run platform:init    # create baseline
npm run platform:fetch   # fetch latest
```

**Expected outcome:**
- `[ ]` A file in `workspace/` (e.g. `workspace/platform-last-fetched.txt` or similar) was created or updated
- `[ ]` The file contains a timestamp (ISO-8601 or similar human-readable date)
- `[ ]` Terminal output includes `platform:fetch` in the entry (to distinguish from other writes)

**Pass:** Timestamp file created with non-empty timestamp content.
**Fail:** No timestamp file, or timestamp is empty.

### Scenario 5 — No new runtime npm dependencies (AC5)

**Command:**
```
cd /tmp/test-repo-4
cat package.json | python3 -c "import json,sys; pkg=json.load(sys.stdin); deps=pkg.get('dependencies',{}); print('Runtime deps:', list(deps.keys()))"
```

**Expected outcome:**
Output is either `Runtime deps: []` (empty) or lists only dependencies that existed before i1.2 was implemented (i.e. no new runtime deps were added by `scripts/platform-init.js` or `scripts/platform-fetch.js`).

**Pass:** No new runtime dependencies.
**Fail:** A new entry appears in `dependencies` (not `devDependencies`).
