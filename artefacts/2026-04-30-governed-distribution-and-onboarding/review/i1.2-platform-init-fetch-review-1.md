## FINDINGS

### 1-L1 — LOW — AC2 "available as slash commands" describes VS Code IDE behaviour, not a directly testable script output

AC2 states: "then all pipeline skills (including /orient, /discovery, /definition, etc.) are available as slash commands and the consumer can invoke any skill without additional setup." Slash command availability is a VS Code Copilot extension behaviour — it depends on the extension reading the installed skill files, not on `platform:init` itself. The directly testable outcome of `platform:init` is that skill files are present at the correct paths. The AC as written is not independently verifiable by a Node.js unit test without a live VS Code instance.

**Recommended action:** Rephrase AC2 to describe the verifiable file-system outcome — e.g. "then all SKILL.md files are present under `.github/skills/[skill-name]/SKILL.md` in the target repo" — and add a note that VS Code slash command availability follows from correct file placement. This keeps the test author working from observable, automatable criteria.

---

### 1-L2 — LOW — AC4 test path requires live network access with no offline fallback defined

AC4 states: "when they run `npm run platform:fetch`, then the command fetches the current HEAD of the platform's `master` branch skill files." No test path is defined for an offline or CI environment where the upstream repo is unavailable. The NFR section notes "clear error output" for network unavailability, but AC4 has no corresponding failure-mode AC.

**Recommended action:** Either add a failure AC for `platform:fetch` when the upstream is unreachable (exits non-zero with a human-readable message naming the failure), or note in the NFR section that the primary AC4 test must be run in a network-enabled environment, with the network-failure case covered as an additional test scenario.
