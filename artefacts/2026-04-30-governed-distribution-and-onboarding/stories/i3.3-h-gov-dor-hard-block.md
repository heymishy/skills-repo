## Story: Add H-GOV hard block to /definition-of-ready SKILL.md

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/initiative-3-governance-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-3-governance-benefit-metric.md

## User Story

As a **platform maintainer**,
I want the `/definition-of-ready` skill to include a hard block `H-GOV` that fails any story whose discovery or benefit-metric artefact has an empty `Approved By` field,
So that no story can reach implementation without attributed governance — enforcing the attribution model at the last and most consequential gate, moving the MM1 enforcement correctness metric to 100%.

## Benefit Linkage

**Metric moved:** MM1 — H-GOV enforcement correctness (target: 100% correct failures on attribution-gap stories, zero false positives)
**How:** Adding H-GOV as a named hard block in the `/definition-of-ready` SKILL.md makes attribution a binary gate: stories with empty `Approved By` fields cannot proceed to implementation. The observable MVP test from the discovery artefact is satisfied: an engineer-only discovery artefact fails H-GOV with a message that names the attribution gap.

## Architecture Constraints

- ADR-011: this story modifies `.github/skills/definition-of-ready/SKILL.md` — a governed skill file. The change must go via PR with platform team review.
- Platform change policy: SKILL.md changes require PR — never direct commit to master.
- The H-GOV block must be added to the existing hard blocks list (H1–H9, H-E2E, H-NFR series) as a named, numbered block — not as a note or a warning.
- Skill files are Markdown only; follow established section headings and the hard block format already used in the file.
- No trailing whitespace; Unix line endings.

## Dependencies

- **Upstream:** i3.1 must be merged before i3.3 is dispatched. H-GOV checks the `Approved By` field that i3.1 adds to the discovery template. Without i3.1 merged, H-GOV would incorrectly block all stories (field absent = pre-i3.1 legacy). The dependency is an ordering constraint, not a technical one — the SKILL.md change is independently writeable, but deployment (merge) must follow i3.1.
- **Downstream:** i1.1 (the `/orient` concierge skill story) and all subsequent I1/I2 stories must pass H-GOV before reaching DoR. This story is the gate that enforces the prerequisite for I1 and I2 implementation.

## Acceptance Criteria

**AC1:** Given the updated `/definition-of-ready` SKILL.md, when the skill runs and the discovery artefact for the feature has an empty or placeholder `Approved By` field, then H-GOV fires as a hard block with a message that: (a) names the field that is empty (`Approved By` in `discovery.md`), (b) states the exact format required (`Name — Role — Date`), and (c) states that the story cannot proceed until the field is populated and the discovery status is updated.

**AC2:** Given the updated `/definition-of-ready` SKILL.md, when the skill runs and the discovery artefact has a substantive `Approved By` value but no non-engineering approver is identifiable from it, then H-GOV passes (the block checks for non-empty, not for role classification) — and the metric signal for M1 (non-engineering attribution rate) is recorded as "present but role unverified".

**AC3:** Given the updated `/definition-of-ready` SKILL.md, when the skill runs and the discovery artefact has a substantive `Approved By` value that contains a name and role that is clearly non-engineering (e.g. "Jane Smith — Product Owner — 2026-04-30"), then H-GOV passes with a positive signal noting non-engineering approver detected.

**AC4:** Given the updated `/definition-of-ready` SKILL.md, when the skill runs and a benefit-metric artefact exists that carries the "Attribution incomplete" note from i3.2 AC5, then H-GOV fires — the benefit-metric note is treated as a trigger-to-check signal, causing H-GOV to perform a live read of the discovery `Approved By` field; if the discovery field is now substantively populated, H-GOV passes on this run (the benefit-metric note alone does not permanently block the story); if the discovery field is still empty or a placeholder, H-GOV fires with the standard AC1 failure message.

**AC5:** Given a story whose discovery artefact predates i3.1 (no attribution section present at all), when H-GOV runs, then the block fires with a message that distinguishes "attribution section absent" from "field present but empty" — giving the operator a specific, actionable remediation path (add the attribution section, then populate `Approved By`).

**AC6:** Given the H-GOV block is added to the SKILL.md, when the full DoR hard blocks list is read, then H-GOV appears as a named block in sequence after the existing H-series blocks — with its block ID, check description, pass condition, and fail message explicitly stated, matching the format of all other hard blocks in the file.

## Out of Scope

- Automated role classification of the `Approved By` value (engineer vs non-engineer by name lookup or org directory lookup) — H-GOV performs a text-presence check only
- H-GOV enforcement on artefact types other than discovery and benefit-metric (story artefacts, test plans, DoR artefacts themselves are out of scope)
- Retroactive H-GOV enforcement on stories that reached DoR before this story merges — existing signed-off stories are not reprocessed
- Changes to `pipeline-state.schema.json` — H-GOV is a SKILL.md instruction change; no schema field is added by this story

## NFRs

- **Actionability:** The H-GOV failure message must be self-contained — an operator reading the message cold must know exactly which artefact to edit, which field to populate, what format to use, and what to do next
- **Zero false positives:** A correctly attributed discovery artefact must never trigger H-GOV — the pass condition must be unambiguous

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the block criteria are fully specified in the discovery artefact. The format of H-series hard blocks in the DoR SKILL.md is an established pattern.
