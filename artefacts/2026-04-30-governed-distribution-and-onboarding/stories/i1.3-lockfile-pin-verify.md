## Story: Implement `platform-lock.json` schema with `platform:pin` and `platform:verify` CLI commands

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i1-lockfile-cli-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-1-onboarding-benefit-metric.md

## User Story

As a **tech lead on a regulated or enterprise team**,
I want to pin the exact skill file version my team tested against and verify that our installed skills have not drifted,
So that we can provide reproducibility guarantees to our compliance team — the delivery process we validated is exactly the process running in production, and any silent drift is detected immediately with a specific, actionable error.

## Benefit Linkage

**Metric moved:** I1-MM1 (platform:verify SHA mismatch produces specific actionable error), I1-MM2 (CLI commands implemented with zero new npm dependencies)
**How:** `platform:pin` writes a `platform-lock.json` with SHA-256 hashes for each installed skill file, and `platform:verify` checks those hashes at any point — enabling regulated teams to commit the lockfile to version control and verify skill integrity in CI, just as they would for npm dependencies.

## Architecture Constraints

- **No new runtime npm dependencies.** `platform:pin` and `platform:verify` use `crypto.createHash('sha256')` from Node.js built-ins only. This is a hard requirement from I1-MM2.
- **CommonJS modules only.** Follows existing `scripts/` pattern — `require()` syntax, `.js` extension.
- **`platform-lock.json` is a committed file.** It must be designed to be diffable and human-readable in a git diff — one entry per line per skill file, JSON format, sorted by file path.
- **`platform-lock.json` schema must be stable for V1.** Breaking changes to the schema format require a new story. The V1 schema is: `{ "version": "1", "pinnedAt": "<ISO-8601>", "platform": "<semver-or-sha>", "skills": { "<relative-path>": "<sha256-hex>" } }`.
- **ADR-011:** `platform:pin` and `platform:verify` scripts + `platform-lock.json` schema added under `scripts/` and documented via this story artefact.

## Dependencies

- i1.2 (`platform:init` + `platform:fetch`) must be merged first — `platform:pin` hashes the files that `platform:init` installs.

## Acceptance Criteria

**AC1**
Given a repo where `platform:init` or `platform:fetch` has been run and skill files are installed,
when a consumer runs `npm run platform:pin`,
then a `platform-lock.json` file is created (or updated) in the repo root containing: the schema version ("1"), the ISO-8601 timestamp of the pin operation, the platform identifier (git SHA or version string from the fetched source), and the SHA-256 hash of each installed skill file keyed by its relative file path.

**AC2**
Given a `platform-lock.json` exists and all installed skill files match their recorded hashes,
when `npm run platform:verify` runs,
then the command exits with code 0 and outputs a one-line confirmation: "All N skill files match pinned version [platform-identifier]. ✓"

**AC3**
Given a `platform-lock.json` exists and one or more installed skill files have been modified since the last `platform:pin`,
when `npm run platform:verify` runs,
then the command exits with a non-zero exit code and outputs an error that: (a) names each file that has drifted (relative path), (b) shows the expected SHA-256 hash vs the actual current hash for each drifted file, and (c) tells the operator the exact command to re-pin (`npm run platform:pin`) or re-fetch (`npm run platform:fetch`).

**AC4**
Given `platform-lock.json` has been committed to the repo and a second developer clones the repo,
when the second developer runs `npm run platform:verify`,
then the verification reads from the committed `platform-lock.json` as the source of truth and produces the same pass/fail result as the original author — confirming the lockfile is portable across environments.

**AC5**
Given a repo with no `platform-lock.json`,
when `npm run platform:verify` runs,
then the command exits with a non-zero code and outputs a specific error message instructing the operator to run `npm run platform:pin` first, rather than silently passing or producing a generic failure.

## Out of Scope

- Automatic re-pinning on fetch — `platform:pin` is always a deliberate operator action, never automatic.
- Lock file migration tooling for schema V1 → V2 — post-I1 scope.
- Verification in CI pipelines (GitHub Actions integration) — the commands work in any Node.js environment; CI integration is a separate configuration story.
- Cryptographic signing of the lockfile — SHA-256 integrity check is sufficient for MVP; signing is a post-I1 security enhancement.

## NFR

- **Actionable error output (AC3):** The three-part error message (file name, expected vs actual SHA, fix command) must be self-contained — the operator must be able to act on it without opening another document or searching the web.
- **Zero false positives (AC2):** A clean repo where no skill files have been modified must always exit with code 0; no flakiness from file encoding, newline normalisation, or OS-specific file metadata.
- **Deterministic hashing:** SHA-256 hashes must be computed on raw file content only (no metadata, no timestamps) so that the same file on two different machines produces the same hash.
- **Zero new npm runtime dependencies** — verified by `package.json` `dependencies` diff.

## Complexity

**2** — crypto.createHash is straightforward; the three-part error message (AC3) requires careful formatting; AC4 (portable across environments) needs attention to path normalisation (forward slashes in the JSON regardless of OS).

## Scope Stability

Stable for MVP. Post-I1: `platform:verify --ci` flag (structured JSON output for CI), lockfile schema V2 (add `checksum-algorithm` field), and auto-verify on `platform:fetch`.
