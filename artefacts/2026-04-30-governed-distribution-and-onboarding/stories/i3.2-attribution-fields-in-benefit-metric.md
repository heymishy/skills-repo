## Story: Add attribution acknowledgement step to /benefit-metric SKILL.md

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/initiative-3-governance-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-3-governance-benefit-metric.md

## User Story

As a **platform maintainer**,
I want the `/benefit-metric` skill to verify that the upstream discovery artefact has a non-empty `Approved By` field before proceeding to metric definition,
So that the attribution gate is enforced at the earliest downstream skill — not deferred to DoR — moving the M1 non-engineering attribution rate metric toward measurability.

## Benefit Linkage

**Metric moved:** M1 — Non-engineering attribution rate (target ≥50% of new discovery artefacts carry a non-engineering contributor within 90 days)
**How:** Adding an explicit attribution check at the `/benefit-metric` entry point means the skill surfaces a gap immediately when an operator starts defining metrics for a discovery that lacks attribution — creating a natural correction point before metric work begins, rather than failing silently at DoR four steps later.

## Architecture Constraints

- ADR-011: this story modifies `.github/skills/benefit-metric/SKILL.md` — a governed skill file. The change must go via PR with platform team review.
- Platform change policy: SKILL.md changes require PR — never direct commit to master.
- Skill files are Markdown only; follow established section headings.
- No trailing whitespace; Unix line endings.

## Dependencies

- **Upstream:** i3.1 must be merged first. The attribution acknowledgement step checks for the `Approved By` field that i3.1 adds to the discovery template. If i3.1 is not merged, the check is described in the SKILL.md but has no data to find in existing artefacts.
- **Downstream:** i3.3 (H-GOV) — the full enforcement chain is i3.1 (field exists) → i3.2 (acknowledged at benefit-metric) → i3.3 (hard block at DoR). i3.3 can proceed without i3.2, but the three together are the complete governance model.

## Acceptance Criteria

**AC1:** Given the updated `/benefit-metric` SKILL.md, when the skill begins and a discovery artefact exists for the feature, then the skill reads the `Approved By` field from the discovery artefact as its first step — before asking any metric-definition questions.

**AC2:** Given a discovery artefact where `Approved By` is empty or contains only the placeholder value, when the `/benefit-metric` skill runs, then the skill surfaces a warning message identifying the gap and asks the operator whether to: (a) pause and get attribution before proceeding, or (b) proceed with a documented acknowledgement that attribution is incomplete.

**AC3:** Given a discovery artefact where `Approved By` contains a substantive value (any non-empty, non-placeholder text), when the `/benefit-metric` skill runs, then the skill proceeds without interruption — the attribution check is a gate, not a speedbump.

**AC4:** Given a discovery artefact where `Approved By` is absent entirely (pre-i3.1 legacy artefact with no attribution section), when the `/benefit-metric` skill runs, then the skill treats this the same as an empty field — surfaces the warning from AC2 — rather than throwing an error or silently proceeding.

**AC5:** Given the operator selects option (b) from AC2 (proceed with acknowledged incomplete attribution), when the skill continues, then the benefit-metric artefact it produces includes a visible note stating: "Attribution incomplete — Approved By was empty at benefit-metric time. Revisit before DoR."

## Out of Scope

- Blocking the `/benefit-metric` skill entirely when attribution is absent — the skill is a gate, not a hard block; that is i3.3's role at DoR
- Automated role classification of the `Approved By` name (whether it is a non-engineer) — this is human judgment
- Modifying the benefit-metric artefact template schema in `pipeline-state.json`
- Attribution checks on benefit-metric artefacts themselves (benefit-metric attribution is MM3 dogfood evidence, not a mechanical check in this story)

## NFRs

- **Non-interruption for correct flow:** When attribution is complete, the check adds zero perceptible overhead — no question to answer, no extra step to confirm
- **Actionability:** The warning message for an empty `Approved By` must name the specific field missing and tell the operator exactly how to fix it

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — the check is well-defined: read one field, branch on empty vs non-empty.
