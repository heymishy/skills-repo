## Story: `/orient` detects existing story artefacts without a discovery and routes to `/tdd` as brownfield Entry A

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i2-brownfield-routing-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-2-brownfield-benefit-metric.md

## User Story

As a **platform consumer (brownfield engineer with an existing set of stories or tasks)**,
I want `/orient` to detect that I have existing story artefacts and route me to the right entry path,
So that I can adopt the platform's delivery model without discarding my existing stories — attaching my in-progress work to the pipeline at the `/tdd` step rather than starting from `/discovery` with nothing.

## Benefit Linkage

**Metric moved:** I2-M1 (Brownfield team reaches supported entry path in one /orient interaction), I2-MM1 (Entry A/B/C routing correctness)
**How:** Entry A routing in `/orient` converts the "I have stories but no discovery" state from a dead end into a supported path, enabling teams who have been working informally (user stories in a backlog, tasks in Jira) to adopt the platform's implementation loop immediately via `/tdd` — which is the inner loop entry point for teams who already know what to build.

## Architecture Constraints

- `/orient` SKILL.md at `.github/skills/orient/SKILL.md` — updated to add Entry A detection logic (story artefact presence without discovery.md).
- Detection uses file-system heuristics only: presence of `.md` files with story-like structure under `artefacts/[feature]/stories/` or in the root. No semantic parsing of story content.
- ADR-011: This story extends the `/orient` SKILL.md (created by i1.1). i1.1 must be merged before this story can be implemented.

## Dependencies

- i1.1 (`/orient` skill SKILL.md) must be merged first — this story extends the routing logic added by i1.1.

## Acceptance Criteria

**AC1**
Given a repo that contains story artefact files (`.md` files under `artefacts/[feature]/stories/` or files matching story naming conventions) but no `discovery.md` or approved discovery artefact for that feature,
when `/orient` runs,
then the output: (a) identifies the state as "Entry A — story-first brownfield" by name, (b) explains that existing stories can be adopted by creating a discovery artefact to capture their intent, (c) names `/tdd` as the inner loop entry point for stories that are already well-understood, and (d) routes to `/discovery` as the outer loop anchor with a note to reference the existing stories as input scope.

**AC2**
Given `/orient` has detected Entry A and the operator follows the routing to run `/discovery`,
when the operator produces a discovery artefact and progresses to the inner loop,
then the `/orient` routing output has explicitly surfaced `/tdd` as the implementation entry — the operator knows they can attach their existing story work to the `/tdd` step without re-deriving it from scratch.

**AC3**
Given `/orient` classifies a repo as Entry A,
then the routing output names the specific detection signal: e.g., "Found: story artefacts under artefacts/[feature]/stories/; no discovery.md found. This looks like Entry A (story-first brownfield)." — so the operator can confirm or correct the classification before acting on it.

## Out of Scope

- Automatically importing or converting existing story files into the platform's story template format — operator-led action only; `/orient` routes, not migrates.
- Handling repos with both stories AND a discovery in progress — that is a standard greenfield state, not a brownfield Entry A state (AC1 requires no discovery.md present).
- Validating the completeness or quality of the detected story artefacts — detection is presence-based only.

## NFR

- **Signal transparency:** The routing output must name the detection signal (AC3) so the operator can verify the classification is correct. A misclassification that the operator cannot see is worse than no classification.
- **No false Entry A classifications:** A repo with a discovery.md present (even unapproved) must not be classified as Entry A — the discovery-in-progress standard routing from i1.1 (AC2) takes precedence.

## Complexity

**1** — file presence detection is straightforward; the main risk is defining the story file heuristic precisely enough to distinguish story artefacts from other `.md` files. Acceptable heuristic: files under `artefacts/[feature]/stories/` with names matching `[story-id]-*.md`.

## Scope Stability

Stable for MVP. Post-I2: Entry A sub-routes for different story maturity levels (stories with acceptance criteria vs stories with just a title), and automated story import tooling.
