## Story: `/orient` detects an existing codebase without pipeline artefacts and routes to `/reverse-engineer` as brownfield Entry B

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i2-brownfield-routing-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-2-brownfield-benefit-metric.md

## User Story

As a **platform consumer (brownfield engineer with an existing codebase and no prior delivery artefacts)**,
I want `/orient` to detect that I have a codebase with no pipeline artefacts and route me to the right starting point,
So that I can adopt the platform by first understanding and documenting what I have built — rather than being told to start fresh with `/discovery` and implicitly discard the knowledge embedded in my existing code.

## Benefit Linkage

**Metric moved:** I2-M1 (Brownfield team reaches supported entry path in one /orient interaction), I2-MM1 (Entry A/B/C routing correctness — Entry B is the hardest detection case; if it works, the assumption about pattern exhaustiveness gains significant evidence)
**How:** Entry B routing converts the "I have code but no artefacts" state from "not supported — start over" into a supported path via `/reverse-engineer → /discovery`, allowing teams to extract their existing system's business rules and design before writing a discovery artefact that reflects reality rather than aspiration.

## Architecture Constraints

- `/orient` SKILL.md at `.github/skills/orient/SKILL.md` — updated to add Entry B detection logic.
- Detection heuristic: presence of `src/`, `app/`, `lib/`, or equivalent code directories containing non-trivial application files (`.js`, `.ts`, `.py`, `.java`, `.cs`, etc.) with no `artefacts/` folder (or an empty `artefacts/` folder) present.
- Heuristic must distinguish Entry B (codebase, no artefacts) from the greenfield state in i1.1 AC1 (no codebase, no artefacts). Both lack artefacts, but Entry B has a populated codebase directory.
- ADR-011: This story extends `/orient` SKILL.md. i1.1 and i2.1 must be merged before implementation.

## Dependencies

- i1.1 (`/orient` skill SKILL.md, routing framework) must be merged.
- i2.1 (Entry A routing) is recommended to be merged first to ensure Entry A/B precedence ordering is correct in the skill.

## Acceptance Criteria

**AC1**
Given a repo that has a non-trivial application codebase (source files present in `src/`, `app/`, or equivalent directories with at least one file containing application logic) and no `artefacts/` folder or only an empty `artefacts/` folder,
when `/orient` runs,
then the output: (a) identifies the state as "Entry B — code-first brownfield" by name, (b) explains that `/reverse-engineer` can extract the existing system's business rules and design to seed a discovery artefact, and (c) lays out the full suggested path: `/reverse-engineer → /discovery → /benefit-metric → /definition`, so the operator understands the journey before committing to it.

**AC2**
Given `/orient` has detected Entry B,
then the routing output names the specific detection signal: e.g., "Found: src/ directory with application files; no artefacts/ folder found. This looks like Entry B (code-first brownfield)." — so the operator can confirm or correct the classification.

**AC3**
Given `/orient` is presented with a repo that has both application code AND story artefacts present (a partial brownfield state),
when `/orient` runs,
then the skill routes to Entry A (stories detected, no discovery) rather than Entry B — Entry A takes precedence over Entry B when story artefacts are present. Entry B applies only when no pipeline artefacts of any kind are found.

## Out of Scope

- Running `/reverse-engineer` automatically — `/orient` routes only; it does not execute other skills.
- Determining the programming language or stack of the detected codebase — detection is presence-based (file extension heuristics), not semantic.
- Handling repos with artefacts present but no codebase — that would be a degenerate state (artefacts without an implementation); not a brownfield entry pattern.

## NFR

- **Entry B detection must not misclassify Entry C:** A repo with only configuration files (`.yml`, `.json`, `Dockerfile`, `Makefile`) but no application logic files must not be classified as Entry B — that is Entry C (no-history). The heuristic must look for application logic file extensions, not just any files in a code directory.
- **Signal transparency:** The routing output must name the detection signal (AC2) so the operator can verify the classification and correct it if the heuristic misfired.

## Complexity

**2** — Entry B is the hardest detection case (as noted in the benefit-metric MM1 minimum signal). The heuristic must reliably distinguish "has application logic" from "has only configuration files" from "has nothing". Using a list of application-logic file extensions (`.js`, `.ts`, `.py`, `.java`, `.rb`, `.go`, `.cs`, `.cpp`, `.rs`, `.swift`, `.kt`) as the detection criterion is the recommended approach — simple and auditable.

## Scope Stability

Stable for MVP. Post-I2: Entry B sub-routes for specific stack detections (`/reverse-engineer` has stack-specific reading plans for Spring Boot, Struts 2, COBOL, etc.); if the detected stack is recognised by `/reverse-engineer`, suggest the relevant reading plan.
