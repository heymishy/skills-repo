## Story: `/orient` detects a no-history or ops-only repo and routes to retrospective discovery as brownfield Entry C

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i2-brownfield-routing-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-2-brownfield-benefit-metric.md

## User Story

As a **platform consumer (brownfield engineer with delivered software but no delivery artefacts and no meaningful application codebase to reverse-engineer)**,
I want `/orient` to detect my situation and route me to a path that documents what was built rather than asking me to start from scratch,
So that I can join the pipeline at a realistic entry point — acknowledging what exists — and begin producing governed artefacts without the fiction of a greenfield discovery.

## Benefit Linkage

**Metric moved:** I2-M1 (Brownfield team reaches supported entry path in one /orient interaction), I2-MM1 (Entry A/B/C routing correctness — Entry C closes the coverage gap for teams with no artefacts and no meaningful codebase)
**How:** Without Entry C, any team that has deployed software stored in configuration files, ops repositories, or packaged binaries (where there is no application source to reverse-engineer) receives a generic "start at /discovery" instruction that ignores their delivery history. Entry C provides an explicit retrospective path, making their existing investment part of the starting context rather than noise to ignore.

## Architecture Constraints

- `/orient` SKILL.md at `.github/skills/orient/SKILL.md` — updated to add Entry C detection logic as the fallback after Entry A and Entry B.
- Detection: Entry C fires when no story artefacts (Entry A signal) AND no application logic files (Entry B signal) are found. It is the default brownfield state when the repo is clearly not greenfield (git history present, or infrastructure/ops files present) but neither A nor B applies.
- Entry C is the fallback in the A → B → C detection priority order. If neither A nor B signal is detected in a brownfield-adjacent repo, Entry C is assumed.
- ADR-011: This story extends `/orient` SKILL.md. i1.1, i2.1, i2.2 should be merged before implementation to ensure the full A → B → C priority chain is in place.
- The retrospective story template at `.github/templates/retrospective-story.md` must exist before this story can be tested. (It is referenced in the implementation.)

## Dependencies

- i1.1 (`/orient` skill SKILL.md) must be merged.
- i2.1 (Entry A routing) and i2.2 (Entry B routing) should be merged before i2.3 so the A → B → C precedence chain is complete and testable.

## Acceptance Criteria

**AC1**
Given a repo that has a git history or infrastructure/ops files (indicating it is not an empty new repo) but has neither story artefacts (Entry A signal) nor application logic source files (Entry B signal),
when `/orient` runs,
then the output: (a) identifies the state as "Entry C — no-history brownfield" by name, (b) explains that `/discovery` can be run as a retrospective exercise to document what was built and why, (c) surfaces `.github/templates/retrospective-story.md` as available for any implementation work that predates the pipeline, and (d) routes to `/discovery` with an explicit note that this is a retrospective discovery, not a greenfield one.

**AC2**
Given `/orient` has detected Entry C,
then the routing output distinguishes Entry C from Entry B in the signal description: e.g., "No application logic files found in the expected source directories (src/, app/, lib/); Entry B (code-first) is not applicable. Treating this as Entry C (no-history brownfield)." — so the operator understands why Entry C was selected over Entry B.

**AC3**
Given a completely empty new repo (no git history, no files),
when `/orient` runs,
then the skill routes to the standard greenfield state (AC1 from i1.1: "no artefacts — run /discovery") rather than Entry C — Entry C requires some evidence that the repo is brownfield (git history, ops files, or infrastructure files). A truly empty repo is greenfield.

## Out of Scope

- Automatically creating the retrospective discovery artefact — `/orient` surfaces the path and template; the operator writes the artefact.
- Distinguishing between "delivered but archived" repos (no active development) and "delivered and active" repos — Entry C applies to both; the operator decides whether to proceed with retrospective discovery.
- Handling repos with a CI/CD history in `.github/workflows/` but no other signals — `.github/workflows/` presence is treated as an ops file and counts as a brownfield signal for Entry C detection purposes.

## NFR

- **Clear Entry C vs greenfield distinction:** An operator running `/orient` on a new empty repo must never see Entry C — they must see the greenfield state (i1.1 AC1). Entry C requires a brownfield signal.
- **Honest routing:** The Entry C output must not imply that the retrospective path is the same as a greenfield discovery. The word "retrospective" must appear in the routing output so the operator knows they are documenting existing work, not discovering new work.

## Complexity

**1** — Entry C is the fallback state after A and B fail to match. The detection logic is simple: if neither Entry A nor Entry B signals are present, and the repo is not empty (has some git history or files), classify as Entry C. The primary challenge is the AC3 boundary: distinguishing "truly empty greenfield" from "no-history brownfield". The greenfield condition is: no git history AND no non-hidden files in the repo root.

## Scope Stability

Stable for MVP. Post-I2: Entry C sub-routes for specific no-history contexts (database-only repo, infra-as-code-only repo, legacy packaged binary), and tooling to auto-populate a retrospective discovery template from existing documentation files.
