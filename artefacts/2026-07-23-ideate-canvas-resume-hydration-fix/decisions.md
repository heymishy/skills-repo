# Decisions: ideate-canvas-resume-hydration-fix

## Root cause investigation (2026-07-23)

**Context:** `tests/e2e/a4-ideate-session-resume.spec.js`'s AC2 & AC3 test failed against real `wuce-staging` in CI run 29996127983 (job 89170114731, PR #568): `locator('#canvas-panel .canvas-block').first()` — element(s) not found — after a browser was closed mid-SSE-stream and the same session reopened in a fresh context. Turn 1 had already rendered canvas blocks successfully before the close (confirmed by the test reaching this assertion rather than skipping at its own `turnBlockedReason()` `test.skip()` gate), so the resumed session's canvas going to zero blocks is a genuine regression from what existed before closing, not a pre-existing "turn never rendered" known blocker.

**Decision: the defect is a missing initial-hydration step, not a data-layer restore bug.**

**Rationale:**
1. `tests/check-a4-session-store-state.js` (5/5 passing, both before and after this fix) independently proves `mergeRedisSessionData()` already correctly restores `session.canvasBlocks` from Redis on session reopen. Per this repo's CLAUDE.md guidance for this investigation, this was treated as settled and `mergeRedisSessionData()` was not modified.
2. Direct code inspection of `src/web-ui/routes/skills.js` found that the `.canvas-block` DOM elements the test's selector targets are created via two independent code paths, and neither reads restored session state on a cold GET:
   - **Live-SSE path (works today, unaffected by this fix):** during an in-progress turn, `handlePostTurnStreamHtml` detects `---CANVAS-JSON:...---` markers in the streamed model output (`parseCanvasBlock`), pushes each parsed block to `session.canvasBlocks`, and emits a `canvasBlock` SSE event; the inline client script's `appendCanvasBlock`/`renderCanvasBlock` functions consume that event and append a `.canvas-block` div to `#canvas-panel`. This only ever fires reactively, during a live stream.
   - **Page-load path (the gap):** `_renderChatPage` (called by `handleGetChatHtml` on every GET to the chat page, including a session-resume reload) builds `draftSections` only from `session.artefactContent` when `session.done` is true (a completed final artefact, a different HTML shape — `.sw-draft-section`, not `.canvas-block`). Nothing in `_renderChatPage` or the inline client script ever read `session.canvasBlocks` to seed either the initial server-rendered HTML or a client-side hydration payload. A page reload is not a live SSE stream, so the canvas rendered empty regardless of what was correctly restored server-side.
3. An equivalent, already-shipped hydration pattern exists in the SAME file for a different panel: `artefactInitScript` sets `window.__SW_INITIAL_ARTEFACT__` server-side when `session.artefactContent` exists, and the inline script's `if(!IS_IDEATE && typeof __SW_INITIAL_ARTEFACT__ !== "undefined" && __SW_INITIAL_ARTEFACT__) { updateDraftPanel(__SW_INITIAL_ARTEFACT__); }` reads it on page load — but this call is explicitly gated `!IS_IDEATE`, deliberately excluding ideate's canvas. (It would not have been the right fix to simply remove that guard: `updateDraftPanel`'s ideate branch rebuilds a `.sw-draft-section` structure, not `.canvas-block` elements — the wrong DOM shape for what the E2E test, and the live-SSE path, actually render.)
4. `/definition`'s own canvas (`_renderDefinitionMapHtml`, `dic.1`/`dic.2`) was checked as a possible "already-working, reusable pattern" per this investigation's brief. Finding: `_renderDefinitionMapHtml` (the server-side pure-function mirror of the client's `renderDefinitionMap`) is exported only for unit testing (`tests/check-dic1-story-cards.js`) and is never actually invoked in `_renderChatPage`'s render path — it is not wired into production HTML output at all. Definition's own cold-reload hydration is, in practice, the SAME `!IS_IDEATE`-gated `updateDraftPanel(__SW_INITIAL_ARTEFACT__)` call described above, which only fires once `session.done && session.artefactContent` (a finished artefact) — meaning `/definition`'s story-map canvas has no true *incremental* mid-session cold-reload hydration either. This matches `b1-formed-idea-outer-loop-story-map.spec.js`'s own test design: its AC4 (story-map-specific session field restoring on resume) is verified only at the Integration level (`tests/check-b1-story-map-session-restore.js`), never by an E2E close/reopen assertion against the live canvas DOM the way `a4`'s test does. **Conclusion: there was no existing, E2E-proven "cold reload restores incremental in-progress canvas content" pattern anywhere in this codebase to port — `a4`'s test is the first to actually exercise this gap, and it found a genuine, previously-undetected defect specific to ideate's `.canvas-block` hydration (which had zero hydration path, weaker even than definition's incomplete one).**

**Decision: fix by adding a new, ideate-scoped init-script + client hydration call, mirroring `__SW_INITIAL_ARTEFACT__`'s existing shape, reusing the existing `appendCanvasBlock` renderer unmodified.**

**Rationale:** This is the smallest change that closes the actual gap, is consistent with this file's own established idiom (three parallel init-script variables now: `artefactInitScript`, `phaseModelInitScript`, `canvasBlocksInitScript`), and does not touch the live-SSE path, the data-layer restore, or introduce any new rendering logic (the same `appendCanvasBlock` function used during a live stream is reused for the one-time hydration pass on page load). An alternative considered — having the client make a separate `fetch()` call on page load to retrieve canvas state — was rejected: it would introduce a new endpoint and a network round-trip/flash-of-empty-canvas for no benefit over embedding the already-fetched session data directly in the initial HTML response, which is also the existing pattern this file already uses for the artefact and phase-model panels.

## Scope boundary decision: definition's own incomplete cold-reload hydration is NOT fixed by this story

**Date:** 2026-07-23
**Context:** Investigation surfaced that `/definition`'s story-map canvas also lacks true incremental cold-reload hydration (see above) — a potentially similar, separate defect.
**Decision:** Out of scope for `icrh-s1`. This story's story/test-plan/DoR are scoped exclusively to `/ideate`'s canvas, matching the CI failure that triggered this investigation (`a4-ideate-session-resume.spec.js`). `/definition`'s gap, if it is a genuine defect, has no failing E2E test today (per the `b1` test design noted above) and should be triaged as its own separate story if/when a test surfaces it, rather than bundled speculatively into this fix.
**Rationale:** Keeps this fix narrowly scoped and verifiable against the specific, live-verified failure it addresses, per this repo's short-track convention and D37/architecture-constraints discipline of not silently expanding scope.
