# Decisions: chat-message-text-truncation-fix

## Root cause investigation (2026-07-24)

**Context:** `tests/e2e/a4-ideate-session-resume.spec.js`'s AC2 & AC3 test ("closing the browser mid-SSE-stream and reopening the session restores canvas/turn history and continues with the restored context") failed against real `wuce-staging` after PR #568 (catc-s1, credits gate fix), PR #569 (icrh-s1, canvas hydration fix), and PR #570 (icv-s1, continue-chain fix) all landed this session. `expect(resumedThreadText).toContain(distinctiveDetail)` failed. Observed failure: the received `#chat-messages` text showed `"...call it Project: an internal tool..."` where the sent message was `"...call it Project-Codename-Falcon-detail-1784829153994-51525: an internal tool..."` — everything from the first hyphen after "Project" up to (not including) the colon had vanished.

**Investigation path:**

1. Located the two candidate rendering functions the task brief pointed at: client-side `lightMd` (`src/web-ui/routes/skills.js`, inline script) and server-side `lightMarkdown` (`src/web-ui/views/chat-view.js`, used by `_renderChatPage`'s `priorQA` build on session-resume GET). Isolated both directly (`node` repro script calling each with the exact failing input string) — **neither reproduces the bug**. Both pass the distinctive-detail string through unmodified; their bold/italic/code regexes require literal `*`/`` ` `` characters, none of which are present in the input.
2. Traced the write path instead: `htmlSubmitTurn()` (`src/web-ui/routes/skills.js` line 1983) calls `sanitiseAnswer(rawAnswer)` and pushes the *result* — not the raw input — onto `session.turns` (line 1986), before any rendering ever happens. This is the point of truth for what a resumed session's `chat-view.js` render sees.
3. Isolated `sanitiseAnswer()` (`src/answer-sanitiser.js`) directly against the exact failing input: reproduces the bug exactly, character-for-character matching the reported failure.

**Root cause: `sanitiseAnswer()`'s `CLI_FLAG` regex, `/--?[a-z][\w-]*/gi`, has no boundary check on what precedes the leading hyphen.** It is meant to strip real CLI flags (`--allow-all`, `-rf`) out of a user's typed answer as an injection defence (NFR3, T4.3 in `tests/skill-launcher.test.js`). Because the regex has no requirement that the hyphen sit at a token boundary, it also matches starting at *any* hyphen inside an ordinary hyphenated compound word/identifier — every character in `-Codename-Falcon-detail-1784829153994-51525` (letters, digits, hyphens) is valid inside the regex's own `[\w-]*` body, so it matches the entire span in one greedy sweep and deletes it, leaving `Project` immediately followed by the (unrelated) colon that came later in the sentence.

**This is a write-time bug, not a render-time bug (closest to the brief's case (c), but more precisely located in `sanitiseAnswer()` — a shared answer-sanitisation utility — rather than in `mergeRedisSessionData()` or any session-merge/restore path, which were not touched and are not implicated.** `sanitiseAnswer()` runs on the raw answer *before* it is ever pushed to `session.turns` (`src/web-ui/routes/skills.js` lines 381, 1983, 3927, 4477 are all call sites). The corruption is baked into the stored turn permanently — every later render (live-SSE echo during the same page load, and the server-side `chat-view.js` re-render on session-resume) faithfully displays the already-corrupted text. This explains why the test's pre-close assertion (`preCloseThreadText`, driven by the live client-side echo of the raw DOM input value, not a re-render of the stored/sanitised copy) passed, while the resumed-session assertion (driven by `chat-view.js`'s re-render of the stored, already-sanitised-and-corrupted `session.turns` content) failed.

**Blast radius: this is a real, shared-function defect affecting every USER-typed answer matching this shape, across every skill session (not model output).** The model's own streamed response text never passes through `sanitiseAnswer()` — only `rawAnswer`/`req.body.answer` does. So case (a) from the investigation brief ("shared function bug affecting the assistant/model's OWN messages") does not apply here; this is closer to case (c) but the defect is in answer sanitisation, not session storage/merge mechanics. It is nonetheless serious: any real user typing a hyphenated identifier, product codename, ticket reference (e.g. "JIRA-1234-fix-something: ..."), or similar phrase followed later by a colon would have that phrase silently and permanently mangled in their own stored conversation history, in production, today — not just in this test.

## Fix decision

**Decision: add a negative lookbehind, `(?<![\w-])`, immediately before the `CLI_FLAG` regex's `--?` so a flag can only be recognised at a genuine token boundary (start of string, or preceded by a non-word/non-hyphen character such as whitespace or punctuation).**

**Rationale:** This is the minimal change that closes the actual gap without weakening the injection defence the regex exists for. `--allow-all` at the start of a string, and `-rf` preceded by a space, both still match and are stripped exactly as before (verified against the existing `T4.3`/`T4.5`/`NFR3` tests in `tests/skill-launcher.test.js`, all of which still pass unmodified). A hyphen embedded mid-word in an ordinary compound identifier (preceded by a letter or digit) no longer matches, because a real CLI flag is never typed as part of a contiguous alphanumeric compound word — it is always its own token, bounded by whitespace or the start of the input. An alternative considered — removing `CLI_FLAG` stripping entirely and relying only on `META_CHARS` (shell metacharacter stripping) — was rejected: `T4.3` explicitly asserts `--allow-all` and `rm -rf` are stripped as literal flag-injection defence, independent of the `;`/`|`/etc. metacharacter set, and removing that coverage would be an unreviewed narrowing of an existing security control, out of scope for this fix.

## Scope boundary

**Decision: this fix touches only `src/answer-sanitiser.js`'s `CLI_FLAG` regex. No change to `lightMd` (`skills.js`), `lightMarkdown` (`chat-view.js`), or `mergeRedisSessionData()`.** All three were directly isolated during investigation and confirmed not to be implicated — changing any of them would not have fixed the actual defect and would have been unreviewed, unjustified scope creep per this repo's Architecture Constraints discipline.
