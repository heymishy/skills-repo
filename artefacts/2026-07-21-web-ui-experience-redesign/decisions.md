**2026-07-21 | SCOPE | discovery /clarify**
**Decision:** Module taxonomy is fully operator-curated per product, with no default/starter set. Every product begins with zero modules.
**Context:** The nine modules confirmed for `skills-framework` this session (Web UI/Product Management, Governance & Gate Enforcement, etc.) are specific to that product's own epic shape.
**Rationale:** A universal starter set risks being either too generic to be useful (e.g. "Core", "Infrastructure") or silently biased toward skills-framework's own structure. Operator-curated with zero defaults keeps the mental model simple: modules mean whatever the person using the product wants them to mean, with no platform assumption baked in.
**Made by:** Hamish King (Founder/Operator), via /clarify, 2026-07-21
**Revisit trigger:** If a second real product is added to the platform and the operator finds recreating module structure from scratch repeatedly painful, reconsider a "copy modules from another product" convenience feature (not a default taxonomy).

**2026-07-21 | ARCH | d1 implementation, mandatory Task 0 technical investigation (session-swap mechanism)**
**Decision:** The impersonation session swap (AC3) is implemented as: (1) preserve the real admin's full identity snapshot (`userId`, `login`, `tenantId`, `role`) in a NEW, dedicated `req.session.impersonation` sub-object — never overwritten into the top-level fields — so exit (D2, out of scope here) can restore it exactly and so a truthy `req.session.impersonation.active` doubles as the nested-impersonation guard (AC5); (2) overwrite the top-level `req.session.tenantId`/`login`/`role` to the target's values, matching AC3's own exact wording (no other fields); (3) do the audit-table `INSERT` first (awaited), and mutate `req.session` fields only in one single synchronous block with no `await` in between and only after that insert resolves — this is what makes AC3/AC4's atomicity and the NFR's "no inconsistent state under concurrent requests during the swap window" hold, since Node's single-threaded event loop cannot interleave another request's handler between two synchronous assignments in the same tick; if the INSERT throws, the catch/propagation happens before any session field is touched, giving AC4 for free from the ordering alone, not a second corrective step.
**Context:** Traced directly against the real code per the story's own mandatory first-task instruction: (a) `middleware/session.js`'s Redis adapter is write-behind/fire-and-forget (`persistSession` is called after mutation and never awaited by callers) — there is no Redis transaction to reason about; the actual atomicity concern is JS-level ordering within one request handler, not a Redis-level concern. (b) `req.session.accessToken` is deliberately NOT part of the swap — it remains the real admin's own GitHub OAuth token throughout, because we have no credential for the target user and cannot fabricate one; AC3's own wording lists only tenantId/login/role, not accessToken, consistent with this. (c) `req.session.userId` is likewise NOT swapped, for the same reason and because several existing handlers (`dashboard.js`'s `handleGetActions` personalised-action-queue fetch, in particular) combine `req.session.userId`/`login` with `req.session.accessToken` to make a live GitHub API call as the admin's real account — swapping userId/login without a real target access token would produce a call attributed to a userId that doesn't match the token making it, which is worse than leaving userId as the admin's own. This is a known, flagged limitation, not fixed here: any feature that derives its data from a live GitHub-API call (as opposed to tenant-scoped Postgres data) will still reflect the admin's own GitHub account while impersonating. Flagged explicitly for D2 (banner/permission-scoped visibility) and D4 (dedicated NFR-security review) — out of D1's own AC scope, not silently patched or hidden. (d) `requireAdmin`'s live per-request role re-check (sec-perf-s2, `setGetCurrentRole`) is wired in `server.js` (line ~380) as `getCurrentRole(tenantId) -> getRoleForTenant(tenantId)` — it is called with ONLY `req.session.tenantId`, never an identityKey/login. Once this story's swap sets `req.session.tenantId` to the target's tenantId, any subsequent `requireAdmin`-gated request during impersonation re-resolves role using the target's tenantId as the lookup key, which is the DESIRED behaviour for the "no privilege leakage" NFR (the admin's real 'admin' role is never consulted while impersonating). This also means `requireAdmin` will self-heal `req.session.role` to the target's role on any such request, which is consistent with, not a threat to, this design (role is meant to reflect the target). Confirmed this is a pre-existing, independently-flagged latent gap in `requireAdmin`'s OWN identityKey plumbing (it never forwards a per-person identity, only tenantId — see `modules/user-roles.js`'s `resolveRoleForPerson`, where `identityKey` defaults to `tenantId` when omitted) that would equally affect any `TENANT_ORG_ALLOWLIST`-shared-tenant admin today, impersonation or not — out of scope for D1 to fix, and orthogonal to impersonation specifically.
**Rationale:** This design satisfies AC3/AC4/AC5 exactly as worded, without expanding scope to fix pre-existing, unrelated gaps in `requireAdmin`'s identityKey plumbing or to fabricate a target access token that does not exist. The accessToken/userId non-swap is the single largest judgment call in this story and is deliberately narrow (matches AC3's literal field list) rather than an attempt to make impersonation a byte-for-byte simulation of a real login — that stronger guarantee is not achievable without the target's own credentials and is flagged, not silently assumed away.
**Made by:** Claude (agent), during d1 implementation Task 0, 2026-07-21 — findings also posted as a PR comment before any implementation code was written, per the DoR Coding Agent Instructions.
**Revisit trigger:** D4's dedicated NFR-security review must re-examine (a) the accessToken/userId non-swap's effect on any GitHub-API-backed feature reachable while impersonating, and (b) whether `requireAdmin`'s tenantId-only live-role recheck should be extended to forward the target's login as identityKey once a real, concrete instance of privilege leakage through that path is found — no such instance is known today, but the plumbing gap is real and pre-existing.
