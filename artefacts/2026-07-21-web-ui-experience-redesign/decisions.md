**2026-07-21 | SCOPE | discovery /clarify**
**Decision:** Module taxonomy is fully operator-curated per product, with no default/starter set. Every product begins with zero modules.
**Context:** The nine modules confirmed for `skills-framework` this session (Web UI/Product Management, Governance & Gate Enforcement, etc.) are specific to that product's own epic shape.
**Rationale:** A universal starter set risks being either too generic to be useful (e.g. "Core", "Infrastructure") or silently biased toward skills-framework's own structure. Operator-curated with zero defaults keeps the mental model simple: modules mean whatever the person using the product wants them to mean, with no platform assumption baked in.
**Made by:** Hamish King (Founder/Operator), via /clarify, 2026-07-21
**Revisit trigger:** If a second real product is added to the platform and the operator finds recreating module structure from scratch repeatedly painful, reconsider a "copy modules from another product" convenience feature (not a default taxonomy).

**2026-07-21 | ARCH | d1 implementation, mandatory Task 0 technical investigation (session-swap mechanism)**
**Decision:** The impersonation session swap (AC3) is implemented as: (1) preserve the real admin's full identity snapshot (`userId`, `login`, `tenantId`, `role`) in a NEW, dedicated `req.session.impersonation` sub-object — never overwritten into the top-level fields — so exit (D2, out of scope here) can restore it exactly and so a truthy `req.session.impersonation.active` doubles as the nested-impersonation guard (AC5); (2) overwrite the top-level `req.session.tenantId`/`login`/`role` to the target's values, matching AC3's own exact wording (no other fields); (3) do the audit-table `INSERT` first (awaited), and mutate `req.session` fields only in one single synchronous block with no `await` in between and only after that insert resolves — this is what makes AC3/AC4's atomicity and the NFR's "no inconsistent state under concurrent requests during the swap window" hold, since Node's single-threaded event loop cannot interleave another request's handler between two synchronous assignments in the same tick; if the INSERT throws, the catch/propagation happens before any session field is touched, giving AC4 for free from the ordering alone, not a second corrective step.
**Context:** Traced directly against the real code per the story's own mandatory first-task instruction: (a) `middleware/session.js`'s Redis adapter is write-behind/fire-and-forget (`persistSession` is called after mutation and never awaited by callers) — there is no Redis transaction to reason about; the actual atomicity concern is JS-level ordering within one request handler, not a Redis-level concern. (b) `req.session.accessToken` is deliberately NOT part of the swap — it remains the real admin's own GitHub OAuth token throughout, because we have no credential for the target user and cannot fabricate one; AC3's own wording lists only tenantId/login/role, not accessToken, consistent with this. (c) `req.session.userId` is likewise NOT swapped, for the same reason and because several existing handlers (`dashboard.js`'s `handleGetActions` personalised-action-queue fetch, in particular) combine `req.session.userId`/`login` with `req.session.accessToken` to make a live GitHub API call as the admin's real account — swapping userId/login without a real target access token would produce a call attributed to a userId that doesn't match the token making it, which is worse than leaving userId as the admin's own. This is a known, flagged limitation, not fixed here: any feature that derives its data from a live GitHub-API call (as opposed to tenant-scoped Postgres data) will still reflect the admin's own GitHub account while impersonating. Flagged explicitly for D2 (banner/permission-scoped visibility) and D4 (dedicated NFR-security review) — out of D1's own AC scope, not silently patched or hidden. (d) `requireAdmin`'s live per-request role re-check (sec-perf-s2, `setGetCurrentRole`) is wired in `server.js` (line ~380) as `getCurrentRole(tenantId) -> getRoleForTenant(tenantId)` — it is called with ONLY `req.session.tenantId`, never an identityKey/login. Once this story's swap sets `req.session.tenantId` to the target's tenantId, any subsequent `requireAdmin`-gated request during impersonation re-resolves role using the target's tenantId as the lookup key, which is the DESIRED behaviour for the "no privilege leakage" NFR (the admin's real 'admin' role is never consulted while impersonating). This also means `requireAdmin` will self-heal `req.session.role` to the target's role on any such request, which is consistent with, not a threat to, this design (role is meant to reflect the target). Confirmed this is a pre-existing, independently-flagged latent gap in `requireAdmin`'s OWN identityKey plumbing (it never forwards a per-person identity, only tenantId — see `modules/user-roles.js`'s `resolveRoleForPerson`, where `identityKey` defaults to `tenantId` when omitted) that would equally affect any `TENANT_ORG_ALLOWLIST`-shared-tenant admin today, impersonation or not — out of scope for D1 to fix, and orthogonal to impersonation specifically.
**Rationale:** This design satisfies AC3/AC4/AC5 exactly as worded, without expanding scope to fix pre-existing, unrelated gaps in `requireAdmin`'s identityKey plumbing or to fabricate a target access token that does not exist. The accessToken/userId non-swap is the single largest judgment call in this story and is deliberately narrow (matches AC3's literal field list) rather than an attempt to make impersonation a byte-for-byte simulation of a real login — that stronger guarantee is not achievable without the target's own credentials and is flagged, not silently assumed away.
**Made by:** Claude (agent), during d1 implementation Task 0, 2026-07-21 — findings also posted as a PR comment before any implementation code was written, per the DoR Coding Agent Instructions.
**Revisit trigger:** D4's dedicated NFR-security review must re-examine (a) the accessToken/userId non-swap's effect on any GitHub-API-backed feature reachable while impersonating, and (b) whether `requireAdmin`'s tenantId-only live-role recheck should be extended to forward the target's login as identityKey once a real, concrete instance of privilege leakage through that path is found — no such instance is known today, but the plumbing gap is real and pre-existing.

**Decision:** `POST /api/admin/impersonate/start` was fixed to trust only `targetId` from the request body. `targetLogin`/`targetTenantId`/`targetRole` are no longer read from the client at all — a new `getImpersonationCandidateById(pool, targetId)` re-derives the real login/tenantId/role server-side from the same `team_memberships`/`person_identities` join `listImpersonationCandidates` uses, and an unknown `targetId` is now rejected with 404 before any session mutation or audit write.
**Context:** Found during a direct pre-merge code read of PR #534 (not caught by `/review` or `/definition-of-ready`, since both focused on higher-level ACs, not this specific trust boundary). The original implementation built `target` directly from `req.body.targetLogin`/`targetTenantId`/`targetRole` — hidden form fields rendered by the GET search page — and passed them straight into both `writeImpersonationAudit` (the audit log write) and the live session mutation (`session.tenantId`/`session.role`), with no server-side check that these values actually matched the DB row for the submitted `targetId`.
**Rationale:** Since `/admin/impersonate` is already `requireAdmin`-gated, the practical risk was not a non-admin escalating to admin — it was that the audit trail itself (this epic's own benefit metric m4, "Impersonation audit completeness") could be made unreliable: an admin could submit a tampered request and have the audit log record an impersonation of a target/role combination that never corresponded to any real DB row, and `session.role`/`session.tenantId` could briefly diverge from DB truth (until `requireAdmin`'s live-role recheck self-healed it on the next request). Re-deriving the target's identity server-side from the one value that actually identifies a real person (`targetId`) closes this cleanly — the client's hidden fields are now purely a rendering convenience, never a trust boundary. Two new tests (T15b, T15c) prove tampered fields are discarded and an unknown `targetId` is rejected outright.
**Made by:** Claude (agent), flagged during independent PR verification before recommending merge; fix applied at the operator's explicit request ("Fix it please"), 2026-07-21.
**Revisit trigger:** None — this closes the gap completely for d1's own scope. D2 (session-swap exit/visibility) and D4 (security review/hardening pass) should re-confirm this pattern (re-derive identity server-side from an id, never trust client-submitted identity fields) is followed consistently in their own new endpoints.

**2026-07-21 | FIX-FORWARD | a1 live staging deploy — journeys.module_id migration failed on a fresh database**
**Decision:** The `journeys.module_id` migration (which has `REFERENCES product_modules(id)`) is now chained inside `product_modules`' own `CREATE TABLE`'s `.then()` callback, rather than fired as a second, independent `_creditsPool.query()` call immediately after it in `server.js`.
**Context:** Operator deployed `2026-07-21-web-ui-experience-redesign` (merged through d1) to `wuce-staging` and observed, in the real boot log: `[a1] journeys.module_id migration failed: relation "product_modules" does not exist`. `pg.Pool` checks out a separate client per `.query()` call; two unchained queries race for their own connection, so on a genuinely fresh database (no pre-existing `product_modules` table from a prior deploy) the `ALTER TABLE journeys ADD COLUMN module_id ... REFERENCES product_modules(id)` could reach Postgres before the `CREATE TABLE product_modules` DDL had actually committed. This also explains the operator's simultaneously-observed product-dashboard 500 (`handleGetProductView`, `products.js:724`) — A4's module-grouped rendering reads `journeys.module_id`, which never got created on this environment because the migration that adds it failed.
**Rationale:** Chaining the dependent migration inside the parent's `.then()` guarantees Postgres has committed the `CREATE TABLE` before the `ALTER TABLE ... REFERENCES` is ever sent — removing the race entirely rather than relying on incidental timing (e.g. this bug likely never surfaced in this session's own testing/dev environment because `product_modules` already existed there from repeated `npm test` runs against a real or persisted dev DB, only becoming visible on a genuinely first-time database). A new regression test (`check-a1-modules-taxonomy-crud.js`) asserts the two migrations are chained (no independent `.catch()` closes `product_modules`' own chain before the `module_id` migration runs) — confirmed to fail against the original buggy code and pass against the fix.
**Made by:** Claude (agent), fixed directly following the operator's live staging deploy report, 2026-07-21. Treated as a fix-forward to a1 (already-merged story), matching this session's own earlier precedent for hotfixes (PR #510, #511, #513) rather than a new story chain, since it is a bug-fix to already-shipped code with no new scope or behaviour.
**Revisit trigger:** If any other future migration introduces a `REFERENCES` dependency on a table created earlier in the same `server.js` startup block, check whether the two are properly chained rather than assuming pg.Pool serializes unchained queries in source order — it does not.

**2026-07-21 | SEC | d2 implementation — exit endpoint fails OPEN on an audit-log write failure (judgment call flagged for operator review)**
**Decision:** `exitImpersonationSession` attempts `endImpersonationAudit(auditId)` (sets `ended_at` on the same row D1's start wrote) but does NOT re-throw if that call fails — the error is logged via an injectable logger and the actual session revert (restoring the real admin's `tenantId`/`login`/`role`/`userId`, deleting `session.impersonation`) proceeds regardless.
**Context:** This is the mirror-image asymmetry of D1's own start flow, which deliberately fails CLOSED on an audit-write failure (a failed audit write blocks the swap entirely, so impersonation can never begin unaudited). Exit has the opposite risk profile: if exit were to fail closed on an audit-UPDATE error, the real admin would be trapped inside the target user's identity/permissions until the DB issue resolved — a worse security and availability outcome than an audit row missing its `ended_at` timestamp (the row still has its full `admin_id`/`target_id`/`reason`/`created_at` from start, so the record of WHO/WHAT/WHY is not lost, only the precise end time).
**Rationale:** This is a genuine judgment call, not dictated explicitly by any AC or the test plan — flagging it here per the epic's High human-oversight rating so the operator can review the asymmetry directly rather than it being an implicit, undocumented choice. No AC required exit to fail in either direction on an audit error; this decision fills that gap conservatively (favouring the admin never being trapped).
**Made by:** Claude (agent), during d2 implementation Task 2, 2026-07-21 — flagged explicitly for operator review given this epic's High oversight rating.
**Revisit trigger:** If a future audit or compliance requirement demands that impersonation sessions can never end without a confirmed, successful audit close-out, this fail-open behaviour would need to change to a retry/queue mechanism instead of silent logging.
