# AC Verification Script: SaaS-GUI, M365-admin, and manual surface adapters (p2.5b)

**Story reference:** artefacts/2026-04-11-skills-platform-phase2/stories/p2.5b-saas-gui-m365-manual-adapters.md
**Date:** 2026-04-11

---

## Overview

This story completes the full five-adapter set (IaC + SaaS-API from p2.5a; SaaS-GUI + M365-admin + manual from p2.5b). The key addition in this story is the `resultPattern: "checklist"` field for the manual adapter, and the requirement that the manual POLICY.md floor contains NO diff/code-review criteria.

---

## Scenarios

### Scenario 1 — SaaS-GUI adapter returns P1.2-conformant result

**AC:** AC1

**Steps:**
1. Construct a minimal context with `surface: "saas-gui"`.
2. Call `execute("saas-gui", context)`.
3. Inspect the result.

**Expected result:** Object has `status` (one of "pass"/"fail"/"error"), `surface: "saas-gui"` (exact value), `findings` (array), `trace` (non-null). No exception thrown.

**Pass/Fail:** Pass if all four fields present and `surface` matches exactly.

---

### Scenario 2 — M365-admin adapter returns P1.2-conformant result

**AC:** AC2

**Steps:**
1. Construct a minimal context with `surface: "m365-admin"`.
2. Call `execute("m365-admin", context)`.
3. Inspect the result.

**Expected result:** Same structure as Scenario 1 with `surface: "m365-admin"`. No Graph API call is made.

**Pass/Fail:** Pass if all fields present and surface matches.

---

### Scenario 3 — Manual adapter result includes resultPattern: "checklist" and checklist findings

**AC:** AC3

**Steps:**
1. Construct a context with `surface: "manual"` and at least one checklist item.
2. Call `execute("manual", context)`.
3. Inspect `result.resultPattern` and `result.findings[0]`.

**Expected result:** `result.resultPattern === "checklist"`. Each finding object has an `item` (string) and `status` ("complete" or "incomplete"). No finding contains diff-analysis-specific fields (e.g. `file`, `line`, `rule`).

**Pass/Fail:** Pass if resultPattern matches and findings conform to checklist structure.

---

### Scenario 4 — SaaS-GUI and M365-admin POLICY.md floors resolve from standards/index.yml

**AC:** AC4

**Steps:**
1. Call the POLICY.md routing function with `surface: "saas-gui"`.
2. Verify the returned path exists on disk.
3. Repeat for `surface: "m365-admin"`.

**Expected result:** Both paths resolve to non-null file paths. Both files exist and contain at least one governance criterion.

**Pass/Fail:** Pass if both files resolve and exist.

---

### Scenario 5 — Manual POLICY.md floor contains NO diff-analysis or code-review criteria

**AC:** AC5

**Steps:**
1. Read `standards/manual/POLICY.md`.
2. Scan the content for any of the following keywords: "diff", "code review", "pull request", "lint", "static analysis", "source code".
3. Check that at least one checklist-type criterion is present.

**Expected result:** No diff-analysis or code-review keywords found. At least one criterion exists in checklist format (e.g. "Change request approved").

**Pass/Fail:** Pass if no forbidden keywords found AND at least one checklist criterion present. Failure if any diff-analysis keyword appears.

---

### Scenario 6 — Unknown surface type returns structured error result (not exception)

**AC:** AC6

**Steps:**
1. Ensure all 5 adapters are registered in the adapter registry.
2. Call `execute("unknown-type", minimalContext)`.
3. Inspect the result and process state.

**Expected result:** Returns `{ status: "error", surface: "unknown-type", findings: [], trace: "<identifier>", error: "unknown surface type: unknown-type" }`. `result.findings` is an empty array (not null). No exception thrown. Process does not crash.

**Pass/Fail:** Pass if result matches structure, status = "error", findings = [], and no exception.

**Expected result:** SaaS-GUI, M365-admin, and manual adapter test cases all green. Suite exits 0.

**Pass/Fail:** Pass if all `[check-surface-adapter]` cases green and overall exit code = 0.
