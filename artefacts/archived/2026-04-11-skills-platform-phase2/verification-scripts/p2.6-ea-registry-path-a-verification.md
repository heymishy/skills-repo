# AC Verification Script: EA registry Path A surface type resolution (p2.6)

**Story reference:** artefacts/2026-04-11-skills-platform-phase2/stories/p2.6-ea-registry-path-a.md
**Date:** 2026-04-11

---

## Overview

This story adds Path A (EA registry HTTP GET lookup) to the surface adapter resolver. Path B (explicit `context.yml` `surface:` declaration) is unchanged. The test data key assumption is ASSUMPTION-02: stub contract fields are `surfaceType` (string), `teamId` (string), `adapterOverride` (string|null). Fixtures use these field names — confirm at DoR H1 before coding begins.

---

## Important Pre-Execution Note

**ASSUMPTION-02 DoR H1 gate:** These test scenarios use `surfaceType`, `teamId`, and `adapterOverride` as EA registry response field names. Before executing any AC1 or AC2 scenario below, confirm that the real EA registry schema uses exactly these field names. If not, update the fixture before each scenario is run.

---

## Scenarios

### Scenario 1 — Path A: HTTP GET request with app_id query parameter

**AC:** AC1

**Steps:**
1. Configure a context.yml with `registry_source: ea-registry`, `registry_url: <stub URL>`, `app_id: app-001`.
2. Intercept outbound HTTP requests (or check resolver logs).
3. Call `resolver.resolve(context)`.

**Expected result:** One HTTP GET request was made to the registry URL. The URL contains `app_id=app-001` as a query parameter. No other network calls made. `registry_url` value matches exactly what is in context.yml (not a hardcoded URL).

**Pass/Fail:** Pass if exactly one GET request made to configured URL with correct query param.

---

### Scenario 2 — Path A: surfaceType from registry drives adapter selection (adapterOverride null)

**AC:** AC2

**Steps:**
1. Configure the EA registry stub to return `{ "surfaceType": "saas-gui", "teamId": "squad-a", "adapterOverride": null }`.
2. Call `resolver.resolve(context)` with Path A config.
3. Check which adapter was selected.

**Expected result:** Adapter `"saas-gui"` is selected. Outcome is identical to a Path B context.yml with `surface: saas-gui` and no registry config.

**Pass/Fail:** Pass if resolved adapter is `"saas-gui"`.

---

### Scenario 3 — Path A: adapterOverride non-null takes precedence over surfaceType

**AC:** AC2

**Steps:**
1. Configure the EA registry stub to return `{ "surfaceType": "saas-gui", "teamId": "squad-b", "adapterOverride": "iac" }`.
2. Call `resolver.resolve(context)`.
3. Check which adapter was selected.

**Expected result:** Adapter `"iac"` is selected (adapterOverride), NOT `"saas-gui"` (surfaceType ignored when adapterOverride is non-null).

**Pass/Fail:** Pass if resolved adapter is `"iac"`.

---

### Scenario 4 — Path A unavailable: fallback to Path B when surface: also declared

**AC:** AC3

**Steps:**
1. Configure the EA registry stub to return an error (timeout or HTTP 503).
2. Set context.yml to have both `registry_source: ea-registry` AND `surface: iac`.
3. Call `resolver.resolve(context)`.
4. Inspect the resolved adapter and the log output.

**Expected result:** Adapter `"iac"` selected (from Path B fallback). Log output contains "EA registry unavailable — falling back to explicit surface declaration". No exception thrown.

**Pass/Fail:** Pass if fallback adapter selected and log message present.

---

### Scenario 5 — Path A unavailable: explicit error when no Path B fallback

**AC:** AC3

**Steps:**
1. Configure the EA registry stub to be unreachable.
2. Use a context.yml with `registry_source: ea-registry` but NO `surface:` declaration.
3. Call `resolver.resolve(context)`.
4. Inspect the result.

**Expected result:** Returns `{ status: "error", error: "EA registry unavailable and no fallback surface declaration found" }`. Result does NOT contain a git-native adapter selection. No uncaught exception.

**Pass/Fail:** Pass if status = "error", error message matches, and no git-native default used.

---

### Scenario 6 — Path B: no HTTP call made; outcome identical to pre-p2.6 behaviour

**AC:** AC4

**Steps:**
1. Use a context.yml with `surface: saas-gui` and no `registry_source`, `registry_url`, or `app_id` fields.
2. Monitor outbound HTTP requests.
3. Call `resolver.resolve(context)`.

**Expected result:** Adapter `"saas-gui"` selected. Zero HTTP requests made. No error about missing `app_id`. Identical behaviour to running the same config before p2.6 was merged.

**Pass/Fail:** Pass if correct adapter selected and zero HTTP calls made.

---

### Scenario 7 — Both paths declared: Path B wins, warning logged

**AC:** AC5

**Steps:**
1. Use a context.yml with both `registry_source: ea-registry` AND `surface: saas-gui`.
2. Monitor outbound HTTP requests.
3. Call `resolver.resolve(context)` and capture log output.

**Expected result:** Adapter `"saas-gui"` selected (Path B wins). Zero HTTP requests (Path A not queried). Log output contains both: "Both Path A (registry_source) and Path B (surface:) declared — Path B (explicit) takes precedence" and "Remove one to resolve." (or equivalent from AC phrasing).

**Pass/Fail:** Pass if Path B adapter selected, no HTTP call, and warning message with removal instruction in log.
