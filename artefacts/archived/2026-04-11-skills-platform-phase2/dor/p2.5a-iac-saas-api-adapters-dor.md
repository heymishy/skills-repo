## Contract Proposal

### What will be built

- **IaC adapter** at `src/surface-adapter/adapters/iac.js`: Implements `execute("iac", context) → result` interface from p1.2. Returns `{ status, surface: "iac", findings: [...], trace: "<adapter-name>@<version>" }`. IaC findings reference change-set items or infrastructure objects (not PR diff line numbers).
- **SaaS-API adapter** at `src/surface-adapter/adapters/saas-api.js`: Implements `execute("saas-api", context) → result`. SaaS-API findings reference API schema changes or endpoint contracts.
- **IaC POLICY.md stub** at `standards/iac/POLICY.md`: Contains sections `## Governance Criteria`, `## Evidence Requirements`, `## Minimum Thresholds` with `[TBD: <description>]` markers. At least one IaC-specific criterion (change-set review, state-file integrity, or drift detection).
- **SaaS-API POLICY.md stub** at `standards/saas-api/POLICY.md`: Same structure. At least one API-specific criterion (breaking-change detection, version pinning, or contract test reference).
- **Registry updates** in `standards/index.yml`: Both new adapters registered under their surface-type keys in the same commit as the adapter files.
- **Schema additions** in `pipeline-state.schema.json`: Any new fields written to `pipeline-state.json` by the adapter result object must be pre-defined (ADR-003 / MC-CORRECT-02).

### What will NOT be built

- SaaS-GUI, M365-admin, manual adapters (p2.5b)
- Live IaC provider connectivity (Terraform Cloud, AWS, Azure) — adapter evaluates declared context only
- SaaS-API contract test execution — adapter references existence, does not execute
- DoD criteria variant generation by surface type
