## Contract Proposal

### What will be built

- **Challenger skill procedure** (invoked from improvement-agent SKILL.md Phase 2 procedure): When a proposal in `workspace/proposals/` has `status: pending_review` and the operator manually approves it (human-assisted — see A3 below), the challenger skill:
  1. Reads the proposal file; validates proposed_diff format (before/after SKILL.md section)
  2. Creates a challenger branch (`improvement/[skill-slug]-[YYYY-MM-DD]`)
  3. Applies the proposed_diff to the target SKILL.md (one atomic commit)
  4. Runs the eval harness (`npm test`) against the patched SKILL.md — full suite
  5. If eval PASSES: writes acceptance decision to `decisions.md` and updates proposal `status: accepted` in `workspace/state.json`
  6. If eval FAILS or DEGRADES (any regression): writes rejection with evidence to `decisions.md`, updates proposal `status: rejected`, reverts patch, records `regression_evidence` in proposal file
- **Single-commit atomicity (AC3):** The diff application + commit is one atomic commit (`git commit --no-verify` prohibited; full hooks must run). The commit message identifies the proposal ID. Manual verification at PR review: the single-commit constraint cannot be fully automated (acknowledged, F-1 LOW from review).
- **Regression guard (AC4):** Before marking accepted, the improvement agent re-runs `npm test` against the unmodified baseline (before patch) to confirm baseline is GREEN. If baseline is RED (pre-existing failure), the challenger run is aborted and a `baseline-failure` warning is written instead. Do not apply diffs to a broken baseline.
- **Decision log entry (AC5):** Every acceptance or rejection is logged to `decisions.md` with: proposal ID, proposal file path, eval outcome (pass/fail), confidence score from proposal, regression_evidence (if any), decision date.
- **Proposal status lifecycle management (AC6):** `workspace/state.json` proposals block is updated at each transition: `pending_review → running | accepted | rejected`.

### What will NOT be built

- Automatic spawning of a new Copilot agent session (A3: human-assisted Phase 2 — challenger is invoked by operator within a Phase 2 inner loop session, not auto-spawned)
- Merging the challenger branch — merge is a human action; the challenger opens a draft PR only
- Cross-repo proposal application
- Multi-proposal clustering or batch acceptance

### A3 architecture decision (human-assisted Phase 2)

The challenger pre-check (spawning a second Copilot agent to independently evaluate) is human-assisted in Phase 2 because one Copilot session cannot spawn another asynchronously in the current platform model. The improvement agent SKILL.md Phase 2 procedure covers this gap by requiring the operator to manually trigger the challenger run within a second Copilot session. This is a documented architectural constraint (A3 in the story), not a deviation from scope.
