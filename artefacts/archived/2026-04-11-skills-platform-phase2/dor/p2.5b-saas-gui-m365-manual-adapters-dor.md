## Contract Proposal

### What will be built

- **SaaS-GUI adapter** at `src/surface-adapter/adapters/saas-gui.js`: Implements `execute("saas-gui", context) → result` interface from p1.2. Returns standard result object with `surface: "saas-gui"`.
- **M365-admin adapter** at `src/surface-adapter/adapters/m365-admin.js`: Implements `execute("m365-admin", context) → result`. Returns standard result object with `surface: "m365-admin"`.
- **Manual adapter** at `src/surface-adapter/adapters/manual.js`: Implements `execute("manual", context) → result`. Returns result with `resultPattern: "checklist"` plus standard interface fields. Findings represent checklist item completions, not code-diff or static-analysis findings.
- **Unknown surface type error path** (AC6): When `execute()` is called with an unrecognised surface type, the adapter framework returns `{ status: "error", surface: "<declared type>", findings: [], trace: "<identifier>", error: "unknown surface type: <declared type>" }` — not a runtime exception, not a silent fall-through.
- **POLICY.md stubs**: `standards/saas-gui/POLICY.md` (change-review approval, screenshot evidence reference), `standards/m365-admin/POLICY.md` (admin audit log reference, change ticket linkage), `standards/manual/POLICY.md` (checklist-only criteria — no code-review or diff-analysis content). Each stub follows the P1.7 three mandatory sections pattern.
- **`resultPattern: "checklist"` in `pipeline-state.schema.json`** (ADR-003 / MC-CORRECT-02): Field defined before adapter writes it.
- **`standards/index.yml` update**: All three new surface types registered.

### What will NOT be built

- IaC and SaaS-API adapters (p2.5a)
- Live authentication to any SaaS platform or M365 tenant
- Screen-capture tooling or browser automation for SaaS-GUI evidence
- M365 Graph API integration
