## Unit Tests

### AC1: queryTraces with storySlug filter returns only traces matching that storySlug

- **Verifies:** AC1
- **Precondition:** 5 synthetic trace fixtures in workspace/traces/ — 3 for slug `story-a`, 2 for slug `story-b`
- **Action:** Call `queryTraces({ storySlug: "story-a" })`
- **Expected result:** Returns exactly 3 traces; all with storySlug `story-a`; the 2 `story-b` traces are excluded
- **Edge case:** No

### AC1: queryTraces with patternLabel filter returns only traces containing that label

- **Verifies:** AC1
- **Precondition:** 6 synthetic trace fixtures — 2 with pattern label `missing-test-coverage`, 4 with `ac-not-testable`
- **Action:** Call `queryTraces({ patternLabel: "missing-test-coverage" })`
- **Expected result:** Returns exactly 2 traces; case-insensitive match; `ac-not-testable` traces excluded
- **Edge case:** No

### AC1: queryTraces with dateRange filter returns only traces within range (createdAt inclusive)

- **Verifies:** AC1
- **Precondition:** 5 synthetic trace fixtures with `createdAt` spread across 180 days
- **Action:** Call `queryTraces({ dateRange: { from: "2026-01-01", to: "2026-03-01" } })`
- **Expected result:** Returns only traces with `createdAt` within range; traces before or after excluded
- **Edge case:** No

### AC1: queryTraces with all three filters combined returns only traces matching all

- **Verifies:** AC1 (combined filter)
- **Precondition:** Trace fixtures with mixed surface types, pattern labels, and dates
- **Action:** Call `queryTraces({ storySlug: "story-a", patternLabel: "ac-ambiguous", dateRange: { from: "2026-01-01", to: "2026-06-01" } })`
- **Expected result:** Only traces where all three conditions hold simultaneously are returned; no partial-match traces included
- **Edge case:** No

### AC2: failure signal detection — exactly 3 occurrences of same label in 10-story window triggers proposal

- **Verifies:** AC2 (threshold boundary)
- **Precondition:** 10 synthetic traces for one surface type; exactly 3 have pattern label `missing-test-coverage` in `failurePattern` field
- **Action:** Run failure signal detection
- **Expected result:** One diff proposal file written to `workspace/proposals/`; filename matches `YYYY-MM-DD-[skill-slug]-failure-proposal.md`; proposal references the 3 trace IDs as evidence
- **Edge case:** exactly-3 threshold (not 2)

### AC2: failure signal detection — 2 occurrences of same label in 10-story window does not trigger proposal

- **Verifies:** AC2 (below threshold — negative path)
- **Precondition:** 10 synthetic traces; exactly 2 have the same pattern label
- **Action:** Run failure signal detection
- **Expected result:** No proposal file written; no `workspace/proposals/` entry created
- **Edge case:** below-threshold negative path

### AC2: failure signal detection — sliding window uses 10 most recent by createdAt (older traces excluded)

- **Verifies:** AC2 (window definition)
- **Precondition:** 15 synthetic traces for one surface type; 3 oldest have label `old-label` (they fall outside the 10-trace window); most recent 10 have label `new-label` exactly 3 times
- **Action:** Run failure signal detection
- **Expected result:** Proposal written for `new-label` (3 occurrences in window); proposal NOT written for `old-label` (outside window)
- **Edge case:** boundary case — sliding window cut-off

### AC2: failure signal detection — pattern label format is kebab-case (uppercase or space labels excluded from match)

- **Verifies:** AC2 (label format)
- **Precondition:** Trace fixtures with labels `Missing Test Coverage` (spaces/uppercase) and `missing-test-coverage` (correct kebab-case); 3 occurrences of each
- **Action:** Run failure signal detection
- **Expected result:** Match fired for `missing-test-coverage`; no match for `Missing Test Coverage` (format invalid); evaluation log notes label format rejection
- **Edge case:** No

### AC3: staleness signal detection — surface with no trace activity for ≥90 days triggers staleness proposal

- **Verifies:** AC3
- **Precondition:** Surface type with last trace `createdAt` = 95 days ago; `improvement_agent.staleness_window_days` = 90 in context.yml
- **Action:** Run staleness detection scan
- **Expected result:** Staleness proposal written to `workspace/proposals/YYYY-MM-DD-[skill-slug]-staleness-proposal.md`; file notes inactive surface, last activity date (95 days ago), suggests explicit deferral or removal of surface entry with rationale
- **Edge case:** No

### AC3: staleness signal — surface active 89 days ago does not trigger staleness proposal (boundary)

- **Verifies:** AC3 (boundary — below staleness threshold)
- **Precondition:** Surface type with last trace `createdAt` = 89 days ago; staleness window = 90
- **Action:** Run staleness detection scan
- **Expected result:** No staleness proposal written; surface considered active
- **Edge case:** boundary test

### AC4: anti-overfitting gate blocks proposal that removes a currently-passing check

- **Verifies:** AC4
- **Precondition:** Proposal would skip a MUST statement from SKILL.md; current 10-trace window for target surface type shows that check as PASS in every trace
- **Action:** Agent applies anti-overfitting gate to proposal
- **Expected result:** Proposal not written to proposals/; instead `YYYY-MM-DD-[skill-slug]-overfitting-warning.md` written; warning states: "Proposed removal affects a currently-passing check ([check label], last N traces all pass). Challenger pre-check required to proceed."
- **Edge case:** No

### AC4: anti-overfitting gate blocks proposal that reduces a threshold (weakening)

- **Verifies:** AC4 (threshold reduction variant)
- **Precondition:** Proposal would lower a pass threshold (e.g. reduce 3-occurrence minimum to 1); target check shows all-pass in the 10-trace window
- **Action:** Anti-overfitting gate applied
- **Expected result:** Proposal blocked; overfitting-warning file written
- **Edge case:** No

### AC4: anti-overfitting gate permits proposal that adds a new check (strengthening — not removal or weakening)

- **Verifies:** AC4 (gate does not over-block)
- **Precondition:** Proposal adds a new MUST statement not currently covered by any existing check; not removing or weakening anything
- **Action:** Anti-overfitting gate applied
- **Expected result:** Proposal not blocked; diff proposal file written normally; no overfitting-warning file created
- **Edge case:** No

### AC5: failure diff proposal file contains all 6 required fields

- **Verifies:** AC5
- **Precondition:** Diff proposal successfully generated (AC2 trigger passed)
- **Action:** Read generated proposal file; check for all 6 fields: `evidence`, `proposed_diff`, `confidence`, `anti_overfitting_gate`, `status: pending_review`, `created_at`
- **Expected result:** All 6 fields present; `status` equals `pending_review`; `created_at` is a valid ISO datetime; `confidence` is one of `high`, `medium`, `low` with a rationale string; `anti_overfitting_gate` is `passed`
- **Edge case:** No

### AC5: staleness proposal file contains all 6 required fields

- **Verifies:** AC5 (staleness variant)
- **Precondition:** Staleness proposal generated (AC3 trigger)
- **Action:** Read staleness proposal file; check for all 6 required fields
- **Expected result:** All 6 fields present as above; `evidence` references the surface type and last activity date; `status: pending_review`

### AC5: overfitting-warning file contains required warning structure

## NFR Tests

### Idempotency: running the improvement agent twice against the same trace set produces no duplicate proposal files

- **NFR addressed:** Idempotency
- **Measurement method:** Generate a proposal in run 1; run improvement agent again against identical trace set; inspect `workspace/proposals/` for duplicate files (same skill-slug + pattern label + date)
- **Pass threshold:** Zero duplicate proposal files; second run skips already-generated proposals; `workspace/proposals/` has the same file count after run 2 as after run 1
- **Tool:** File count check; duplicate-key comparison

### Privacy: trace data containing synthetic sensitive fields is redacted before processing

- **NFR addressed:** Privacy (MC-SEC-02)
- **Measurement method:** Run improvement agent against trace fixtures that contain a synthetic `operator_email` field; complete run; open generated proposal files; search for the synthetic email value
- **Pass threshold:** Synthetic email value not present in any proposal file; agent log or processing output shows redaction step was applied
- **Tool:** grep on proposal files for synthetic sensitive values; agent execution log check

### Auditability: proposal status transitions are recorded as appended history entries, not overwrites

- **NFR addressed:** Auditability
- **Measurement method:** Create a proposal with `status: pending_review`; record an acceptance (p2.12 story — simulate by manual edit); re-read the proposal file; check that original `status: pending_review` is preserved in `status_history` and the new `accepted` status is an appended entry
- **Pass threshold:** Original content intact in `status_history`; no content was overwritten; file can be read as an append-only record
- **Tool:** File content diff (pre- and post-status-transition)
