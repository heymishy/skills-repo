# Story: /review skill improvement — incremental per-story write (D4)

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-1-pipeline-evolution-foundation.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the /review skill to write each story's findings to a review artefact immediately after reviewing that story, rather than batching all findings into a single post-run write,
So that if a review session ends mid-run (context compaction, session timeout, operator interruption), the findings for completed stories are not lost and the next session can resume from the first unreviewed story without re-doing work.

## Benefit Linkage

**Metric moved:** MM5 — Flow findings conversion rate; MM1 — Solo operator outer loop Phase 2 calibration
**How:** D4 is a Phase 1 flow finding — "batched state writes lose progress on mid-session end" — logged in workspace/learnings.md and guarded by the `s-review-incremental-state` eval scenario in workspace/suite.json. Delivering D4 in Phase 2 is a MM5 measurement event (second of five Phase 1 findings actioned). It also saves outer loop wall time when reviews span session boundaries (MM1 — no need to re-review completed stories after session recovery).

## Architecture Constraints

- All changes are to `.github/skills/review/SKILL.md` instruction text only — no new scripts or executable code
- Per-story write must use the established artefact naming convention: `artefacts/[feature]/review/[story-slug]-review-N.md` where N is the review pass number (1 for first review in this feature's cycle, 2 for re-review after revision)
- The `s-review-incremental-state` eval scenario in `workspace/suite.json` is the existing regression guard — the D4 implementation must produce behaviour consistent with this scenario's invariant: "incremental write prevents progress loss on mid-session end"
- No changes to the review artefact template structure or finding format — D4 is a write-timing change only

## Dependencies

- **Upstream:** None — independent pipeline evolution story; can be parallelised with p2.1
- **Downstream:** All Phase 2 review runs benefit from per-story write; p2.2 DoD-complete before p2.4's review run is the natural sequencing (p2.2 improves the tool that will be used to review p2.4)

## Acceptance Criteria

**AC1 (per-story write on completion):** Given the /review skill is reviewing a multi-story feature and has completed evaluation of one story, when the skill produces findings for that story, then it writes those findings to `artefacts/[feature]/review/[story-slug]-review-1.md` immediately before loading the next story — the file is written and persisted, not held in memory until the session ends.

**AC2 (session recovery — no overwrite of completed stories):** Given a review session wrote per-story artefacts for stories S1 and S2 but ended before reviewing S3, when a new session resumes /review for the same feature, then the skill detects existing review artefacts for S1 and S2 by checking `artefacts/[feature]/review/` for their slugs — and begins from S3 without re-reviewing or overwriting S1 and S2 findings, unless the operator explicitly requests a re-review.

**AC3 (write completeness — no partial writes):** Given the per-story write triggers after a story's review completes, when the written artefact is inspected, then it contains all required finding fields from the review template: at minimum — story slug, review date, findings list (may be empty), severity levels (HIGH/MEDIUM/LOW), and recommended actions for any finding. An empty findings list is a valid complete write; a file with missing required fields is not.

**AC4 (eval scenario alignment):** Given the `s-review-incremental-state` scenario in `workspace/suite.json` describes the invariant that "incremental write prevents progress loss on mid-session end", when the D4-updated /review skill is evaluated against this scenario's described input conditions, then the scenario's invariant is satisfied — per-story artefacts are present for each story reviewed before session end, with no findings lost.

## Out of Scope

- Changes to the review finding template structure or severity classification system — format is unchanged by D4
- Automated session recovery detection (e.g. reading session logs to identify last reviewed story) — the skill detects existing artefacts by file presence check; it does not read session logs or timestamps to infer resume point
- Re-review triggering logic (when to produce a pass-2 review) — that is existing /review SKILL.md behaviour unchanged by this story
- Applying the incremental write pattern to other skills (e.g. /definition, /test-plan) — those are separate improvement candidates not in this story's scope

## NFRs

- **Durability:** The per-story write must be file-system persistent — each artefact must be readable in a fresh session on the same machine without any session state from the writing session
- **Auditability:** Each per-story review artefact must include the story slug and the review date in its file header — enabling traceability of when each story in the feature set was reviewed

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — single write-timing change to one skill's instruction text; the naming convention and filing structure are already established by Phase 1; the eval scenario is already written
