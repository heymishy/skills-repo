# Story: /definition skill improvements — dependency chain validation, testability filter, and learnings exit step (D1/D2/D3)

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-1-pipeline-evolution-foundation.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the /definition skill to validate dependency chains between stories, apply a testability filter to ACs as they are written, and prompt for a learnings write at skill exit,
So that stories leave /definition with traceable dependencies, verifiable ACs, and flow learnings captured before the session ends — reducing rework at /review and /definition-of-ready and ensuring Phase 1 learnings are carried forward into the pipeline.

## Benefit Linkage

**Metric moved:** MM5 — Flow findings conversion rate; MM1 — Solo operator outer loop Phase 2 calibration
**How:** D1, D2, and D3 are three of the five Phase 1 flow findings logged at Phase 1 /levelup. Delivering them in Phase 2 is the primary measurement event for MM5 (≥50% of logged findings actioned within 2 features). The improvements also reduce outer loop blocking time by catching dependency and testability issues during story decomposition rather than at /review or /definition-of-ready (MM1 — fewer rework loops).

## Architecture Constraints

- All changes are to `.github/skills/definition/SKILL.md` instruction text only — no new executable code, no new scripts
- ADR-004: any references to story locations use artefact-path conventions (`artefacts/[feature]/stories/[slug].md`), not pipeline-state.json field names — the /definition skill does not read pipeline-state.json directly
- The D2 testability filter is a warning prompt to the operator, not a hard block — hard blocks are reserved for /definition-of-ready H-checks; /definition must not stop story creation for testability warnings
- Skill files are content governed by pipeline process — changes to SKILL.md do not trigger check-governance-sync.js updates; however any new H-check or gate introduced must follow MC-CONSIST-02 if it appears in governance sync

## Dependencies

- **Upstream:** None — first Phase 2 story; no dependency on other Phase 2 stories
- **Downstream:** p2.3 (D8) consumes well-formed dependency declarations produced by the D1 improvement; the cross-story schema check in D8 validates fields that D1 requires to be declared at /definition time

## Acceptance Criteria

**AC1 (D1 — dependency chain validation):** Given the /definition skill is writing a story with at least one upstream dependency declared in the Dependencies block, when the story is saved to `artefacts/[feature]/stories/`, then the skill checks whether each named upstream story slug resolves to an existing artefact path (`artefacts/[feature]/stories/[slug].md`) — and if any slug does not resolve, surfaces a prompt before proceeding to the next story: "Upstream story [slug] not found at expected path. Options: (1) add the missing story, (2) confirm this is an external dependency, (3) remove the reference."

**AC2 (D1 — external dependency acknowledgement):** Given the /definition skill surfaces a missing-upstream-slug prompt, when the operator selects option 2 (external dependency) and provides a description, then the skill records `[External: <description> — confirmed by operator on <date>]` in the story's Dependencies block and does not re-surface the warning for that slug in the same session.

**AC3 (D2 — testability filter warning):** Given the /definition skill produces an AC for a story, when the AC is written, then the skill applies a testability check — flagging any AC that: (a) uses "should" or "would" instead of asserting current-state observable behaviour, (b) describes internal system state not visible to a test runner or human reviewer, or (c) cannot be evaluated independently without first running a prior AC. Flagged ACs are surfaced as: "AC[N] may not be independently testable — [reason]. Revise the AC or accept as-is?"

**AC4 (D2 — testability filter does not block):** Given the /definition skill flags an AC as potentially not independently testable, when the operator accepts the AC as-is, then story writing continues without the AC being removed or altered — the filter is advisory. The accepted AC is annotated `[Testability: accepted by operator on <date>]` in the artefact.

**AC5 (D3 — learnings exit step):** Given the /definition skill has completed decomposition of all stories for a feature and the scope accumulator check has passed, when the skill produces its Completion Output, then the completion output includes an explicit prompt before closing: "Before ending this session — are there any learnings from this decomposition to write to `workspace/learnings.md`? Reply with the learning text, or `skip` to proceed to /estimate."

## Out of Scope

- Changes to any skill other than `.github/skills/definition/SKILL.md` (D4 is p2.2; D8/D9/B1-enforce are p2.3)
- Hard-blocking story progression when a dependency slug is missing — that gate belongs at /definition-of-ready (H1: clear and testable ACs)
- Automated ML-based testability scoring — D2 is a heuristic prompt based on known AC anti-patterns ("should", internal state, AC coupling)
- Backfilling D1/D2/D3 checks onto Phase 1 story artefacts — new stories from Phase 2 forward only

## NFRs

- **Consistency:** The D1 dependency check must produce the same warning for the same input regardless of session order — non-deterministic warnings cause confusion in multi-story sessions
- **Auditability:** The `[Testability: accepted by operator]` annotation must be preserved by downstream skills (/review, /definition-of-ready) — it is a decision record, not a transient comment

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — modifications are to SKILL.md instruction text; skill boundaries are well-understood from Phase 1 dogfood; only D2 heuristic definition has any ambiguity (which AC patterns to flag)
