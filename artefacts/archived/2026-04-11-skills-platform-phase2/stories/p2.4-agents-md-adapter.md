# Story: AGENTS.md adapter in assembly script — agent instruction format driven by vcs.type (ADR-005)

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-2-full-surface-adapter-model.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the assembly script to emit `AGENTS.md` at the repository root when `vcs.type` in `context.yml` is non-GitHub, and `.github/copilot-instructions.md` when `vcs.type` is GitHub,
So that squads on Bitbucket, GitLab, or AAIF-compliant enterprise platforms receive the correct agent instruction format without manual format conversion after each distribution update.

## Benefit Linkage

**Metric moved:** M1 — Second squad outer loop unassisted
**How:** Non-GitHub squads currently hit a hard blocker at distribution: the assembly script always emits `.github/copilot-instructions.md`, which is inert in non-GitHub inner loop tooling. Resolving the format means a Bitbucket or enterprise-AAIF squad can complete the outer loop unassisted — the AGENTS.md output is the distribution vehicle they need to start using the platform. This story is the prerequisite for M1 applying to non-GitHub squads.

## Architecture Constraints

- ADR-005: this story IS the ADR-005 revisit trigger implementation — the ADR records "At Phase 2, implement the `agent_instructions.format` adapter in `context.yml` and update the assembly script to branch on `vcs.type`"
- ADR-004: `vcs.type` and `agent_instructions.format` are `context.yml` fields — NOT hardcoded strings in the assembly script; the script must read them from the config, not compare against literal "github" inline. The exact key path is `vcs.agent_instructions.format` (nested under the `vcs:` block). Permitted values: `copilot-instructions` (default, emits `.github/copilot-instructions.md`) and `agents-md` (emits `AGENTS.md` at repo root). Any value other than these two is an error.
- MC-CONSIST-01/02: if this story introduces a new `agent_instructions.format` field in `context.yml`, its expected values must be documented in a schema comment or the `contexts/` template files in the same commit — no fields used by the script without documented expected values
- `check-assembly.js` pre-commit test must be updated to cover both output paths in the same commit as the assembly script change (ADR-003 pattern: tests updated simultaneously with implementation)
- MC-SEC-02: the assembly script must not log or persist any credential values from `context.yml` during the format-branch logic

## Dependencies

- **Upstream:** None — assembly script is standalone; no dependency on other Phase 2 stories
- **Downstream:** p2.5a, p2.5b — non-git-native adapters require the AGENTS.md output path to be operational before a non-GitHub inner loop squad can use them via distribution; p2.5a/b can be written in parallel but their end-to-end adoption path is complete only after p2.4 is merged

## Acceptance Criteria

**AC1 (GitHub default — no regression):** Given `context.yml` has `vcs.type: github`, when the assembly script runs, then the assembled output is written to `.github/copilot-instructions.md` — existing behaviour preserved exactly; no `AGENTS.md` file is created.

**AC2 (non-GitHub — AGENTS.md output):** Given `context.yml` has `vcs.type` set to any non-github value (e.g. `bitbucket`, `gitlab`, `generic`), when the assembly script runs, then the assembled output is written to `AGENTS.md` at the repository root — and `.github/copilot-instructions.md` is NOT created in the same run.

**AC3 (explicit format override):** Given `context.yml` contains `vcs.agent_instructions.format: agents-md` explicitly set (nested under the `vcs:` block), when the assembly script runs, then `AGENTS.md` is emitted regardless of `vcs.type` value — explicit format declaration takes precedence over `vcs.type` inference. Permitted values for `vcs.agent_instructions.format` are `agents-md` and `copilot-instructions`; any other value is an assembly-time error.

**AC4 (content parity):** Given the assembly script runs with any valid configuration, when both `.github/copilot-instructions.md` (GitHub path) and `AGENTS.md` (non-GitHub path) outputs are compared, then the assembled content is identical byte-for-byte — output file path is the only variable; no content is added, removed, or transformed based on format.

**AC5 (unconfigured repo default):** Given `context.yml` does not declare `vcs.type` or `agent_instructions.format`, when the assembly script runs, then it defaults to `.github/copilot-instructions.md` output — backward-compatible default for existing repos that were configured before this story was delivered.

**AC6 (check-assembly.js covers both paths):** Given the assembly script is updated to branch on `vcs.type`, when `check-assembly.js` runs in the pre-commit hook, then the test suite covers both output paths — at minimum: one test asserting `.github/copilot-instructions.md` output for `vcs.type: github`, and one test asserting `AGENTS.md` output for a non-github `vcs.type`. Both tests pass on a clean checkout.

## Out of Scope

- Content transformation by VCS type (e.g. adding Bitbucket-specific preamble) — content is format-agnostic; the same assembled instructions are delivered regardless of output path (AC4)
- AGENTS.md conflict resolution for repos where both `.github/copilot-instructions.md` and `AGENTS.md` exist from a prior run — the script overwrites the correct output path; stale files from a prior config are not automatically cleaned up; this is a manual maintenance task
- Bitbucket Pipelines CI adapter validation — that is p2.10; this story is assembly script format only
- Supporting multiple simultaneous output paths (emit both files at once for hybrid environments) — single output path per `vcs.type` is the Phase 2 model; multi-path is a Phase 3 decision

## NFRs

- **Performance:** The `vcs.type` branch logic adds at most one conditional check to the assembly script; the overall script execution time must not increase by more than 100ms on a standard developer machine
- **Security:** The assembly script must not read, log, or expose credential sections of `context.yml` during the format-branch decision (MC-SEC-02)
- **Auditability:** The output file path chosen by the script must be logged to stdout during assembly — enabling the CI log to confirm which format was emitted for that run

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the ADR-005 decision provides clear implementation direction; the only ambiguity is the exact `context.yml` schema shape for `agent_instructions.format`, which must be agreed before the check-assembly.js tests can be written
