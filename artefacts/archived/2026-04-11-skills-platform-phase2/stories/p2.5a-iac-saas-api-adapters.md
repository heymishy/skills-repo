# Story: IaC and SaaS-API surface adapters with POLICY.md floor variants (p2.5a)

**Epic reference:** artefacts/2026-04-11-skills-platform-phase2/epics/epic-2-full-surface-adapter-model.md
**Discovery reference:** artefacts/2026-04-11-skills-platform-phase2/discovery.md
**Benefit-metric reference:** artefacts/2026-04-11-skills-platform-phase2/benefit-metric.md

> **Scope note:** p2.5 was split into p2.5a (IaC + SaaS-API) and p2.5b (SaaS-GUI + M365-admin + manual) per operator pre-authorisation during /definition — AC count for all five adapter types in one story exceeded the threshold of 8. Decision recorded in artefacts/2026-04-11-skills-platform-phase2/decisions.md SCOPE-01.

## User Story

As a **platform maintainer**,
I want IaC and SaaS-API surface adapters implemented with their respective POLICY.md floor variants,
So that squads delivering infrastructure as code or API-first SaaS products can run the assurance gate against surface-appropriate governance standards — not the git-native floor that assumes code diffs and pull requests.

## Benefit Linkage

**Metric moved:** M2 — Non-git-native surface adapter assurance verdict (partial delivery — IaC and SaaS-API); M1 — Second squad outer loop unassisted
**How:** M2 requires at least one non-git-native adapter to produce a passing assurance verdict on a real PR. p2.5a delivers the IaC and SaaS-API adapters; the first inner loop PR for a story with `surface: iac` or `surface: saas-api` is the M2 acceptance vehicle. M1: IaC and API-first squads currently cannot adopt the platform at all; p2.5a removes that adoption blocker.

## Architecture Constraints

- Adapters implement the `execute(surface, context) → result` interface contract established in p1.2 — the interface signature must not be changed; new adapters are registered alongside the git-native adapter, not replacing it
- **Stub POLICY.md creation — in scope for this story:** This story creates stub POLICY.md floor files for both IaC (`standards/iac/POLICY.md`) and SaaS-API (`standards/saas-api/POLICY.md`) surface types as deliverables. Each stub follows the P1.7 structural pattern and contains the three mandatory floor sections (`## Governance Criteria`, `## Evidence Requirements`, `## Minimum Thresholds`) with `[TBD: <description>]` content markers. Stubs must be registered in `standards/index.yml` in the same commit as the adapters. Stub content is populated by the platform maintainer before the adapters are used in a live squad delivery — this is not gated as part of the story ACs.
- POLICY.md floor files for IaC and SaaS-API must follow the `standards/index.yml` routing pattern established in p1.7 — surface type is the routing key; `standards/index.yml` must be updated in the same commit as adapter registration
- MC-CORRECT-02/03 (schema-first): any new fields added to the adapter result object (e.g. `changeset`, `apiDiff`) that are written to `pipeline-state.json` must be defined in `pipeline-state.schema.json` first
- ADR-003: the adapter result schema for IaC and SaaS-API findings vocabulary must be documented (as a YAML comment or schema file) before the adapters write results — "changeset items" and "API schema changes" are result types, not arbitrary JSON
- MC-SEC-03: the adapters read local context only — no external fetch to IaC providers (Terraform Cloud, AWS, Azure) or SaaS-API registries; the adapter evaluates the governance criteria against the declared surface context, not against a live system state

## Dependencies

- **Upstream:** p1.2 (Phase 1, DoD-complete) — the `execute(surface, context) → result` interface is the contract this story extends; p1.2 must be merged for the interface contract to exist
- **Upstream:** p2.4 — AGENTS.md adapter must be merged before the end-to-end adoption path for non-GitHub IaC squads is complete; p2.5a can be written and merged independently, but the full squad adoption flow requires both stories
- **Downstream:** p2.6 — EA registry Path A resolver uses all five registered adapter types to route surface type to the correct adapter; p2.6 depends on p2.5a and p2.5b both being DoD-complete

## Acceptance Criteria

**AC1 (IaC adapter — result interface conformance):** Given `context.yml` declares `surface: iac`, when `execute("iac", context)` is called, then the IaC adapter returns a result object conforming to the P1.2 interface: `{ status: "pass" | "fail" | "error", surface: "iac", findings: [...], trace: "<identifier>" }` — no required field is absent.

**AC2 (SaaS-API adapter — result interface conformance):** Given `context.yml` declares `surface: saas-api`, when `execute("saas-api", context)` is called, then the SaaS-API adapter returns a result conforming to the P1.2 interface with `surface: "saas-api"` — no required field is absent.

**AC3 (IaC POLICY.md floor — surface-specific standards):** Given a `standards/index.yml` routing lookup with `surface: "iac"`, when the POLICY.md floor is resolved, then an IaC-specific POLICY.md floor file exists at its declared path — it is returned by the routing lookup instead of the git-native POLICY.md floor, and the file contains at least one IaC-specific governance criterion (e.g. change-set review, state-file integrity, drift detection).

**AC4 (SaaS-API POLICY.md floor — surface-specific standards):** Given a `standards/index.yml` routing lookup with `surface: "saas-api"`, when the POLICY.md floor is resolved, then a SaaS-API-specific POLICY.md floor file exists at its declared path and is returned — containing at least one API-specific criterion (e.g. breaking-change detection, version pinning, contract test reference).

**AC5 (multi-surface context — no shadowing):** Given `context.yml` declares both `surface: iac` and `surface: saas-api` simultaneously, when `execute()` resolves both adapters and runs both, then the result contains separate entries for each surface — IaC findings and SaaS-API findings are distinct; neither adapter shadows or overrides the other's result.

**AC6 (findings vocabulary — surface-appropriate):** Given the IaC adapter returns findings for a failed governance check, when the findings array is inspected, then IaC findings reference change-set items or infrastructure objects (not PR diff line numbers) — and SaaS-API findings reference API schema changes or endpoint contracts (not file diffs). The findings vocabulary is surface-appropriate, not borrowed from the git-native adapter.

## Out of Scope

- SaaS-GUI, M365-admin, and manual adapter implementations — those are p2.5b
- IaC provider connectivity (Terraform Cloud API, AWS CloudFormation, Azure ARM) — the adapter evaluates governance criteria against declared context; it does not make live calls to infrastructure providers (MC-SEC-03)
- SaaS-API contract test execution — the adapter references contract test existence and results by convention; it does not execute the tests itself
- DoD criteria variant generation by surface type (producing IaC-specific DoD gates) — the adapter identifies surface type and applies the POLICY.md floor; generating bespoke DoD variants from that result is a Phase 3 capability

## NFRs

- **Performance:** Each adapter's `execute()` call must complete within 5 seconds — it runs inline in the CI gate budget (P1.3 gate total budget: 10 minutes; adapter runtime is a sub-step)
- **Security:** Adapters must not log, persist, or transmit credential values from `context.yml` (MC-SEC-02); no external network calls from inside `execute()` (MC-SEC-03)
- **Auditability:** Each adapter result's `trace` field must include the adapter name and version — enabling the CI trace to record which adapter implementation produced the result

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — the interface contract is defined by p1.2; the POLICY.md floor pattern is defined by p1.7; the main unknowns are IaC and SaaS-API findings vocabulary definitions, which must be agreed before ACs can be fully verified
