## Story: Spike C — Validate the distribution model for the governance package

**Epic reference:** artefacts/2026-04-19-skills-platform-phase4-opus/epics/e2-distribution-zero-commit-install.md
**Discovery reference:** artefacts/2026-04-19-skills-platform-phase4/discovery.md
**Benefit-metric reference:** artefacts/2026-04-19-skills-platform-phase4-opus/benefit-metric.md

## User Story

As a **tech lead at work (Craig's context)**,
I want to **see a working prototype of the distribution model — including the publishing layer and commit provenance behaviour — demonstrated end-to-end**,
So that **I know the distribution mechanism (M1) works before the platform commits to building the full install and sync tooling**.

## Benefit Linkage

**Metric moved:** M1 (Distribution sync — zero-commit install and sync success rate)
**How:** This spike validates the distribution architecture before implementation begins. The discovery explicitly states the architectural question cannot be resolved by decision alone — the provenance behaviour must be shown working. A PROCEED verdict unblocks all Theme B implementation stories. A REDESIGN verdict forces a rethink before any distribution code is written.

## Architecture Constraints

- **C1 (update channel never severed):** The prototype must demonstrate that the distribution model preserves the update channel — consumers can receive updates without forking
- **C5 (hash-verified skill files):** The prototype must show that hash verification works across the distribution boundary — skills delivered to consumers are verifiable against a lockfile
- **ADR-012 (platform-agnostic):** The distribution model must not depend on GitHub-specific features (e.g. GitHub Packages, GitHub Actions artifacts) as the sole distribution path
- **Upstream authority constraint:** The discovery states that whether `heymishy/skills-repo` is the direct authoritative upstream or whether an intermediate publishing layer exists is a Spike C deliverable. This spike must resolve that question.

## Dependencies

- **Upstream:** Spike A (governance extractability) — the distribution model must know what is being distributed; Spike A defines the governance package boundary
- **Downstream:** All Theme B implementation stories (design-package-manifest, implement-zero-commit-install, implement-sync-command, implement-lockfile-hash-verification, implement-context-yml-seeding, validate-install-sync-e2e)

## Acceptance Criteria

**AC1:** Given the governance package boundary defined by Spike A, When I prototype a distribution mechanism (one of: npm package, git subtree, installer script, or another mechanism identified during the spike), Then the prototype demonstrates end-to-end: package is published from the source repo, consumer installs it into a fresh repo, and the installed files match the source files by hash.

**AC2:** Given the prototype from AC1, When the consumer repo's git history is inspected after installation, Then zero commits were generated by the install process — the governance package files exist in the consumer's working tree without appearing in the consumer's commit history.

**AC3:** Given the prototype from AC1, When a simulated update is published from the source repo (a file in the governance package is modified), Then the consumer can sync the update and the updated files arrive with correct hashes — demonstrating the update channel is not severed by the initial install.

**AC4:** Given the prototype from AC1, When the upstream authority question is assessed, Then the spike artefact documents: (a) whether `heymishy/skills-repo` serves as the direct upstream or whether an intermediate publishing layer is needed, (b) the rationale for this decision, and (c) the commit provenance behaviour of the chosen model.

**AC5:** Given the evidence from AC1 through AC4, When the spike verdict is written, Then it is one of PROCEED (distribution model works end-to-end with zero-commit install and update channel preservation), REDESIGN (the model has specific documented failures that require architectural changes), or DEFER (the model works but the implementation cost exceeds Phase 4 capacity).

## Out of Scope

- Building the full install and sync CLI tooling — this spike produces a prototype, not production tooling
- Evaluating enforcement mechanisms — that is Spikes B1 and B2
- Cross-VCS distribution (Bitbucket, GitLab) — Phase 4 validates on GitHub; other VCS adapters are Phase 5 scope
- Teams bot or non-technical surface distribution — that is Epic 3
