## Scenario 1 — AC1: git-native adapter returns a complete result

**What you are checking:** When the surface adapter is called with `surface: "git-native"` and a valid context, it returns a result that has all the fields the platform needs to operate.

**Steps:**
1. Invoke the `execute("git-native", context)` function (or observe a CI gate run that does this)
2. Capture or inspect the result object returned

**Expected outcome:** The result contains at minimum these fields — all non-empty:
- `status` — one of `pass`, `fail`, or `error`
- `surface` — must say `git-native` (echoed from input)
- `findings` — an array (may be empty, but the field must exist)
- `trace` — a non-empty identifier string

**Fail if:** Any of the four fields is missing, null, or empty.

## Scenario 2 — AC2: Path B declaration in context.yml selects the git-native adapter

**What you are checking:** When a consuming squad's `context.yml` declares `type: "git-native"` under Path B (`explicit-declaration`), the platform selects the git-native adapter — without doing any EA registry lookups.

**Steps:**
1. Find or create a `context.yml` file with this shape:
   ```
   surface:
     path: explicit-declaration
     type: git-native
   ```
2. Run the surface adapter resolution against this file
3. Observe which adapter is selected

**Expected outcome:** The git-native adapter is selected. No EA registry call is made (no outbound call to an EA registry service should occur during this resolution).

**Fail if:** A different adapter is selected, or any EA registry lookup occurs during Path B resolution.

**What you are checking:** When `context.yml` lists more than one surface type, the platform selects the correct adapter for each — not just the first one, and not the same adapter for all.

## Scenario 4 — AC4: Interface is extensible for Path A without signature change 🔴 (Design review)

**What you are checking:** The `execute` function has been designed so that a Phase 2 developer can add EA registry lookup (Path A) by writing a new resolver — without needing to change `execute`'s inputs or outputs.

**This scenario requires a developer or technical reviewer.**

**Steps:**
1. Open the `execute` function implementation in your editor
2. Check the function signature — does it take exactly `(surface, context)` as its parameters? Is there no `registryUrl` or `resolutionPath` baked into it?
3. Check the function body — does it operate on the `surface` string it receives, without branching internally on *how* that string was determined?
4. Consider: could someone write a separate function that calls an EA registry, gets back a surface type string, and passes it to the existing `execute` call? If yes, Path A addition would require no changes to `execute`.

**Expected outcome:** The function signature is `execute(surface, context) → result`. The function body does not contain Path A/Path B branching. A Path A resolver can be added as a wrapper that calls `execute` with its output — `execute` itself does not change.

**Fail if:** Adding Path A would require changing `execute`'s function signature, or `execute` contains internal branching that would need modification for Path A.

## Scenario 5 — AC5: Phase 2 surface types are not foreclosed by the interface 🔴 (Design review)

**What you are checking:** The interface has been designed so that any of the five Phase 2 surface types (IaC, SaaS-API, SaaS-GUI, M365-admin, manual) can be added later without changing the core interface or the git-native adapter.

**This scenario requires a developer or technical reviewer.**

**Steps:**
1. Open the `SurfaceType` definition in the source code
2. Check: is `SurfaceType` a string type or an extensible string union — not a closed enum that would require a code change to add a new value?
3. Check the `execute` body: does it contain a switch or if/else that handles each surface type explicitly? If it does, adding a new surface type would require modifying `execute`.
4. Check how adapters are registered: can a new adapter be written as a separate module and registered without modifying any existing adapter?

**Expected outcome:** Adding any of the five Phase 2 surface types means writing a new adapter module and registering it — no changes to the `execute` function, no changes to the `SurfaceType` definition (or the definition allows addition by extension), no changes to the `git-native` adapter.

**Fail if:** Adding a new surface type would require modifying `execute`, modifying the `SurfaceType` definition in a breaking way, or modifying any existing adapter.

## Scenario 6 — AC6: A single context.yml file works for both assembly and surface resolution

**What you are checking:** There should be one `context.yml` file per consuming squad — it should work as both the assembly configuration (for P1.1 distribution) and the surface type input (for P1.2 resolution). There should not be two separate files.

**Steps:**
1. Take a `context.yml` file that conforms to the Path B format agreed between P1.1 and P1.2
2. Pass it through P1.1's assembly logic — does it produce a valid assembled context without errors?
3. Pass the same file through P1.2's surface resolution — does it select the correct adapter without errors?

**Expected outcome:** The same file is accepted by both without modification. P1.1 and P1.2 each read the fields they need from the same `context.yml`. No format conflict, no second file required.

**Fail if:** P1.1 and P1.2 require different `context.yml` shapes and a single file cannot satisfy both.
