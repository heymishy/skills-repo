## Setup

**Before you start:**
1. You need access to: (a) a consuming squad repository that uses the distribution mechanism, (b) the assembled `copilot-instructions.md` in that repository, (c) the skills-repo
2. For scenarios 2a/b/c and 6 (the M1 acceptance test), you need the ability to commit a controlled change to the skills-repo and observe its propagation — allow up to 72 hours for a push model, or one session open for a pull model
3. For scenario 3, you need to open a fresh Copilot Chat session using the assembled `copilot-instructions.md`
4. Have a text editor ready to inspect the assembled `copilot-instructions.md` file directly

## Scenario 1 — AC1: Skills available without copying files into the squad repo

**What you are checking:** When a squad engineer uses the platform, they do not need a copy of the skills-repo in their repo. They can load platform skills using `/load [skill-name]` via the assembled context file.

**Steps:**
1. Open a fresh Copilot Chat session in a consuming squad repository
2. The session should be using an assembled `copilot-instructions.md` produced by the distribution mechanism (not an ad-hoc copy)
3. Type `/load discovery` and observe whether the discovery skill is available
4. Open the squad repo's file browser and check whether any `SKILL.md` files from the skills-repo have been copied into the repo

**Expected outcome:** The discovery skill is available after `/load discovery`. The squad repo's files do not contain any copied `SKILL.md` files from the platform.

**Fail if:** Skills are not available via `/load`, or platform skill files are copied directly into the squad repo.

## Scenario 2a — AC2a: Change present in assembled file after one distribution cycle

**What you are checking:** When a platform maintainer makes a change in the skills-repo, that change reaches the squad's assembled `copilot-instructions.md` within one distribution cycle — automatically.

**Steps:**
1. Add a clearly identifiable string to a core skill in the skills-repo (e.g. add a comment like `<!-- ACCEPTANCE TEST 2026-04-10 -->` to one SKILL.md file)
2. Commit and push this change to the skills-repo
3. Wait one distribution cycle (the implementation documentation will state the expected time — up to 72 hours for a push model, or until the next session open for a pull model)
4. Open the consuming squad's `copilot-instructions.md` and search for the test string

**Expected outcome:** The test string appears in the assembled file. The change propagated without any action from the squad team.

**Fail if:** The string is not found in the assembled file after one distribution cycle has elapsed.

**What you are checking:** The squad team did not need to run a merge, resolve a conflict, or take any manual step to receive the update.

**Expected outcome:** No squad team member took any action. No merge commits appear in the squad repo's log from the distribution period.

## Scenario 3 — AC3: Only discovery-phase skills loaded at session start

**What you are checking:** When you open a session in the outer loop (at discovery phase), inner loop skills like TDD and implementation planning are not loaded. You only get the skills you need right now.

**Steps:**
1. Open a fresh Copilot Chat session using the assembled `copilot-instructions.md`
2. At the session start (before invoking any skills), try to use an inner loop skill — for example, type `/tdd`

**Expected outcome:** The TDD skill is not available at session start. Copilot should indicate the skill is not loaded (e.g. "TDD skill not loaded in current phase — use `/load tdd` to load it"). Outer loop skills like `/discovery`, `/review`, and `/test-plan` should be available.

**Fail if:** Inner loop skills are loaded at session start in an outer loop context, or none of the available skills work as expected.

## Scenario 5 — AC5: Assembled file has three-layer composition with a readable header

**What you are checking:** The assembled `copilot-instructions.md` is built from three layers in a declared order, and you can tell from the file header which layers were used and which platform version was assembled.

**Steps:**
1. Open the assembled `copilot-instructions.md` in a text editor
2. Read the header of the file (the first section or comment block at the top)
3. Look for references to: (a) the core platform layer, (b) a domain layer, (c) a squad layer
4. Check whether the composition order is stated
5. Check whether a platform version reference (commit hash, tag, or version number) is present

**Expected outcome:** The header clearly identifies three layers in order. If the domain layer is absent for this squad, the header notes that. The header contains a platform version reference so you can tell which version of the platform skills this file was assembled from.

**Fail if:** The header is absent, the layer composition is not documented, or there is no version reference.

## Scenario 6 — AC6: M1 acceptance test records all required fields

**What you are checking:** The M1 acceptance test — the formal record of P1.1's delivery — captures all the information needed to verify the distribution mechanism delivers on the core metric.

**Steps:**
