## Setup

**Before you start:**
1. You need access to the repository on GitHub and the ability to open a pull request
2. For scenarios 1–5, you need a PR open against the repository (you can open a draft PR with a trivial change)
3. Ensure `workspace/traces/` directory exists in the repository

---

## Scenario 1 — AC1: Gate fires automatically when a PR is opened

**What you are checking:** When you open or update a pull request, the assurance gate starts on its own — you do not need to manually trigger it.

**Steps:**
1. Open a new pull request in the repository (or push a commit to an existing open PR)
2. Go to the Actions tab on GitHub (or your CI platform's equivalent)
3. Look for a gate run that started within 60 seconds of your push

**Expected outcome:** A gate run appears in the CI actions list. When you click into the run, you should see that it was triggered by a "pull_request" event — not by a manual trigger. You should not have had to press any buttons to start it.

**Fail if:** No gate run appears after 2 minutes, or the run was triggered by "workflow_dispatch" (manual trigger), or you had to start it yourself.

## Scenario 2 — AC2: "In progress" trace entry written before evaluation begins

**What you are checking:** Even early in the gate run — before it has evaluated anything — a trace entry showing "in progress" has already been written.

**Steps:**
1. After the gate starts running (you can observe this in CI while it is still running, or check after it completes)
2. Open `workspace/traces/` in the repository
3. Find the trace file for the PR you opened
4. Look at the trace entries in chronological order

**Expected outcome:** The first entry in the trace file is an "inProgress" entry, timestamped at the start of the run — before any evaluation results appear. This entry should exist even if the gate run subsequently fails or is cancelled.

**Fail if:** No "inProgress" entry exists, or it appears after evaluation results, or the trace file only exists if the run completes successfully.

## Scenario 3 — AC3: "Completed" trace entry follows "in progress"

**What you are checking:** After a full gate run, the trace shows a clean and complete picture: started, then finished — in that order.

**Steps:**
1. Wait for a gate run to complete (pass or fail)
2. Open the trace file in `workspace/traces/` for that run
3. Read the entries in order

**Expected outcome:** The trace file contains two entries in sequence:
- First: an "inProgress" entry with a start timestamp
- Second: a "completed" entry with a finish timestamp, gate verdict (pass or fail), and trace hash

There should be nothing between these two entries and both should be present.

**Fail if:** Only one entry exists, or the "completed" entry appears before "inProgress", or the entries are out of order.

## Scenario 4 — AC4: Gate verdict visible in the PR without opening any other tool

**What you are checking:** After the gate finishes, you can see the result directly in the pull request — you should not need to open a log file, a dashboard, or any other system.

**Steps:**
1. After a gate run completes, open the pull request on GitHub
2. Scroll to the checks area at the bottom of the PR (the section that shows CI status)
3. Look for the assurance gate result

**Expected outcome:** You can see:
- Whether the gate passed or failed — clearly labelled
- A trace hash or reference ID (so the full trace can be found if needed)

You should not need to click into CI logs, open `workspace/traces/` manually, or use any other tool to find this information.

**Fail if:** The gate result is not visible in the PR interface, or you can only find it by opening the CI log or trace files manually.

## Scenario 5a — AC5a: Both manual and CI trace entries coexist

**What you are checking:** If the assurance agent was run manually in a prior session for this story, and then a CI run also happens, both trace entries should be preserved — one for the manual run, one for the CI run.

**Steps:**
1. Find a story that had a manual assurance agent run before the CI gate was set up (or simulate this by checking for a prior trace entry)
2. After a CI-triggered gate run on the same story, open `workspace/traces/`
3. Look at all trace entries for that story

**Expected outcome:** You can see at least two separate trace entries — one labelled "manual" and one labelled "ci". The CI run has not written over or replaced the manual run entry.

**Fail if:** Only one trace entry exists, or the CI run replaced the prior manual run entry.

## Scenario 5b — AC5b: CI-triggered trace entry is labelled as "ci"

**What you are checking:** The trace file correctly records how the gate was triggered — so you can tell at a glance whether a gate run was done automatically by CI or manually by an operator.

**Steps:**
1. Open the trace file for a CI-triggered gate run in `workspace/traces/`
2. Find the "completed" entry for the CI-triggered run
3. Look for a "trigger" field in the entry

**Expected outcome:** The "trigger" field contains the value `ci` — not `manual` or blank.

**Fail if:** The "trigger" field is missing, blank, or says `manual` on a run that was triggered by a PR event.
