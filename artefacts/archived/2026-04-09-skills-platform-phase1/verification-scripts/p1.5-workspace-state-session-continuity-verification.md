## Setup

**Before you start:**
1. You need access to the repository root on your local machine
2. Ensure `workspace/state.json` exists and is populated (it is written at each phase boundary — if it is missing, the implementation is not complete)
3. You need to be able to open a fresh VS Code Copilot Chat window with no prior conversation history for scenarios AC1 and AC4
4. Have a stopwatch ready for scenario AC3a

## Scenario 1 — AC1: New session identifies correct phase, no prior work repeated

**What you are checking:** When you open a brand new session and Copilot reads `workspace/state.json`, it correctly identifies where the team is in the pipeline and does not re-ask questions already answered.

**Steps:**
1. Close all existing Copilot Chat windows
2. Open a fresh Copilot Chat window (no prior conversation history)
3. Do not type anything — just wait for Copilot's first response

**Expected outcome:** Copilot's first message tells you the last completed phase (e.g. "Last completed phase: review") and asks whether to resume or start fresh. It does not ask you to re-run the review, re-describe the feature, or confirm decisions that are already in `state.json`.

**Fail if:** Copilot's first message asks a question that is already answered in `state.json`, or begins working on a phase that is already complete.

## Scenario 3a — AC3a: /checkpoint terminates within 60 seconds 🔴

**What you are checking:** When you invoke `/checkpoint` mid-phase, Copilot finishes and the session ends (or confirms completion) within 60 seconds.

**Steps:**
1. During a live session, at a natural mid-phase break point, type `/checkpoint`
2. Start your stopwatch as soon as you press Enter
3. Watch for Copilot's closing confirmation ("Pipeline state updated ✅" or equivalent)
4. Stop your stopwatch when the closing message appears

**Expected outcome:** The closing message appears within 60 seconds.

**Fail if:** 60 seconds passes with no closing message, or the session ends via context compaction before Copilot finishes writing the checkpoint.

**Note:** This scenario requires a live session — it cannot be tested against a committed file. Record the elapsed time as the evidence value for MM3.

## Scenario 3b — AC3b: Checkpoint-written state.json contains all required fields

**What you are checking:** After `/checkpoint` runs, the resulting `state.json` file has everything it needs — nothing is blank or missing.

**Steps:**
1. After running `/checkpoint` (from scenario 3a, or a dedicated test invocation), open `workspace/state.json` in your editor
2. Check each of the following fields is present and not blank:
   - `currentPhase` — a non-empty string naming the current phase
   - `lastUpdated` — an ISO date/time value (looks like `2026-04-10T14:00:00Z`)
   - A cycle block for the current phase, containing at minimum `status` and `artefact` path

**Expected outcome:** All three items are present and non-empty.

**Fail if:** Any field is missing, null, or an empty string `""`.

## Scenario 3c — AC3c: New session resumes from checkpoint without verbal priming

**What you are checking:** After a `/checkpoint` run, opening a new session picks up exactly where you left off — without you having to explain anything.

**Steps:**
1. After a `/checkpoint` write (use the `state.json` from scenario 3b)
2. Close the session
3. Open a fresh Copilot Chat window
4. Do not type anything — observe Copilot's opening message
5. Confirm it identifies the correct resume point

**Expected outcome:** Copilot tells you where you are (correct phase, correct story if applicable) without you describing it. It should match what was in `state.json` at the time of the checkpoint.

**Fail if:** Copilot asks you to describe what you were doing, or identifies the wrong phase.

**What you are checking:** After any phase boundary skill completes (e.g. benefit-metric, definition, review), `workspace/state.json` has all three mandatory elements.

## Scenario 6 — AC6: /checkpoint documentation gives clear invocation timing guidance

**What you are checking:** The `/checkpoint` instructions in `copilot-instructions.md` tell the operator to invoke it *before* the session reaches compaction pressure — not at it.

**Steps:**
1. Open `copilot-instructions.md` (or wherever the `/checkpoint` convention is documented)
2. Find the section describing when to invoke `/checkpoint`
3. Check that the text explicitly says to invoke it before compaction pressure — with enough headroom for the write to finish

**Expected outcome:** The documentation contains clear wording that `/checkpoint` should be invoked early enough that the write can complete — not at the exact moment context is full. Something like: "invoke before the session reaches compaction pressure, not at it."

**Fail if:** The documentation only says "invoke at 75%" or gives a percentage threshold without explaining that headroom is needed for the write to complete.
