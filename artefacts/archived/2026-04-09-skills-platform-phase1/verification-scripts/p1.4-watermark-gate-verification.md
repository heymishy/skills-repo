## Scenario 1 — AC1: First gate run creates the watermark file

**What you are checking:** When the gate is run for the very first time (no prior results exist), it creates `workspace/results.tsv` and records the first run as the starting watermark — it does not block anything yet.

**Steps:**
1. If `workspace/results.tsv` already exists and has content, skip to scenario 2 (the baseline has already been established)
2. Trigger a gate run against a skill/surface combination that has no prior record
3. After the run completes, open `workspace/results.tsv`

**Expected outcome:** The file exists and contains exactly one row with these six columns (tab-separated):
- A date and time stamp (e.g. `2026-04-10T14:00:00Z`)
- A skill-set hash (a string of letters and numbers)
- The surface type (e.g. `git-native`)
- A pass rate value between 0 and 1 (e.g. `0.9`)
- A full test score (a whole number, e.g. `18`)
- Verdict: `baseline`

There should be no rejection — this first run always passes.

**Fail if:** The file is not created, the verdict is not `baseline`, any column is missing, or the gate rejects the first run.

## Scenario 3 — AC3: Gate blocks automatically when quality drops — no approval needed

**What you are checking:** When the gate detects a regression, it blocks immediately — it does not wait for a person to approve or review the run.

**Steps:**
1. You cannot easily simulate this without modifying the suite (use scenario 5 for the full test); for this scenario, check the gate's behaviour in a recent blocked run if one exists
2. Open `workspace/results.tsv` and find a row with verdict `blocked`
3. Check the row's reason/trigger column

**Expected outcome:** The blocked row shows the trigger condition — one of: `pass-rate-below-watermark`, `score-below-best`, or `both`. The gate did not pause for human input — it just blocked and recorded the reason.

**Fail if:** No reason is recorded in the blocked row, or you can observe that a human approval was requested during the run.

## Scenario 4 — AC4: Blocked verdict visible in the PR interface

**What you are checking:** When the gate blocks a PR, you can see the blocked result directly in the pull request — you do not need to open any other tool to find out what happened.

**Steps:**
1. Find a PR where the gate has produced a `blocked` verdict (or trigger one via scenario 5)
2. Open the pull request on GitHub
3. Scroll to the checks section at the bottom of the PR

**Expected outcome:** The gate check shows a failed or blocked status. The reason for the block (which condition was below the watermark) is visible in the PR — either in the check description, a PR comment, or the check annotations. You do not need to open `results.tsv`, the CI log, or any dashboard to see this.

**Fail if:** The PR shows a failing check but gives no reason, or you have to leave the PR to find out why it was blocked.

## Scenario 5 — AC5: Controlled acceptance test — add failure, observe block, remove, observe pass

**What you are checking:** The full end-to-end acceptance test for the watermark gate — confirming the gate works correctly in both directions.

**Steps:**

**Part A — introduce a regression:**
1. Open `workspace/suite.json` in your editor
2. Add a deliberately failing scenario — a test entry that you know will fail (e.g. set `expected: pass` but configure it so the actual result will be `fail`)
3. Commit this change to a branch and open or update a PR
4. Wait for the gate to run

**Expected outcome (Part A):** Gate verdict is `blocked`. The PR checks section shows the block. `results.tsv` has a new row with `blocked` and the trigger condition recorded.

**Part B — remove the regression:**

5. Remove the deliberately failing scenario from `suite.json`
6. Commit this change to the same branch
7. Wait for the gate to run again

**Expected outcome (Part B):** Gate verdict is `pass`. The PR checks section shows the gate has cleared. `workspace/results.tsv` now has one more row — the blocked run row from Part A is still there (it has not been deleted), and a new `pass` row follows it.

**Fail if:** Part A does not produce `blocked`, or Part B does not restore `pass`, or the blocked row from Part A is deleted from `results.tsv` after Part B.

## Scenario 6 — AC6: Blocked row records which condition triggered the block

**What you are checking:** When you read a blocked row in `results.tsv`, it tells you exactly why the gate blocked — so you can diagnose the regression without opening any other tools.

**Steps:**
1. Open `workspace/results.tsv` in a text editor
2. Find one or more rows with verdict `blocked`
3. Read the trigger/reason column

**Expected outcome:** The reason column contains one of these three values:
- `pass-rate-below-watermark` — the percentage of scenarios that passed was too low
- `score-below-best` — the full test score was lower than the best ever seen
- `both` — both conditions failed at the same time

**Fail if:** The reason column is blank, missing, or contains a value other than the three listed above.
