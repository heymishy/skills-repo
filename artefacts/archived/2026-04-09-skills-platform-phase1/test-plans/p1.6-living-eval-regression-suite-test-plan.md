## AC3 split — 1-L6

AC3 original then-clause: "the proposed scenario is reviewed against the outcome-orientation diagnostic before being merged — and a record of the review decision is appended to `workspace/suite.json` in the scenario's `reviewNote` field."

Split assertions:
- **AC3a:** The proposed scenario is reviewed against the outcome-orientation diagnostic before being merged (process gate — verified by PR review checklist and merge policy; no executable test)
- **AC3b:** The `reviewNote` field is populated with the review decision record — tested by U4, U5, and N3

AC3a is a process gate (outcome: the merge did not occur without the diagnostic step) — it is verifiable by inspection of the PR merge policy and the `reviewNote` field content, not by an isolated executable test. The `reviewNote` field content (AC3b) is the observable artefact of AC3a having occurred.

---

## AC6 — outcome-orientation diagnostic

AC6 requires each scenario in `workspace/suite.json` to pass the diagnostic: "if the model behaviour improved past this behaviour, would this scenario still be correct?" This is a human authoring review at the time of P1.6 DoD, not an executable unit test. AC6 is verified at the DoD stage by the platform maintainer reviewing each scenario against the diagnostic. The verification script captures this as a manual confirmation step.

## Test data dependencies

- **Mock gate run output:** Integration tests I2 and I3 require a mock gate run output containing a known `expectedOutcome` string. This is a fixture created at test-authoring time and does not require a live CI run.
- **M4 controlled acceptance test (AC5):** Adding a failing scenario → observing `blocked` → removing it → observing `pass` restoration requires `p1.4-watermark-gate` to be DoD-complete. This acceptance test is noted as a dependency-blocked test and is scheduled after p1.4 and p1.3 are both DoD-complete. It does not block P1.6 DoR.
