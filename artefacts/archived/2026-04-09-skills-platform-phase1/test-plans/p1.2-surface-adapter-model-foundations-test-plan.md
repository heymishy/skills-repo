## Unit Tests

### `git-native-adapter-returns-status-field` (AC1)

- **Verifies:** AC1 — result object contains `status` field
- **Precondition:** Valid `context` object with required fields
- **Action:** Call `execute("git-native", context)`; inspect returned object
- **Expected result:** `result.status` is one of `"pass"`, `"fail"`, or `"error"` — not undefined or null
- **Edge case:** No

### `git-native-adapter-returns-surface-echoed` (AC1)

- **Verifies:** AC1 — result echoes the input surface type
- **Precondition:** Same as above
- **Action:** Call `execute("git-native", context)`; inspect `result.surface`
- **Expected result:** `result.surface === "git-native"`
- **Edge case:** No

### `path-b-selects-git-native-adapter` (AC2)

- **Verifies:** AC2 — Path B declaration `type: "git-native"` selects the git-native adapter
- **Precondition:** A `context.yml` fixture with `surface: { path: "explicit-declaration", type: "git-native" }`
- **Action:** Run the surface adapter resolution function against the fixture
- **Expected result:** Function returns the git-native adapter (or a reference to it); Path A resolution is not triggered — no EA registry lookup occurs
- **Edge case:** No

### `path-b-does-not-trigger-path-a` (AC2)

- **Verifies:** AC2 — Path B resolution short-circuits before any EA registry lookup
- **Precondition:** Same fixture as above; Path A registry lookup function is observable (spy or stub)
- **Action:** Run resolution; assert Path A lookup was not called
- **Expected result:** Path A lookup call count is zero
- **Edge case:** No

### `multi-surface-selects-correct-adapter-per-type` (AC3)

- **Verifies:** AC3 — multi-surface declaration selects the correct adapter for each type
- **Precondition:** A `context.yml` fixture with two surface types declared (e.g. `["git-native", "manual"]`)
- **Action:** Run Path B resolution against the fixture; collect the selected adapter for each declared type
- **Expected result:** `git-native` → git-native adapter; `manual` → manual adapter; no cross-selection
- **Edge case:** Yes — also test with an unsupported surface type in the list: resolution returns an error for that type without affecting the valid types

## Integration Tests

### `git-native-adapter-end-to-end-result-contract` (AC1)

- **Verifies:** AC1 end-to-end — full adapter invocation returns a complete, conforming result object
- **Components involved:** `execute` function, git-native adapter, result schema
- **Precondition:** A valid `context` object and `surface: "git-native"`
- **Action:** Call `execute("git-native", context)` through the full invocation path; validate result against schema
- **Expected result:** Result contains: `status` (enum), `surface` (string = "git-native"), `findings` (array), `trace` (non-empty string), `adapterVersion` (string) — all fields present and typed correctly

### `multi-surface-integration-all-adapters-invoked` (AC3)

- **Verifies:** AC3 — multi-surface resolution invokes the correct adapter for each type, end-to-end
- **Components involved:** Path B resolver, all declared adapters
- **Precondition:** `context.yml` fixture with two valid surface types
- **Action:** Run full resolution and adapter selection; assert both adapters are invoked
- **Expected result:** Two adapter invocations; each returns a result with the correct `surface` value matching the declaration

### `context-yml-valid-for-p11-and-p12` (AC6)

- **Verifies:** AC6 — the shared Path B `context.yml` format is accepted by both P1.1 assembly and P1.2 resolution
- **Components involved:** P1.1 assembly function, P1.2 resolution function, shared `context.yml` fixture
- **Precondition:** A `context.yml` file conforming to the agreed Path B format (shared schema)
- **Action:** (1) Pass the file to P1.1's assembly/distribution function; assert it is accepted without error. (2) Pass the file to P1.2's surface resolution function; assert it selects the correct adapter.
- **Expected result:** Both functions accept the same `context.yml` without modification — one file, two consumers, no conflict

## Design Review — AC4 and AC5

> These items cannot be verified by automated tests. They are assessed by a designated reviewer inspecting the interface implementation against the criteria below. The reviewer records Pass/Fail in the verification script.

### Design review item 1 — AC4: Path A extensibility

**Reviewer instruction:** Open the `execute` function implementation. Check that:
1. The function signature is `execute(surface, context) → result` — it does not have a `resolutionPath` or `registryUrl` parameter baked in
2. The `surface` parameter is resolved *before* the function body — the `execute` body does not branch on whether to use Path A or Path B internally
3. Adding Path A resolution means providing a surface type string to `execute` via a different resolver — not modifying `execute` itself

**Pass criterion:** A developer can implement Path A by writing a new resolver function that calls an EA registry and returns a surface type string, then passes that string to the existing `execute(surface, context)` call — with no changes to `execute`'s signature or body.

### Design review item 2 — AC5: Phase 2 surface types not foreclosed

**Reviewer instruction:** Open the interface definition for `SurfaceType`. Check that:
1. The type is an extensible string union or string type — not a closed enum that would require a code change in the `execute` function to add a new value
2. The `execute` body does not contain branches like `if surface === "git-native" { ... } else { throw "unsupported" }` that would need modification for each new adapter
3. Each surface adapter is a separate, independently-addable module — adding `iac` as a new adapter does not require touching the `git-native` adapter or the `execute` function

**Pass criterion:** All five Phase 2 surface types (IaC, SaaS-API, SaaS-GUI, M365-admin, manual) can be added by writing a new adapter module and registering it — without modifying the interface contract or any existing adapter.

## NFR Tests

### `nfr-path-b-resolution-completes-within-1s`

- **Verifies:** NFR — Performance: Path B surface type resolution must complete within 1 second
- **Precondition:** A valid `context.yml` fixture on disk
- **Action:** Time the resolution function from file-read start to adapter-selected output; assert elapsed time < 1000ms
- **Expected result:** Resolution completes within 1 second on a modern CI runner

### `nfr-adapter-result-no-credential-values`

- **Verifies:** NFR — Security: git-native adapter must not log or include credential values in the result
- **Precondition:** A `context` object containing a secrets store reference (not an actual credential value)
- **Action:** Invoke `execute("git-native", context)`; scan the full result object JSON for credential-like patterns (`ghp_`, `Bearer `, `password:`, `token:`)
- **Expected result:** Zero matches in the result object

### `nfr-result-contains-surface-and-adapter-version`

- **Verifies:** NFR — Audit: `result` object includes `surface` and `adapterVersion`
- **Precondition:** Standard git-native invocation
- **Action:** Call `execute("git-native", context)`; assert `result.surface` and `result.adapterVersion` are both non-empty strings
- **Expected result:** Both fields present and non-empty
