### Credential handling — AC circuit break

The CI configuration file must reference secret names (e.g. `${{ secrets.COPILOT_TOKEN }}`) and never inline credential values. The credential-pattern test (see NFR tests) scans the committed file. Any match is an automatic test failure regardless of other test outcomes.

## Unit Tests

### `trace-inprogress-written-before-evaluation` (AC2)

- **Verifies:** AC2 — `inProgress` trace entry is written before any evaluation check runs
- **Precondition:** A writable `workspace/traces/` directory; gate run function instrumented to capture write order
- **Action:** Call the gate execution function with a synthetic PR event payload; intercept write calls; assert that the `inProgress` write occurs before any check evaluation call
- **Expected result:** `inProgress` write is the first I/O operation; no evaluation function is called before it
- **Edge case:** No

### `trace-completed-follows-inprogress` (AC3)

- **Verifies:** AC3 — `completed` trace entry directly follows `inProgress` in the trace file
- **Precondition:** A fixture trace file containing only an `inProgress` entry; gate run completes successfully
- **Action:** Run the full gate against a synthetic PR payload; read the resulting trace file
- **Expected result:** File contains exactly two entries in order: `inProgress` then `completed`; no other entries between them
- **Edge case:** Yes — test also verifies that an error mid-run still produces a `completed` entry (with `status: failed`) rather than leaving the trace with only `inProgress`

### `trace-ci-trigger-metadata` (AC5b)

- **Verifies:** AC5b — CI-triggered run writes `trigger: ci` in trace metadata
- **Precondition:** A synthetic CI event payload (not a manual invocation)
- **Action:** Run the gate with the synthetic CI payload; parse the resulting `completed` trace entry
- **Expected result:** `trigger` field equals `"ci"` — not `"manual"` or undefined
- **Edge case:** No

## Integration Tests

### `ci-trigger-starts-gate-without-manual-action` (AC1)

- **Verifies:** AC1 — gate starts from `pull_request` event, not `workflow_dispatch`
- **Components involved:** CI configuration file, CI runtime (GitHub Actions or equivalent local simulation)
- **Precondition:** A synthetic `pull_request` event payload conforming to GitHub Actions event schema
- **Action:** Invoke the CI workflow with a simulated `pull_request` event; confirm the gate job starts; inspect the event metadata captured in the run
- **Expected result:** Workflow run log shows `event: pull_request`; no manual trigger required; gate invocation is recorded in the trace with `trigger: ci`
- **Note:** If CI simulation is not available in the test environment, this test is performed manually (see verification script Scenario 1)

### `gate-writes-inprogress-then-completed` (AC2, AC3)

- **Verifies:** AC2 and AC3 end-to-end — full gate run against a synthetic PR produces both trace entries in correct order
- **Components involved:** Gate runner, `workspace/traces/` write layer, trace schema
- **Precondition:** Empty `workspace/traces/` directory; valid synthetic PR event payload
- **Action:** Run the full gate against the payload; read the trace file produced
- **Expected result:** Trace file contains `inProgress` entry (timestamp earlier than evaluation result), followed by `completed` entry with `status`, `verdict`, `traceHash`, `prRef`, `commitSha`, and `trigger: ci`

### `gate-verdict-surface-to-pr` (AC4 integration proxy)

- **Verifies:** AC4 — gate result is surfaced to the PR via the CI check mechanism
- **Components involved:** Gate runner, CI status reporting step, GitHub Checks API (or equivalent)
- **Precondition:** A completed gate run with a known verdict; CI configuration includes a step to post the result
- **Action:** After a gate run completes, verify that the CI step responsible for posting the verdict executed and produced output (check run, PR comment, or status check)
- **Expected result:** The CI log shows successful execution of the verdict-posting step; the step output includes the verdict value and trace hash

### `two-trace-entries-on-ci-trigger` (AC5a)

- **Verifies:** AC5a — when a CI-triggered gate fires on a PR that also has a prior manual trace entry, both entries exist independently in `workspace/traces/`
- **Components involved:** `workspace/traces/` directory, gate runner
- **Precondition:** `workspace/traces/` contains a pre-existing manual trace entry for the same story (fixture); CI-triggered gate run executes
- **Action:** Run the CI-triggered gate; read all trace entries for the story
- **Expected result:** Two trace entries found — one with `trigger: manual`, one with `trigger: ci`; the CI run did not overwrite or replace the manual run entry

## NFR Tests

### `nfr-gate-completes-within-10-minutes`

- **Verifies:** NFR — Performance: gate run must complete within 10 minutes
- **Note:** This is tested in CI as a timeout check. The CI job configuration must set a timeout of 10 minutes. If the gate times out, the CI system marks the run as failed — equivalent to a governance check failure.
- **Precondition:** Gate job in CI configuration has `timeout-minutes: 10` (or equivalent)
- **Action:** Read the CI configuration file; assert that a timeout of ≤10 minutes is configured on the gate job
- **Expected result:** Timeout value found and is ≤600 seconds / 10 minutes
- **Edge case:** No

### `nfr-no-credential-values-in-ci-config`

- **Verifies:** NFR — Security: CI configuration must reference credential names only, no credential values
- **Precondition:** The committed CI configuration YAML file
- **Action:** Scan the full file content for patterns: `ghp_[A-Za-z0-9]{36}`, `Bearer [A-Za-z0-9]{20,}`, `token: [A-Za-z0-9]{16,}`, `password: [^\$]`
- **Expected result:** Zero matches
- **Edge case:** Yes — test must use a fixture that would trigger a false positive if it matched, to avoid a vacuous pass on an empty or minimal file

### `nfr-trace-audit-fields-present`

- **Verifies:** NFR — Audit: every gate run trace entry must have ISO timestamps, PR number + commit SHA, and trigger type
- **Precondition:** A completed gate run trace file
- **Action:** Parse `completed` entry; assert presence and non-empty values for: `startedAt` (ISO 8601), `completedAt` (ISO 8601), `prRef` (non-empty), `commitSha` (40-char hex), `trigger` (`ci` or `manual`)
- **Expected result:** All five fields present and matching format constraints
