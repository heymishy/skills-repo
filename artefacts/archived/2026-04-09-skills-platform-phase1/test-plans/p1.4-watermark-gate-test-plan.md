## Unit Tests

### `first-run-creates-results-tsv-with-baseline` (AC1)

- **Verifies:** AC1 — first gate run creates `results.tsv` with a single baseline row
- **Precondition:** No `results.tsv` file exists in `workspace/`
- **Action:** Run the gate against a minimal synthetic suite fixture; check for file creation
- **Expected result:** `results.tsv` is created; it contains exactly one row; the row has: ISO timestamp, non-empty skill-set hash, surface type, pass rate (0–1 float), full test score (integer), and verdict value `baseline`
- **Edge case:** No

### `first-run-baseline-row-schema` (AC1)

- **Verifies:** AC1 — schema completeness: all six required columns present and correctly typed
- **Precondition:** `results.tsv` created by the first-run test above (or an identical fixture)
- **Action:** Parse the TSV row; assert presence and type of: timestamp (ISO 8601), skillSetHash (non-empty string), surfaceType (non-empty string), passRate (float 0–1), fullTestScore (non-negative integer), verdict (`baseline`)
- **Expected result:** All six columns present and correctly typed
- **Edge case:** No

### `gate-pass-when-at-watermark` (AC2)

- **Verifies:** AC2 — gate returns `pass` when both pass rate and score exactly equal the watermark
- **Precondition:** `results.tsv` fixture with baseline row: passRate=0.9, fullTestScore=18; incoming result: passRate=0.9, fullTestScore=18
- **Action:** Run the gate comparison logic against the fixture
- **Expected result:** Verdict is `pass`; a new row with `pass` is appended to `results.tsv`
- **Edge case:** Yes — equal-to threshold must be treated as pass (≥, not >)

### `gate-pass-when-above-watermark` (AC2)

- **Verifies:** AC2 — gate returns `pass` when results improve on the watermark
- **Precondition:** Same fixture as above; incoming result: passRate=1.0, fullTestScore=20
- **Action:** Run gate comparison; assert verdict
- **Expected result:** Verdict is `pass`; new row appended
- **Edge case:** No

### `gate-blocks-when-pass-rate-drops` (AC3)

- **Verifies:** AC3 — gate blocks without human step when pass rate drops below watermark
- **Precondition:** `results.tsv` fixture with baseline: passRate=0.9, fullTestScore=18; incoming: passRate=0.7, fullTestScore=18
- **Action:** Run gate comparison
- **Expected result:** Verdict is `blocked`; no human review step invoked; new row appended with reason field set
- **Edge case:** No

### `gate-blocks-when-full-score-drops` (AC3)

- **Verifies:** AC3 — gate blocks when full test score drops, even if pass rate holds
- **Precondition:** `results.tsv` fixture: passRate=0.9, fullTestScore=18; incoming: passRate=0.9, fullTestScore=15
- **Action:** Run gate comparison
- **Expected result:** Verdict is `blocked`; reason field indicates score trigger
- **Edge case:** No

### `blocked-row-reason-pass-rate` (AC6)

- **Verifies:** AC6 — blocked row records the specific trigger condition
- **Precondition:** Same fixture as `gate-blocks-when-pass-rate-drops`
- **Action:** Parse the blocked row appended to `results.tsv`; inspect the trigger/reason field
- **Expected result:** Field value is `pass-rate-below-watermark`
- **Edge case:** No

### `blocked-row-reason-score` (AC6)

- **Verifies:** AC6 — trigger field correctly identifies score drop
- **Precondition:** Same fixture as `gate-blocks-when-full-score-drops`
- **Action:** Parse the blocked row; inspect the trigger field
- **Expected result:** Field value is `score-below-best`
- **Edge case:** No

### `blocked-row-reason-both` (AC6)

- **Verifies:** AC6 — trigger field records `both` when both conditions fail
- **Precondition:** `results.tsv` fixture: passRate=0.9, fullTestScore=18; incoming: passRate=0.6, fullTestScore=12
- **Action:** Run gate; parse blocked row trigger field
- **Expected result:** Field value is `both`
- **Edge case:** No

## Integration Tests

### `gate-pass-appends-only-one-row` (AC2)

- **Verifies:** AC2 (end-to-end) — gate run appends exactly one new row on pass; no rows deleted or modified
- **Components involved:** Gate runner, `workspace/results.tsv`
- **Precondition:** `results.tsv` fixture with two existing rows (baseline + one prior pass); incoming score above watermark
- **Action:** Run full gate; read `results.tsv` after run
- **Expected result:** File now has three rows; first two rows are unchanged; third row is new with verdict `pass`; row count is the only change (append-only verified)

### `gate-blocks-no-human-input-required` (AC3)

- **Verifies:** AC3 end-to-end — gate blocks and no human confirmation step is triggered
- **Components involved:** Gate runner, CI pipeline integration
- **Precondition:** `results.tsv` fixture with a watermark above the incoming score
- **Action:** Run full gate; observe execution flow; confirm the gate exits with a non-zero status code without pausing for input
- **Expected result:** Gate exits `blocked` immediately; process exit code is non-zero; no prompt or wait state triggered

### `gate-blocked-verdict-visible-in-pr` (AC4)

- **Verifies:** AC4 — blocked verdict appears in the PR interface when viewed on GitHub
- **Components involved:** Gate runner, CI status reporting step, GitHub PR interface
- **Precondition:** A gate run that returns `blocked` has completed in CI on an open PR
- **Action:** Open the PR on GitHub; observe the checks section
- **Expected result:** Blocked verdict is visible in the PR checks section, with the reason (which condition triggered the block) — no other tool required

### `controlled-acceptance-failing-scenario-then-removal` (AC5)

- **Verifies:** AC5 — the controlled acceptance test: add failure, observe block, remove, observe pass
- **Components involved:** `workspace/suite.json`, gate runner, `workspace/results.tsv`, CI reporting
- **Precondition:** A passing baseline row exists in `results.tsv`; `suite.json` is in passing state
- **Action:** (1) Add a deliberately failing scenario to `suite.json` that lowers the pass rate below the watermark. (2) Run the gate. Assert verdict is `blocked`, reason is captured in `results.tsv`. (3) Remove the failing scenario. (4) Run the gate again. Assert verdict is `pass`. (5) Inspect `results.tsv` — all rows should still be present (the blocked row from step 2 is not deleted).
- **Expected result:** Step 2 verdict: `blocked`. Step 4 verdict: `pass`. Step 5: `results.tsv` contains all rows including the blocked run record — no rows removed.

## NFR Tests

### `nfr-gate-completes-within-2-minutes`

- **Verifies:** NFR — Performance: gate verdict computation must complete within 2 minutes
- **Note:** Tested as a timeout constraint in the CI job configuration (same approach as p1.3).
- **Precondition:** CI job configuration for the watermark gate component
- **Action:** Assert the step or job timeout is ≤120 seconds
- **Expected result:** Timeout configured at or below 2 minutes

### `nfr-results-tsv-no-credentials`

- **Verifies:** NFR — Security: `results.tsv` contains only scores, hashes, and timestamps — no credentials or PII
- **Precondition:** A populated `results.tsv` fixture (multiple rows including baseline, pass, blocked)
- **Action:** Scan the full file for credential-like patterns: `ghp_`, `Bearer `, `password`, any value matching a PAN pattern (`\d{12,19}`)
- **Expected result:** Zero matches

### `nfr-results-tsv-append-only-no-modifications`

- **Verifies:** NFR — Audit: rows are never deleted or modified, only appended
- **Precondition:** `results.tsv` with two rows; run three gate operations (pass, blocked, pass again)
- **Action:** After all three runs, read the file; verify row count is 5; verify the content of the original two rows is byte-for-byte unchanged
- **Expected result:** Five rows; first two rows identical to original; last three correspond to the new gate runs
