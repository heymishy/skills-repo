## User Story

As a **platform maintainer**,
I want a structured `workspace/state.json` file written at each phase boundary, and a `/checkpoint` operator convention documented in `copilot-instructions.md`,
So that I can resume a session interrupted at any phase boundary in a new session using only the state file, without verbal priming or re-running any prior stage.

## Acceptance Criteria

**AC1:** Given a new session is opened with no verbal priming after a natural session boundary, when the agent reads `workspace/state.json` via the SESSION START hook declared in `copilot-instructions.md`, then the correct current pipeline phase is identified, no prior work is re-run or re-asked, and no decision already recorded in `state.json` is re-surfaced as a question.

**AC2:** Given the SESSION START hook is triggered, when `state.json` contains a `currentPhase` value and a populated cycle block for that phase, then the agent reports: the last completed phase, any in-progress story execution, and any pending improvement proposals or human input items â€” before asking whether to resume or restart.

**AC3:** Given the operator invokes `/checkpoint` at a mid-phase point during a definition or earlier phase, when the current phase's section of `state.json` is written, then (a) the session terminates within 60 seconds without hanging; (b) all required fields for the current phase section are present and non-null; (c) a new session opened from that `state.json` identifies the correct resume point without operator verbal description of where the session ended.

**AC4:** Given `copilot-instructions.md` contains the SESSION START block, when a new session begins from a clean terminal (no history), then the agent's first action is to check for `workspace/state.json`, read it if it exists, and report its contents to the operator before proceeding with any task.

**AC5:** Given `state.json` is written at the end of a phase boundary skill (e.g. `/benefit-metric` complete), when the file is inspected, then it contains a `lastUpdated` ISO date, the `currentPhase` string, and the cycle block for each completed phase populated with at minimum `status` and `artefact` path.

**AC6:** Given the `/checkpoint` convention documentation in `copilot-instructions.md`, when the documentation is read, then it explicitly states that `/checkpoint` must be invoked before the session reaches compaction pressure â€” not at it â€” with enough context headroom for the write to complete before compaction fires.
