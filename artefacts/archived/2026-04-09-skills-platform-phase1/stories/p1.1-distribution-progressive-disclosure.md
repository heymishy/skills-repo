# Story: Implement versioned skill distribution and progressive skill disclosure

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-3-distribution-surface-model.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **dev / engineer (consuming squad)**,
I want to load skills from a versioned platform package without forking the skills-repo, and have platform skill updates reach my assembled `copilot-instructions.md` without any merge action on my part,
So that my squad can adopt the skills platform without severing the update channel from the platform as it evolves.

## Benefit Linkage

**Metric moved:** M1 — Update channel latency; MM1 — Solo operator outer loop
**How:** This story is the direct delivery of M1 — the distribution mechanism either exists and the acceptance test passes, or it does not and P1.1 cannot be marked done. It also reduces MM1 session overhead by delivering the progressive disclosure pattern (phase-sequenced skill loading), which the dogfood finding identified as responsible for a 27% fixed overhead contribution.

## Architecture Constraints

- The distribution model choice (push vs pull) is an open design decision at story entry — it must be resolved and logged in `decisions.md` before implementation begins. Push model targets ≤72 hours; pull model targets ≤one session open. Both service M1 but with different SLAs and implementation approaches.
- The `context.yml` MCP credentials section must not be modified by this story — governed by product constraint #12. The distribution mechanism references credential names from the secrets store; it never reads or writes credential values.
- Progressive skill disclosure depends on phase declarations being present in every SKILL.md — the authoring standard (outcome-oriented instructions, phase declaration requirement) is defined in `ref-skills-platform-standards-model.md`; this story must enforce it as a PR review requirement for all skill files touched.

## Dependencies

- **Upstream:** None — this story is the entry point for Epic 3 with no predecessors
- **Downstream:** `p1.2-surface-adapter-model-foundations` references the assembled `copilot-instructions.md` base layer and the `context.yml` Path B structure; it can proceed in parallel but depends on the assembly format being stable

## Acceptance Criteria

**AC1:** Given a consuming squad repository that has not forked the skills-repo, when a squad engineer opens a session with `copilot-instructions.md` assembled using the platform distribution mechanism, then all core platform skills are available via `/load [skill-name]` without any files from the skills-repo being copied into the squad repo.

**AC2:** Given a platform maintainer commits a controlled skill change to the skills-repo, when one distribution cycle elapses (≤72 hours for push model; ≤one session open for pull model — whichever matches the implemented mechanism), then the change is present in the consuming squad's assembled `copilot-instructions.md` with zero merge steps authored or resolved by the squad — evidenced by the squad's `copilot-instructions.md` showing the changed skill content without a merge commit.

**AC3:** Given a session is opened with the assembled `copilot-instructions.md` base layer, when the session is at the discovery phase, then only discovery-phase and session-start skills are loaded into the context — implementation-phase skills (tdd, implementation-plan, subagent-execution, verify-completion) are not loaded until the inner loop phase is declared.

**AC4:** Given the progressive disclosure pattern is active, when an operator types `/load [skill-name]` for a skill that is not loaded in the current phase context, then the skill is loaded and available for the remainder of the session without requiring a session restart.

**AC5:** Given the assembled `copilot-instructions.md`, when it is inspected, then it is composed from at minimum three layers in declared order: (1) core platform layer (from versioned skills-repo package), (2) domain layer (from domain repo if present, otherwise absent), (3) squad layer (from squad repo `copilot-instructions.md` if present); and the composition order is documented in the file header.

**AC6:** Given the distribution mechanism acceptance test at P1.1 DoD, when the acceptance test is run, then the following are recorded: change reference, distribution mechanism type (push/pull), elapsed time from commit to availability in squad context, change present in assembled context (Y/N), squad merge action required (Y/N) — and the M1 minimum validation signal is satisfied (change present, no squad merge step).

## Out of Scope

- EA registry integration — the distribution mechanism operates without the EA registry; Path A surface type resolution is Phase 2 and must not be foreclosed by this story's design decisions
- Automated drift detection (detecting that a squad's assembled context has diverged from the platform baseline without a distribution cycle) — Phase 3
- Non-git-native surface adapter implementations — Phase 2; this story establishes the base layer, not surface-specific adapters
- Squad override model conflict resolution tooling — squads can override core layer settings in their squad layer; this story documents the override model but does not build tooling to detect or flag conflicts
- Token usage telemetry for the progressive disclosure pattern — the 27% fixed overhead finding is a dogfood observation; formal token telemetry is not in scope

## NFRs

- **Performance:** Assembled `copilot-instructions.md` base layer must not exceed 8,000 tokens in the progressive disclosure pattern's session-start state (outer loop skills only), measured against a representative set of 10 core skills at their deployed SKILL.md sizes
- **Security:** Distribution mechanism configuration must reference credential names only — no credential values or PAT tokens in any committed configuration file (product constraint #12)
- **Audit:** The assembly composition order and source versions must be readable from the assembled `copilot-instructions.md` file header — a reviewer must be able to identify which platform version a session was running against
