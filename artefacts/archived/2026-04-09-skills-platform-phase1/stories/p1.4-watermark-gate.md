# Story: Implement watermark gate for eval regression detection

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-2-core-loop-infrastructure.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want a watermark gate that tracks eval suite scores in `workspace/results.tsv` and automatically blocks a run when the score drops below the best previously recorded score for the same skill/surface combination,
So that regressions in harness quality are surfaced automatically before human review rather than slipping through undetected.

## Benefit Linkage

**Metric moved:** M4 — Watermark gate blocks regression
**How:** This story is the direct delivery of M4. Both M4 sub-conditions (automatic rejection with no human review step, and rejection visible in PR) are acceptance criteria for this story. The controlled acceptance test — add a failing scenario, run the gate, observe rejection, remove the scenario — is the M4 delivery evidence.

## Architecture Constraints

- `workspace/results.tsv` is written by this gate and read by the CI gate (P1.3); the file format must be agreed between this story and P1.3's implementation — the schema (timestamp, skill-set hash, surface type, eval suite pass rate, full test score, gate verdict) is defined in the discovery MVP scope (P1.4 entry) and must not deviate without logging a decision
- No persistent agent runtime — the gate runs in CI like P1.3; same platform constraint applies (Product constraint #11)

## Dependencies

- **Upstream:** `p1.3-assurance-agent-ci-gate` — the CI trigger mechanism exists before the watermark gate is integrated into it; the gate can be developed and tested standalone, but the AC for PR visibility requires P1.3's CI runner to be in place
- **Upstream:** `p1.6-living-eval-regression-suite` — the watermark gate scores against `workspace/suite.json`; P1.6 must be DoD-complete before the gate has a real suite to score (the gate can be integration-tested against a minimal fixture suite during development)
- **Downstream:** Nothing is blocked by this story at Phase 1

## Acceptance Criteria

**AC1:** Given `workspace/results.tsv` does not yet exist, when the watermark gate runs for the first time against a skill/surface combination, then a new `results.tsv` file is created with one row containing: ISO timestamp, skill-set hash, surface type, eval suite pass rate, full test score, and gate verdict of `baseline` — establishing the first watermark with no rejection.

**AC2:** Given a prior watermark row exists in `results.tsv` for a skill/surface combination, when the gate runs and the current eval suite pass rate is at or above the recorded watermark pass rate AND the current full test score is at or above the recorded best score for that combination, then the gate verdict is `pass` and a new row is appended to `results.tsv`.

**AC3:** Given a prior watermark row exists for a skill/surface combination, when the gate runs and either the eval suite pass rate OR the full test score is below the recorded watermark for that combination, then the gate verdict is `blocked` — with no human review step triggered — and a new row with verdict `blocked` is appended to `results.tsv` recording the trigger condition (pass rate dropped / score dropped).

**AC4:** Given the gate returns a `blocked` verdict on a PR run, when a reviewer opens the PR, then the blocked verdict and the reason (score dropped below watermark for this skill/surface combination) are visible in the PR interface without opening a separate tool.

**AC5:** Given a controlled acceptance test where a deliberately failing scenario is added to `workspace/suite.json`, when the gate runs against that modified suite, then the gate produces a `blocked` verdict (AC3 applies), the rejection is visible in the PR (AC4 applies), and after removing the deliberately failing scenario and re-running, the gate returns to `pass` — leaving no permanent artefact from the acceptance test other than the `results.tsv` rows.

**AC6:** Given the two-check gate logic (eval suite pass rate ≥ threshold AND full score ≥ best), when either condition fails independently, then the `results.tsv` blocked row records which condition triggered the block — `pass-rate-below-watermark`, `score-below-best`, or `both` — enabling diagnosis without opening the full run log.

## Out of Scope

- Automated remediation when a regression is detected — the gate blocks and surfaces the reason; fixing the regression is a human decision
- Rolling-window watermark or percentile watermark — Phase 1 uses the best recorded score (high watermark model); statistical variants are Phase 2
- Per-story watermarks — the watermark is per skill/surface combination, not per story
- Alert routing (PagerDuty, Slack, email) on regression detection — Phase 3 observability
- Watermark decay or planned-regression support — if a scenario is intentionally removed from the suite, the watermark must be manually reset; automated decay is not in scope

## NFRs

- **Performance:** Gate verdict computation (reading `results.tsv`, scoring the suite, comparing watermarks) must complete within 2 minutes — the total P1.3 gate run budget is 10 minutes; the watermark gate is one component
- **Security:** `results.tsv` contains no credentials, secrets, or PII — only scores, hashes, and timestamps; it is safe to commit to the repository
- **Audit:** Every gate run appends a row to `results.tsv` — rows are never deleted or modified, only appended. `results.tsv` is the immutable run history.
