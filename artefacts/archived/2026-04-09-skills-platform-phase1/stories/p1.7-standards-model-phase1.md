# Story: Implement standards model Phase 1 ‚Äî three anchor disciplines, standards/index.yml, POLICY.md floors

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-4-standards-eval-compliance.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want three anchor discipline standards (software engineering, security engineering, quality assurance) authored as composable, hash-verifiable files with a `standards/index.yml` routing table and POLICY.md floors,
So that the assurance agent can inject the correct standards into an inner loop story trace and the T3M1 trace readability test can be validated against real standards content rather than placeholder stubs.

## Benefit Linkage

**Metric moved:** M3 ‚Äî Standards injection in trace; T3M1 ‚Äî Trace readability for risk review
**How:** M3 requires standardsInjected hashes in a real inner loop trace that match the hash of the standards files at the PR's commit SHA. P1.3 (Epic 2) provides the injection mechanism (trace write); this story provides the content. Without the standards files present and indexed, the injection mechanism has nothing to hash and inject. T3M1 also depends on the trace being populated with real standards references for a non-engineering reviewer to verify.

## Architecture Constraints

- `standards/index.yml` schema must adopt the structure illustrated in `ref-skills-platform-standards-model.md` as the Phase 1 starting point ‚Äî the schema is provisional and logged as a decision; it must be extensible to accommodate 8 additional disciplines and domain-tier entries in Phase 2 without a breaking change. The `disciplines:` ‚Üí `core:`, `policy-floor:`, `surface-variants:` structure is the canonical starting schema.
- Standards files must be outcome-oriented, not workaround-oriented ‚Äî the authoring principle from `ref-skills-platform-standards-model.md` applies. A standard that encodes a current model weakness will produce staleness signal and become dead weight; a standard that encodes required outcomes ages well. Each standards file must pass the diagnostic checklist before merge.
- POLICY.md floor pattern: squads cannot weaken POLICY.md floors defined at a higher tier. Phase 1 establishes core-tier floors only; the enforcement model (preventing weakening at lower tiers) is declarative at Phase 1 and tooled at Phase 2.
- Hash computation: the assurance agent computes hashes of standards files at gate time against the PR's commit SHA ‚Äî not self-reported by the dev agent. Standards file content must be committed, not generated at gate time.

## Dependencies

- **Upstream:** `p1.3-assurance-agent-ci-gate` ‚Äî the injection mechanism exists before the standards content; this story can be written and reviewed before P1.3 is DoD-complete, but the M3 acceptance test requires P1.3's CI gate to hash and inject the standards files into a real trace
- **Upstream:** `p1.2-surface-adapter-model-foundations` ‚Äî `standards/index.yml` uses surface type as a routing key (the `surface-variants:` field in the index schema); the surface type vocabulary from P1.2 is the input to the routing table's surface-variant entries
- **Downstream:** `p1.8-model-risk-documentation` ‚Äî `MODEL-RISK.md` references the standards injection mechanism as one of the eight audit questions; P1.8 cannot be completed until P1.7 is DoD-complete

## Acceptance Criteria

**AC1:** Given the `standards/` directory in the platform repository, when P1.7 is DoD-complete, then three standards files exist at `standards/software-engineering/core.md`, `standards/security-engineering/core.md`, and `standards/quality-assurance/core.md` ‚Äî each with a human-readable title, owning discipline, last-reviewed date, and at least five outcome-oriented requirements in the format "MUST / SHOULD / MAY [observable outcome]".

**AC2:** Given each anchor discipline's `core.md` exists, when the corresponding `POLICY.md` file is reviewed at `standards/[discipline]/POLICY.md`, then it declares the minimum floor requirements that no domain or squad override can relax ‚Äî with each floor item stated as a binary requirement (pass/fail, not a gradient) so the assurance agent can evaluate it programmatically.

**AC3:** Given `standards/index.yml` exists in the platform repository, when it is read by the assurance agent's standards injection step, then it returns the correct `core.md` and `POLICY.md` paths for each of the three anchor discipline IDs (`software-engineering`, `security-engineering`, `quality-assurance`) ‚Äî and the schema is parseable without modification after adding a fourth discipline entry.

**AC4:** Given a real inner loop story runs through the P1.3 CI gate after P1.7 is DoD-complete, when the gate run completes and the trace file is inspected, then the `standardsInjected` array contains at minimum one entry with: discipline ID, file path, and hash of the standards file at the PR's commit SHA ‚Äî computed by the assurance agent, not self-reported by the dev agent.

**AC5:** Given the `standardsInjected` array in the trace contains entries for all three anchor disciplines for a story tagged as legitimately requiring all three, when the platform maintainer recomputes the hash of each standards file at the PR's commit SHA, then the recomputed hashes match the values in the `standardsInjected` array ‚Äî validating that the injected standards were the ones in effect at the time of the gate run.

**AC6:** Given each standards file authored in this story, when it is reviewed using the outcome-orientation diagnostic checklist from `ref-skills-platform-standards-model.md`, then each requirement passes the diagnostic ("if the model improved past this behaviour, would this requirement still be correct?") ‚Äî and the review outcome is recorded in the standards file's `reviewedBy` and `reviewDate` fields.

## Out of Scope

- The eight remaining discipline standards beyond the three anchor disciplines ‚Äî Phase 2; this story delivers the three anchor disciplines and the extensible index schema only
- Domain-tier and squad-tier standards layers ‚Äî Phase 1 establishes core tier only; domain extension and squad specification are Phase 2
- Automated POLICY.md floor enforcement (tooling that prevents a squad override from weakening a core floor at runtime) ‚Äî Phase 2; Phase 1 enforcement is by PR review
- Standards file versioning and change history beyond git history ‚Äî the git commit log is the standards audit trail at Phase 1; separate versioning tooling is Phase 2
- Surface-variant standards files for non-git surfaces ‚Äî `standards/index.yml` includes the `surface-variants:` key for future use; no surface-variant files are authored at Phase 1 (git-native is the only implemented surface)

## NFRs

- **Performance:** Standards file injection (reading three files and computing their hashes) must complete within 30 seconds as part of the P1.3 CI gate run
- **Security:** Core security engineering `POLICY.md` floor must include at minimum: no high or critical OWASP findings without an accepted risk entry; secrets never committed to source control. These are non-negotiable floors inherited from the platform design principles.
- **Audit:** Each standards file must include `lastReviewedBy` and `lastReviewedDate` fields ‚Äî a standards file without a review record cannot be injected into a trace that will be used as a T3M1 audit evidence artefact
