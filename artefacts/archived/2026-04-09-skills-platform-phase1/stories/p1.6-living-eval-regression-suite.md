# Story: Implement workspace/suite.json living eval regression suite

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-4-standards-eval-compliance.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want a living `workspace/suite.json` regression suite where each real failure pattern discovered in operation can be added as a scenario, and where once a scenario is added it must pass on every subsequent assurance gate run,
So that harness quality is measured against actual failure patterns rather than theoretical ones, and regressions are caught automatically before they reach production.

## Benefit Linkage

**Metric moved:** M4 — Watermark gate blocks regression
**How:** The watermark gate (P1.4) scores against `workspace/suite.json`. Without a populated suite, the gate has nothing to score — a watermark established against an empty suite is meaningless. This story populates the suite with at least one real scenario per core skill, making the watermark a genuine regression anchor.

## Architecture Constraints

- Eval scenarios must be outcome-oriented, not procedure-oriented — this is the authoring principle defined in `ref-skills-platform-standards-model.md` (SKILL.md authoring principles section). A scenario that passes only when the agent follows a specific procedure is workaround-oriented and will become dead weight as the model improves. Scenarios must test whether the required outcome was achieved, not whether a specific sequence was followed. This constraint applies to the suite growth mechanic: proposed additions must pass the outcome-orientation diagnostic before being merged.
- `workspace/suite.json` schema must be compatible with the watermark gate's scoring mechanism (P1.4) — the schema fields (`taskId`, `description`, `skillSurfaceCombination`, `expectedOutcome`, `failurePatternGuarded`) are defined in the discovery MVP scope (P1.6 entry) and must not deviate without logging a decision

## Dependencies

- **Upstream:** `p1.3-assurance-agent-ci-gate` — the suite is scored by the assurance gate; the gate must exist before the suite's regression guarantee can be validated in CI. The suite can be written and structurally validated before P1.3 is DoD-complete, but the regression guarantee AC (AC5) requires the CI gate to be running.
- **Upstream:** `p1.4-watermark-gate` — the watermark is scored against this suite; the watermark gate must exist before the M4 acceptance test can be run. Suite structure goes first; watermark gate integration follows.
- **Downstream:** `p1.7-standards-model-phase1` — once standards are live, scenarios testing standards injection correctness can be added via the growth mechanic; the suite must accommodate those additions without schema revision

## Acceptance Criteria

**AC1:** Given `workspace/suite.json` does not yet exist, when the initial suite is authored, then the file contains at least one scenario per core skill currently in the skills platform — where each scenario has the required fields: `taskId` (unique string), `description` (human-readable), `skillSurfaceCombination` (skill name + surface type), `expectedOutcome` (observable outcome statement, outcome-oriented not procedure-oriented), and `failurePatternGuarded` (description of the regression class the scenario guards against).

**AC2:** Given a scenario exists in `workspace/suite.json`, when the assurance gate runs the suite, then the scenario is evaluated against the `expectedOutcome` — pass if the outcome is present, fail if absent — and the result is recorded in the gate run output without requiring the gate to know the specific procedure the agent used to produce the outcome.

**AC3:** Given a new failure pattern is discovered during an assurance gate run (a real failure not currently covered by any scenario in the suite), when the platform maintainer proposes adding a scenario to guard against it, then the proposed scenario is reviewed against the outcome-orientation diagnostic (ref-skills-platform-standards-model.md) before being merged — and a record of the review decision is appended to `workspace/suite.json` in the scenario's `reviewNote` field.

**AC4:** Given a scenario is accepted and merged into `workspace/suite.json`, when any subsequent assurance gate run executes the suite, then that scenario must pass — it cannot be silently removed, disabled, or skipped. If it fails, the gate returns a `blocked` verdict. If it needs to be retired, it must be explicitly retired with a reason logged in the scenario's `retiredAt` and `retiredReason` fields — never silently deleted.

**AC5:** Given the complete suite exists and `p1.4-watermark-gate` is DoD-complete, when the controlled M4 acceptance test is run (add a failing scenario, run gate, observe blocked verdict, remove scenario), then the regression guarantee holds — the failing scenario triggers a `blocked` verdict, and removing it restores the `pass` verdict — validating that the suite and the watermark gate together form a closed regression loop.

**AC6:** Given the suite is reviewed for outcome-orientation at the time of P1.6 DoD, when each scenario is assessed using the diagnostic test ("if this model behaviour improved tomorrow, would this scenario still be correct?"), then all scenarios pass the diagnostic — no scenario is workaround-oriented at the time of initial delivery.

## Out of Scope

- Automated scenario generation from failure logs — Phase 2 (improvement agent responsibility)
- Domain-tier and squad-tier suite extensions — Phase 2; Phase 1 establishes the core suite only
- Performance benchmarking scenarios — P1.6 focuses on correctness regression, not performance regression (performance scenarios are Phase 2)
- Cross-skill dependency scenarios (scenarios that test the handoff between two skills) — Phase 2; Phase 1 scenarios are bounded to single skill outcomes
- Suite visualisation or dashboard — Phase 3; the suite is a JSON file consumed by the watermark gate; human readability comes from the `description` and `expectedOutcome` fields

## NFRs

- **Performance:** Evaluating the full suite against a gate run must complete within 8 minutes — the total P1.3 gate run budget is 10 minutes; the suite evaluation is the dominant component
- **Security:** `workspace/suite.json` contains no credentials, secrets, or PII — scenarios describe harness behaviour, not user data
- **Audit:** Every scenario merge (addition or retirement) must have a `reviewNote` field populated — a scenario with a null `reviewNote` is a gap in the outcome-orientation audit trail
