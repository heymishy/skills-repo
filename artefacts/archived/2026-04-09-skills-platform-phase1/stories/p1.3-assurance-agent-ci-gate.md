# Story: Deploy assurance agent as automated CI gate on PR open/update

**Epic reference:** artefacts/2026-04-09-skills-platform-phase1/epics/epic-2-core-loop-infrastructure.md
**Discovery reference:** artefacts/2026-04-09-skills-platform-phase1/discovery.md
**Benefit-metric reference:** artefacts/2026-04-09-skills-platform-phase1/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the assurance agent to fire automatically on PR open or update events and surface a gate verdict and trace hash in the PR,
So that governance is verified without manual initiation and the audit evidence is visible in the PR without opening a separate tool.

## Benefit Linkage

**Metric moved:** M2 — CI-triggered assurance gate; T3M1 — Trace readability for risk review
**How:** This story is the direct delivery of M2 — all four M2 sub-conditions are its acceptance criteria. It also produces the trace that T3M1 is tested against: the first inner loop PR after P1.3 DoD is the T3M1 acceptance test vehicle.

## Architecture Constraints

- Credentials are structural and never in the agent's environment — the CI configuration must reference secret names from the secrets store, not credential values. (Product constraint #12.)
- No persistent agent runtime — the gate must operate on standard CI/CD infrastructure with no hosted agent service or proprietary orchestration. (Product constraint #11.)

## Dependencies

- **Upstream:** `p1.5-workspace-state-session-continuity` — `state.json` schema must be stable before the gate writes its verdict section to state
- **Downstream:** T3M1 acceptance test uses the first PR opened after this story's DoD — plan the target story accordingly

## Acceptance Criteria

**AC1:** Given a PR is opened or a push is made to an open PR in the prototype repository, when the CI event fires, then the assurance agent starts without any manual operator action — evidenced by a CI log entry showing the trigger event type as `pull_request` opened or synchronised, not a manual `workflow_dispatch`.

**AC2:** Given the assurance agent has started on a PR event, when the agent begins execution, then it writes an `inProgress` trace entry to `workspace/traces/` before any evaluation check runs — the `inProgress` entry must be present even if the agent subsequently errors or stops mid-run.

**AC3:** Given the assurance agent has completed a gate run against a PR, when all structural and audit checks have executed, then a `completed` trace entry is written that directly follows the `inProgress` entry from AC2 — producing a valid `inProgress`→`completed` transition in the trace file.

**AC4:** Given the `completed` trace entry exists after an assurance gate run, when a reviewer opens the PR in GitHub (or equivalent), then the gate verdict (pass/fail) and the trace hash are visible in the PR interface without opening a separate tool — surfaced as a CI check status, PR comment, or check run annotation.

**AC5:** Given the assurance gate fires on a PR for a story that was produced with human initiation of the assurance agent in a prior session, then the gate run for the PR event is distinct from — and does not replace — the manually-run session; both entries exist in the trace, and the CI-triggered run is identified in the trace metadata as `trigger: ci` not `trigger: manual`.

## Out of Scope

- Challenger pre-check automation — requires the Phase 2 improvement agent; this story delivers the basic gate, not the challenger
- Cross-team trace registry — Phase 3; traces are written locally to `workspace/traces/` only
- Gate results pushed to a monitoring dashboard — Phase 3 observability
- Staleness checks on the standards model — requires P1.7 to be DoD-complete; staleness flags are a Phase 2 gate enhancement
- GitHub Checks API deep integration (custom check run UI, annotations on specific lines) — the gate verdict must be visible in the PR by the simplest available mechanism; advanced annotation is not in scope

## NFRs

- **Performance:** Gate run must complete within 10 minutes on the CI platform — a run that times out before writing `completed` is a gate failure equivalent to a governance check failure
- **Security:** CI configuration must reference credential names only — no credential values in any committed file. Secret scanning must not trigger on the CI configuration file.
- **Audit:** Every gate run must write both `inProgress` and `completed` trace entries with ISO timestamps, the PR reference (number + commit SHA), and the trigger type (`ci` or `manual`)

## Complexity Rating

**Rating:** 3
**Scope stability:** Unstable — trace schema for PR metadata (`inProgress`/`completed` structure, `trigger` field values) is an open design decision at story entry
