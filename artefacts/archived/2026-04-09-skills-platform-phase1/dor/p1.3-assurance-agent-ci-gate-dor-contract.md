# Contract Proposal — Deploy assurance agent as automated CI gate on PR open/update

**Story:** artefacts/2026-04-09-skills-platform-phase1/stories/p1.3-assurance-agent-ci-gate.md
**DoR artefact:** artefacts/2026-04-09-skills-platform-phase1/dor/p1.3-assurance-agent-ci-gate-dor.md
**Contract status:** Approved — 2026-04-10

---

## What will be built

- `.github/workflows/assurance-gate.yml` — GitHub Actions workflow file; triggers on `pull_request` events (`opened`, `synchronize`); sets `timeout-minutes: 10`; references secret names only for credentials; includes a step to post the verdict (trace hash + pass/fail) to the PR via CI check status or comment
- Gate runner logic — on CI trigger: (1) writes `inProgress` trace entry to `workspace/traces/` before any evaluation check runs; (2) executes structural and audit checks; (3) writes `completed` trace entry containing `verdict`, `traceHash`, `prRef`, `commitSha`, `trigger: "ci"`, `startedAt`, `completedAt`

## What will NOT be built

- Challenger pre-check automation — Phase 2 (requires Phase 2 improvement agent)
- Cross-team trace registry — Phase 3 (traces written locally to `workspace/traces/` only)
- Gate results pushed to a monitoring dashboard — Phase 3 observability
- Staleness checks on the standards model — requires P1.7 DoD first
- GitHub Checks API custom annotations on specific code lines — verdict by simplest available mechanism only

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Integration: simulate `pull_request` event payload, assert `trigger: ci` in trace; Manual: open real PR, observe gate fires automatically without manual action | Integration + Manual |
| AC2 | Unit: intercept write order — assert `inProgress` write precedes all evaluation calls; Integration: full gate run, inspect trace entry timestamps | Unit + Integration |
| AC3 | Unit: run full gate against synthetic payload, assert exactly two entries in `inProgress`→`completed` order; integration run confirms | Unit + Integration |
| AC4 | Manual: open completed PR on GitHub — gate verdict (pass/fail) and trace hash visible in PR interface without opening any other tool | Manual |
| AC5a | Integration: pre-existing manual trace entry in `workspace/traces/`; CI gate fires; assert both `trigger: manual` and `trigger: ci` entries present and independent | Integration |
| AC5b | Unit: CI-triggered run trace `completed` entry asserts `trigger` field equals `"ci"` | Unit |

## Assumptions

- GitHub Actions is the CI platform (`context.yml: ci: github-actions`); `github.event_name` is the trigger discriminator
- `workspace/traces/` is the canonical trace directory; no path redesign in scope
- "Visible in PR without opening a separate tool" is satisfied by CI check status (green/red icon) with verdict and trace hash in the check run summary — no deep line annotation required
- The gate runner is a Node.js script consistent with existing `.github/scripts/` conventions (CommonJS, no external npm dependencies)

## Estimated touch points

Files: `.github/workflows/assurance-gate.yml` (new), gate runner script under `.github/scripts/` (new or extended), `workspace/traces/` (write target)
Services: GitHub Actions (CI trigger), GitHub Checks API or PR status API (verdict surfacing)
APIs: GitHub Actions workflow API (trigger), GitHub REST API (check run or PR comment for AC4)
