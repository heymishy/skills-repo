## What will be built

- `standards/software-engineering/core.md` — human-readable title, owning discipline, last-reviewed date, `lastReviewedBy`, `lastReviewedDate`, at least five outcome-oriented requirements in `MUST/SHOULD/MAY [observable outcome]` format
- `standards/software-engineering/POLICY.md` — binary pass/fail floor requirements that no domain or squad override can relax
- `standards/security-engineering/core.md` — same structure; at minimum includes: no OWASP high/critical findings without an accepted risk entry; secrets never committed to source control
- `standards/security-engineering/POLICY.md` — binary floor; minimum floors: OWASP check (no high/critical without RISK-ACCEPT) and secrets check (secrets never committed)
- `standards/quality-assurance/core.md` — same structure
- `standards/quality-assurance/POLICY.md` — binary floor requirements for QA discipline
- `standards/index.yml` — discipline ID lookup table mapping `software-engineering`, `security-engineering`, `quality-assurance` to their `core.md` and `POLICY.md` paths; schema uses `disciplines:` → `core:`, `policy-floor:`, `surface-variants:` structure from `ref-skills-platform-standards-model.md`; extendable to a fourth discipline entry without parser change

## What will NOT be built

- The eight remaining discipline standards beyond the three anchor disciplines — Phase 2
- Domain-tier and squad-tier standards layers — Phase 2
- Automated POLICY.md floor enforcement tooling — Phase 2; enforcement at Phase 1 is by PR review
- Standards file versioning beyond git history — Phase 2
- Surface-variant standards files for non-git surfaces — `standards/index.yml` includes the `surface-variants:` key for future use; no content populated at Phase 1

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 | File existence check + metadata parse: title, discipline, last-reviewed date, lastReviewedBy, lastReviewedDate, 5+ requirements in MUST/SHOULD/MAY format | Integration (I1, I2) |
| AC2 | POLICY.md floor parse: every requirement is binary (pass/fail); no gradient language | Unit (U1) + Integration (I1) |
| AC3 | index.yml lookup for all three discipline IDs returns correct paths; adding fourth entry parses without modification | Unit (U2–U5) |
| AC4 | Assurance agent reads index.yml + core.md files, computes hashes, writes standardsInjected array to trace with: discipline ID, file path, hash | Integration (I3) — dependency-gated on P1.3 DoD |
| AC5 | Platform maintainer recomputes hashes of core.md files at PR commit SHA; values match standardsInjected array in trace exactly | Integration (I4) — dependency-gated on P1.3 DoD |
| AC6 | Each requirement passes outcome-orientation diagnostic at DoD time; review outcome recorded in lastReviewedBy/lastReviewedDate fields | Manual authoring review at DoD |

## Assumptions

- Hash computation at gate time is the assurance agent's responsibility (P1.3 implementation) — this story only ensures the standards files are committed with correct content
- AC4/AC5 joint acceptance tests are scheduled after P1.3 and P1.7 are both DoD-complete; they do not block P1.7 DoR or DoD for the standards files themselves
- `lastReviewedBy` and `lastReviewedDate` fields are populated by the platform maintainer at authoring time; in solo operator context this is self-review documented in the standards file
