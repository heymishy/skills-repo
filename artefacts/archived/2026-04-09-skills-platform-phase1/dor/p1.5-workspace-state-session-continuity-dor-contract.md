# Contract Proposal ‚Äî Implement workspace/state.json cross-session continuity and /checkpoint convention

**DoR artefact:** artefacts/2026-04-09-skills-platform-phase1/dor/p1.5-workspace-state-session-continuity-dor.md
**Contract status:** Approved ‚Äî 2026-04-10

---

## What will be built

- `workspace/state.json` ‚Äî schema defined with `currentPhase`, `lastUpdated`, and per-phase `cycle[]` blocks (discovery, benefitMetric, definition, etc.) each containing at minimum `status` and `artefact` path
- SESSION START block in `copilot-instructions.md` ‚Äî reads `state.json` on every new session, reports last completed phase, any in-progress story execution, any pending improvement proposals or human input items, then asks: "Resume from last session state, or start fresh?"
- `/checkpoint` convention in `copilot-instructions.md` ‚Äî documents when to invoke (before compaction pressure, with context headroom), what is written, and the 60-second completion expectation

## What will NOT be built

- `workspace/learnings.md` rendered view ‚Äî deferred to Phase 2 (decisions.md SCOPE 2026-04-10)
- `/checkpoint` as a standalone SKILL.md file ‚Äî stays as a documented convention unless MM3 Invocation 2 fails (decisions.md SCOPE 2026-04-09)
- Improvement agent that reads `state.json` for task prioritisation ‚Äî Phase 2
- Automated push-based state distribution ‚Äî state.json is written at phase boundaries by operator-initiated phase skills, not streamed

## How each AC will be verified

| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Pre-populate `state.json` fixture with defined `currentPhase` and completed cycle block ‚Üí fresh session ‚Üí verify correct phase identified, no prior work re-run | Integration + Manual |
| AC2 | Unit tests against fixtures with `execution.status: "inProgress"` and `pendingHumanInput` entries; manual session start observation confirms all three items surface | Unit + Manual |
| AC3a | Stopwatch during live `/checkpoint` invocation ‚Äî closing message within 60s recorded as MM3 measurement | Manual only (untestable-by-nature) |
| AC3b | Parse written `state.json`; assert `currentPhase`, `lastUpdated`, and cycle block fields all non-null | Unit |
| AC3c | Open fresh session from checkpoint `state.json`; verify correct resume point identified without operator verbal description | Manual |
| AC4 | Fresh Copilot Chat window with no prior history ‚Äî first agent action is to check for `state.json`, read it, report contents | Manual |
| AC5 | 3 unit tests: `lastUpdated` matches ISO 8601 pattern; `currentPhase` is non-empty string; cycle block contains `status` and `artefact` keys after phase boundary write | Unit + Integration |
| AC6 | Unit: assert `copilot-instructions.md` contains wording that `/checkpoint` must be invoked before compaction pressure (not at it), with context headroom; manual read-through | Unit + Manual |

## Assumptions

- SESSION START hook is a Markdown block in `copilot-instructions.md` that Copilot reads automatically ‚Äî no runtime component required
- "Session terminates" in AC3a means the agent produces its closing confirmation message; the 60-second clock starts at `/checkpoint` invocation
- `state.json` schema is defined descriptively in `product/tech-stack.md` and used as the fixture reference for unit tests
- `state.json` and `copilot-instructions.md` are the only two files touched by this story
