## What will be built

- Watermark gate script under `.github/scripts/` — reads `workspace/results.tsv`; on first run, creates the file with one `baseline` row; on subsequent runs, compares current pass rate and full test score against the best recorded values for the `skillSetHash + surfaceType` composite key; appends a `pass` or `blocked` row with trigger field set to `pass-rate-below-watermark`, `score-below-best`, or `both` on blocked runs
- `workspace/results.tsv` — append-only TSV file; columns (tab-separated): ISO timestamp, skillSetHash, surfaceType, passRate (float 0–1), fullTestScore (int), verdict (baseline | pass | blocked), trigger (empty on baseline/pass; reason value on blocked)
- Integration step in `.github/workflows/assurance-gate.yml` invoking the watermark gate as a component of the overall assurance gate run

## What will NOT be built

- Automated remediation on regression — gate blocks and surfaces reason; fixing is a human decision
- Rolling-window or percentile watermark — Phase 1 uses best-recorded high watermark only; statistical variants are Phase 2
- Per-story watermarks — per skill/surface combination only
- Alert routing (PagerDuty/Slack/email) on regression — Phase 3 observability
- Watermark decay or planned-regression support — manual reset only; automated decay not in scope

## Assumptions

- `workspace/results.tsv` is the canonical file path — no alternative paths in scope
- Composite key for watermark lookup: `skillSetHash + surfaceType`; same combination across runs refers to the same harness context
- Gate script is Node.js CommonJS, no external npm dependencies — consistent with `.github/scripts/` conventions
- Equal-to threshold is treated as pass (≥), not strictly greater than (>)
- Integration with P1.3 CI runner: watermark gate is invoked as a named step in the existing assurance gate workflow, not a separate workflow
