## What will be built

1. **`.github/workflows/attestation-publisher.yml`** (new file) — post-merge GitHub Actions workflow (AC1 + AC2 + AC3):
   - Triggers on `push` to `main` for regulated story merges (same classification used in p3.2a AC5 discriminator).
   - Uses OIDC token (`id-token: write`) to authenticate as a CI service identity — no personal access token (Security NFR).
   - Computes SHA-256 hash of the trace file produced for the merged story (`workspace/traces/[story-slug]-trace.jsonl`).
   - Calls GitHub Artifact Attestation API (Option A per DEC-P3-002) to publish the hash.
   - Confirms publication in the workflow run log.

2. **`workspace/traces/` write-back** — trace file updated post-attestation (AC4):
   - The post-merge workflow adds (or updates) a `tamperEvidence` object in the trace file containing:
     - `registryType`: `"github-artifact-attestation"`
     - `registryRef`: URL or attestation ID referencing the published registry entry
     - `publishedAt`: ISO 8601 timestamp of publication

3. **`docs/MODEL-RISK.md`** — Q8 entry updated (AC5, manual documentation):
   - Registry type in use: `github-artifact-attestation`
   - How an auditor recomputes the hash (`sha256sum` of the trace file)
   - How an auditor retrieves the registry entry for comparison (HTTP GET to the Artifact Attestation endpoint)
   - Access-separation evidence: delivery team write access is limited to repository contents; Artifact Attestation records are controlled by the GitHub API — delivery agents cannot delete or modify published attestations

## What will NOT be built

- Option B (read-only registry repo) — DEC-P3-002 selected Option A. Option B is documented in decisions.md as a Bitbucket/Jenkins fallback but is not in scope for this implementation.

- Retroactive attestation of existing trace files — only traces produced after this story merges are attested.

- New npm packages in the test runner — any test helpers use plain Node.js CommonJS.
