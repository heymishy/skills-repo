# DoR Contract — Eval suite anti-gaming controls — traceId, failurePattern, human-approval staging

**Story reference:** p3.4-eval-anti-gaming-controls.md
**Date:** 2026-04-18
**Status:** BLOCKED — awaiting p3.3 DoD-complete

---

## What will be built

A validation script (`scripts/validate-suite-entry.js`) that:
- Checks any new `workspace/suite.json` or `platform/suite.json` entry for mandatory `traceId` and `failurePattern` fields (exits 1 if missing or null)
- Resolves the `traceId` value to an existing committed file at `workspace/traces/[traceId]` or `platform/traces/[squad]/[traceId]` (exits 1 if file not found)
- Applies grandfathering logic — entries that predate this story pass without traceId/failurePattern
- Invoked by `npm test` and/or a pre-commit hook

Improvement agent modification to:
- Write new scenario proposals to `workspace/proposals/suite-additions/[proposal-id].json` (not directly to suite.json)
- Proposal file includes `traceId`, `failurePattern`, and `justification` fields

---

## What will NOT be built

- Automated proposal promotion — human maintainer promotes manually by committing to platform-infrastructure; no code for this
- Validation of grandfathered entries (entries predating this story)
- Retroactive traceId/failurePattern backfill of existing entries
- Any feature of p3.12 (squad contribution flow) — that is a downstream story
