---

## What will be built

1. **`src/trace-registry/getTraces.js`** — new Node.js CommonJS module. Exports `getTraces({ squadId, filter })`. Supported filter fields: `squadId`, `dateRange`, `storySlug`, `passRate` (threshold). Returns an array of matching trace entries from `platform/traces/[squadId]/`. Returns `[]` when no match or when the registry is empty — never throws an error in the empty-registry or no-match case.

2. **`.github/workflows/trace-aggregation.yml`** — new CI workflow file. Triggered by `schedule:` (daily minimum) or `workflow_dispatch:`, not by `pull_request` or `push`. Uses a repository-scoped read-only token — not the same token as the delivery pipeline. Aggregates JSONL trace files from squad repos to `platform/traces/[squadId]/[storySlug].jsonl`.

---

## What will NOT be built

- No real-time aggregation (sub-minute); 24-hour SLA is the target
- No schema changes to JSONL trace format (that is p3.2a scope)
- No aggregation of non-JSONL artefacts
- No peer query access (squads cannot call getTraces for other squads directly)

## Assumptions

- `platform/traces/` directory is in this repo (the platform repo).
- Squad repos are not actually queried in tests — mock fixture data is used for all tests.
- The `schedule:` trigger minimum cadence is daily (`0 2 * * *` or equivalent).
- Read-only token for aggregation workflow uses `permissions: contents: read` and no write scope.
