# Story: Resolve permanently-skipped Bitbucket DC auth tests

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the four Bitbucket DC auth tests (app-password, OAuth, SSH key, PAT) to move from permanent skip to either a scheduled CI run with a Docker service container or a formal manual test record,
So that the test suite no longer contains permanently-skipped tests with no documented resolution path, and the governance coverage status of Bitbucket DC auth is unambiguous.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: governance failure reduction
**How:** Permanently-skipped tests represent undocumented coverage gaps. Formalising the resolution (either Docker-based execution or a dated manual record) converts an ambiguous "skipped" status into a documented, auditable coverage decision. The governance check count either increases (Option A: tests now run) or the skip rationale is on record (Option B).

## Architecture Constraints

- If Option A (scheduled CI workflow): must follow ADR-009 (separate trigger/permission scope) and ADR-010 (audit records to main post-merge). The scheduled auth test workflow must not commit back to the feature branch it evaluates.
- If Option B (manual test record): `tests/smoke-tests.md` entry must include test name, last-run date, result (pass/fail), runner identity, and the Docker dependency rationale that explains the skip.
- Plain Node.js only — no external npm dependencies added.

## Dependencies

- **Upstream:** None.
- **Downstream:** None within Phase 3. The resolution chosen here informs how future auth-level tests are handled across all surface adapters.

## Acceptance Criteria

**AC1:** Given the four Bitbucket DC auth tests (`app-password`, `OAuth`, `SSH key`, `PAT`) are currently annotated with `// PREREQ-DOCKER` or equivalent skip reason, when this story is DoD-complete, then each of the four tests is resolved via exactly one of: (a) the test runs in a scheduled `bitbucket-dc-auth.yml` CI workflow with a Bitbucket DC service container, or (b) a dated manual test record entry exists in `tests/smoke-tests.md` for each test with last-run date, result, and runner identity.

**AC2:** Given Option A is chosen, when the scheduled `bitbucket-dc-auth.yml` workflow is added, then it triggers on a schedule (e.g. weekly) rather than on `pull_request` or `push` — preventing auth tests from blocking standard PR merges when the Docker service is unavailable.

**AC3:** Given Option B is chosen, when `tests/smoke-tests.md` is inspected, then each of the four tests has an entry containing: test name, last-run date (ISO 8601), result (`pass` or `fail`), runner identity (operator name or CI job reference), and a note confirming the Docker dependency justification.

**AC4:** Given the chosen option is implemented, when `npm test` runs on a standard CI runner (without Docker), then the four tests either pass (Option A environment) or are formally skipped with a `// MANUAL: see smoke-tests.md` annotation (Option B) — in neither case are they permanently skipped with no resolution path.

**AC5:** Given the decision between Option A and Option B, when this story reaches DoR, then the decision is logged in `artefacts/2026-04-14-skills-platform-phase3/decisions.md` with the rationale.

## Out of Scope

- Implementing Bitbucket DC auth logic itself — only the test resolution method is in scope, not adding new auth capabilities.
- Resolving auth tests for any other Git host besides Bitbucket DC.
- Removing the Docker dependency permanently — if Option B is chosen, Docker remains the correct long-term solution; this story only formalises the current state.

## NFRs

- **Audit:** The chosen resolution method must produce an auditable record (workflow run log or `smoke-tests.md` entry) that can be inspected without running tests interactively.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — two clearly defined resolution paths; choice is made at DoR.
