# Story: Register agent behaviour observability as Phase 4 backlog item

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want a documented candidate analysis of agent behaviour observability approaches registered as a Phase 4 backlog item,
So that the governance gap is acknowledged with effort estimates on record and the Phase 4 team does not need to rediscover or re-scope it from scratch.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: governance failure reduction
**How:** Closing a known backlog obligation (identified in Phase 2 adversarial audit) with a documented, on-record candidate analysis prevents this gap from being silently carried forward as undocumented technical debt. The improvement cycle is closed even though implementation is deferred.

## Architecture Constraints

- No code changes in this story. Output is a single Markdown document at `docs/agent-behaviour-observability.md`.
- The Phase 4 story registration goes to `workspace/` (consistent with existing backlog registration pattern: `workspace/phase3-backlog-*.md`).

## Dependencies

- **Upstream:** None.
- **Downstream:** None within Phase 3. The registered Phase 4 story is an input to Phase 4 discovery.

## Acceptance Criteria

**AC1:** Given this story is DoD-complete, when `docs/agent-behaviour-observability.md` is read, then it contains exactly three candidate approaches for agent behaviour observability — each with: a descriptive name, a one-paragraph mechanism description, an effort estimate (Low/Medium/High), and a noted tradeoff.

**AC2:** Given `docs/agent-behaviour-observability.md` exists, when a platform maintainer reads the document, then it explicitly states that no implementation is included in Phase 3 and identifies the Phase 4 story registration as the only Phase 3 deliverable.

**AC3:** Given the three candidate approaches are documented, when `workspace/` is inspected, then a Phase 4 backlog file exists (following the `workspace/phase4-backlog-*.md` naming pattern) containing the agent behaviour observability story with: title, problem statement, the three candidates from the docs file (by reference), and a "NOT STARTED — Phase 4" status marker.

## Out of Scope

- Implementation of any agent behaviour observability approach — this is a documentation and registration story only.
- Selecting one candidate approach over the others — the document presents all three; selection is a Phase 4 decision.
- Changes to any skill, gate script, or schema file.

## NFRs

- None specific to this story — documentation artefact only.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — output is a documentation file and a backlog registration entry.
