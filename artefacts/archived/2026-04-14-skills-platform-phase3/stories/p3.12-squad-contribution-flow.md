# Story: Define and document the governed squad contribution flow for new skills and eval additions

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **squad tech lead** that has identified a reusable skill pattern during delivery,
I want a documented, governed contribution flow for proposing new skills and eval additions to the platform repo — including the required artefacts (SKILL.md + EVAL.md + performance evidence) and the human approval gate — so that I can contribute without bypassing the quality controls established by p3.4.

## Benefit Linkage

**Metric moved:** MM2 — Outer loop self-sufficiency: zero blocking lookups
**How:** Without a documented contribution process, a squad lead must ask the platform team how to contribute. The governed flow document removes that blocking question. The dependency on p3.4 ensures the contribution pipeline cannot be used as a vector for gaming the eval suite before anti-gaming controls are in place.

## Architecture Constraints

- This story depends on p3.4 (eval anti-gaming controls) being DoD-complete. The contribution flow document must reference the anti-gaming requirements (traceId + failurePattern) as mandatory fields in any EVAL.md contribution.
- The contribution flow is documented in `docs/squad-contribution-guide.md` and cross-referenced from `CONTRIBUTING.md` (or that file is created if it does not exist).
- PR review threshold for platform contributions is higher than for delivery PRs: requires explicit sign-off from a designated platform team reviewer (named or by GitHub team alias), not just any code-owner auto-assignment.
- The contribution PR must contain: SKILL.md for the new skill, EVAL.md with at least 3 test cases (each with traceId + failurePattern per p3.4 requirements), and performance evidence linking the skill to a real delivery outcome (story reference or metric movement evidence).

## Dependencies

- **Upstream:** p3.4 (eval anti-gaming controls) — must be DoD-complete. The contribution flow references the EVAL.md traceId + failurePattern requirements that p3.4 establishes.
- **Downstream:** None within Phase 3.

## Acceptance Criteria

**AC1:** Given `docs/squad-contribution-guide.md`, when it is reviewed, then it describes the complete contribution flow: (1) propose via feature branch, (2) required PR contents (SKILL.md + EVAL.md + performance evidence), (3) EVAL.md traceId + failurePattern requirements per p3.4, (4) the human approval gate (designated platform reviewer sign-off), and (5) the merge path (platform team review → merge, not the contributing squad).

**AC2:** Given the EVAL.md requirements section in the guide, when a contributor produces a new EVAL.md, then each test case entry must contain a `traceId` field (resolving to a committed trace per p3.4 AC1) and a `failurePattern` field — the same requirements as new `suite.json` entries in p3.4. A contribution PR with EVAL.md entries missing these fields is non-compliant.

**AC3:** Given a contributing squad's PR, when it is opened, then the PR description template (from `docs/squad-contribution-guide.md`) guides the contributor to include: skill name, use case rationale, delivery story reference, at least 3 EVAL.md test cases with traceId+failurePattern, and the name of the platform reviewer requested for sign-off.

**AC4:** Given `CONTRIBUTING.md` at the repo root, when it is reviewed, then it contains a section titled "Contributing to the Skills Platform" with a link to `docs/squad-contribution-guide.md` and a note that delivery repo changes (stories, artefacts, workspace) do not require this process — only platform SKILL.md and eval additions do.

**AC5:** Given both documents are complete, when the PR is reviewed by the platform team, then no new permissions, scripts, or CI configuration are required — the contribution flow is a documented process, not a new automated gate (automation can be added in a Phase 4 story if adoption warrants it).

## Out of Scope

- Automating the contribution review process — Phase 4.
- Reviewing or incorporating existing external SKILL.md contributions — Phase 4.
- Changing the CODEOWNERS file or GitHub branch protection rules — referenced only, not modified.
