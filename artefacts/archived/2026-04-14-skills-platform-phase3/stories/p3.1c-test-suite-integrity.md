# Story: Test suite integrity — import path, passRate floor, anti-overfitting counter

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the test suite to import production code at its real source paths, enforce a minimum pass-rate floor before watermarking a baseline, and count check removals separately from additions in the anti-overfitting gate,
So that governance check results reflect the actual production code and a weakening proposal cannot pass the anti-overfitting gate by adding a new check to conceal a removed one.

## Benefit Linkage

**Metric moved:** MM3 — Priority 1 hardening: governance failure reduction
**How:** Fixing the import path ensures test failures reflect real production bugs. The passRate floor prevents a degraded suite from being watermarked as a new baseline. The anti-overfitting counter change closes the "add one, remove one" bypass.

## Architecture Constraints

- Plain Node.js only — no external npm dependencies in test or gate scripts.
- ADR-008 (DoR touch-point contract binding at pre-merge): the anti-overfitting gate change must not alter the evaluation logic in a way that retroactively flags previously-accepted proposals. Only new proposals evaluated post-merge are subject to the new counter logic.

## Dependencies

- **Upstream:** None — this story does not depend on p3.1a or p3.1b at the code level, though it should be reviewed before p3.1b to ensure a clean test baseline.
- **Downstream:** p3.4 (eval anti-gaming controls) builds on the anti-overfitting gate; the counter fix in this story is a prerequisite.

## Acceptance Criteria

**AC1:** Given `tests/check-definition-skill.js`, when the test file is inspected, then it imports the definition skill checker from `src/` production source path (e.g. `require('../src/...')`) — no inline re-implementation of production logic appears in the test file itself.

**AC2:** Given `src/improvement-agent/watermark-gate.js` (or equivalent path), when a new baseline is being created from a set of scenarios where fewer than 70% pass, then `watermark-gate.js` throws an error (or exits 1) and does not write the baseline — the minimum pass-rate floor is 0.70 by default and the floor value is documented in the source file as a named constant.

**AC3:** Given a watermark baseline has been written and a new proposal is evaluated that adds 1 new passing check and removes 1 previously-passing check (net zero change in total check count), when `failure-detector.js` evaluates the proposal, then it exits 1 and reports that a removal of a passing check was detected — the pass/add/remove counts are reported separately in the output.

**AC4:** Given a watermark baseline has been written and a new proposal is evaluated that adds 1 new passing check and removes 0 previously-passing checks (net +1 total), when `failure-detector.js` evaluates the proposal, then it exits 0 (proposal passes the anti-overfitting gate) — confirming that legitimate additions without removals still pass.

**AC5:** Given `npm test` is run after all three changes, then the full suite passes with no regressions — the test count is equal to or greater than the pre-story baseline.

## Out of Scope

- Increasing the passRate floor above 0.70 — the floor is 0.70 in this story. Tuning is a Phase 4 improvement.
- Retroactive re-evaluation of previously accepted proposals — only proposals evaluated after this story merges are subject to the new anti-overfitting counter logic.
- Changes to the watermark baseline file format — no format changes, only floor enforcement at baseline creation time.

## NFRs

- **Performance:** No increase in `npm test` runtime beyond what is introduced by AC5 test additions (if any). The import path fix and counter change should have negligible runtime impact.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — three specific gaps identified by adversarial audit with clear pass/fail criteria.
