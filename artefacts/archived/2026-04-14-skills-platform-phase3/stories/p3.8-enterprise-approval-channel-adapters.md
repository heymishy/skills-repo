# Story: Deliver Teams and Jira enterprise approval channel adapters

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e6-enterprise-adapters-agents-validation.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **non-engineering approver** in an enterprise organisation using Microsoft Teams or Jira for workflow approvals,
I want to approve a DoR sign-off by clicking a button in my Teams channel or transitioning a Jira workflow — without needing a GitHub account — so that the pipeline proceeds without the platform team acting as a manual intermediary.

## Benefit Linkage

**Metric moved:** M4 — Non-engineer approval without platform team intermediary
**How:** Phase 2 established the persona-routing adapter contract (ADR-006). Phase 3 delivers the first two enterprise adapters in that contract: Teams (via Adaptive Card + webhook) and Jira (via workflow transition or `/approve-dor` comment). Measured as: non-engineering approver can complete approval in their native tooling, and `pipeline-state.json` is updated without platform team involvement.

## Architecture Constraints

- ADR-006 applies: each adapter must conform to the approval-channel interface defined in `src/approval-channel/`. A new adapter is a new file in `src/approval-channel/adapters/` — it does not modify the existing interface contract.
- No personal access tokens (PATs) — both adapters must use service credentials (Teams: webhook URL with rotating secret; Jira: service account OAuth token or API token issued to the integration service identity, not an individual developer's credentials). MC-SEC-01 applies.
- The Teams adapter must use an Adaptive Card with an "Approve DoR" action that posts to a webhook — the webhook handler updates `pipeline-state.json` and calls the existing `process-dor-approval.js` script.
- The Jira adapter must trigger the `pipeline-state.json` write via a Jira workflow transition callback or a `/approve-dor` comment on the linked Jira issue — matching the GitHub issue channel pattern from Phase 2.
- Both adapters must be independently unit-testable using mock channel APIs — no real Teams or Jira credentials required to pass the tests.

## Dependencies

- **Upstream:** Phase 2 p2.8 (persona-routing) must be DoD-complete — the adapter interface it defines is the contract this story implements for two new channels. (Phase 2 is complete.)
- **Downstream:** None within Phase 3.

## Acceptance Criteria

**AC1:** Given a configured Teams channel with the DoR approval Adaptive Card, when a non-engineering approver clicks "Approve DoR" in Teams, then the webhook handler calls `process-dor-approval.js` with the correct story ID and writes `dorStatus: "signed-off"` to `pipeline-state.json` for that story — verified by unit test against a mock Teams webhook payload.

**AC2:** Given a configured Jira issue linked to a DoR sign-off request, when a Jira approver either (a) transitions the issue to the "Approved" state or (b) posts `/approve-dor` as a comment, then the Jira adapter calls `process-dor-approval.js` and writes `dorStatus: "signed-off"` to `pipeline-state.json` — verified by unit test against a mock Jira webhook or comment payload.

**AC3:** Given both adapters, when their integration tests run, then no real Teams webhook URL, no real Jira API credentials, and no real `pipeline-state.json` writes happen — tests use mock interfaces only. The test suite must not require external network access to pass.

**AC4:** Given an adapter failure (invalid payload, missing story ID, `pipeline-state.json` write error), when the webhook handler encounters the failure, then it returns an appropriate HTTP error status (4xx or 5xx) and logs the error — it does not silently swallow the failure or write a partial state.

**AC5:** Given both adapters' credentials configuration, when the integration is deployed, then credentials are read from environment variables or a secrets manager reference — no credentials are hardcoded in source files or in `context.yml` (MC-SEC-01 compliance).

## Out of Scope

- ServiceNow adapter — Phase 4.
- Slack adapter — Phase 4.
- Email-based approval — not in scope for any phase (identified as anti-pattern in identity verification context).
- Changes to the approval-channel interface contract itself — only adapters implementing the existing contract are delivered.

## NFRs

- **Security:** Webhook payloads must be authenticated (Teams: HMAC signature; Jira: service account token verification) — unauthenticated webhook calls are rejected with 401.
- **Test isolation:** No external network calls in CI. Mock adapters must be the default for all tests.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — adapter interface is fixed by ADR-006; only implementation is required.
