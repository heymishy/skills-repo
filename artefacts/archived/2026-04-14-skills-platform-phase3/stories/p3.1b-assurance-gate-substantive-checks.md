# Story: Extend assurance gate to substantive content checks

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want the assurance gate to verify governance substance — not just file existence — so that production traces record a `completedAt` duration that reflects real evaluation time and a reviewer cannot infer "pass" from an empty repository with four minimal files.

## Benefit Linkage

**Metric moved:** M2 — Assurance gate substantive signal (completedAt duration)
**How:** Extending `run-assurance-gate.js` from file-existence checks to content-level checks means real evaluation work is performed during each gate run, producing a `completedAt` delta measurably greater than 1–2ms.

## Architecture Constraints

- ADR-003 (Schema-first): any new fields added to the gate's output trace (e.g. `checksRun`, `failedChecks`) must be added to `pipeline-state.schema.json` in the same commit.
- ADR-002 (Gates must use evidence fields): the new checks must evaluate at least one evidence field per file — not merely assert the file exists.
- Plain Node.js only — no external npm dependencies.

## Dependencies

- **Upstream:** p3.1c (test suite integrity) recommended before this story's PR opens — the anti-overfitting gate is more meaningful once the test import paths are correct, but there is no hard code dependency.
- **Downstream:** p3.3 (gate structural independence) — this story's output is what gets moved to the infra repository. Proceed in order.

## Acceptance Criteria

**AC1:** Given `workspace/state.json` exists and is valid JSON, when `run-assurance-gate.js` checks it, then the gate verifies that `currentPhase` is a non-empty string AND `lastUpdated` is a string matching ISO 8601 format (`YYYY-MM-DDTHH:mm:ssZ` or `YYYY-MM-DD`) — the gate fails if either field is absent, null, or does not match the format.

**AC2:** Given `workspace/state.json` has `status: "completed"` in any cycle block and `completedAt: null` in the same block, when `run-assurance-gate.js` runs, then `check-workspace-state.js` exits 1 and reports the specific cycle block where the inconsistency was found.

**AC3:** Given `.github/pipeline-state.json` exists and is valid JSON, when `run-assurance-gate.js` checks it, then the gate verifies that a `version` field is present (non-empty) AND `features` is an array (may be empty but must be an array type) — the gate fails if either field is absent or wrong type.

**AC4:** Given the `artefacts/` directory exists, when `run-assurance-gate.js` checks it, then the gate verifies at least one immediate subdirectory (feature slug directory) exists inside `artefacts/` — the gate fails if `artefacts/` is empty (no subdirectories).

**AC5:** Given `.github/governance-gates.yml` exists, when `run-assurance-gate.js` checks it, then the gate verifies the parsed YAML contains a `gates:` key with at least one entry — the gate fails if the key is absent or the array is empty.

**AC6:** Given all five new content checks pass, when the gate writes the completion trace, then the `completedAt` timestamp in the trace is greater than `startedAt` by at least 5ms on any hardware (measured at test time using a synthetic fixture with artificial delay if needed).

## Out of Scope

- Content validation of story or epic artefact files (e.g. checking that AC count ≥ 3) — that is a future gate expansion, not in this story.
- Validation of the `pipeline-state.schema.json` file itself — schema validity is checked by the existing pre-commit hook.
- Changes to governance gate thresholds in `governance-gates.yml` — content of the gates file is not modified by this story.

## NFRs

- **Performance:** The full gate (including all five new checks) must complete within 30 seconds on a standard CI runner.
- **Security:** Gate reads files only — no writes, no network calls. Read-only filesystem operations.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — five specific checks identified by adversarial audit, each with clear pass/fail criteria.
