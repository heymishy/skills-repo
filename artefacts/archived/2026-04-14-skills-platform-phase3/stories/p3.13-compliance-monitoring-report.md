# Story: Produce automated compliance monitoring report from cross-team trace registry

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **risk reviewer / audit function** responsible for model-risk governance,
I want the improvement agent to sample the cross-team trace registry on a monthly cadence and produce an attestation report confirming that gate verdicts are present, no squad exceeds the policy-defined gap threshold, and all T3M1 mandatory fields are populated — so that I can conduct an independent audit without asking the platform team to manually extract data.

## Benefit Linkage

**Metric moved:** CR1 — Independent audit completion without platform team involvement
**How:** CR1 is satisfied when a risk reviewer can access a third-party-verifiable compliance report without requesting raw data from the delivery team. This story produces exactly that: a structured report sampled from the cross-team registry by the improvement agent on a schedule, with tamper-evidence references where applicable.

## Architecture Constraints

- Hard dependency: p3.7 (cross-team trace registry) must be DoD-complete — the `getTraces()` API is the data source for this report.
- Hard dependency: p3.2a (T3M1 trace fields) must be DoD-complete — the report validates the presence of the four standardised fields (`standardsInjected`, `watermarkResult`, `stalenessFlag`, `sessionIdentity`) introduced in p3.2a.
- The report is generated by the improvement agent's audit sampling module — not by a standalone script.
- Output: `workspace/compliance-reports/compliance-[YYYY-MM].md` — monthly cadence, human-readable markdown with a machine-readable summary block.
- The audit agent must not have write access to the trace registry — read-only via `getTraces()`.
- ADR-009 applies: the compliance report generation CI job uses a separate trigger (scheduled monthly) and read-only token — not the delivery pipeline job.

## Dependencies

- **Upstream (hard):** p3.7 — cross-team trace registry and `getTraces()` must be DoD-complete.
- **Upstream (hard):** p3.2a — T3M1 trace fields (standardsInjected, watermarkResult, stalenessFlag, sessionIdentity) must be defined and populated.
- **Upstream (recommended):** p3.2b — if tamper-evidence registry is DoD-complete, the report includes `tamperEvidence.registryRef` verification; if not, the report notes the absence of tamper-evidence verification in the current cycle.
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given the improvement agent's audit sampling module, when it runs on its monthly schedule, then it calls `getTraces({ filter: { period: 'last-30-days' } })` across all participating squads and produces a compliance report at `workspace/compliance-reports/compliance-[YYYY-MM].md`.

**AC2:** Given a generated compliance report, when it is reviewed, then it contains for each sampled squad: squad ID, trace count sampled, gate verdicts present: yes/no, T3M1 fields populated for regulated stories: yes/no (with count of missing-field traces if any), gap count against the policy threshold, and overall compliance status: PASS/FAIL.

**AC3:** Given a squad with a gap pattern exceeding the policy threshold (more than 10% of sampled traces missing a gate verdict), when the compliance report is generated, then that squad's entry is marked FAIL and the specific gap pattern (missing field name, trace slugs with missing fields) is listed.

**AC4:** Given the compliance report CI job, when it runs, then it uses a separate GitHub Actions `schedule:` trigger and a read-only registry token — not the delivery pipeline trigger or the delivery team's write token (ADR-009 compliance).

**AC5:** Given the report is published, when a risk reviewer reads it, then the report is self-contained — it includes the policy threshold used for gap detection, the sample size, the period covered, and a reference to the trace registry path — the reviewer does not need to ask the platform team for context to interpret the findings.

## Out of Scope

- Real-time compliance monitoring — monthly cadence is the Phase 3 target.
- Automated remediation actions triggered by FAIL status — the report is advisory; remediation is a human action.
- Integration with external governance platforms (e.g. ServiceNow, Archer) — Phase 4.
- Squads that have not opted into the cross-team registry (p3.7) — absent squads are not reported as FAIL; they are listed as "not registered" in the report.

## NFRs

- **Data classification:** Compliance reports are internal governance documents — not customer data, not PII.
- **Retention:** Monthly compliance reports must be committed to the platform repo and not overwritten — `compliance-[YYYY-MM].md` filename ensures no collision.
