# Story: Implement tamper-evidence registry for traceHash (T3M1 Q8)

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e2-t3m1-audit-gap-closure.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **risk reviewer / audit function**,
I want each delivery trace hash published to an append-only external store that the delivery team cannot write to or modify, so that I can independently verify that a trace has not been tampered with since it was recorded — closing MODEL-RISK audit question Q8.

## Benefit Linkage

**Metric moved:** M1 — T3M1 audit question coverage; CR1 — T3M1 independent validation on record
**How:** Publishing `traceHash` to an external attestation store that is outside the delivery repository closes Q8 — an auditor can recompute the hash from the trace file and compare against the registry entry. If they match, the trace is confirmed unmodified. The delivery team's inability to write to the registry makes the attestation independently credible.

## Architecture Constraints

- ADR-009 (Separate workflow permission scopes): the workflow that publishes the traceHash must be a separate trigger from the gate evaluation workflow. The publisher fires post-merge on `push` to main with write access only to the external registry — not to the delivery repository.
- ADR-010 (CI audit records to main post-merge): the traceHash publication timestamp and registry entry reference must be committed back to `workspace/traces/` on main as part of the post-merge workflow.
- ASSUMPTION-02: two implementation options are available — (a) GitHub Artifact Attestation with OIDC-signed workflow identity (preferred when repository is on GitHub), or (b) a dedicated read-only registry repository (fallback for Bitbucket/Jenkins environments). The DoR must confirm which option is in scope before coding begins.

## Dependencies

- **Upstream:** p3.2a (T3M1 trace fields) must be DoD-complete — the `traceHash` is computed over the complete trace record including the four new fields. Starting this story before p3.2a changes the hash definition.
- **Downstream:** p3.3 (gate structural independence) — the tamper-evidence registry is a complement to structural gate separation. Both must be complete for a regulated-enterprise adoption claim.

## Acceptance Criteria

**AC1:** Given a regulated story trace is written to `workspace/traces/` on the `origin/traces` branch post-merge, when the post-merge attestation workflow runs, then a SHA-256 hash of the trace file is computed and published to the selected external registry (Artifact Attestation or read-only registry repo) — the publication is confirmed within the workflow run log.

**AC2:** Given a traceHash has been published to the external registry, when an independent auditor retrieves the trace file and recomputes `sha256sum` of the file content, then the computed hash matches the registry entry exactly — no difference due to line endings, encoding, or metadata.

**AC3:** Given the delivery repository, when a delivery agent or team member with standard write access attempts to modify or delete the registry entry (Option A: Artifact Attestation record; Option B: read-only registry repo commit), then the operation is denied by the access control of the external registry — confirmed by role/permission documentation in the DoD artefact.

**AC4:** Given the post-merge workflow has published a traceHash, when `workspace/traces/[story-slug]-trace.jsonl` is inspected, then it contains a `tamperEvidence` object with: `registryType` (string: `"github-artifact-attestation"` or `"registry-repo"`), `registryRef` (URL or commit SHA of the registry entry), and `publishedAt` (ISO 8601 timestamp).

**AC5:** Given `MODEL-RISK.md`, when this story is DoD-complete, then the Q8 entry documents: the registry type in use, how an auditor recomputes the hash, how an auditor retrieves the registry entry for comparison, and the access-separation evidence (who cannot write to the registry).

## Out of Scope

- Tamper-evidence for non-regulated story traces — the attestation workflow triggers only for regulated story merges (same classification used in p3.2a AC5).
- Retroactive attestation of existing trace files — only traces produced after this story merges are attested.
- Real-time tamper detection alerting — the registry is an append-only audit log, not an active monitoring system.

## NFRs

- **Security:** The workflow identity used to publish to the external registry must be a CI service identity (OIDC or service account) — not a personal access token. Personal tokens must not be stored in repository secrets for this workflow.
- **Audit:** The registry entry must be retrievable without special tooling — a plain HTTP GET or `git log` command must be sufficient for an auditor to retrieve the entry.

## Complexity Rating

**Rating:** 3
**Scope stability:** Unstable until ASSUMPTION-02 is confirmed at DoR. Once the implementation option is selected, scope is well-defined.
