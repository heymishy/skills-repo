# Story: Add trace-commit observability to npm test and skill guidance

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e1-governance-chain-integrity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer**,
I want `npm test` to fail when the `origin/traces` branch has not received a commit within the last 24 hours after a story merge,
So that post-merge workflow failures surface within the next CI cycle rather than going undetected for hours or days.

## Benefit Linkage

**Metric moved:** M3 — Post-merge silent failure detection rate
**How:** The new `check-trace-commit.js` script added to `npm test` exits 1 when the traces branch is stale, converting a previously-invisible failure mode into a blocking governance check that appears on the next PR.

## Architecture Constraints

- ADR-010: CI audit records must be persisted to main post-merge, not to feature branches. The check must read `origin/traces`, not a local branch.
- ADR-009: The check script is read-only — it never writes to any branch. Write-back is handled by the separate post-merge workflow.
- Plain Node.js only — no external npm dependencies (pre-commit scripts must run with only `node` available).

## Dependencies

- **Upstream:** None — this story can begin immediately.
- **Downstream:** p3.1b (assurance gate substantive checks) benefits from a clean test baseline; 1A should precede 1B in the review queue even though there is no hard code dependency.

## Acceptance Criteria

**AC1:** Given the `tests/` directory, when `npm test` is run, then a script named `check-trace-commit.js` is present in `tests/` and is executed as part of the test suite — its result (pass or fail) is reported in the `npm test` output.

**AC2:** Given `origin/traces` exists and its most recent commit timestamp is more than 24 hours ago, when `check-trace-commit.js` runs, then it exits with code 1 and prints a message identifying the stale branch, the last commit timestamp, and the hours elapsed since the last commit.

**AC3:** Given `origin/traces` exists and its most recent commit timestamp is within the last 24 hours, when `check-trace-commit.js` runs, then it exits with code 0 and prints a confirmation of the branch health and last commit timestamp.

**AC4:** Given `origin/traces` does not exist (branch has never been created), when `check-trace-commit.js` runs, then it exits with code 1 and prints a message indicating the branch is absent.

**AC5:** Given the `/verify-completion` SKILL.md, when a maintainer reads the post-merge verification section, then it contains explicit diagnostic commands to check `origin/traces` branch health and interpret a stale-branch `npm test` failure — including the `git log origin/traces` command and the expected output format.

**AC6:** Given the `/trace` SKILL.md, when the skill is run on a master branch validation, then the output includes a "Traces Branch Health" section reporting whether `origin/traces` is current or stale, with last commit timestamp.

## Out of Scope

- Alerting or notification to the operator outside of `npm test` output — the check is a passive CI gate, not an active alert system.
- Auto-remediation (e.g. automatically re-triggering the post-merge workflow) — fix is a human action.
- Adjusting the 24-hour threshold to a configurable value — the threshold is 24 hours and is not configurable in this story. Configuration can be added as a Phase 4 improvement.
- Changes to `pipeline-viz.html` — no viz changes required.

## NFRs

- **Performance:** `check-trace-commit.js` must complete within 5 seconds on a standard CI runner (single `git ls-remote` call).
- **Security:** The script must not require write access to any repository. Read-only git operations only.
- **Audit:** The script's exit code and printed output are sufficient audit record — no additional logging required.

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable
