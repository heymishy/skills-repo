# Story: Add estimation calibration as a feedback dimension in the improvement agent

**Epic reference:** artefacts/2026-04-14-skills-platform-phase3/epics/e7-platform-maturity.md
**Discovery reference:** artefacts/2026-04-14-skills-platform-phase3/discovery.md
**Benefit-metric reference:** artefacts/2026-04-14-skills-platform-phase3/benefit-metric.md

## User Story

As a **platform maintainer** managing long-term estimation accuracy,
I want the improvement agent to generate a calibration proposal when >30% underestimation is detected across 3 or more completed features in `workspace/estimation-norms.md`, so that the E1/E2 estimate guidance is updated before the estimation model drifts further from delivery actuals.

## Benefit Linkage

**Metric moved:** MM1 — Outer loop unassisted replication rate (improvement agent quality)
**How:** Estimation calibration is the weakest feedback loop in the current platform — estimates are recorded but not used to improve future estimates unless a maintainer manually reviews `estimation-norms.md`. Adding an improvement agent detection dimension closes this loop: the agent identifies the pattern, proposes a calibrated update, and a human approves via PR.

## Architecture Constraints

- The improvement agent reads `workspace/estimation-norms.md` and `workspace/results.tsv` to compute the underestimation rate. It does not call external estimation tools.
- The calibration proposal must be staged to `workspace/proposals/estimation-calibration-[date].md` — same staging pattern as other improvement agent proposals. It is not applied automatically.
- The human approval path is: review the proposal → open a PR updating the `/estimate` SKILL.md E1/E2 guidance section → merge via platform team review.
- The detection threshold is >30% underestimation (actual > estimate by more than 30%) across ≥3 features. The threshold is a named constant (`ESTIMATION_UNDERRUN_THRESHOLD = 0.30`, `ESTIMATION_MIN_FEATURE_COUNT = 3`) — not hardcoded inline.

## Dependencies

- **Upstream:** Requires at least 3 features with completed E3 actuals in `workspace/estimation-norms.md`. In the current repo this is satisfied by Phase 1, 2, and 3 delivery records.
- **Downstream:** None within Phase 3.

## Acceptance Criteria

**AC1:** Given `workspace/estimation-norms.md` contains E3 actuals for 3 or more completed features where actual focus hours exceeded estimated hours by more than 30%, when the improvement agent runs its calibration detection dimension, then it generates a calibration proposal at `workspace/proposals/estimation-calibration-[YYYY-MM-DD].md`.

**AC2:** Given a calibration proposal file, when it is reviewed, then it contains: the detected underestimation rate (as a percentage), the list of feature slugs contributing to the signal, the current E1/E2 guidance extract that is affected, a proposed updated guideline (e.g. adjusted complexity multiplier or base hour range), and the rationale for the adjustment.

**AC3:** Given the improvement agent's calibration dimension, when fewer than 3 features have E3 actuals or the underestimation rate is ≤30%, then no calibration proposal is generated — the dimension runs silently without error or warning.

**AC4:** Given a generated calibration proposal, when the maintainer has not yet reviewed it, then no automatic changes are made to any SKILL.md, `estimation-norms.md`, or `context.yml` file — the proposal is read-only until a human opens a PR.

**AC5:** Given the detection threshold constants (`ESTIMATION_UNDERRUN_THRESHOLD`, `ESTIMATION_MIN_FEATURE_COUNT`), when they are reviewed in the source, then they are defined as named constants in the improvement agent's calibration module — not as inline magic numbers.

## Out of Scope

- Automated application of calibration proposals — human review and PR are always required.
- Overestimation detection — only underestimation is in Phase 3 scope. Overestimation detection can be added as a Phase 4 proposal.
- Changes to how `/estimate` records actuals — that is the `/estimate` skill's E3 mode, unchanged by this story.

## NFRs

None — improvement agent computation is offline batch; no SLA applies.

## Complexity Rating

**Rating:** 2
**Scope stability:** Stable — detection logic and proposal format are well-defined.
