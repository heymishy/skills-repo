## Test Data Strategy

**Source:** Synthetic — mock trace registry fixtures for `getTraces()`:
- `fixtures/traces/squad-compliant/` — 100 traces, all with gate verdicts + all T3M1 fields → PASS
- `fixtures/traces/squad-failing/` — 100 traces, 15 missing gate verdicts (15% gap, >10% threshold) → FAIL
- `fixtures/traces/squad-boundary/` — 100 traces, 10 missing gate verdicts (exactly 10% — on threshold, not over) → PASS
- `fixtures/traces/squad-t3m1-missing/` — 50 traces, all gate verdicts present but 8 missing `standardsInjected` field
- `fixtures/traces/` — all fixtures conform to trace.schema.json; T3M1 fields as defined in p3.2a

**PCI/sensitivity:** No — delivery metadata only
**External network:** None — `getTraces()` mocked or reads from local fixtures

## Unit Tests

### report-generated-with-correct-filename-format

- **Verifies:** AC1 (output path and filename format)
- **Test file:** `src/improvement-agent/compliance-report.test.js`
- **Precondition:** Compliance report generation module exists at `src/improvement-agent/compliance-report.js`; `getTraces()` mocked to return fixture data
- **Action:** Run the compliance report generation function with a fixed current date (e.g. 2026-04-01); check the output filename
- **Expected result:** Output file created at `workspace/compliance-reports/compliance-2026-04.md` (YYYY-MM format matching the provided date)
- **RED step:** Module does not exist; filename format incorrect

### report-contains-per-squad-fields

- **Verifies:** AC2 (per-squad report fields)
- **Test file:** `src/improvement-agent/compliance-report.test.js`
- **Precondition:** Module exists; `getTraces()` returns fixture data for `squad-compliant` and `squad-failing`
- **Action:** Generate the compliance report; read the generated markdown; verify each squad block contains: squad ID, trace count sampled, gate verdicts present (yes/no), T3M1 fields populated (yes/no with count of missing), gap count, compliance status (PASS/FAIL)
- **Expected result:** All six fields present in each squad's block
- **RED step:** One or more fields are missing from the squad block

### squad-above-gap-threshold-is-marked-fail

- **Verifies:** AC3 (>10% gap threshold → FAIL)
- **Test file:** `src/improvement-agent/compliance-report.test.js`
- **Precondition:** Module exists; `squad-failing` fixture has 15/100 traces missing gate verdict (15% > 10%)
- **Action:** Generate report with `squad-failing` fixture; find squad-failing's entry in the report
- **Expected result:** `squad-failing` entry is marked FAIL; the 15 trace slugs missing gate verdicts are listed; the specific missing field name ("gate verdict") is named
- **RED step:** Squad marked PASS despite 15% gap; or FAIL without gap details

### squad-at-gap-threshold-is-not-marked-fail

- **Verifies:** AC3 (boundary — exactly 10% does not trigger FAIL)
- **Test file:** `src/improvement-agent/compliance-report.test.js`
- **Precondition:** Module exists; `squad-boundary` fixture has 10/100 traces missing gate verdict (exactly 10%)
- **Action:** Generate report with `squad-boundary` fixture
- **Expected result:** `squad-boundary` entry is marked PASS (10% is not greater than the 10% threshold)
- **RED step:** Threshold uses `>=` instead of `>` and incorrectly marks 10% as FAIL

### report-is-self-contained-with-context-fields

- **Verifies:** AC5 (self-contained — policy threshold, sample size, period, registry path)
- **Test file:** `src/improvement-agent/compliance-report.test.js`
- **Precondition:** Module exists; report generated
- **Action:** Read generated report; verify report header or footer contains: (a) the policy threshold value (10%), (b) the sample size (total traces reviewed), (c) the period covered (e.g. "last 30 days" or "2026-03-02 to 2026-04-01"), (d) a reference to the trace registry path (`platform/traces/`)
- **Expected result:** All four context fields present in the report
- **RED step:** Report lacks context fields; reviewer must ask platform team for interpretation context

## Integration Tests

### compliance-ci-job-uses-separate-schedule-trigger

- **Verifies:** AC4 (ADR-009 — separate trigger from delivery pipeline)
- **Test file:** `tests/check-trace-commit.js` (extend) or `tests/check-compliance-report.js`
- **Precondition:** Compliance monitoring CI workflow YAML exists
- **Action:** Read the compliance monitoring CI workflow YAML; verify it uses a `schedule:` trigger with monthly cadence (e.g. `cron: '0 0 1 * *'` or similar); verify it is a separate workflow file from the delivery pipeline and from the trace aggregation workflow (p3.7)
- **Expected result:** `schedule:` trigger present; monthly cadence confirmed; workflow is a separate file from delivery and aggregation workflows
- **RED step:** Workflow does not exist; or uses delivery pipeline trigger

### compliance-ci-job-uses-read-only-token

- **Verifies:** AC4 (read-only token constraint)
- **Test file:** `tests/check-compliance-report.js`
- **Precondition:** Compliance monitoring CI workflow YAML exists
- **Action:** Read the workflow YAML `permissions:` block; verify `contents: read` (or read-only equivalent); no write permissions to registry or delivery repos
- **Expected result:** Read-only token confirmed; no write permissions present
- **RED step:** Write permission granted in CI job
