### `dor-h-gov-fail-message-includes-remediation-step` — H-GOV fail message includes a remediation instruction

- **Verifies:** AC1 (part 3 — remediation)
- **Precondition:** H-GOV block contains fail message
- **Action:** Read SKILL.md H-GOV fail message; assert it includes a remediation instruction (e.g. "Return to the discovery artefact and populate Approved By before re-running /definition-of-ready")
- **Expected result:** Remediation step is present — operator knows what to do next
- **Edge case:** No

### `dor-h-gov-pass-condition-is-text-presence` — H-GOV pass condition is text-presence, not role classification

- **Verifies:** AC2 (pass on substantive text — role classification not required)
- **Precondition:** H-GOV block specifies pass condition
- **Action:** Read SKILL.md H-GOV pass condition; assert it specifies "text is present and non-placeholder" as the passing bar — not "role has been verified as non-engineering"
- **Expected result:** Pass condition is text-presence only; role classification is not a hard requirement
- **Edge case:** Yes — role classification would over-block valid approvals

### `dor-h-gov-m1-signal-for-role-unverified` — H-GOV records M1 metric signal when text present but role not clearly non-engineering

- **Verifies:** AC2 (M1 signal for unverified role)
- **Precondition:** H-GOV block is present
- **Action:** Read SKILL.md H-GOV block; assert it contains an instruction to record a metric signal (M1 — attribution quality) when Approved By has text but the role is unclear or potentially engineering
- **Expected result:** M1 signal instruction present for ambiguous role case
- **Edge case:** No

### `dor-h-gov-positive-m1-signal-for-clear-non-engineer` — H-GOV records positive M1 signal when approver is clearly non-engineering

- **Verifies:** AC3
- **Precondition:** H-GOV block is present
- **Action:** Read SKILL.md H-GOV block; assert it contains an instruction to record a positive M1 metric signal when the approver's title/role clearly identifies them as non-engineering (e.g. PM, business stakeholder, product owner)
- **Expected result:** Positive M1 signal instruction present — distinct from the ambiguous-role signal
- **Edge case:** No

### `dor-h-gov-benefit-metric-note-is-trigger-to-check` — H-GOV treats "Attribution incomplete" benefit-metric note as a trigger-to-check signal only

- **Verifies:** AC4 (note is not permanent block)
- **Precondition:** H-GOV block is present
- **Action:** Read SKILL.md H-GOV block; assert it contains an instruction that, when a benefit-metric "Attribution incomplete" note is detected, H-GOV performs a live read of the discovery `Approved By` field rather than automatically firing
- **Expected result:** "trigger-to-check" (or equivalent) instruction is present; note alone does not equal permanent block
- **Edge case:** Yes — confusing note-triggered block with permanent block was the original AC4 ambiguity (resolved in review run 2)

### `dor-h-gov-passes-when-discovery-field-corrected-despite-note` — H-GOV passes when discovery Approved By is corrected, regardless of benefit-metric note

- **Verifies:** AC4 (live read determines outcome)
- **Precondition:** H-GOV block contains trigger-to-check instruction
- **Action:** Read SKILL.md H-GOV block; assert it explicitly states that if the discovery `Approved By` field is now substantively populated, H-GOV passes on the current DoR run — and does not hold the story blocked merely because the benefit-metric note still exists
- **Expected result:** Pass condition based on live discovery field state — not on note presence
- **Edge case:** Yes — stale note from before correction should not permanently block

### `dor-h-gov-absent-section-message-distinct-from-empty-field` — H-GOV has distinct messaging for absent attribution section vs empty Approved By field

- **Verifies:** AC5 (absent section distinct from empty field)
- **Precondition:** H-GOV block is present
- **Action:** Read SKILL.md H-GOV block; assert it contains distinct message text or branching instructions for: (a) attribution section absent from discovery artefact entirely, (b) attribution section present but Approved By field empty or placeholder
- **Expected result:** Two distinct message paths — the operator knows specifically which condition applies
- **Edge case:** Yes — absent section is a stronger signal (likely old artefact) requiring clearer guidance

### `dor-h-gov-absent-section-references-old-artefact` — H-GOV absent-section message references the likelihood of a pre-i3.1 artefact

- **Verifies:** AC5 (contextualisation for absent section)
- **Precondition:** H-GOV block has absent-section message
- **Action:** Read SKILL.md H-GOV absent-section message; assert it provides context indicating the artefact may pre-date the attribution fields requirement (i.e. was created before i3.1 was merged)
- **Expected result:** Absent-section message is contextualised — not a generic "section missing" error
- **Edge case:** No
