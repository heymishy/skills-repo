## FINDINGS

### 1-M1 — MEDIUM — AC2 dual-outcome routing with unspecified branching condition

AC2 states: "then the skill routes to either /clarify (if the discovery text is vague or incomplete) or back to /discovery (if approval is the only remaining step)." Two distinct outcomes are bundled into one AC with a parenthetical condition that is not observable from the outside. A test cannot determine which branch fires without knowing the skill's internal evaluation logic. The branching condition ("vague or incomplete" vs. "approval is the only remaining step") is qualitative and not independently testable.

**Recommended action:** Split AC2 into two ACs — one for the "discovery exists but is vague/incomplete → route to /clarify" path and one for the "discovery exists, approval is the only remaining step → route back to /discovery" path. Each must specify a concrete detectable signal that triggers it.

---

### 1-M2 — MEDIUM — AC6 dual-outcome routing with ambiguous branch condition

AC6 states: "routes to /verify-completion (if a branch exists with test results) or /branch-setup (if no branch has been set up yet)." The condition "if a branch exists with test results" is not defined — which branch? detected how? what constitutes test results? A SKILL.md instruction must give the agent a reliable detection criterion so the routing is deterministic. As written, either outcome could be claimed as correct by the executing agent for the same repo state.

**Recommended action:** Define the observable detection criterion for each branch: e.g. "a branch whose name matches the story slug exists in the repo and contains test result artefacts under `workspace/`" for the /verify-completion path, or specify the simpler rule (e.g. "no worktree exists under `workspace/`" → /branch-setup).
