## Story: Add required attribution fields to /discovery SKILL.md output template

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/initiative-3-governance-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-3-governance-benefit-metric.md

## User Story

As a **platform maintainer**,
I want the `/discovery` skill to require `Contributors`, `Reviewers`, and `Approved By` fields in every discovery artefact it produces,
So that the attribution of non-engineering involvement is structurally captured at discovery — moving the MM2 attribution completeness metric from 0% to a measurable state.

## Benefit Linkage

**Metric moved:** MM2 — Attribution field completeness rate (currently 0% — fields do not exist in the template)
**How:** Adding the three attribution fields to the discovery output template means every discovery artefact produced after this story merges will have placeholder slots for Contributors, Reviewers, and Approved By — creating the artefact-level data structure that MM2 measures. Without this story, H-GOV (i3.3) has no field to check.

## Architecture Constraints

- ADR-011: this story modifies `.github/skills/discovery/SKILL.md` — a governed skill file. The change must go via PR with platform team review. The coding agent opens a draft PR; no merge until reviewed.
- Platform change policy: SKILL.md changes require PR — never direct commit to master.
- Skill and template files are Markdown only; no embedded HTML except HTML comments.
- Follow established section headings in existing skill files.
- No trailing whitespace; Unix line endings.

## Dependencies

- **Upstream:** None — this is the first story in Initiative 3; no prerequisite stories exist.
- **Downstream:** i3.2 (benefit-metric attribution acknowledgement) and i3.3 (H-GOV enforcement) both depend on the attribution fields existing in discovery artefacts. i3.3 cannot be meaningfully tested without i3.1 merged first.

## Acceptance Criteria

**AC1:** Given the `/discovery` SKILL.md, when the skill produces a discovery artefact, then the artefact template includes a `## Attribution` section with three sub-fields: `Contributors` (one or more names and roles), `Reviewers` (one or more names and roles), and `Approved By` (name — role — date, or "Pending").

**AC2:** Given the updated `/discovery` SKILL.md, when an operator completes the discovery skill and the `Approved By` field is left as the default placeholder (empty or "Pending"), then the skill's completion message includes an explicit reminder that `Approved By` must be filled before the discovery status is set to "Approved".

**AC3:** Given the updated discovery template, when an operator fills in `Approved By` with a non-empty value in the format `Name — Role — Date`, then the discovery artefact is considered attribution-complete for the purposes of the H-GOV block (i3.3).

**AC4:** Given an existing discovery artefact created before this story merges, when the operator views it, then it continues to render correctly — this story does not retroactively modify existing artefacts.

**AC5:** Given the `/discovery` SKILL.md's Step 1 (context gathering / operator prompting), when the skill runs in a session, then the skill prompts for contributor names and roles before finalising the artefact — not as an afterthought at the end.

## Out of Scope

- Automated validation that `Approved By` contains a non-engineering name (role classification is human judgment; mechanical checks are H-GOV's responsibility in i3.3)
- Attribution fields on any artefact type other than `discovery.md` — benefit-metric attribution is i3.2's scope
- Section-level or per-AC attribution tracking — artefact-level only
- Changes to the discovery artefact schema in `pipeline-state.json` — attribution fields are SKILL.md instruction changes only; no state schema change is needed

## NFRs

- **Consistency:** The attribution section must follow the same Markdown formatting conventions used elsewhere in the discovery template (heading level, field format)
- **Readability:** Field labels and format hints must be clear enough that a non-technical operator can fill them in without guidance

## Complexity Rating

**Rating:** 1
**Scope stability:** Stable — the exact fields (Contributors, Reviewers, Approved By) and format (`Name — Role — Date`) are specified in the discovery artefact.
