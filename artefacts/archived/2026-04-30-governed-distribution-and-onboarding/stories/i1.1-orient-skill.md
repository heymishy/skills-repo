## Story: Create `/orient` concierge SKILL.md that reads pipeline state and routes operator to next skill

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i1-orient-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-1-onboarding-benefit-metric.md

## User Story

As a **platform consumer (new or returning, including non-technical outer loop participants)**,
I want to run `/orient` at any point in a repo and receive a clear, specific instruction for what to do next,
So that I can progress through the pipeline without needing to read the pipeline documentation or ask the platform team — moving me from "what do I do?" to "running my next skill" in under a minute.

## Benefit Linkage

**Metric moved:** I1-M1 (Time-to-first-skill-run), I1-M2 (Zero orientation contacts), I1-MM3 (/orient routing coverage)
**How:** `/orient` eliminates the "which skill do I run?" stall that causes most of the 15–45 minute observed onboarding delay. When a consumer's first action after `platform:init` is `/orient`, they receive a concrete next step immediately — the orientation contact that previously required the platform team is handled by the skill.

## Architecture Constraints

- New SKILL.md file at `.github/skills/orient/SKILL.md` — follows the standard SKILL.md format used by all skills in this repo (see `.github/skills/*/SKILL.md` for reference).
- Skill files are Markdown only — no Node.js scripts, no tooling dependencies. The skill is executed by an AI agent in VS Code context.
- ADR-011: A new SKILL.md under `.github/skills/` requires a story artefact (this file) + test plan + DoR before the skill file is committed. This story satisfies ADR-011 for the `/orient` skill.
- `/orient` is distinct from `/start` (greenfield only, single-use) and `/workflow` (diagnostic for experienced operators). The distinction must be explicit in the skill description.

## Dependencies

- i3.1/i3.2/i3.3 must be merged before i1.1 is dispatched (H-GOV enforcement must be active before I1 begins).

## Acceptance Criteria

**AC1**
Given a repo with no `artefacts/` folder and no `pipeline-state.json`,
when a consumer runs `/orient`,
then the skill output: (a) identifies the current state as "no artefacts found — this is a new repo", (b) routes to `/discovery` as the next step, (c) explains in one sentence what `/discovery` produces, and (d) tells the operator the exact command to invoke (`/discovery`).

**AC2a**
Given a repo where `discovery.md` exists under `artefacts/[feature]/` but its `Status` field is not "Approved", and the discovery text is missing required sections or contains placeholder content (e.g. "[FILL IN]" markers or fewer than 3 completed sections),
when `/orient` runs,
then the skill routes to `/clarify` and names the specific incompleteness detected (e.g. "discovery exists but problem statement section is incomplete — run /clarify to sharpen it before seeking approval").

**AC2b**
Given a repo where `discovery.md` exists under `artefacts/[feature]/`, all required sections are populated, but its `Status` field is not "Approved",
when `/orient` runs,
then the skill routes back to `/discovery` and states that the discovery is complete but awaiting approval (e.g. "discovery exists and appears complete — approval is the remaining step; run /discovery to request approval or mark Status: Approved").

**AC3**
Given a repo where `discovery.md` status is "Approved" but no benefit-metric artefact exists for that feature,
when `/orient` runs,
then the skill routes to `/benefit-metric` and explains that the next step is defining measurable targets for the discovery.

**AC4**
Given a repo where a benefit-metric artefact is active but no story artefacts exist for that feature,
when `/orient` runs,
then the skill routes to `/definition` and explains that the next step is breaking the discovery into epics and stories.

**AC5**
Given a repo where stories exist but at least one story has no test plan,
when `/orient` runs,
then the skill identifies the first story without a test plan by name and routes to `/test-plan` for that story specifically.

**AC6a**
Given a repo where stories exist with `dorStatus: "signed-off"` but no merged PR, and a worktree or branch matching the story slug exists under `workspace/` or in git branches,
when `/orient` runs,
then the skill routes to `/verify-completion` and states that implementation appears to be in progress (e.g. "branch found for [story-slug] — run /verify-completion to check whether all ACs are met before opening a PR").

**AC6b**
Given a repo where stories exist with `dorStatus: "signed-off"` but no merged PR, and no worktree or branch matching the story slug exists,
when `/orient` runs,
then the skill routes to `/branch-setup` and states that no implementation branch has been created yet (e.g. "story [story-slug] is DoR signed-off but no branch found — run /branch-setup to create an isolated worktree and begin implementation").

## Out of Scope

- Brownfield codebase detection (Entry A/B/C) — that is i2.x scope delivered after I1.
- Executing any pipeline skill on the operator's behalf — routing only.
- Artefact quality validation (checking that discovered artefacts are well-formed) — that is `/workflow` and `/trace` scope.
- Multi-repo or fleet-level orientation — single-repo context only in MVP.

## NFR

- **Routing completeness:** All 8 routing states (AC1, AC2a, AC2b, AC3, AC4, AC5, AC6a, AC6b) must be covered without gaps — an operator in any supported pipeline state must receive a specific routing recommendation, never a generic "I'm not sure" response.
- **Conciseness:** Each routing output must be actionable in one read — state identified, next skill named, one-sentence rationale. No routing output should require the operator to read more than 3–4 lines before knowing what to do.
- **Consistency:** Field names and skill names referenced by `/orient` must match the canonical names used in the pipeline documentation (`pipeline-state.json`, `copilot-instructions.md`).

## Complexity

**2** — moderate. The routing logic is straightforward (read state, match to routing rule) but there are 6 distinct states, and the skill must handle ambiguous states gracefully (e.g. partially completed features with mixed story stages). The file-read heuristics for detecting pipeline state need to be reliable across the variety of repo configurations that real consumers will have.

## Scope Stability

Stable for MVP (6 routing states). Post-I1 extension: brownfield states (handled by I2), multi-feature repos (one feature in progress + one pending), and fleet routing (handled by fleet-state.json in a separate story if needed).
