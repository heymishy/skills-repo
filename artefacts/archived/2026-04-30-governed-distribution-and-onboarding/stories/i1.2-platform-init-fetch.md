## Story: Implement `platform:init` and `platform:fetch` CLI commands for one-command repo bootstrap

**Epic reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/epics/i1-lockfile-cli-epic.md
**Discovery reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/discovery.md
**Benefit-metric reference:** artefacts/2026-04-30-governed-distribution-and-onboarding/benefit-metric/initiative-1-onboarding-benefit-metric.md

## User Story

As a **platform consumer (engineer adopter on a new or existing repo)**,
I want to run a single command (`npm run platform:init`) to install the complete platform skill set into my repo,
So that I can start using the pipeline immediately without navigating documentation, copying files manually, or understanding the repository layout — reducing setup time from 15–45 minutes to under 2 minutes.

## Benefit Linkage

**Metric moved:** I1-M1 (Time-to-first-skill-run), I1-M2 (Zero orientation contacts), I1-MM2 (CLI commands implemented with zero new npm dependencies)
**How:** `platform:init` replaces the current onboarding process (read README, clone or copy files, understand directory structure, configure VS Code) with a single command that performs all those steps automatically, making the 2-minute target achievable.

## Architecture Constraints

- **No new runtime npm dependencies.** `platform:init` and `platform:fetch` must be implemented using Node.js built-ins only (`fs`, `path`, `crypto`, `child_process` for git operations). This constraint is a hard requirement from I1-MM2 — post-init, the consumer's `package.json` must show zero new `dependencies` entries from this story's implementation. (Existing devDependencies for CI/testing are not affected.)
- **CommonJS modules only.** All scripts follow the existing pattern in `scripts/` — `require()` syntax, `.js` extension, no TypeScript, no transpilation step.
- **Non-destructive default.** `platform:init` must not overwrite files the consumer has customised without warning. Safe default: skip files that already exist; report skipped files to the operator.
- **ADR-011:** This story implements CLI scripts under `scripts/` and adds entries to `package.json` — both governed by the artefact-first rule. This story artefact satisfies ADR-011 for `platform:init` and `platform:fetch`.

## Dependencies

- i1.1 (`/orient` skill) should be delivered first so that `platform:init` can recommend running `/orient` as its post-install next step.
- i3.1/i3.2/i3.3 must be merged (I3 prerequisite gate).

## Acceptance Criteria

**AC1**
Given a new repo with Node.js and npm installed and no existing platform files,
when a consumer runs `npm run platform:init`,
then the command copies all `.github/skills/`, `.github/templates/`, and the core `scripts/` platform files into the repo, creates or updates `.github/copilot-instructions.md` with the standard platform instructions, and outputs a summary confirming what was installed — without requiring the consumer to clone the platform repo manually or understand the source directory layout.

**AC2**
Given `platform:init` completes successfully,
when the consumer opens VS Code in that repo,
then all pipeline skills (including `/orient`, `/discovery`, `/definition`, etc.) are available as slash commands and the consumer can invoke any skill without additional setup.

**AC3**
Given a repo where platform files already exist (a previous `platform:init` was run),
when `platform:init` runs again,
then files that already exist are not overwritten by default; the command outputs a list of skipped files and offers a `--force` flag to overwrite if the consumer explicitly wants to refresh.

**AC4**
Given a consumer wants to retrieve the latest skill files from the platform upstream,
when they run `npm run platform:fetch`,
then the command fetches the current HEAD of the platform's `master` branch skill files and copies them to the repo, recording the fetch timestamp in `workspace/` — without requiring the consumer to have the platform repo as a git remote or understand git fetch mechanics.

**AC5**
Given `platform:init` or `platform:fetch` runs to completion,
when the consumer inspects their repo's `package.json` `dependencies` field,
then no new runtime npm packages have been added — all implementation uses Node.js built-in modules only.

## Out of Scope

- `platform:pin` and `platform:verify` (lockfile commands) — that is i1.3 scope.
- Automated conflict resolution when existing customised files differ from the new version — the `--force` flag in AC3 is sufficient for MVP.
- Support for alternative package managers (yarn, pnpm) — `npm run` only in MVP.
- Windows PowerShell-specific install path — the script runs via `node scripts/platform-init.js`; PowerShell compatibility is handled by Node.js, not a separate script.

## NFR

- **Zero new npm runtime dependencies** — verified by inspecting `package.json` `dependencies` diff pre/post implementation.
- **Idempotent** — running `platform:init` twice on a clean repo produces the same result as running it once; no duplicate writes, no errors on second run.
- **Clear error output** — if the command cannot complete (e.g., missing Node.js version, network unavailable for `platform:fetch`), it exits with a non-zero code and a human-readable error message naming the specific failure.

## Complexity

**2** — file-copy logic is well-understood, but the `platform:fetch` command requires a network call (git fetch or HTTPS download) and the non-destructive default for existing files adds edge-case handling. Node.js built-ins make this achievable without dependencies.

## Scope Stability

Stable for MVP. Post-I1 extension: interactive prompts for `platform:init` (choose which skill packs to install), multi-version fetch (fetch a specific release tag), and fleet-level init (init across all repos in `fleet-state.json`).
