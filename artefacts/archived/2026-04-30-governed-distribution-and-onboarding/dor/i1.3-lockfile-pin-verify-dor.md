## Contract Proposal — i1.3

**What will be built:**
Two new Node.js CLI scripts, two new `package.json` entries, and a `platform-lock.json` schema:
1. `scripts/platform-pin.js` — computes SHA-256 hash for each installed skill file and writes `platform-lock.json` with V1 schema: `{ "version": "1", "pinnedAt": "<ISO-8601>", "platform": "<semver-or-sha>", "skills": { "<relative-path>": "<sha256-hex>" } }`
2. `scripts/platform-verify.js` — reads `platform-lock.json`, recomputes SHA-256 hashes, exits 0 with one-line confirmation if all match; exits non-zero with three-part error (file path, expected SHA, actual SHA, fix command) if any drifted; exits non-zero with "run platform:pin first" if no lockfile
3. `package.json` — adds `"platform:pin": "node scripts/platform-pin.js"` and `"platform:verify": "node scripts/platform-verify.js"`

**What will NOT be built:**
- Automatic re-pinning on fetch
- Lock file migration tooling (schema V1 only)
- CI pipeline integration
- Cryptographic signing (SHA-256 only)

**How each AC will be verified:**

| AC | Test approach | Type |
|----|---------------|------|
| AC1 — pin creates platform-lock.json with version, timestamp, platform ID, per-file SHA | Assert script exists; package.json entry exists; running pin on temp dir creates lock with all required fields; SHA-256 values are correct | Unit |
| AC2 — verify exits 0 + one-line confirmation when all files match | Assert verify exits 0 on clean state; output matches "All N skill files match…" format | Unit + Integration |
| AC3 — verify exits non-zero + names drifted file + shows SHA diff + names fix command | Assert non-zero exit; output contains drifted file path, expected SHA, actual SHA, npm run platform:pin or platform:fetch | Unit + Integration |
| AC4 — lockfile is portable across environments | Assert paths in platform-lock.json use forward slashes regardless of OS; verify runs cleanly on second temp dir with copied lockfile | Integration |
| AC5 — no lockfile → exits non-zero with "run platform:pin first" message | Assert non-zero exit; output contains "platform:pin" instruction | Unit |
| NFR-actionable | Assert three-part error (name, expected vs actual SHA, fix command) is self-contained in one output block | Unit |
| NFR-deterministic | Assert same content produces identical SHA on repeated runs; assert metadata (mtime, ctime) changes do not affect hash | Unit |
| NFR-paths | Assert platform-lock.json keys use forward slashes on Windows (path normalisation) | Unit |

**Assumptions:**
- Skill files to hash are those under `.github/skills/` (relative paths from repo root)
- `platform` field in lockfile is derived from git SHA of the HEAD at pin time (`git rev-parse HEAD`)
- `platform-lock.json` is sorted by file path for diffability
- Hashing is computed on raw file content read as Buffer, no normalisation

**Estimated touch points:**
Files created: `scripts/platform-pin.js`, `scripts/platform-verify.js`
Files modified: `package.json` (2 new script entries)
Runtime-created file (not committed with code): `platform-lock.json` (created at consumer run time)
Services: None
APIs: None (git CLI via `child_process` for SHA)

### Implementation guidance

1. `scripts/platform-pin.js`:
   - Find all skill files under `.github/skills/` recursively
   - Use `crypto.createHash('sha256').update(fs.readFileSync(filePath)).digest('hex')` for each file
   - Normalise paths to forward slashes using `.replace(/\\/g, '/')`
   - Sort keys alphabetically for diffability
   - Derive `platform` field from `child_process.execSync('git rev-parse HEAD').toString().trim()`
   - Write `platform-lock.json` to repo root with JSON.stringify(lock, null, 2)

2. `scripts/platform-verify.js`:
   - If `platform-lock.json` does not exist: exit non-zero, print "No platform-lock.json found. Run npm run platform:pin first."
   - Read lockfile; re-hash each listed file
   - If all match: exit 0, print "All N skill files match pinned version [platform-id]. ✓"
   - If any drifted: exit 1, for each drifted file print the file path, expected SHA, actual SHA, and "Run npm run platform:pin to re-pin or npm run platform:fetch to refresh."

3. Both scripts: Node.js built-ins only — `fs`, `path`, `crypto`, `child_process`. Zero new npm deps.
