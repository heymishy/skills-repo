## Contract Proposal — i1.1

**What will be built:**
A new SKILL.md at `.github/skills/orient/SKILL.md` implementing the `/orient` concierge skill. The skill will:
1. Read the current repo's artefact state (presence/absence of `artefacts/`, `discovery.md`, `benefit-metric`, stories, test plans, `dorStatus` values, and branch/worktree markers)
2. Route to the correct next skill for all 8 pipeline states: new repo → /discovery; incomplete discovery → /clarify; complete unapproved discovery → /discovery for approval; no benefit-metric → /benefit-metric; no stories → /definition; story missing test plan → /test-plan; DoR signed-off with branch → /verify-completion; DoR signed-off no branch → /branch-setup
3. Include a description that explicitly distinguishes /orient from /start (greenfield only) and /workflow (experienced operator diagnostic)

**What will NOT be built:**
- Brownfield Entry A/B/C routing — that is i2.x scope
- Automated skill execution — /orient routes only, does not run skills
- Artefact quality validation — /workflow and /trace scope
- Multi-repo or fleet-level orientation — single-repo context only

**How each AC will be verified:**

| AC | Test approach | Type |
|----|---------------|------|
| AC1 — new repo routes to /discovery with 4-part output | Assert SKILL.md contains new-repo state language, /discovery command, one-sentence explanation, exact command string | Unit |
| AC2a — incomplete discovery routes to /clarify naming gap | Assert SKILL.md contains /clarify routing + incompleteness detection description | Unit |
| AC2b — complete unapproved discovery routes to /discovery for approval | Assert SKILL.md contains approval-pending routing language | Unit |
| AC3 — approved + no benefit-metric → /benefit-metric | Assert SKILL.md contains /benefit-metric routing for post-approval state | Unit |
| AC4 — benefit-metric active + no stories → /definition | Assert SKILL.md contains /definition routing for post-benefit-metric state | Unit |
| AC5 — story missing test plan → /test-plan naming the story | Assert SKILL.md contains /test-plan routing with story-name reference | Unit |
| AC6a — DoR signed-off + branch exists → /verify-completion | Assert SKILL.md contains /verify-completion routing for branch-found state | Unit |
| AC6b — DoR signed-off + no branch → /branch-setup | Assert SKILL.md contains /branch-setup routing for no-branch state | Unit |
| NFR — all 8 states covered | Integration: verify all 8 routing outputs are present; no generic fallback response | Integration |
| NFR — /orient distinct from /start and /workflow | Assert file references both /start and /workflow with distinguishing language | Unit |
| NFR — canonical field names (dorStatus, pipeline-state.json) | Assert canonical names appear in routing descriptions | Unit |

**Assumptions:**
- The SKILL.md format follows the standard used by all skills in `.github/skills/*/SKILL.md` (YAML frontmatter + Markdown body)
- File-system heuristics (presence of `artefacts/`, `discovery.md`, `dorStatus` values) are sufficient for routing; no semantic parsing of artefact content is needed
- A SKILL.md is executed by an AI agent reading it as instructions — routing logic is expressed as natural language instructions, not executable code
