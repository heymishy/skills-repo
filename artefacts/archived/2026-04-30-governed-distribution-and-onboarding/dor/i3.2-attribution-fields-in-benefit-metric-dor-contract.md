# DoR Contract: i3.2 — Attribution acknowledgement in /benefit-metric SKILL.md

**Feature:** 2026-04-30-governed-distribution-and-onboarding
**Story:** i3.2-attribution-fields-in-benefit-metric
**Date:** 2026-04-30

---

## What will be built

Edit `.github/skills/benefit-metric/SKILL.md` to add:

1. A first-step attribution check: read discovery artefact's `Approved By` field before asking any metric-definition questions
2. A warning branch for empty / placeholder / absent `Approved By`: surface a named warning with two options — (a) pause and get attribution, (b) proceed with documented acknowledgement
3. A clean-pass branch for populated `Approved By`: proceed without any interruption
4. An "absent section = empty" rule: pre-i3.1 legacy artefacts without `## Attribution` are treated the same as empty — triggers the warning from case 2
5. A visible note in the benefit-metric artefact when option (b) is chosen: "Attribution incomplete — Approved By was empty at benefit-metric time. Revisit before DoR."
6. Field name used: `Approved By` — matching i3.1 cross-story consistency

## What will NOT be built

- A hard block preventing the skill from proceeding (soft gate only — i3.3 enforces the hard block at DoR)
- Automated role classification of the `Approved By` value
- Changes to `pipeline-state.schema.json`
- Attribution checks on benefit-metric artefacts themselves (MM3 scope, out of scope for i3.2)
