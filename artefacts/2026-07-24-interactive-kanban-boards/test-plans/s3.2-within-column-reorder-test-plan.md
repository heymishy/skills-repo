## Unit Tests

### nonDragReorderAlternativeExists_OR_exceptionLogged

- **Verifies:** AC4
- **Precondition:** Rendered column with 2+ cards
- **Action:** Check for a non-drag reorder control (e.g. up/down buttons) in the rendered markup
- **Expected result:** EITHER the control exists and is keyboard-activatable, OR `decisions.md` contains a logged exception for this story dated before DoR sign-off (this test checks for the control; the exception-logged case is verified by DoR's own check, not this test — if the control is absent, this test should fail until the exception is logged, which is verified separately at DoR)
- **Edge case:** No

---

## Integration Tests

### reorderedPositionPersistsAcrossReload

- **Verifies:** AC2
- **Components involved:** Whichever persistence mechanism the coding agent chooses (server field or client-only)
- **Precondition:** A column with 2+ cards, one reordered
- **Action:** Simulate a page reload (re-fetch board data / re-read client storage)
- **Expected result:** The reordered position is reflected in the re-fetched/re-read state — not reset to the original order

---

## E2E Tests

### withinColumnDragReordersNoServerCall

- **Verifies:** AC1
- **Precondition:** A column with 2+ cards
- **Action:** Real browser drag (Playwright) of one card to a different position within the same column
- **Expected result:** Cards visually reorder; network interception confirms zero calls to S1.1's advance endpoint

### betweenColumnDragStillUsesS31Logic

- **Verifies:** AC3
- **Precondition:** A board with 2+ columns, a ready card
- **Action:** Drag the card between columns (not within one)
- **Expected result:** This is handled by S3.1's logic (advance attempt occurs) — confirms the two drag types are correctly distinguished, not confused with each other
