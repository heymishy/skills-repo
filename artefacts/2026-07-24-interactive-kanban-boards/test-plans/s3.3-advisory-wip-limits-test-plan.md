## Unit Tests

### overLimitColumnShowsDistinctIndicator

- **Verifies:** AC1
- **Precondition:** A column configured with limit 4, containing 5 cards
- **Action:** Render the column via `_renderKanbanColumns`
- **Expected result:** The `kb-wip--over`-equivalent class/styling is applied, badge text reads "5/4" (count/limit)
- **Edge case:** No

### atLimitColumnShowsPlainCount

- **Verifies:** AC2
- **Precondition:** A column configured with limit 4, containing 4 cards (at limit, not over)
- **Action:** Render the column
- **Expected result:** Plain count badge ("4"), no over-limit styling — "at limit" is not treated as "over limit"
- **Edge case:** Yes — exactly-at-limit boundary condition, distinct from over-limit

### limitConventionConsistentAcrossScopes

- **Verifies:** AC4
- **Precondition:** The same stage (e.g. `review`) rendered on product, org, and tenant board data
- **Action:** Compare the configured limit value used for `review` across all three renders
- **Expected result:** Identical limit value applied — no per-scope divergence without a documented reason
- **Edge case:** No

## Integration Tests

### advanceIntoOverLimitColumnStillSucceeds

- **Verifies:** AC3
- **Components involved:** S1.1's advance endpoint, WIP-limit check
- **Precondition:** A target column already over its WIP limit; a ready card elsewhere
- **Action:** Advance the ready card into the over-limit column (via S1.1's endpoint)
- **Expected result:** The advance succeeds exactly as it would into a non-over-limit column — WIP-limit state has zero effect on the real action's success

### dragIntoOverLimitColumnStillSucceeds

- **Verifies:** AC3
- **Components involved:** S3.1's drag mechanism, WIP-limit check
- **Precondition:** Same as above, via drag instead of click
- **Action:** Drag the ready card into the over-limit column
- **Expected result:** The drop succeeds exactly as it would into a non-over-limit column
