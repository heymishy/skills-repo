**AC2 flagged:** "visually match ... same token values" — this is a rendered-visual-comparison assertion. A DOM-simulation test (jsdom) cannot verify actual rendered colour/contrast; only a real browser or a human eye can. No visual-regression tooling (e.g. Playwright screenshot-diff) is configured in this repo today (confirmed 2026-07-24).

**Handling decision (per CLAUDE.md's B2 rule and Review finding s2.1-1-M1):** Manual verification, written to the higher quality bar, marked 🔴 in the verification script. AC1 (no raw hex remaining) and AC3 (dark-theme token override reuse) ARE automatable — those are static code/CSS-source assertions, not rendered-visual assertions, so they don't require this treatment.

## Unit Tests

### noRawHexColoursInLiveRenderer

- **Verifies:** AC1
- **Precondition:** `_renderKanbanColumns({columns: [...]})` called with representative test data
- **Action:** Regex-scan the returned HTML string's `<style>` block for hex colour patterns (`#[0-9a-fA-F]{3,8}`)
- **Expected result:** Zero matches — every colour reference is a `var(--token-name)`, confirmed via a companion assertion that `var(--` appears the expected number of times
- **Edge case:** No

### darkThemeOverrideBlockPresent

- **Verifies:** AC3 (static part)
- **Precondition:** N/A — static source check
- **Action:** Confirm `_renderKanbanColumns`'s CSS references the same token names already defined in `html-shell.js`'s dark-theme override block (no NEW dark-theme CSS authored specifically for the kanban board)
- **Expected result:** No kanban-board-specific dark-theme override exists — it relies entirely on the shared `html-shell.js` tokens cascading through
- **Edge case:** No
