## Unit Tests

### readinessLookupIsBatchedNotLinear

- **Verifies:** AC3
- **Precondition:** A board-data-fetch function is called with N=5 cards, then N=50 cards, each requiring a session-readiness check
- **Action:** Count the number of session-store read calls (via a spy/mock on the session-store's read function) for each N
- **Expected result:** Read count does not grow linearly with N — e.g. a single batched call (or a small constant number of calls) regardless of N, not N separate calls
- **Edge case:** No

### notReadyCardOffersNoAdvanceAction

- **Verifies:** AC2
- **Precondition:** A card's underlying journey has `session.done !== true`
- **Action:** Render the card's server-side HTML for the Advance action
- **Expected result:** The Advance action markup (e.g. the button/link) is absent or disabled — not merely hidden via CSS (a real, server-computed absence)
- **Edge case:** No

## Integration Tests

### readyCardAdvanceCallsRealGateConfirm

- **Verifies:** AC1
- **Components involved:** New board-action endpoint, `handlePostGateConfirm`, journey store
- **Precondition:** A real test journey with a `done: true` session at stage `discovery`
- **Action:** POST to the new board-action endpoint for this card
- **Expected result:** The journey's `activeSkill`/stage advances exactly as it would via the existing chat-session gate-confirm button; response reflects the new stage

### readyCardAdvanceUpdatesCompletedStages

- **Verifies:** AC1
- **Components involved:** Same as above
- **Precondition:** Same as above
- **Action:** Same POST, then GET the journey state
- **Expected result:** `completedStages` includes the just-completed stage, matching `handlePostGateConfirm`'s existing behaviour exactly (no new/different completedStages logic introduced)

### crossTenantAdvanceRejected404

- **Verifies:** AC4
- **Components involved:** New board-action endpoint, tenant-ownership check
- **Precondition:** Journey belongs to tenant B; request authenticated as tenant A
- **Action:** POST to the board-action endpoint for tenant B's journey, as tenant A
- **Expected result:** HTTP 404 (not 403); no change to the journey's stage or `completedStages` (verified via a follow-up GET as an authorized tenant-B request)

### realGateConfirmFailureSurfacesActualReason

- **Verifies:** AC5
- **Components involved:** New board-action endpoint, `_validate()`
- **Precondition:** A `done: true` session whose artefact content fails `_validate()` for the target gate
- **Action:** POST to the board-action endpoint
- **Expected result:** Response body contains the real validation error detail (matching `handlePostGateConfirm`'s existing 422 `{error: 'validation-failed', exitCode, detail}` shape) — not a generic "advance failed" message

## E2E Tests

### boardAdvanceButtonMovesCardToNextColumn

- **Verifies:** AC1 (visual confirmation layer, in addition to the Integration test's API-level assertion)
- **Precondition:** A logged-in test session viewing the product-level kanban board, one ready card
- **Action:** Click the card's Advance action in a real browser (Playwright, `tests/e2e/`, local `NODE_ENV=test` harness)
- **Expected result:** After the click, the card visually appears in its new stage's column on page reload/re-render
- **Edge case:** No

### notReadyCardHasNoClickableAdvanceInBrowser

- **Verifies:** AC2 (visual confirmation)
- **Precondition:** A not-ready card on the board
- **Action:** Attempt to locate and click an Advance control for this card in a real browser
- **Expected result:** No clickable Advance control exists for this card (Playwright locator find fails/returns 0 elements)
- **Edge case:** No

## NFR Tests

### sessionReadinessLookupPerformanceBound

- **NFR addressed:** Performance
- **Measurement method:** Same as Unit Test `readinessLookupIsBatchedNotLinear` above — this NFR test IS that unit test (one test satisfies both the AC3 functional requirement and this NFR; not duplicated).
- **Pass threshold:** Session-store read count for N=50 cards is not more than 2x the read count for N=5 cards (a concrete numeric bound, resolving Review finding 1-M1's request for a fixed threshold rather than a fuzzy "linear vs not" judgment)
- **Tool:** Node test runner (`scripts/run-all-tests.js`, per `package.json`'s configured `npm test`)

### crossTenantAuthorizationBound

- **NFR addressed:** Security
- **Measurement method:** Same as Integration Test `crossTenantAdvanceRejected404` above — this NFR test IS that integration test.
- **Pass threshold:** 404 response, zero state mutation
- **Tool:** Node test runner
