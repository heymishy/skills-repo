## Story: Redesign the product/org/tenant kanban board CSS to use the shared design-system tokens

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-2-board-visual-redesign.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want **the kanban boards to visually match the rest of the platform (same neutrals, accent, and type), instead of their current raw-hex, non-themed styling**,
So that **the board — my highest-traffic daily view — no longer looks like a visibly older, disconnected part of the product**.

## Benefit Linkage

**Metric moved:** M2 (Visual consistency with the current design system)
**How:** Confirmed by direct code read (2026-07-24): `_renderKanbanColumns` (`src/web-ui/views/kanban-view.js`, the live renderer for all three board scopes) hardcodes raw hex values (`#22c55e`, `#f59e0b`, `#ef4444`, etc.) rather than the shared CSS custom properties (`--green`, `--amber`, `--red` + soft variants) already defined in `src/web-ui/utils/html-shell.js`. This story replaces every hardcoded value with the shared token, directly closing M2's 0-of-3 baseline gap.

## Architecture Constraints

- **Anti-pattern guardrail (architecture-guardrails.md):** "Any change to shared surface modules (html-shell.js, design tokens, navigation structure, shared CSS) is a story — even a small one." This story IS that story for the kanban board's CSS — confirmed appropriately scoped, not a shadow change.
- **Use the existing token set verbatim** — `--bg`, `--surface`, `--ink`, `--ink-2`, `--muted`, `--muted-2`, `--line`, `--line-2`, `--accent`, `--accent-soft`, `--accent-ink`, `--green`/`--green-soft`, `--amber`/`--amber-soft`, `--red`/`--red-soft`, `--serif`, `--sans`, `--mono` (all defined in `html-shell.js`'s `:root` and its dark-theme override block). Do not invent new custom properties for this story unless a genuine gap is found (e.g. no existing token for a board-specific need) — if so, add the new token to `html-shell.js`'s root block, not locally to `kanban-view.js`, so it's available platform-wide.
- **Preserve the accessibility pattern already present:** health is never colour-only — `_cardHealthLabel()`'s text label must remain alongside any colour treatment; do not regress this while restyling.
- **Do not touch `_renderKanbanColumns`'s data-shape logic** (`buildProductKanbanColumns`, `buildOrgKanbanColumns`, `buildTenantKanbanColumns`, `_aggregateJourneysByStage`) — this story is CSS-only; card/column data flow is unchanged.

## Dependencies

- **Upstream:** None.
- **Downstream:** S2.2 (dark-theme parity check builds on this story's token substitution); Epic 1's/Epic 3's UI additions (Advance action, drag handles, WIP badges, detail-view trigger) should visually integrate with whatever this story establishes, though those stories can proceed independently of this one (different epics, sequenced by risk not by strict dependency).

## Acceptance Criteria

**AC1:** Given `_renderKanbanColumns`'s current CSS block, When this story is complete, Then zero raw hex colour values remain in it — every colour reference uses a `var(--token-name)` from the shared set already defined in `html-shell.js`.

**AC2:** Given the product-level kanban board (`/products/:id/kanban`) rendered in light mode, When compared side-by-side against the published product-view redesign mockup, Then the neutral background/surface/ink colours and accent colour visually match (same token values, not just "similar").

**AC3:** Given the same board rendered in dark mode (`prefers-color-scheme: dark` or the equivalent theme toggle), When compared against light mode, Then contrast remains legible and the accent/semantic colours (green/amber/red) still read correctly against the dark surface — using `html-shell.js`'s existing dark-theme override block, not a new one authored from scratch.

**AC4:** Given the existing `psh-s6-product-kanban.spec.js` and `psh-s7-org-kanban.spec.js` E2E tests (structural checks: scroll width, column count), When this story's CSS changes ship, Then both tests still pass unmodified — this story does not regress existing structural test coverage, only restyles.

## Out of Scope

- New interactive elements (drag handles, WIP badges, Advance buttons) — those stories (Epic 1, Epic 3) apply their own styling once this story's token foundation exists.
- Redesigning the legacy `renderKanban({features, ideas})` path — confirmed dead code per `kbc-s1` (no live route calls it); not touched here.
- Any change to card/column data-fetching, health computation, or the underlying Postgres queries.

## NFRs

- **Performance:** No new runtime cost — CSS-only change, same inline `<style>` block pattern already used.
- **Security:** None — no new user input rendered; existing `escHtml` usage is unchanged.
- **Accessibility:** Colour-plus-label pattern preserved (not colour-only); contrast checked in both themes (AC3).
- **Audit:** Not applicable.

## Complexity Rating

**Rating:** 1 — mechanical token substitution against an existing, already-published token set.
**Scope stability:** Stable.
