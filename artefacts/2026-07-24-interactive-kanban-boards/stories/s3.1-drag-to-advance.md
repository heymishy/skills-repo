## Story: Drag a ready card to its next column to advance its stage

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-3-board-productivity-features.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want to **drag a ready-to-advance card into its next column**,
So that **stage advancement feels like a natural board interaction, not a separate click action tucked in a menu — the lower-friction interaction model discovery's MVP scope confirmed as the primary path, in addition to S1.1's click action**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity for real stage transitions)
**How:** Drag-and-drop is the interaction model discovery's operator confirmed as primary; this story is expected to be the dominant contributor to M1's real-usage measurement, more than the click-only path S1.1 provides alone.

## Architecture Constraints

- **Calls the exact same endpoint S1.1 built — no new backend mechanism.** This story is a client-side drag-and-drop UI layer only; the server-side call (and its `session.done` precondition, tenant-ownership check, and error handling) is entirely S1.1's, reused verbatim.
- **Native HTML5 drag-and-drop API preferred over a new npm dependency**, per this repo's established "no new npm dependencies without justification" pattern (seen in `kfd1`) and this discovery's own Constraints section. Only introduce a library if native drag-and-drop proves genuinely insufficient for cross-browser or accessibility reasons — and if so, justify this explicitly in `decisions.md` before adding the dependency.
- **A drag to a column that isn't the card's real, valid next stage must be rejected client-side before even attempting the server call** — e.g. dragging a card two columns ahead, or to a column with no defined transition from its current stage. The board only allows drops onto the single real "next stage" column per the existing `STAGE_COLUMNS`/gate-confirm sequence — not an arbitrary column.
- **Accessibility floor (already established by discovery's MVP scope item #6 / S1.1's click action):** drag-and-drop is an enhancement layer on top of the click action, never the ONLY way to trigger the same outcome — a keyboard-only operator can always use S1.1's click action instead.
- **Failure/revert UX:** if the server call (S1.1's endpoint) rejects the drag (not ready, tenant mismatch, validation failure), the card must visibly revert to its original column, and the reason (from S1.1's error-surfacing AC5, or S1.2's not-ready explanation) must be shown — not a silent snap-back with no explanation.

## Dependencies

- **Upstream:** S1.1 (this story's entire backend call is S1.1's endpoint, reused) and S1.2 (this story's failure UX reuses S1.2's not-ready explanation pattern where applicable).
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given a card whose journey is ready to advance (per S1.1's readiness data), When the operator drags it onto its single valid next-stage column and drops it, Then the same real gate-confirm call S1.1 defines executes, and the card's new position reflects the real, confirmed new stage — not an optimistic UI update that could be wrong if the server call fails.

**AC2:** Given a card is dragged onto any column OTHER than its single valid next-stage column, When dropped, Then the drop is rejected client-side (no server call attempted) and the card reverts to its original column with a clear indication that this isn't a valid transition.

**AC3:** Given a card whose journey is NOT ready to advance (per S1.1's readiness data), When the operator attempts to drag it, Then the drag either doesn't initiate at all, or initiates but is visibly rejected on drop with S1.2's not-ready explanation — the operator is not left wondering why nothing happened.

**AC4:** Given a real gate-confirm call fails for a reason other than readiness (S1.1's AC5 — e.g. artefact validation error), When this occurs during a drag-drop, Then the card reverts to its original column and the real failure reason is shown — matching S1.1's non-generic error surfacing, not a bare "drop failed" message.

**AC5:** Given an operator who cannot or does not use drag-and-drop (keyboard-only, assistive technology), When they need to advance a ready card, Then S1.1's click action remains fully available and produces the identical outcome — drag-and-drop is never the only path.

## Out of Scope

- Reordering cards within the same column via drag (that's a separate, non-gated interaction — S3.2).
- Any visual drag-preview styling beyond what's needed for basic usability (e.g. elaborate custom drag ghost images) — functional drag feedback only.
- Touch-screen/mobile drag gesture support — this platform's real usage today is desktop-only (per discovery's Constraints); if genuinely needed later, a follow-up story.

## NFRs

- **Performance:** No additional cost beyond S1.1's already-established readiness/advance mechanism.
- **Security:** None new — reuses S1.1's tenant-ownership enforcement entirely.
- **Accessibility:** Drag-and-drop is an enhancement, not the only path (AC5) — this is the binding accessibility floor for this story.
- **Audit:** Not applicable beyond S1.1's existing logging.

## Complexity Rating

**Rating:** 2 — the interaction itself (native HTML5 drag-and-drop) is well-understood, but correct client-side validation of "which column is actually valid" (AC2) and honest failure/revert UX (AC3/AC4) require real care to avoid a worse experience than the click-only alternative.
**Scope stability:** Stable.
