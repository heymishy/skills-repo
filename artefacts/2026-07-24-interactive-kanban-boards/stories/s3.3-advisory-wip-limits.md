## Story: Show an advisory (soft) WIP-limit warning per column

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-3-board-productivity-features.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want **a column to visibly warn me when it's carrying more work than its configured comfortable limit**,
So that **I notice a real bottleneck (e.g. too many stories stuck in review) at a glance, without the board ever blocking me from actually moving work there — advisory only, per discovery's explicit scope decision**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity) — indirectly, by making the board a genuinely more useful daily tool (surfacing bottleneck signal), increasing the odds the operator actually uses it as their primary view rather than defaulting to the CLI/session UI.

## Architecture Constraints

- **Reuse the ALREADY-BUILT WIP-badge logic from the legacy renderer.** Confirmed via direct code read (2026-07-24): `kanban-view.js`'s `lane(laneConfig, cards, wipLimit)` function already implements exactly this — a `kb-wip`/`kb-wip--over` badge showing `count/limit` when exceeded, styled with a distinct over-limit treatment. This logic currently only runs in the legacy `renderKanban({features,ideas})` path with a hardcoded default (`{ delivery: 5, review: 4 }`). Port this pattern into `_renderKanbanColumns` — do not reimplement WIP-badge logic from scratch.
- **No hard blocking anywhere** — confirmed by operator (2026-07-24, discovery.md): exceeding a WIP limit shows a warning but never blocks a drop (S3.1) or an advance (S1.1). This story must not introduce any code path that rejects an action based on WIP-limit state.
- **Configuration surface is a genuinely new decision** — the legacy renderer's limits were hardcoded (`{ delivery: 5, review: 4 }`); this story needs a way for the operator to set a limit per column per board scope. The simplest honest option: a sensible hardcoded default per `STAGE_COLUMNS` stage (mirroring the legacy defaults' spirit — e.g. flag a limit only on stages known to be common bottlenecks, like `review`/`test-plan`), with a documented path to make it operator-configurable later if the default proves wrong in practice. Do not over-build a full settings UI for this MVP unless the operator explicitly asks for it at review.

## Dependencies

- **Upstream:** S2.1 (this story's badge styling should use the token-based redesign S2.1 establishes, not new raw-hex styling).
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given a column whose card count exceeds its configured WIP limit, When the board renders, Then the column visibly shows a distinct "over limit" indicator (count/limit, styled distinctly from the normal count badge) — reusing the existing `kb-wip--over` pattern's visual language, restyled with S2.1's tokens.

**AC2:** Given a column at or under its WIP limit, When the board renders, Then it shows a plain count badge with no warning styling.

**AC3:** Given a column is over its WIP limit, When the operator attempts to drag a new card into it (S3.1) or advance a card into it (S1.1), Then the action still succeeds — over-limit state is never a blocker, confirmed by a test that drags/advances into an already-over-limit column and asserts success.

**AC4:** Given the WIP-limit configuration (whatever form it takes per this story's Architecture Constraints), When applied across all three board scopes (product/org/tenant), Then the same limit-per-stage convention applies consistently — an operator doesn't see different limits for the same stage on different board scopes without a documented reason.

## Out of Scope

- Hard/enforced WIP limits (blocking drops) — explicitly excluded per discovery's operator decision.
- A full settings UI for per-column, per-scope limit configuration — a sensible default is sufficient for this MVP; a configuration UI is a follow-up if the default proves wrong in practice.
- WIP limits on the legacy `renderKanban({features,ideas})` path — already has this; not touched (dead code, per Epic 2's own scope note).

## NFRs

- **Performance:** Trivial — a count comparison per column, no new query cost.
- **Security:** None.
- **Accessibility:** Over-limit indicator must not be colour-only — reuse the existing `kb-wip--over` pattern's text content (`count/limit`), which already satisfies this.
- **Audit:** Not applicable.

## Complexity Rating

**Rating:** 1 — the core WIP-badge logic already exists and is proven (legacy renderer); this story ports and lightly configures it.
**Scope stability:** Stable.
