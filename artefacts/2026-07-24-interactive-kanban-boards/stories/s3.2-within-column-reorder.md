## Story: Reorder cards within a column for local prioritisation

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-3-board-productivity-features.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want to **drag a card to a different position within the same column**,
So that **I can signal my own priority order among several features sitting at the same stage, without that action triggering any real pipeline-state change**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity) — indirectly. This story doesn't itself move a real stage transition, but it makes the board a more complete substitute for the operator's own mental prioritisation, which discovery's problem statement names as part of why the board should be more than read-only.

## Architecture Constraints

- **No existing priority/order field exists anywhere in the current schema** (`journeys` table, `pipeline-state.json`) — confirmed via direct code search (2026-07-24). This story requires a genuinely new persistence decision: either (a) a new `display_order` column on the `journeys` table (or an equivalent per-card ordering store), or (b) a client-only (e.g. `localStorage`) ordering that doesn't persist across devices/sessions. Given this repo's own operator is the sole real user today (single-device, single-operator per discovery's Constraints), a client-only approach is an acceptable, honest first-pass scope — but the coding agent must explicitly choose and document which approach was taken in `decisions.md`, not silently pick one.
- **This action must NEVER call the gate-confirm/advance mechanism** (S1.1) — reordering within a column is explicitly non-gated, per discovery's MVP scope item #3. A within-column drag must be distinguishable, client-side, from a between-column drag (S3.1) before any action is taken.
- **Reuses the same native HTML5 drag-and-drop mechanism as S3.1** — do not introduce a second drag-and-drop implementation for within-column vs between-column dragging; one drag-and-drop system handles both, differentiated by drop target.

## Dependencies

- **Upstream:** None strictly required, though implementing alongside S3.1 (same drag-and-drop mechanism) is the practical sequencing — both may be developed together if convenient.
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given two or more cards in the same column, When the operator drags one card to a different position within that same column and drops it, Then the cards visually reorder to reflect the new position, and no server-side stage-advance call is made.

**AC2:** Given a card has been reordered within a column, When the operator reloads the board (or, if a client-only approach is chosen, revisits it in the same browser/session), Then the reordered position persists — the exact persistence mechanism (server-side field vs client-only storage) is a coding-agent decision, documented in `decisions.md`, but SOME persistence beyond the current page view is required (a reorder that resets on every reload provides no real value).

**AC3:** Given the operator attempts to drag a card between two columns (not within the same column), When dropped, Then this is handled by S3.1's between-column logic, not this story's within-column logic — the two drag types are mutually exclusive and correctly distinguished by the drop target.

## Out of Scope

- Any server-side schema migration beyond what's minimally needed if the coding agent chooses server-side persistence — a full "priority" feature with cross-device sync, audit history, or multi-operator conflict resolution is not in scope (matches discovery's explicit exclusion of multi-operator collaborative editing).
- Reordering across board scopes (e.g. a product-level reorder affecting the tenant-level aggregate view) — each board scope's ordering is independent.

## NFRs

- **Performance:** Reorder must feel immediate (optimistic UI update) even if underlying persistence is asynchronous.
- **Security:** If server-side persistence is chosen, it must respect the same tenant-ownership checks as S1.1's endpoint.
- **Accessibility:** A non-drag alternative (e.g. up/down move buttons) should be considered if drag-and-drop-only reordering would leave keyboard-only operators with no way to reorder — the coding agent should assess this against S1.1's/S3.1's established accessibility-floor precedent (drag-and-drop is never the only path to an action) and either provide an alternative or explicitly note in `decisions.md` why this specific, low-stakes action is an acceptable exception.
- **Audit:** Not applicable — no real state mutation.

## Complexity Rating

**Rating:** 2 — the drag mechanics are shared with S3.1, but the genuinely open persistence-approach decision (no existing schema field) introduces real design work.
**Scope stability:** Unstable — the persistence approach (server field vs client-only) may reasonably change once implementation starts; this is acceptable given the low-stakes nature of the feature (pure UX preference, no data-integrity risk either way).
