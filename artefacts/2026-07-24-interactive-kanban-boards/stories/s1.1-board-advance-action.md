## Story: Add a board action that advances a ready journey to its next pipeline stage via the real gate-confirm route

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-1-board-stage-advance-core.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want to **click an action on a kanban card whose current stage session has already completed, and have it move to the next real pipeline stage without leaving the board**,
So that **I don't have to open the chat session UI just to confirm a gate I've already finished, directly moving M1's board-to-CLI parity metric toward its target**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity for real stage transitions)
**How:** Before this story, 100% of real stage transitions require leaving the board (chat-session UI or CLI). This story is the entire mechanism by which a board-driven transition becomes possible at all — every subsequent board-driven interaction (drag-and-drop in Epic 3) reuses this same action.

## Architecture Constraints

- **Reuse the existing, real `POST /api/journey/:journeyId/gate-confirm` route (`handlePostGateConfirm`, `src/web-ui/routes/journey.js`) — do not build a parallel or lighter-weight transition mechanism.** Confirmed via direct code read (2026-07-24, `decisions.md`): this route already validates the artefact, persists to disk/Postgres, calls `completeStage()`, and writes governance fields via the injected `pipelineStateWriterFactory` adapter. The board action is a new CALLER of this existing route, not a new mechanism.
- **`session.done` precondition (real, load-bearing):** `handlePostGateConfirm` returns HTTP 400 "Session not complete yet" if the journey's `activeSessionId`'s session isn't done. The board action's endpoint must check this same precondition before even attempting gate-confirm, and the UI must not offer the advance action on a card that isn't ready (see S1.2 for the "not ready" UX).
- **Session-readiness lookup is NOT part of the current card data shape.** `_aggregateJourneysByStage` (`src/web-ui/routes/products.js`) currently returns `{id, title, stage, health, healthLabel}` per card — no session-done state. Determining readiness requires a per-journey lookup of its `activeSessionId`'s session (`getGetHtmlSession()`), which is a separate store from the `journeys` Postgres table. **Do not add N+1 per-card lookups on every board render** — batch this (e.g. one bulk session-store read keyed by the board's full set of `activeSessionId`s, or extend the existing `journeys` query/session store to expose a `done` flag queryable in bulk) — the exact batching approach is an implementation decision for the coding agent, but the N+1 risk must be explicitly avoided, not discovered at DoD.
- **Multi-tenancy (ADR-025):** the new board-action endpoint must apply the same tenant-ownership check `handleGetProductKanban` already applies (confirmed via `bri-s3.4`'s fix — `_ownerRow.tenant_id !== tenantId` → 404) — an operator must not be able to advance a journey belonging to a product/tenant they don't own via this new endpoint, even if they know its journey ID.
- **ADR-018 (Playwright):** new E2E coverage for this story belongs in `tests/e2e/`, using the existing local NODE_ENV=test mocked-harness convention for its Playwright spec (this is an in-app UI action, not a real-staging-only E2E case).

## Dependencies

- **Upstream:** None — `handlePostGateConfirm` already exists and is already proven (used daily by the real chat-session gate-confirm button).
- **Downstream:** S1.2 (this story's own UI needs the "not ready" case it defines); Epic 3's drag-and-drop stories (S3.1 onward) call this same board-action endpoint as their underlying mechanism.

## Acceptance Criteria

**AC1:** Given a kanban card (any of product/org/tenant scope) whose journey's current active session has `done === true`, When the operator clicks the card's "Advance" action, Then the real `gate-confirm` mechanism runs (artefact validated, persisted, stage advanced), and the card moves to its new stage's column on the next board render.

**AC2:** Given a kanban card whose journey's current active session has `done !== true` (still in progress), When the board renders, Then the "Advance" action is not offered as clickable on that card (see S1.2 for the exact non-clickable/disabled presentation).

**AC3:** Given a board with N cards across potentially multiple journeys, When the board renders and needs to determine each card's advance-readiness, Then this is done without N separate per-card session-store round-trips at render time (batched lookup) — verified via a test asserting the number of session-store reads does not scale linearly with card count for a fixed board render.

**AC4:** Given an operator viewing a board for a product/tenant they do NOT own, When they attempt to call the advance action directly (e.g. via a crafted request with a known journey ID from another tenant), Then the request is rejected with 404 (not 403, matching this repo's existing FORBIDDEN-vs-NOT_FOUND policy), and no state is written — verified by a test using a real cross-tenant journey ID.

**AC5:** Given a card is successfully advanced (AC1), When the underlying `gate-confirm` call itself fails for a reason other than the `session.done` precondition (e.g. artefact validation error), Then the board surfaces the real error reason from the response, not a generic failure message — the operator can see why the advance didn't happen.

## Out of Scope

- Drag-and-drop UI for triggering this same action — Epic 3.
- Visual/CSS redesign of the "Advance" action's appearance to match the design system — Epic 2 (this story's UI is functional-minimum only).
- Any change to WIP limits interacting with this action (e.g. blocking an advance because the target column is at its WIP limit) — WIP limits are advisory-only per discovery scope and do not gate this action.

## NFRs

- **Performance:** Board render time must not scale linearly with card count due to session-readiness lookups (AC3) — batched lookup required.
- **Security:** Tenant ownership enforced on the new endpoint (AC4) — no new bypass of existing multi-tenancy guards (ADR-025).
- **Accessibility:** The "Advance" action must be operable via keyboard (not mouse-only), consistent with this repo's existing accessibility guardrail for interactive elements.
- **Audit:** No new audit requirement beyond what `handlePostGateConfirm` already logs — this story is a new caller, not a new state-mutation path.

## Complexity Rating

**Rating:** 2 — the core mechanism (calling an existing route) is low-risk, but the session-readiness batching (AC3) and the "not ready" UX (S1.2) introduce real design work not present in any existing pattern in this codebase.
**Scope stability:** Stable.
