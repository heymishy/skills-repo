## Story: Open an item detail view directly from any kanban card

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-3-board-productivity-features.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want to **click a card and see enough of its underlying state (current stage, DoR/DoD-equivalent status, linked artefact files) to decide my next action**,
So that **I don't need to leave the board and separately look up a journey's state before deciding whether to advance, prioritise, or just check on it**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity) — indirectly, by reducing the need to leave the board for information-gathering, one of the two reasons (the other being action-taking, covered by Epic 1/S3.1) an operator currently has to drop to the CLI/session UI.

## Architecture Constraints

- **The live product/org/tenant boards' cards currently render as plain `<div>` elements with no link at all** (`_renderKanbanColumns`, confirmed via direct code read 2026-07-24) — unlike the legacy `featureCard()`, which links to `/features/:slug`. This story adds a real navigation target to these cards for the first time.
- **Confirm the correct existing detail-page route before building anything new.** `GET /features/:slug` (`features.js`) still exists and is live (confirmed 2026-07-24 — `kbc-s1` removed the `/features` board-LIST route, not this per-feature detail route) and is `kfd1`'s design-system-styled destination. However, cards on these boards are keyed by `journey_id` (a UUID), not directly by `feature_slug` — the card data (`_aggregateJourneysByStage`) does not currently expose `feature_slug` as its own field (only bundled into the display title). Before implementing, the coding agent must confirm: (a) whether `/features/:slug` is actually the right destination for a journey-scoped card (as opposed to a journey-specific page, e.g. `handleGetJourney`/`GET /journey?id=...`), and (b) what identifier the card needs to carry to link there correctly. Document the confirmed route and identifier mapping in `decisions.md` before writing the AC-verifying tests.
- **Do not redesign the destination detail page itself** — this story only wires a new entry point (a link/click target from the card); the destination page's own styling is out of scope (already handled by `kfd1` if `/features/:slug` is confirmed as the right target, or is its own separate concern if a journey-specific page is the right target instead).

## Dependencies

- **Upstream:** None strictly, though sequencing after S2.1/S2.2 (so the new link/click-target styling matches the redesigned card) is the practical order.
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given the coding agent has confirmed the correct destination route and identifier (per Architecture Constraints), When a card renders on any of the three board scopes, Then it is a real navigable link (not a plain `<div>`) to that destination, carrying the correct identifier for that specific journey/feature.

**AC2:** Given the operator clicks (or keyboard-activates) a card, When the destination page loads, Then it shows genuinely useful state for deciding next action — at minimum, the current stage and a link back to the artefact files/story state already present (per `kfd1`'s existing detail page pattern, if that's the confirmed destination).

**AC3:** Given the operator is on the detail view, When they want to return to the board, Then a clear way back exists (e.g. a back link or breadcrumb) — the operator is not left without a path back to the board they came from.

**AC4:** Given a card's underlying journey belongs to a product/tenant the operator does NOT own, When they attempt to reach its detail view directly (e.g. via a crafted URL), Then access is denied per this repo's existing tenant-ownership enforcement pattern (404, not 403) — the new navigation entry point does not become a new way to bypass existing access control.

## Out of Scope

- Redesigning the destination detail page's own visual style — already handled by `kfd1` if that's the confirmed target.
- Any new data surfaced on the detail page beyond what already exists there — this story is a navigation entry point, not a detail-page feature expansion.
- A modal/inline detail panel as an alternative to full navigation — full-page navigation to the existing detail route is the confirmed approach unless the coding agent's route investigation (Architecture Constraints) finds a compelling reason otherwise, documented in `decisions.md`.

## NFRs

- **Performance:** Navigation only — no new data-fetching cost beyond whatever the destination page already does.
- **Security:** Tenant-ownership enforcement (AC4) — matches the existing pattern, no new bypass.
- **Accessibility:** Cards become real, keyboard-activatable links (AC1) — an improvement over the current non-interactive `<div>` cards.
- **Audit:** Not applicable.

## Complexity Rating

**Rating:** 2 — the exact destination route/identifier mapping is a genuinely open question (not yet confirmed) that must be resolved before implementation, even though the underlying mechanism (a link) is trivial once resolved.
**Scope stability:** Unstable — pending the route-confirmation investigation; low risk either way since the destination page itself is out of scope regardless of which route is confirmed.
