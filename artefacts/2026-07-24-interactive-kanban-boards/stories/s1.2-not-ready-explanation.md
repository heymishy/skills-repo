## Story: Show a clear, plain-language reason when a card's journey isn't ready to advance

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-1-board-stage-advance-core.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want **a card whose current stage isn't finished yet to visibly explain why it can't be advanced, in plain language, rather than silently having no action or showing a raw error**,
So that **I don't waste time trying to advance a card that isn't ready, and I understand what state it's actually in without switching to the chat session to check**.

## Benefit Linkage

**Metric moved:** M1 (Board-to-CLI parity for real stage transitions)
**How:** A confusing or silent "not ready" state would undermine trust in the board as a genuine alternative to the CLI/chat-session path — if the operator can't tell why a card won't advance, they'll default back to opening the session directly, which is exactly the behaviour M1 is meant to reduce.

## Architecture Constraints

- **Plain-language requirement (CLAUDE.md convention):** the explanation shown must describe what's actually happening in operator-facing terms ("this feature's current [stage name] turn isn't finished yet" or similar), not raw internal state ("session.done === false") or a bare HTTP status.
- **Reuses S1.1's session-readiness data** — this story is purely the presentation layer on top of S1.1's batched readiness lookup; it does not introduce a second data-fetching mechanism.
- **Accessibility (existing guardrail):** the "not ready" indicator must not rely on colour alone — a text label or icon-plus-label must accompany any visual treatment, consistent with `kanban-view.js`'s existing `healthLabel` pattern (colour + accompanying text, never colour-only).

## Dependencies

- **Upstream:** S1.1 — this story presents the "not ready" case S1.1's AC2 defines; it cannot be built independently of S1.1's readiness data.
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given a card whose journey's current session is not done, When the operator views the board, Then the card visibly shows a "not ready to advance" indicator (not colour-only) with a plain-language reason (e.g. naming the current stage and that its turn is still in progress).

**AC2:** Given the same not-ready card, When the operator hovers or focuses it (keyboard), Then a more detailed explanation is available (e.g. via a tooltip or expandable detail) without requiring a page navigation.

**AC3:** Given a card whose journey IS ready to advance (session done), When the operator views the board, Then no "not ready" indicator appears, and the Advance action (S1.1) is visibly available and distinguishable from a not-ready card.

**AC4:** Given S1.1's AC5 (a real gate-confirm failure for a non-readiness reason, e.g. artefact validation error), When this occurs, Then this story's "not ready" indicator styling/language is NOT reused for that case — a validation failure gets its own distinct, honestly-labelled treatment (they are different failure modes and must not look identical to the operator).

## Out of Scope

- Any mechanism to notify the operator in real time when a card BECOMES ready (e.g. push notification, polling) — this story is about the board's state at render/view time, not live updates.
- Retrying or auto-resuming an in-progress session from the board — the not-ready explanation is informational only; resuming the session still happens via the existing chat-session UI link (already present per `kfd1`'s detail-page pattern).

## NFRs

- **Performance:** No additional cost beyond S1.1's already-batched readiness data — this is pure presentation.
- **Security:** None beyond S1.1's existing tenant-ownership enforcement — this story does not introduce a new data-access path.
- **Accessibility:** Keyboard-focusable detail (AC2); colour-plus-label, never colour-only (AC1) — matches existing `kanban-view.js` convention.
- **Audit:** Not applicable — presentation-only, no state mutation.

## Complexity Rating

**Rating:** 1 — well understood; this is a presentation layer on data S1.1 already computes, following an existing accessibility pattern already proven elsewhere in `kanban-view.js`.
**Scope stability:** Stable.
