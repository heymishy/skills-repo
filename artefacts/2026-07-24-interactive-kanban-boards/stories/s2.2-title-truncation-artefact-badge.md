## Story: Port title truncation and artefact-count badge to the live kanban boards

**Epic reference:** artefacts/2026-07-24-interactive-kanban-boards/epics/epic-2-board-visual-redesign.md
**Discovery reference:** artefacts/2026-07-24-interactive-kanban-boards/discovery.md
**Benefit-metric reference:** artefacts/2026-07-24-interactive-kanban-boards/benefit-metric.md

## User Story

As **Hamish King (Founder/Operator)**,
I want **card titles on the live product/org/tenant boards to truncate sensibly and show an artefact count, the same way `kfd1` originally delivered for the board UX**,
So that **I can scan WIP at a glance without long titles distorting card height or layout — the outcome `kfd1` already promised, now actually reaching the boards I use today**.

## Benefit Linkage

**Metric moved:** M3 (WIP visibility — title truncation and artefact-count badge reach the live boards)
**How:** Confirmed via direct code and git-history read (2026-07-24, `benefit-metric.md`): `kfd1` (2026-06-17) built `truncateTitle()` and artefact-count-badge logic, but only wired it into the legacy `renderKanban({features,ideas})` path. `kbc-s1` (2026-07-19) later introduced `_renderKanbanColumns` as the new live renderer for product/org/tenant scope, without inheriting this UX. This story closes that gap directly.

## Architecture Constraints

- **Reuse `kfd1`'s existing `truncateTitle(title, max)` function** (`kanban-view.js`) — same 48-char bound, not a new truncation implementation.
- **Artefact-count data availability is a genuine open question, not assumed solved:** `getArtefactsForJourney(journeyId)` already exists in `src/web-ui/adapters/journey-store-pg.js`, but `products.js`'s `handleGetProductKanban`/`handleGetOrgKanban`/`buildTenantKanbanColumns` currently query only `journey_id, feature_slug, activeSkill` — no artefact count. Adding this requires either (a) a batched/joined query extending the existing journeys query, or (b) N parallel `getArtefactsForJourney()` calls (acceptable only if bounded — most operators' boards will have a small number of cards; if this becomes a real N+1 concern at scale, batch it, following the same batching discipline as S1.1's session-readiness lookup).
- **If artefact-count data cannot be obtained within this story's reasonable scope** (e.g. requires a schema change), the coding agent should implement title truncation alone and explicitly flag the artefact-count gap as a follow-up, per this story's minimum validation signal (benefit-metric.md) — do not silently drop the AC or force an over-scoped schema change to satisfy it.

## Dependencies

- **Upstream:** S2.1 (this story's card styling builds on the token-based redesign S2.1 establishes) — sequence after S2.1, not before.
- **Downstream:** None.

## Acceptance Criteria

**AC1:** Given a card with a title longer than 48 characters, When rendered on any of the three live board scopes (product/org/tenant), Then the title is truncated using the existing `truncateTitle()` logic (48-char bound, ellipsis), matching `kfd1`'s original behaviour.

**AC2:** Given a card with a title of 48 characters or fewer, When rendered, Then the full title displays unmodified — truncation does not over-trigger on short titles.

**AC3:** Given the truncated title is shown, When the operator hovers (or focuses via keyboard) the card, Then the full, untruncated title is available (e.g. via a `title` attribute or equivalent), so no information is lost, only visually deferred.

**AC4:** Given artefact-count data IS obtainable within this story's scope (per the Architecture Constraints investigation), When a card is rendered, Then it shows an artefact-count badge matching `kfd1`'s existing badge styling/logic (`0 artefacts` state distinct from `n artefacts`).

**AC5:** Given artefact-count data is NOT obtainable within reasonable scope, When this story ships, Then title truncation (AC1-AC3) still ships on its own, and the artefact-count gap is explicitly logged as a follow-up in this feature's `decisions.md` — not silently dropped.

## Out of Scope

- Any change to `getArtefactsForJourney()`'s own implementation — reused as-is if used at all.
- A new schema/migration to pre-compute artefact counts — if the existing per-journey query approach proves too costly, that's a follow-up story's scope, not this one's.
- The legacy `renderKanban({features,ideas})` path's own title-truncation behaviour — already correct, not touched.

## NFRs

- **Performance:** If artefact-count is added via a per-journey query (AC4), it must not introduce unbounded N+1 cost on boards with many cards — apply the same batching discipline flagged in S1.1, or explicitly bound/defer per AC5.
- **Security:** None new — no new user input, `escHtml` usage unchanged.
- **Accessibility:** Truncated title's full text remains available via hover/focus (AC3) — not permanently hidden information.
- **Audit:** Not applicable.

## Complexity Rating

**Rating:** 2 — title truncation itself is trivial (rating 1), but the artefact-count data-availability question (AC4/AC5) introduces genuine, not-yet-resolved technical uncertainty.
**Scope stability:** Stable — AC5's fallback path means this story ships real value even if the harder half (AC4) proves infeasible within scope.
