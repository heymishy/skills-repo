# Decisions: Interactive Kanban Boards

## CORRECTION — the real stage-advance mechanism is `handlePostGateConfirm`, not the `skills` CLI (2026-07-24)

**Context:** Discovery's `/clarify` pass resolved an assumption about whether "the existing `skills gate-advance`/`advance` machinery" could be safely called from a web handler, concluding yes based on `cli-advance.js`/`cli-gate-advance.js` being pure, `require()`-able functions.

**The correction:** That check answered the wrong question. `node bin/skills advance`/`gate-advance` (`src/enforcement/cli-advance.js`, `cli-gate-advance.js`) is **this repository's own meta-governance tool** — it reads/writes `.github/pipeline-state.json` to track how the skills-platform repo delivers features to itself (the exact mechanism used earlier this session for `bssm-s1`, `eatrl-s1`, and now this very kanban feature's own bookkeeping).

The kanban boards in scope for this feature (`/products/:id/kanban`, `/org/kanban`, `/dashboard?view=board`) show a completely different thing: **real customer/product journeys**, stored in the deployed web-app's own Postgres `journeys` table (`src/web-ui/adapters/journey-store-pg.js`), each with a `data->>'activeSkill'` stage field. Advancing one of these journeys to its next stage already has a real, existing, web-callable mechanism: `POST /api/journey/:journeyId/gate-confirm` (`handlePostGateConfirm`, `src/web-ui/routes/journey.js:1712`) — the same route the skill-chat UI's own "gate confirm" button already calls today. It validates the artefact (`_validate()`), persists it to disk and Postgres, calls `completeStage()`, and (via the injected `pipelineStateWriterFactory` adapter) writes the equivalent governance fields to the **customer's own** target repo's `pipeline-state.json` — not this repo's.

**Real, load-bearing precondition found:** `handlePostGateConfirm` requires `session.done === true` on the journey's current active session — if the operator's current skill turn hasn't completed (mid-conversation, no artefact produced yet), the route returns HTTP 400 "Session not complete yet." This means a kanban card cannot simply be dragged to any arbitrary next column at any time — only once its current stage's session has genuinely finished. This is a real, must-encode constraint for the drag/click "move to next stage" stories, not an edge case to gloss over.

**Consequences:**
- All references to "the real governed stage-advance machinery ... skills gate-advance/advance" in `discovery.md` and `benefit-metric.md` refer, in implementation reality, to `POST /api/journey/:journeyId/gate-confirm`, not the CLI tool.
- `/definition`'s stories for drag/click stage-advance must explicitly handle the `session.done !== true` case: either disable the drag/action for a card whose journey isn't ready, or handle the 400 response with a clear, plain-language explanation (per CLAUDE.md's abbreviation/plain-language writing convention) — "revert and explain why," not "gate-advance failed."
- No change needed to the discovery's MVP scope or benefit metrics — this correction is about implementation mechanism accuracy, not scope.

**Verification performed:** Direct code read of `src/web-ui/routes/journey.js` (`handlePostGateConfirm`, lines 1712–1786) and `src/web-ui/routes/products.js` (`handleGetProductKanban`, confirming `journey_id`/`activeSkill` come from the real `journeys` Postgres table, not `.github/pipeline-state.json`). Confirmed `setPipelineStateWriter`'s wiring in `server.js` (`pipelineStateWriterFactory(repoRootForAdapter)`) targets a per-deployment target repo root, not this repo specifically.

## RESOLVED — S2.2 AC4 investigation: batched artefact-count query is feasible within scope (2026-07-24)

**Context:** The RISK-ACCEPT immediately above accepted that S2.2's AC4 (Acceptance Criterion 4: artefact-count badge on kanban cards) depended on an unresolved implementation-time investigation — whether `getArtefactsForJourney(journeyId)` (`src/web-ui/adapters/journey-store-pg.js`) could be wired into the board query without unbounded per-card query cost.

**Investigation performed:** Direct code read of `src/web-ui/adapters/journey-store-pg.js`'s `migrateSchema()` (lines 71-82) confirmed a real `artefacts` Postgres table already exists, with `journey_id` as a foreign key into `journeys` and an index (`artefacts_journey_id_idx`) already created on that column. This means artefact counts for an arbitrary set of journeys can be obtained with a single `SELECT journey_id, COUNT(*) AS count FROM artefacts WHERE journey_id = ANY($1) GROUP BY journey_id` query — one round-trip for the whole board render, regardless of card count — rather than N calls to the existing per-journey `getArtefactsForJourney()`. This is option (a) named in the story's Architecture Constraints ("a batched/joined query extending the existing journeys query"), not option (b) (N parallel per-journey calls).

**Decision:** Implemented via a new `getArtefactCountsForJourneys(journeyIds)` function in `journey-store-pg.js` (the single batched GROUP BY query above), wired into `products.js` through a new injectable seam (`_getArtefactCountsBulk`/`setGetArtefactCountsBulk`) that mirrors the existing `_getHtmlSessionsBulk` pattern from `s1.1` — same shape, same testability contract (tests spy on call count without a real `DATABASE_URL`/pg pool). A new `_enrichColumnsWithArtefactCounts(columns)` helper in `products.js` collects every card's journey ID across the already-built `STAGE_COLUMNS`-shaped columns, makes exactly one bulk call, and annotates each card's `artefactCount`. It is called from all three board-scope handlers (`handleGetProductKanban`, `handleGetOrgKanban`, `buildTenantKanbanColumns`) after their existing (unchanged) synchronous column-building step.

**Why this keeps the shared column-builders (`_aggregateJourneysByStage`, `buildProductKanbanColumns`, `buildOrgKanbanColumns`) fully synchronous and unchanged:** an earlier design considered threading a `pool` parameter through `_aggregateJourneysByStage` itself and making it async, but that would force `buildProductKanbanColumns`/`buildOrgKanbanColumns` (currently synchronous, called directly and without `await` by several existing tests — e.g. `tests/check-psh-s6-product-kanban.js`, `tests/check-psh-s7-org-kanban.js`, `tests/check-s1.1-board-advance-action.js`) to become async too, breaking their existing call sites' synchronous-array-return contract. Doing the enrichment as a separate post-processing step on the already-built columns avoids that breaking change entirely — zero existing tests needed modification.

**Resilience (AC5):** `_enrichColumnsWithArtefactCounts` wraps its bulk read in a try/catch — if the artefact-count read fails for any reason (adapter not wired, DB error), the board render still succeeds with title truncation intact; cards simply render without a badge (identical to a caller that never supplies `artefactCount` today). AC4 and AC5 are therefore not mutually exclusive branches in the shipped code — AC4's happy path is implemented, and AC5's fallback behaviour is also implemented as the failure-mode safety net for the same code path, per the story's framing that AC5 is "the fallback" if AC4 proves infeasible OR fails at runtime.
**Rationale:** The existing `artefacts` table and its index made the batched-query approach the same cost as a normal indexed lookup — no schema change, no new migration, and no unbounded N+1 pattern. This falls squarely inside this story's "reasonable scope" per its own Architecture Constraints, so AC4 is implemented in full rather than descoped to AC5's fallback-only path.
**Verified by:** `tests/check-s2.2-title-truncation-artefact-badge.js` — unit tests for AC1/AC2/AC4, integration tests for AC3/AC4/AC5, and an NFR test (`artefactCountLookupNotUnboundedN1`) asserting exactly one bulk call regardless of card count (N=5 and N=50).
