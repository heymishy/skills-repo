# Contract Proposal: Show a clear, plain-language reason when a card is not ready to advance

**What will be built:**
Presentation-layer additions to `_renderKanbanColumns`'s card markup in `kanban-view.js`: a not-ready indicator (text label + distinct styling, not colour-only) shown when `card.ready === false` (the field S1.1 adds), a hover/focus-accessible detail mechanism (e.g. a `title` attribute with fuller text), and distinct styling for a real gate-confirm validation failure (S1.1's AC5 case) versus the routine not-ready case.

**What will NOT be built:**
Any live-update/notification mechanism for when a card becomes ready. Any change to S1.1's own readiness-computation logic — this story only presents what S1.1 already computes.

**How each AC will be verified:**
| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Unit test on rendered not-ready card markup, confirming text label + distinct class | Unit |
| AC2 | Integration test confirming hover/focus detail mechanism is present and populated | Integration |
| AC3 | Unit test confirming ready card shows no not-ready indicator, Advance IS visible | Unit |
| AC4 | Unit test comparing validation-failure markup/classes against not-ready markup/classes | Unit |

**Assumptions:**
None beyond what S1.1 already establishes (this story is purely additive presentation on S1.1's data).

**Estimated touch points:**
Files: `src/web-ui/views/kanban-view.js` only.
Services: None new.
APIs: None new.
