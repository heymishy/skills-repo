# Contract Proposal: Reorder cards within a column for local prioritisation

**What will be built:**
Extends S3.1's drag-and-drop mechanism to handle a within-column drop target distinctly from a between-column one. Persists the new order via either a new `display_order`-style field or client-only storage (coding agent's documented choice, per the story's Architecture Constraints). Adds a non-drag reorder alternative (e.g. up/down buttons) OR logs an explicit accepted exception in `decisions.md` before this story reaches DoR sign-off completion (AC4 is binding, not optional).

**What will NOT be built:**
Cross-device/multi-operator ordering sync. Any server-side schema migration beyond a single new column if that path is chosen.

**How each AC will be verified:**
| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Playwright E2E: within-column drag reorders, zero calls to S1.1's endpoint | E2E |
| AC2 | Integration test: reordered position persists across a simulated reload | Integration |
| AC3 | Playwright E2E: between-column drag still uses S3.1's logic | E2E |
| AC4 | Unit test: non-drag control exists and is keyboard-activatable (or decisions.md exception logged) | Unit |

**Assumptions:**
The persistence mechanism (server field vs client-only) is a genuinely open, low-stakes decision — documented as accepted in `decisions.md` regardless of which is chosen.

**Estimated touch points:**
Files: `src/web-ui/views/kanban-view.js` (shared drag mechanism, extended), possibly `src/web-ui/adapters/journey-store-pg.js` and a schema addition if server-side persistence is chosen.
Services: journey store (if server-side persistence chosen).
APIs: A new small PATCH/POST endpoint if server-side persistence is chosen; none if client-only.
