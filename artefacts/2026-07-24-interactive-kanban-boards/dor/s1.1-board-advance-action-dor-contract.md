# Contract Proposal: Add a board action that advances a ready journey via the real gate-confirm route

**What will be built:**
A new POST endpoint (e.g. `/api/board/journeys/:journeyId/advance`) in `src/web-ui/routes/products.js` or a new small module, that: (1) looks up the journey's tenant ownership (reusing the existing `_ownerRow.tenant_id !== tenantId` → 404 pattern from `handleGetProductKanban`), (2) checks the journey's current active session's `done` state via a NEW batched/bulk session-readiness lookup function (not N individual `getGetHtmlSession()` calls), (3) if ready, calls the existing `handlePostGateConfirm` logic (either by direct internal call or by proxying to `POST /api/journey/:journeyId/gate-confirm`), and (4) returns the real success/failure response including validation-failure detail. Also extends `_aggregateJourneysByStage`'s card shape to include a `ready: boolean` field computed from the new batched lookup, and adds a rendered "Advance" action/button to `_renderKanbanColumns`'s card markup when `ready === true`.

**What will NOT be built:**
Drag-and-drop triggering this same endpoint (S3.1, separate story). Any change to `handlePostGateConfirm`'s own validation or state-write logic — this story only adds a new caller.

**How each AC will be verified:**
| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Integration test posting to the new endpoint for a ready journey, confirming stage advance | Integration |
| AC2 | Unit test rendering a not-ready card, confirming no clickable Advance markup | Unit |
| AC3 | Unit test comparing session-store read count at N=5 vs N=50 cards | Unit |
| AC4 | Integration test: cross-tenant advance attempt returns 404, no state written | Integration |
| AC5 | Integration test: malformed artefact triggers real validation-failure detail in response | Integration |

**Assumptions:**
The batched session-readiness lookup can be implemented as either (a) a new bulk-read function against the existing session store, or (b) an extension to the `journeys` query itself if session-done state can be denormalized there — the coding agent chooses based on what's actually feasible in the existing session-store implementation (not yet fully read at DoR time; this is real, bounded implementation-time discovery, not a DoR blocker per H3/H8 since the AC's test still specifies the OBSERVABLE requirement — bounded read count — regardless of mechanism).

**Estimated touch points:**
Files: `src/web-ui/routes/products.js`, `src/web-ui/views/kanban-view.js`, `src/web-ui/server.js` (route registration), possibly a new small module for the batched session-readiness lookup.
Services: journey store (Postgres), session store.
APIs: new `POST /api/board/journeys/:journeyId/advance` (or equivalent name), reusing `handlePostGateConfirm`'s logic.
