# Contract Proposal: Open an item detail view directly from any kanban card

**What will be built:**
First implementation step: confirm the correct destination route (`/features/:slug` vs a journey-specific page like `GET /journey?id=...`) and the identifier the card needs to carry to link there (the card currently only has `journey_id`, not `feature_slug`, as its own field). Then wire cards in `_renderKanbanColumns` as real `<a>` elements to the confirmed destination, applying the same tenant-ownership check pattern as S1.1/`handleGetProductKanban`.

**What will NOT be built:**
Any redesign of the destination page's own visual style. A modal/inline detail panel alternative to full navigation.

**How each AC will be verified:**
| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Unit test, conditional on route-confirmation investigation | Unit |
| AC2 | Integration test at the confirmed destination | Integration |
| AC3 | Integration test confirming a back-link/breadcrumb exists | Integration |
| AC4 | Integration test: cross-tenant access denied 404 | Integration |

**Assumptions:**
The exact destination route/identifier mapping is genuinely unconfirmed at DoR time — this is the RISK-ACCEPT logged in `decisions.md`. The coding agent's first task is resolving this, documented before AC1/AC2's tests are finalized.

**Estimated touch points:**
Files: `src/web-ui/views/kanban-view.js` (card markup), possibly `src/web-ui/routes/products.js` (if card data needs to expose `feature_slug` or an equivalent identifier).

APIs: None new — links to an existing route.
