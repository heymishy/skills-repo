# Contract Proposal: Drag a ready card to its next column to advance its stage

**What will be built:**
Client-side native HTML5 drag-and-drop wiring in the board's page script (wherever `_renderKanbanColumns`'s output is served, e.g. via an inline `<script>` block similar to the pattern already used in the legacy `renderKanban`'s idea-form script), constrained to a card's single valid next-stage column as the only accepted drop target, calling S1.1's existing advance endpoint on drop, with revert-on-failure/rejection handling reusing S1.2's not-ready explanation where applicable.

**What will NOT be built:**
Within-column reordering (S3.2, shares the same drag mechanism but different drop-target handling). Any new backend endpoint — this story is a client-side caller of S1.1's existing endpoint only.

**How each AC will be verified:**
| AC | Test approach | Type |
|----|---------------|------|
| AC1 | Playwright E2E: drag ready card to valid column, confirm real advance | E2E |
| AC2 | Playwright E2E: drag to invalid column, confirm no server call + revert | E2E |
| AC3 | Playwright E2E: drag not-ready card, confirm rejection + explanation | E2E |
| AC4 | Playwright E2E: drag card whose gate-confirm will fail validation, confirm revert + real reason | E2E |
| AC5 | Integration test confirming S1.1's click endpoint is unaffected | Integration |

**Assumptions:**
Native HTML5 drag-and-drop API is sufficient for cross-browser/accessibility needs at this repo's current scale (Chromium-only Playwright testing, per existing `tests/e2e` convention) — a library is only justified if this proves insufficient, documented in `decisions.md` if so.

**Estimated touch points:**
Files: `src/web-ui/views/kanban-view.js` (client-side script addition), possibly `src/web-ui/routes/products.js` (if any server-side rendering hook is needed for drag-related data attributes).
Services: None new — reuses S1.1's endpoint.
APIs: None new.
