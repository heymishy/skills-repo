## Setup

**Before you start:**
1. Log in to the local test environment and open a product's kanban board (`/products/:id/kanban`).
2. Have one feature whose current stage's chat session has already finished a turn (fully "done" — no pending question), and one feature whose session is still mid-conversation.

**Reset between scenarios:** Reload the board page between scenarios.

---

## Scenarios

### Scenario 1: A ready card can be advanced directly from the board

**Covers:** AC1

**Steps:**
1. Find the card for the feature whose session is already done.
2. Click the "Advance" button/link on that card.

**Expected outcome:** The card disappears from its current column and reappears in the next column (the next real pipeline stage) — the same outcome as if you'd clicked "gate confirm" inside the chat session itself.

**Pass / Fail:** ___

---

### Scenario 2: A not-ready card doesn't offer the Advance action

**Covers:** AC2

**Steps:**
1. Find the card for the feature whose session is still mid-conversation.
2. Look for an "Advance" button/link on that card.

**Expected outcome:** No clickable "Advance" action exists on this card — it's either absent or clearly shown as unavailable.

**Pass / Fail:** ___

---

### Scenario 3: Advancing many cards doesn't slow the board down

**Covers:** AC3

**Steps:**
1. If a test board with many cards (20+) is available, load it and time how long the page takes to appear.

**Expected outcome:** The board loads at roughly the same speed whether it has 5 cards or 50 — it shouldn't visibly slow down as more cards are added. (This is best confirmed by the automated test in the technical plan; this manual check is a sanity spot-check only.)

**Pass / Fail:** ___

---

### Scenario 4: You can't advance a feature that belongs to someone else

**Covers:** AC4

**Steps:**
1. Ask someone with access to a second, separate account/tenant for one of their feature's card IDs.
2. Try to trigger an advance for that card ID while logged in as yourself.

**Expected outcome:** You get a "not found" response — not a permission error, and nothing about that other tenant's feature changes.

**Pass / Fail:** ___

---

### Scenario 5: A real failure (not "not ready") shows the actual reason

**Covers:** AC5

**Steps:**
1. Arrange (with engineering help) for a feature's underlying artefact file to be invalid in a way that would fail validation.
2. Click "Advance" on that feature's card.

**Expected outcome:** You see a specific error message describing what's actually wrong with the artefact — not a generic "something went wrong" message.

**Pass / Fail:** ___
