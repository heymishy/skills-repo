# EXP-009 — Focused Model Routing Validation

<!-- DO NOT include API keys, access tokens, or any credentials in this file (MC-SEC-02). -->

## Purpose and economic motivation

The GitHub Copilot billing model changed on 1 June 2026, shifting from per-request (PRU) metering to per-token metering. Claude Sonnet 4.6 — the current default for all invocations — is now the dominant cost driver at a 1x Layer 1 credit multiplier. A single heavy outer-loop pipeline session consumed 4,471 credits (~$44.71) on a single day against a 7,000-credit ($70) monthly pool. At this burn rate, the platform cannot sustain sustained pipeline delivery without depleting the monthly budget within the first week of operation.

EXP-008 demonstrated that Haiku 4.5 (0.33x Layer 1 multiplier) achieves CPF = 1.00 and AQ = 0.77–0.80 as a pipeline worker (definition, review, test-plan, DoR) when Sonnet orchestrates discovery. The open question is whether Haiku can also take over orchestration — running the full outer loop, handling general platform usage, and directing pipeline sessions — without falling below acceptable quality thresholds. A successful routing shift to Haiku as the default conversational model would reduce session credit consumption by approximately two-thirds.

EXP-009 is not a comprehensive corpus regression suite. It is the minimum experiment needed to make a routing policy decision with confidence within the remaining monthly credit pool.

## Hypotheses

### H1 — Haiku as pipeline orchestrator

**Full statement:** Haiku 4.5 can replace Sonnet as the primary model across the full five-stage outer-loop pipeline (discovery through definition-of-ready) while maintaining CPF = 1.00 and AQ ≥ 0.75 on medium-high and high difficulty regulated corpus stories.

**What this tests:** Whether the CPF and AQ losses observed when Haiku runs discovery (vs EXP-008 Config C where Sonnet ran discovery) are tolerable. EXP-003 established that Haiku without Sonnet scaffolding produced CPF 0.33 on regulated constraints in the S1 scenario (F5 — constraints dropped at source). EXP-008 Config C achieved CPF = 1.00 with Sonnet at discovery. H1 determines whether the EXP-003 failure mode recurs with context injection active at Step 4a.

**Pass criteria:** CPF = 1.00 (all constraints propagated including regulated) AND AQ ≥ 0.75 on both corpus stories.

**Failure modes to watch:** Regulatory constraint dropped in discovery (EXP-003 F1/F5 pattern); vertical-slice definition decomposition that omits C2 from architectural stories (EXP-003 F6 pattern); AQ drop below 0.75 due to weaker problem framing or scope discipline at discovery stage.

### H2 — MAI-Code-1-Flash as definition-tier worker

**Full statement:** MAI-Code-1-Flash can perform /definition on a medium-high difficulty regulated corpus story while maintaining CPF = 1.00 and AQ ≥ 0.75 without the GPT-4o-style format compliance failure mode (schema drift, hallucinated SKILL.md fields, collapsed sections) observed in EXP-008 Config D.

**What this tests:** Whether MAI-Code-1-Flash has the same SKILL.md format compliance properties as Haiku, or whether it exhibits the OpenAI-pattern failure mode. This is a smoke test, not a full corpus run. A single definition-stage run on S8 is sufficient to detect format compliance failure — if MAI produces a definition artefact that drops sections from the SKILL.md template, hallucinated fields, or structurally non-conformant output, it is ruled out without further testing.

**Pass criteria:** CPF = 1.00 for constraints present in the discovery artefact AND AQ ≥ 0.75 for the definition output specifically AND no SKILL.md format compliance failures (no dropped sections, no hallucinated fields).

**Failure modes to watch:** Schema drift (fields present in output that are not in the SKILL.md template); section collapse (SKILL.md sections omitted or merged without basis); hallucinated story structure (ACs or NFRs in formats incompatible with the downstream test-plan skill).

**Scope:** Definition stage only. If H2 passes, extend to review and test-plan on the same S8 story before committing to full pipeline adoption. Full pipeline validation of MAI is deferred to EXP-010 if the smoke test passes.

**Prerequisites:** MAI-Code-1-Flash model string must be confirmed before this run begins. The model string is not known with certainty at design time. Confirm via the GitHub Copilot model selector or Copilot API model list before starting the H2 run. Do not assume a model string — use a confirmed value. Candidates: `mai-code-1-flash` (likely), `microsoft/mai-code-1-flash` (alternative). Record the confirmed string in the run record before executing.

### H3 — Haiku for general usage and conversational orchestration

**Full statement:** Haiku 4.5 can handle general platform usage — skill selection questions, architectural queries, multi-constraint design discussions, ambiguous requests, and multi-turn context retention — while maintaining AQ ≥ 0.80 across at least four of five representative prompts, without the CDG gate as a safety net.

**What this tests:** Whether Haiku is safe as the default conversational model for interactions that are not pipeline-stage invocations. These interactions have no downstream quality gate — the CDG assurance gate and DoR hard blocks only apply to pipeline skill outputs, not to conversational responses. An incorrect Haiku response to an architectural question goes undetected unless the operator recognises it.

**Pass criteria:** AQ ≥ 0.80 on at least 4/5 prompts in the H3 prompt suite, scored by independent Sonnet judge session.

**Failure modes to watch:** Incorrect skill identification; schema misquotation; failure to ask for clarification on ambiguous requests (incorrect assumption instead); context loss across turns.

## Scenario selection

### H1 corpus selection

**S8 — Regulatory Reporting Pipeline (medium-high difficulty, primary):**
Selected because: (1) confirmed Sonnet reference artefact from EXP-008 Config A with AQ=0.80 and CPF=1.00; (2) medium-high difficulty with 5 constraints including 2 regulated (C1 RBNZ Prudential Reporting, C2 FMA Financial Reporting Act) and a high-difficulty hidden constraint (C5 — normalisation logic governance gap); (3) context injection files already exist at `workspace/experiments/EXP-003-pipeline-eval/corpus/context-injection/S8-*.md`; (4) EXP-008 Config C S8 achieved CPF=1.00 with Haiku workers, providing evidence that Haiku can handle the constraint set — the question for H1 is whether Haiku also handles discovery.

**S2 — Lending Origination (high difficulty, conditional):**
Selected because: (1) confirmed Sonnet reference from EXP-008 Config A with AQ=0.90 and CPF=1.00; (2) high difficulty with 3 regulated constraints (C1 CCCFA, C2 FMA algorithmic fairness, C5 FMA enforcement risk from undisclosed disparity finding) — the hardest regulated CPF test available in the corpus; (3) context injection files exist. S2 is run only if H1 passes on S8 — it is the confidence-confirming run, not the primary gating run. If H1 S8 is marginal (CPF=1.00 but AQ barely above 0.75), run S2. If H1 S8 is clear (AQ ≥ 0.85), the continue/stop decision may favour stopping to preserve budget.

**Why not S13 (Trans-Tasman Payments):**
S13 has multi-jurisdiction scoring complexity (NZ-leg, AU-leg, cross-border CPF) that would add scoring overhead without adding signal for the H1 routing question. S2 and S8 provide sufficient difficulty range. S13 is reserved as a tie-breaker if S8 passes and S2 fails unexpectedly.

**Why not three stories:**
EXP-008 Config C already ran S2, S8, and S13 with Sonnet at discovery and Haiku as workers. Config B (all Haiku) adds a single variable: whether Haiku at discovery drops constraints that Sonnet at discovery captured. Two stories at different difficulty levels are sufficient to detect this failure mode. Running all three would consume ~225 credits in H1 alone, exceeding the total budget.

### H2 corpus selection

**S8 only, definition stage only.**
S8 is selected because it has a confirmed EXP-008 Config A discovery artefact and its C5 normalisation logic governance gap is a well-characterised test of whether a model carries forward a B1 blocker correctly into /definition. The format compliance failure mode (GPT-4o pattern) is detectable at definition stage alone — a model that hallucinated fields, dropped sections, or produced structurally non-conformant stories would produce visible evidence within the first definition run.

### H3 corpus selection

Five representative general usage prompts designed to cover the interaction range a practitioner encounters in a typical platform session. Full prompts are in the H3 Prompt Suite section below. Selection rationale: one prompt per interaction pattern — skill selection, architectural query, multi-constraint design, ambiguous request, multi-turn retention. The suite is intentionally narrow enough to complete within ~60 credits.

### H3-P1 — Skill selection (simple)

**Prompt to send verbatim:**

> We've completed /test-plan for a story. Our product owner wants assurance that the ACs are unambiguous enough for a coding agent to implement without operator involvement. Which skill provides this check, what hard blocks does it enforce, and what are the minimum inputs it needs to run?

**What a high-quality response contains:**
Names `/definition-of-ready` as the correct skill. Describes the hard blocks H1 through H9 (ACs exist, test plan exists, review complete, no open HIGH findings, scope contract present, NFRs testable as coded assertions, scope confirmed by product owner, Approved By field populated, architecture guardrails consulted) — not necessarily all nine by name, but the framing of gate checks rather than quality suggestions. ### H3-P2 — Architectural query (medium)

**Prompt to send verbatim:**

> Explain the relationship between pipeline-state.json and the CDG assurance gate workflow (assurance-gate.yml). When the workflow runs on a PR, which fields in pipeline-state.json does it read, and what specific conditions cause it to post a blocking audit comment? Include what happens when a story that is recorded as 'dor-signed-off' in pipeline-state.json has no artefact path entry.

**What a high-quality response contains:**
Explains that assurance-gate.yml reads pipeline-state.json to cross-check whether the PR's story has reached the correct phase before merge is allowed. Names the fields that matter: feature slug, story id/slug, dorStatus (must be `signed-off`), artefact path (must be present), prStatus. Explains that a missing artefact path on a dor-signed-off story causes a crosscheck failure — the gate cannot verify the DoR artefact exists on disk, which blocks the audit comment from confirming chain integrity. Notes that the gate is designed to catch stories that were state-advanced in pipeline-state.json without the corresponding artefact file being committed. Does not hallucinate field names not present in the schema.

**Failure indicators:** Invents field names; describes the workflow as blocking for style/quality reasons rather than traceability gap reasons; conflates the CDG gate with /definition-of-ready; omits the artefact path cross-check as the specific failure condition.

### H3-P3 — Multi-constraint design (complex)

**Prompt to send verbatim:**

> We're designing a capability that allows a second-line compliance officer to annotate a DoR-signed-off story with regulatory constraints discovered after sign-off — for example, a new RBNZ supervisory notice received after the coding agent has been dispatched but before the first PR opens. The annotation must not reset the story to an earlier pipeline phase, must not allow scope additions disguised as regulatory constraints, and must be auditable in pipeline-state.json. What are the core design constraints on this capability, what changes does it require in the DoR artefact format, and what guardrails prevent misuse?

**What a high-quality response contains:**
Identifies the primary tension: the DoR sign-off is a timestamped, scope-locked gate — any post-sign-off change risks invalidating the signed artefact and the coding agent instructions block that depends on it. Proposes a separation: the DoR contract is immutable post-sign-off; annotations are a separate overlay field with its own metadata (timestamp, author, regulatory-basis, scope-neutral assertion). Notes that the annotation mechanism must enforce that annotations are scope-neutral — they can flag that a constraint exists but cannot introduce new ACs, change the scope boundary, or modify the implementation plan without a full DoR revision. Identifies the pipeline-state.json impact: needs a `post-dor-annotations` array with `id`, `timestamp`, `author`, `regulatory_basis`, `scope_neutral: true/false`, `status: pending|acknowledged|incorporated`. Identifies the guardrail: a `scope_neutral: false` annotation must trigger a phase reset to pre-sign-off, effectively voiding the current DoR and requiring a re-cycle. Does not conflate annotation with story update.

**Failure indicators:** Proposes modifying the existing DoR contract directly (violates sign-off integrity); ignores the scope creep risk; proposes an audit mechanism that does not distinguish annotation from sign-off; does not address what happens when a constraint requires a scope change.

### H3-P4 — Ambiguous request (clarification required)

**Prompt to send verbatim:**

> We want to update the model routing.

**What a high-quality response contains:**
Recognises this as underspecified and asks for clarification before proposing any routing change. The response should identify the ambiguity dimensions that need resolution: (1) which routing slot — orchestration (all-invocations default), specific skill, or general usage? (2) which model change — adding a new model, changing an existing assignment, or removing an entry? (3) what experiment or evidence basis supports this change — routing policy changes must cite `experiment_id` (routing-policy-framework.md governance rule)? (4) is this a Layer 1 (VS Code model selector) change, a Layer 2 (MODEL_ROUTING map in run-model-sweep.js) change, or both? Does not assume the intent and proceed with an unsupported routing change. Does not refuse to engage — it asks questions, not blocks.

**Failure indicators:** Proceeds to propose specific routing changes without asking any questions; asks only one of the four ambiguity dimensions; treats the request as complete and describes the MODEL_ROUTING map without engaging with the governance requirement.

## CPF and AQ scoring methodology

### CPF (Constraint Propagation Fidelity)

Applies to H1 and H2. Inherits from EXP-003-pipeline-eval manifest (constraint propagation fidelity section). Summary:

```
CPF = propagated_count / total_constraints_in_discovery
```

A constraint is propagated if it appears by name or clear unambiguous reference in either the DoR contract scope section OR the test plan NFR/acceptance section. Paraphrase with no material detail loss = propagated. Material detail omitted = dropped. Ambiguous = dropped for regulated constraints, discretionary for non-regulated.

**EXP-009 regulated CPF threshold:** 1.00. This is tighter than EXP-003's 0.80 floor. Rationale: EXP-008 demonstrated that CPF = 1.00 is achievable across all tested configs on S2, S8, and S13. Any regulated CPF below 1.00 in EXP-009 indicates a model-configuration failure, not a structural pipeline limitation. The threshold reflects the capability ceiling established by prior experiments.

For H2 (definition stage only), CPF is scored on the definition output alone — check whether all constraints present in the S8 discovery artefact appear in the definition stories' ACs and Architecture Constraints sections. Do not score downstream stages for H2.

### AQ (Artefact Quality)

Applies to H1 and H2 using the EXP-008 AQ rubric (five dimensions, max 2 each, scored 0.00–1.00). Judge: claude-sonnet-4-6 in a separate session. Never self-scored.

Applies to H3 using the H3 AQ rubric defined in the judge prompt template above (five dimensions adapted for conversational responses — domain_accuracy, reasoning_quality, completeness, scope_discipline, operator_safety).

**H1/H2 pass threshold:** AQ ≥ 0.75.
**H3 pass threshold:** AQ ≥ 0.80 (tighter — no CDG gate as safety net for conversational outputs).
